# 策略开发与回测体验原型 v1

> 本文档从策略开发者 / 回测用户视角定义 Vortex 的策略体验，目标是明确“研究结果如何进入策略层、策略层如何形成可比较回测、以及如何把结果交给交易层”。

---

## 一、背景

策略层最容易失控的地方，不在策略逻辑本身，而在输入输出边界：

1. 输入可能是研究员手工导出的信号文件。
2. 回测参数可能散落在脚本参数里。
3. 输出可能只是一张收益图，而没有稳定的 target portfolio 或调仓差异。

因此策略产品原型要优先冻结一件事：

策略开发者不是在“跑某个回测脚本”，而是在“基于某个 snapshot 和某个 signal snapshot，运行一份 StrategyProfile，并得到可比较的 backtest report 与可下游交付的 target portfolio”。

---

## 二、策略开发者真正关心什么

策略开发者真正关心的是：

1. 这次回测到底使用了哪版 snapshot 和哪版 signal snapshot。
2. 手续费、滑点、换手约束、调仓规则是什么。
3. 回测结果是否可复现。
4. 同一个策略不同版本之间到底谁更优。
5. 这次结果能否直接交给交易层生成订单预览。

也就是说，策略开发者不应该把时间浪费在：

1. 自己手工拼信号路径。
2. 自己记住成本模型配置。
3. 自己从回测结果里手工提炼目标组合。

---

## 三、策略层对用户暴露的核心对象

建议策略开发者主要围绕下面 6 个对象工作：

1. `StrategyProfile`：策略模板，定义信号消费方式、约束、成本假设和编译规则。
2. `snapshot`：本次回测和编译所依赖的数据版本。
3. `signal snapshot`：研究层交付的信号版本。
4. `backtest run`：一次完整回测记录。
5. `target portfolio`：策略层编译后的目标组合。
6. `strategy report`：回测表现、成本、暴露变化和失败原因的统一摘要。

这层最重要的产品边界是：

1. 策略层输入必须是标准 `signal snapshot`，而不是随手文件。
2. 策略层输出必须是标准 `target portfolio`，而不是只停留在绩效图表。

---

## 四、策略开发者真正需要的动作

建议策略域先固定下面 5 个动作：

1. `backtest`：运行一次回测。
2. `inspect`：查看某次回测的参数、输入和结果。
3. `compare`：比较两次回测结果。
4. `compile`：把 signal snapshot 编译成 target portfolio。
5. `status`：查看最近回测状态和当前可交付组合。

这样做的原因是：

1. 回测是验证动作。
2. 编译是交付动作。
3. 二者都属于策略域，但不能混成一个模糊命令。

---

## 五、建议的 CLI 语义

建议策略开发者看到的入口像这样：

```bash
vortex strategy backtest --profile dividend_classic --snapshot 2026-04-03 --signal signal_20260403_a
vortex strategy inspect --run-id backtest_20260403_223118
vortex strategy compare --left backtest_20260331_220801 --right backtest_20260403_223118
vortex strategy compile --profile dividend_classic --snapshot 2026-04-03 --signal signal_20260403_a
vortex strategy status --profile dividend_classic
```

这里需要强调：

1. `backtest` 是为了验证策略版本。
2. `compile` 是为了生成可交付组合。
3. `inspect` 和 `compare` 是为了让策略开发者做决策，而不是盲目看收益曲线。

---

## 六、一条策略用户旅程应该怎么走

### 6.1 选择 snapshot 和 signal snapshot

策略开发者先确认：

1. 使用哪版数据 snapshot。
2. 使用哪版 signal snapshot。

这一步的目标是锁定版本，而不是找路径。

### 6.2 选择 StrategyProfile

策略开发者选择的是：

1. 成本模型
2. 调仓频率
3. 约束规则
4. 组合编译规则

这一步应该是选模板或改少量差异，而不是全量重写配置。

### 6.3 运行 backtest

系统应自动记录：

1. `run_id`
2. 使用的 snapshot / signal snapshot
3. profile 配置快照
4. 成本与滑点口径
5. 绩效结果与工件位置

### 6.4 决定是否 compile 组合

只有当回测结果与风险暴露可接受时，策略开发者才应该执行 `compile`，生成标准 `target portfolio` 给交易层使用。

也就是说，交易层接到的应该是“已冻结版本的组合”，而不是回测用户脑中记住的一套规则。

---

## 七、策略开发者需要看到什么反馈

一次 `strategy backtest` 或 `strategy compile` 至少应返回：

1. `status`
2. `run_id`
3. `profile`
4. `snapshot`
5. `signal_snapshot`
6. `artifact_manifest`
7. `key_metrics`
8. `blocking_reason`（如果失败）

其中 `key_metrics` 建议至少包含：

1. 收益率
2. Sharpe
3. 最大回撤
4. 换手率
5. 预估成本
6. 暴露变化摘要

这些信息才是真正帮助策略开发者判断“能不能交付”的关键反馈。

---

## 八、策略域必须产出哪些标准工件

建议策略域默认产出并展示：

1. `target_portfolio.json`
2. `rebalance_diff.csv`
3. `order_plan_preview.csv`
4. `backtest_report.json`

这里的意义是：

1. 策略层不仅输出“历史表现”，还输出“可执行组合”。
2. 交易层拿到的是标准交付物，而不是再从回测结果里反推。

---

## 九、当前实现与目标设计的边界

### 9.1 当前实现

当前仓库里已经有：

1. 回测与策略运行相关代码
2. 若干因子和策略实验脚本

### 9.2 目标设计

下一阶段要补齐的是：

1. 统一的 `StrategyProfile` 用户视图
2. 标准 `signal snapshot -> target portfolio` 交付链
3. `backtest / inspect / compare / compile / status` 入口冻结
4. 面向交易层的稳定交付对象与说明

所以现在更像“能力已经部分存在”，但“产品体验还没有收敛成统一动作”。
