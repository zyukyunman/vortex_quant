## 二、角色与核心对象

### 2.1 四类角色

| 角色 | 关注点 | 主要域 |
|------|--------|--------|
| 数据运营 / 平台维护者 | 数据同步成功率、质量门禁、snapshot 发布 | Data |
| 研究员 / 因子开发者 | snapshot 版本、特征标签、IC 指标、signal 产出 | Research |
| 策略开发者 | 回测复现、成本口径、target portfolio 交付 | Strategy |
| 交易运营 / 执行同学 | 组合来源可追溯、风控结论、对账 | Trade |

### 2.2 七个核心对象

这些是用户在整个系统中接触的一等对象。

| 对象 | 说明 | 生产者 | 消费者 |
|------|------|--------|--------|
| `profile` | 用户/角色的工作模板（按域分型） | 平台 + 用户 | 全域 |
| `preset` | 系统内置的一揽子技术方案 | 平台 | profile 展开 |
| `pack` | 可替换规则包（quality / pit / publish / storage） | 平台 | preset 引用 |
| `snapshot` | 某个时点可供消费的数据版本 | Data 域 | Research / Strategy / Trade |
| `signal snapshot` | 研究域冻结发布的一版选股/打分结果，不是因子库，也不是最终持仓 | Research 域 | Strategy 域 |
| `target portfolio` | 策略编译后的目标组合 | Strategy 域 | Trade 域 |
| `report` | 任意 run 的结构化结果摘要 | 全域 | 用户审阅 |

### 2.3 对象生命周期链

```text
DataProfile
  ─→ bootstrap / update / backfill / repair
    ─→ snapshot (通过 publish 发布)
      ─→ ResearchProfile + snapshot
        ─→ research run
          ─→ signal snapshot (通过 publish-signal 发布)
            ─→ StrategyProfile + snapshot + signal snapshot
              ─→ backtest / compile
                ─→ target portfolio
                  ─→ TradeProfile + target portfolio
                    ─→ paper / live rebalance
                      ─→ execution report + reconcile report
```

## 三、Data 域动作规格

### 3.1 动作总表

Data 域对运营者暴露 **6 个主动作**。

| # | 动作 | CLI 入口 | 一句话说明 |
|---|------|---------|-----------|
| D1 | `bootstrap` | `vortex data bootstrap` | 第一次建库：从 `history_start` 一次性补齐到最新可用交易日 |
| D2 | `update` | `vortex data update` | 日常增量：从上次成功日期自动前进到最新可用交易日 |
| D3 | `backfill` | `vortex data backfill` | 指定缺口区间补数据 |
| D4 | `repair` | `vortex data repair` | 只重跑失败单元，不重跑已成功部分 |
| D5 | `publish` | `vortex data publish` | 质量门禁 → PIT 对齐 → 研究视图构建 → snapshot 发布 |
| D6 | `status` | `vortex data status` | 查看最近成功日期、失败任务、当前可用 snapshot、整体健康 |

> **为什么要收敛到这 6 个？**
> 运营者不应该每天手工 `sync --as-of YYYY-MM-DD`。`data sync` 是底层原语 / 调试入口；运营主入口应当是声明式动作，系统自动计算时间范围。

> **本轮收敛的运行形态**
> 这 6 个动作首先是 Data 域的标准能力和管理入口。
> 默认日常形态应是后台服务按计划自动执行 `update -> quality -> pit -> publish`；
> 人主要负责首次 `bootstrap`、查看 `status`、处理 `repair/backfill` 和异常审计。

### 3.2 D1 — bootstrap

| 项目 | 规格 |
|------|------|
| **作用** | 第一次建库，或更换 `history_start` 后全量重建 |
| **输入参数** | `--profile <name>` |
| **可选参数** | `--dry-run`（只显示计划，不执行） |
| **前置条件** | ① profile 合法且已解析 ② `.env` 中有有效 Token ③ 权威存储目录为空或显式 `--force` |
| **执行逻辑** | ① 解析 profile → 获取 `history_start` 和 `datasets` ② 获取交易日历 ③ 按数据集 × 日期区间分 chunk 并行/串行拉取 ④ 每 chunk 落盘 Parquet（ODS → DWD） ⑤ 记录进度和失败 chunk |
| **输出工件** | `sync_manifest.json`（每日分区）、Parquet 文件 |
| **成功条件** | 所有数据集的所有 chunk 全部落盘，manifest 完整 |
| **失败处理** | ① 中断时自动记录已完成 / 待续 chunk ② 失败 chunk 记入 `failed_units.json` ③ 可通过 `repair` 续跑 |
| **幂等性** | 是。同一 chunk 重复执行覆盖写入，不产生重复记录 |

### 3.3 D2 — update

| 项目 | 规格 |
|------|------|
| **作用** | 日常增量同步；默认由后台服务按调度自动触发 |
| **输入参数** | `--profile <name>` |
| **可选参数** | `--dry-run` |
| **前置条件** | ① 至少已成功执行过一次 `bootstrap` ② 最近一次成功日期可从 manifest 推断 |
| **执行逻辑** | ① 读取最近成功日期 ② 获取当前最新可用交易日 ③ 计算增量范围 ④ 按 `bootstrap` 相同逻辑分 chunk 拉取并落盘 |
| **输出工件** | 增量 `sync_manifest.json`、增量 Parquet 文件 |
| **成功条件** | 增量范围内所有 chunk 全部落盘 |
| **失败处理** | 同 `bootstrap`，失败 chunk 记入 `failed_units.json` |
| **幂等性** | 是 |

### 3.4 D3 — backfill

| 项目 | 规格 |
|------|------|
| **作用** | 手工指定日期区间补缺 |
| **输入参数** | `--profile <name>` `--start YYYY-MM-DD` `--end YYYY-MM-DD` |
| **可选参数** | `--datasets <list>`（只补指定数据集）、`--dry-run` |
| **前置条件** | profile 合法 |
| **执行逻辑** | ① 解析区间 ② 只拉取指定区间的数据 ③ 落盘覆盖 |
| **输出工件** | 同 `bootstrap` |
| **成功条件** | 指定区间内所有 chunk 全部落盘 |
| **失败处理** | 同 `bootstrap` |
| **幂等性** | 是 |

### 3.5 D4 — repair

| 项目 | 规格 |
|------|------|
| **作用** | 从 `failed_units.json` 续跑失败单元 |
| **输入参数** | `--profile <name>` |
| **可选参数** | `--run-id <id>`（指定哪次 run 的失败单元；默认最近一次）、`--dry-run` |
| **前置条件** | 存在 `failed_units.json` |
| **执行逻辑** | ① 读取失败单元清单 ② 只重试这些 chunk ③ 成功后从失败清单移除 |
| **输出工件** | 更新后的 `sync_manifest.json`、`failed_units.json`（如仍有失败） |
| **成功条件** | 所有失败单元全部补齐 |
| **失败处理** | 仍然有失败的继续留在 `failed_units.json`，可再次 `repair` |
| **幂等性** | 是 |

### 3.6 D5 — publish

| 项目 | 规格 |
|------|------|
| **作用** | 在数据同步完成后，执行质量门禁 → PIT 对齐 → 研究视图构建 → snapshot 冻结发布；默认由日常调度自动接续 |
| **输入参数** | `--profile <name>` |
| **可选参数** | `--as-of YYYY-MM-DD`（默认自动取最新可用日期）、`--dry-run`、`--force`（跳过门禁，仅限调试，日志留审计记录） |
| **前置条件** | ① 对应日期的数据已同步完成 ② 不存在未处理的失败 chunk（或已显式接受） |
| **执行逻辑** | ① 运行 QualityGate 全部规则 ② **任一 critical 规则失败 → 阻断，不发布** ③ 运行 PITAligner ④ 构建 DWS 研究视图 ⑤ 构建 Qlib 增量视图 ⑥ 计算并落盘衍生分析指标（估值分位数、市场宽度、资金面等） ⑦ 写入 snapshot 描述对象并标记为可消费 |
| **输出工件** | `data_health_report.json`、`pit_report.json`、`derived_metrics/`（衍生指标目录）、`snapshot_descriptor.json`、Qlib 增量文件 |
| **成功条件** | 质量门禁通过 + PIT 通过 + snapshot 描述对象写入完成 |
| **失败处理** | ① 阻断时明确输出哪条规则失败 ② 不发布 snapshot ③ 不允许 Research 消费 |
| **幂等性** | 是（重复发布同一 `as_of` 会覆盖，不产生多版本） |

> **为什么仍然保留独立 `publish` 动作？**
> ① 默认日常链路应自动串联 `update -> publish`。② 但在异常处理、调试、补数后重发版等场景下，仍然需要手工 `publish` 入口。③ 所以 `publish` 既是后台服务的标准阶段，也是管理入口。

> **Data 域与 Research 域的边界**
> `publish` 流程中自动计算的衍生分析指标（估值分位数、市场宽度、资金面等）属于 Data 域产出，是数据快照的一部分。Research 域的因子分析（IC 评测、多空检验等）由研究员主动发起，产出独立的 experiment 和 signal snapshot。简单说：Data 域回答"市场现在是什么状态"，Research 域回答"这个因子能不能赚钱"。

### 3.7 D6 — status

| 项目 | 规格 |
|------|------|
| **作用** | 查看当前数据线的整体状态 |
| **输入参数** | `--profile <name>` |
| **可选参数** | `--format json|table`（默认 table） |
| **前置条件** | 无 |
| **输出信息** | ① 最近同步成功日期 ② 最新失败任务（如有） ③ 当前可用 snapshot 列表 ④ 待修复 chunk 数量 ⑤ 最近一次 publish 状态 |
| **成功条件** | 正常输出状态 |
| **失败处理** | 如果 manifest 目录不存在，提示需要先 `bootstrap` |

### 3.8 Data 域进度模型

数据下载不是原子操作，需要分层进度：

| 层级 | 粒度 | 说明 |
|------|------|------|
| **run 级** | 1 次 bootstrap / update / backfill / repair | 记录 run_id、开始时间、结束时间、整体状态 |
| **阶段级** | sync → quality → pit → publish | 当前执行到哪个阶段 |
| **数据集级** | 例如 bar_1d / income / disclosure_calendar | 每个数据集的完成 / 失败状态 |
| **chunk 级** | 例如某个 ts_code 区间 / 某个日期区间 | 最细粒度，支持 repair 续跑 |

## 四、Research 域动作规格

### 4.1 动作总表

Research 域对研究员暴露 **5 个主动作**。

| # | 动作 | CLI 入口 | 一句话说明 |
|---|------|---------|-----------|
| R1 | `run` | `vortex research run` | 运行一次研究 workflow（因子评测 / 模型训练 / 信号生成） |
| R2 | `inspect` | `vortex research inspect` | 查看某次 experiment 的参数、指标和产物 |
| R3 | `compare` | `vortex research compare` | 比较两次 experiment 结果 |
| R4 | `publish-signal` | `vortex research publish-signal` | 把实验结果固化为 signal snapshot |
| R5 | `status` | `vortex research status` | 查看最近实验状态和当前可用 signal snapshot |

### 4.2 R1 — run

| 项目 | 规格 |
|------|------|
| **作用** | 基于 snapshot 运行一份可复现的研究 workflow |
| **输入参数** | `--profile <name>` `--snapshot <ref>` |
| **可选参数** | `--set <key=value>`（运行时 override）、`--dry-run` |
| **前置条件** | ① ResearchProfile 合法 ② 指定 snapshot 已经 publish 且可消费 |
| **执行逻辑** | ① 解析 profile + snapshot ② 装配 Qlib handler / workflow config ③ 执行 workflow ④ 产出 experiment 记录 + 指标 + 产物 |
| **输出工件** | `pred.pkl`、`ic_metrics.csv`、`signal_analysis.csv`、`workflow_config_snapshot.yaml`、`research_report.json` |
| **反馈指标**（第一屏） | IC、RankIC、ICIR、分组收益、多空收益 |
| **成功条件** | workflow 执行完成，所有指标正常写入 |
| **失败处理** | ① 如 snapshot 不可用 → 阻断，提示先 `data publish` ② 如 workflow 异常 → 记录 blocking_reason ③ 不产出 signal snapshot |
| **幂等性** | 否（每次 run 产出独立 experiment，但相同输入应可复现相同结果） |

### 4.3 R2 — inspect

| 项目 | 规格 |
|------|------|
| **作用** | 查看指定 experiment 的完整信息 |
| **输入参数** | `--run-id <id>` |
| **输出信息** | ① run_id ② 使用的 snapshot ③ profile 快照 ④ workflow 类型 ⑤ key_metrics ⑥ 工件位置清单 ⑦ blocking_reason（如有） |

### 4.4 R3 — compare

| 项目 | 规格 |
|------|------|
| **作用** | 并排比较两次 experiment |
| **输入参数** | `--left <run-id>` `--right <run-id>` |
| **输出信息** | ① 两次的 snapshot / profile diff ② 指标对比表（IC delta、RankIC delta 等） ③ 是否值得发布的建议（可选） |

### 4.5 R4 — publish-signal

| 项目 | 规格 |
|------|------|
| **作用** | 把一次 experiment 产出的股票评分、排序或入选结果冻结为 signal snapshot，供 Strategy 消费 |
| **输入参数** | `--run-id <id>` |
| **可选参数** | `--tag <label>` |
| **前置条件** | 对应 experiment 状态为成功 |
| **执行逻辑** | ① 把本次 experiment 的信号结果复制 / 链接到 signal snapshot 目录 ② 写入 signal_snapshot_descriptor.json ③ 标记为可消费 |
| **输出工件** | `signal_snapshot_descriptor.json` |
| **成功条件** | descriptor 写入完成，信号文件完整 |
| **幂等性** | 是（同一 run_id 重复 publish 覆盖） |

### 4.6 R5 — status

| 项目 | 规格 |
|------|------|
| **作用** | 查看研究线状态 |
| **输入参数** | `--profile <name>` |
| **输出信息** | ① 最近 N 次 experiment 列表 ② 当前可用 signal snapshot 列表 ③ 各实验 key_metrics 概览 |

### 5.2 S1 — backtest

| 项目 | 规格 |
|------|------|
| **作用** | 基于 snapshot + signal snapshot 运行一次完整回测 |
| **输入参数** | `--profile <name>` `--snapshot <ref>` `--signal <signal-ref>` |
| **可选参数** | `--set <key=value>`（覆盖成本 / 滑点等）、`--dry-run` |
| **前置条件** | ① StrategyProfile 合法 ② snapshot 已 publish ③ signal snapshot 已 publish |
| **执行逻辑** | ① 解析 profile + 版本 ② 装配 Backtrader adapter ③ 注入成本模型、滑点模型、调仓规则 ④ 执行回测 ⑤ 产出 backtest 记录 + 绩效工件 |
| **输出工件** | `backtest_report.json`、`equity_curve.csv`、`trade_log.csv` |
| **反馈指标**（第一屏） | 收益率、年化收益、Sharpe、最大回撤、换手率、预估成本 |
| **成功条件** | 回测执行完成，指标和工件正常写入 |
| **失败处理** | ① snapshot / signal 不可用 → 阻断 ② 回测异常 → 记录 blocking_reason |
| **幂等性** | 否（每次 run 独立记录） |

### 5.3 S2 — inspect

| 项目 | 规格 |
|------|------|
| **作用** | 查看指定 backtest run 的完整信息 |
| **输入参数** | `--run-id <id>` |
| **输出信息** | ① run_id ② snapshot + signal snapshot ③ profile 快照 ④ 成本/滑点口径 ⑤ key_metrics ⑥ 工件位置清单 |

### 5.4 S3 — compare

| 项目 | 规格 |
|------|------|
| **作用** | 并排比较两次回测 |
| **输入参数** | `--left <run-id>` `--right <run-id>` |
| **输出信息** | ① 输入版本 diff ② 指标对比表 ③ 成本口径 diff |

### 5.5 S4 — compile

| 项目 | 规格 |
|------|------|
| **作用** | 把 signal snapshot 编译成可交付给 Trade 的 target portfolio |
| **输入参数** | `--profile <name>` `--snapshot <ref>` `--signal <signal-ref>` |
| **可选参数** | `--tag <label>`、`--dry-run` |
| **前置条件** | ① signal snapshot 已 publish ② 回测验证过（建议但不强制） |
| **执行逻辑** | ① 读取 signal snapshot ② 应用 portfolio_constraints / rebalance_rule ③ 生成目标持仓 ④ 计算与当前持仓（如有）的调仓差异 ⑤ 产出交付对象 |
| **输出工件** | `target_portfolio.json`、`rebalance_diff.csv`、`order_plan_preview.csv` |
| **成功条件** | target portfolio 写入完成，字段完整 |
| **失败处理** | signal 不可用 → 阻断 |
| **幂等性** | 是（相同输入产出相同组合） |

### 5.6 S5 — status

| 项目 | 规格 |
|------|------|
| **作用** | 查看策略线状态 |
| **输入参数** | `--profile <name>` |
| **输出信息** | ① 最近 N 次 backtest 列表 ② 当前可交付 target portfolio 列表 ③ key_metrics 概览 |

## 六、Trade 域动作规格

### 6.1 动作总表

Trade 域对交易运营暴露 **6 个主动作**。

| # | 动作 | CLI 入口 | 一句话说明 |
|---|------|---------|-----------|
| T1 | `paper rebalance` | `vortex trade paper rebalance` | 模拟盘调仓 |
| T2 | `live rebalance` | `vortex trade live rebalance` | 实盘调仓 |
| T3 | `inspect` | `vortex trade inspect` | 查看本次执行的计划、成交和异常 |
| T4 | `retry` | `vortex trade retry` | 对失败单元发起重试 |
| T5 | `reconcile` | `vortex trade reconcile` | 日终对账 |
| T6 | `status` | `vortex trade status` | 查看账户、最近执行状态、当前阻断项 |

### 6.2 T1 — paper rebalance

| 项目 | 规格 |
|------|------|
| **作用** | 在模拟盘环境执行调仓 |
| **输入参数** | `--profile <name>` `--target <tp-ref>` |
| **可选参数** | `--dry-run`（只看 order plan，不提交） |
| **前置条件** | ① TradeProfile 合法 ② target portfolio 已存在 ③ 模拟盘网关可用 |
| **执行逻辑** | ① 读取 target portfolio ② 从网关获取当前持仓 ③ 计算调仓差异 ④ 生成 order plan ⑤ 执行事前风控 ⑥ **风控通过 → 提交订单** ⑦ **风控不通过 → 阻断，明确输出原因** ⑧ 监听成交回报 |
| **输出工件** | `order_plan.json`、`pre_trade_result.json`、`execution_report.json` |
| **成功条件** | 所有订单已成交或已按规则处理（撤单 / 超时） |
| **失败处理** | ① 风控阻断 → 不提交任何订单 ② 部分失败 → 记录失败订单，可 `retry` |
| **幂等性** | 否（重复执行会重新计算差异并下单） |

### 6.3 T2 — live rebalance

与 `paper rebalance` 规格完全相同，仅以下差异：

| 差异项 | paper | live |
|--------|-------|------|
| 网关 | 模拟盘 | miniQMT 实盘 |
| 风控级别 | 标准 | 严格（额外 pre-trade 规则） |
| 确认 | 无二次确认 | 执行前要求人工确认或 `--confirm` |
| 审计 | 标准日志 | 强制审计日志 + 通知 |

### 6.4 T3 — inspect

| 项目 | 规格 |
|------|------|
| **作用** | 查看指定执行的完整记录 |
| **输入参数** | `--run-id <id>` |
| **输出信息** | ① 来源 target portfolio ② order plan ③ pre_trade_result ④ 成交明细 ⑤ 拒单 / 超时订单 ⑥ 是否需要 retry |

### 6.5 T4 — retry

| 项目 | 规格 |
|------|------|
| **作用** | 对指定执行中的失败订单重试 |
| **输入参数** | `--run-id <id>` |
| **前置条件** | 存在失败订单记录 |
| **执行逻辑** | ① 读取失败订单 ② 重新执行事前风控 ③ 通过则重新提交 |
| **幂等性** | 否 |

### 6.6 T5 — reconcile

| 项目 | 规格 |
|------|------|
| **作用** | 日终对账 |
| **输入参数** | `--profile <name>` `--date YYYY-MM-DD` |
| **执行逻辑** | ① 从网关获取实际持仓与成交 ② 与 target portfolio + execution report 对比 ③ 计算持仓偏差、成交差异 ④ 输出对账结果 |
| **输出工件** | `reconcile_report.json` |
| **成功条件** | 偏差在允许范围内 |
| **失败处理** | 偏差超阈值 → 标记异常并通知 |

### 6.7 T6 — status

| 项目 | 规格 |
|------|------|
| **作用** | 查看交易线总体状态 |
| **输入参数** | `--profile <name>` |
| **输出信息** | ① 账户状态 ② 最近执行列表 ③ 当前持仓 ④ 未处理阻断项 ⑤ 最近对账结果 |

### 6.8 Trade 域风控三层模型

| 层级 | 时机 | 阻断行为 |
|------|------|---------|
| **事前（Pre-Trade）** | order plan 生成后、提交前 | 不通过 → 不提交订单，输出结构化原因 |
| **事中（Intra-Trade）** | 成交回报回来时 | 触发规则 → 撤回未成交订单 / 暂停执行 |
| **事后（Post-Trade）** | 日终对账 | 偏差超限 → 标记异常、通知、归入审计 |

## 八、Profile 字段冻结表（四类 × 三层）

### 8.0 分层原则

| 层 | 谁维护 | 谁可见 | 变更频率 |
|----|--------|--------|---------|
| **Profile（用户层）** | 运营者 / 研究员 / 策略开发者 / 交易同学 | 全部可见 | 低（选择 + 少量差异） |
| **Preset（方案层）** | 平台开发者 | 用户可查看、通常不编辑 | 很低 |
| **Pack（规则包层）** | 平台开发者 | 用户一般不直接接触 | 低 |

### 8.1 DataProfile

#### 用户层字段（运营者直接维护）

| 字段 | 类型 | 必填 | 说明 | 是否常改 |
|------|------|------|------|---------|
| `name` | string | 是 | 数据线唯一名称 | 否 |
| `type` | literal "data" | 是 | 固定为 data | 否 |
| `preset` | string | 是 | 内置数据方案名，如 `cn_stock_daily_core` | 偶尔 |
| `extends` | string | 否 | 继承另一个 profile | 否 |
| `description` | string | 否 | 用途说明 | 否 |
| `history_start` | date | 是 | 首次建库起始日 | 首次常改，之后很少 |
| `schedule` | string | 否 | 自动更新时间策略，如 `weeknights_2200` | 偶尔 |
| `publish` | string | 否 | 更新后是否自动 publish，如 `research_daily` | 偶尔 |
| `notify` | string | 否 | 运行结果通知对象 | 偶尔 |
| `enabled` | bool | 否 | 是否启用（默认 true） | 偶尔 |
| `tags` | list[string] | 否 | 检索标签 | 偶尔 |

#### Preset 展开字段（由 preset 提供，用户通常不改）

| 字段 | 类型 | 说明 |
|------|------|------|
| `market` | string | 市场，如 `cn_stock` |
| `universe` | string | 标的范围，如 `all_a` |
| `freqs` | list[string] | 频率，如 `["1d"]` |
| `provider` | string | 数据源，如 `tushare` |
| `datasets` | list[string] | 同步数据集列表 |
| `timezone` | string | 时区 |
| `calendar` | string | 交易日历 |
| `quality_pack` | string | 引用质量规则包名 |
| `pit_pack` | string | 引用 PIT 规则包名 |
| `publish_pack` | string | 引用发布规则包名 |
| `storage_pack` | string | 引用存储规则包名 |

#### Pack 层字段（平台内部，用户不直接维护）

| Pack 类型 | 包含字段示例 |
|-----------|-------------|
| `quality_pack` | `mode: fail_closed`、`rules: [not_empty, no_all_nan, date_continuity, ...]`、`critical_rules: [...]` |
| `pit_pack` | `mode: strict`、`fallback: block`、`missing_ann_date_policy: block_or_review` |
| `publish_pack` | `publish_time: "22:00"`、`selector: latest_success`、`retention: 30d` |
| `storage_pack` | `backend: parquet_duckdb`、`partition_by: [market, as_of]`、`compression: snappy` |

```text
    └── status           # 交易线状态
```

## 十一、未决问题与待讨论项

以下为撰写本文档时识别的尚未冻结的点，需在确认冻结前明确决定。

| # | 问题 | 涉及域 | 建议方向 |
|---|------|--------|---------|
| 1 | 手工 `publish` 的权限边界和使用场景要不要进一步收窄？ | Data | 默认后台自动串联 `update -> publish`；手工 `publish` 仅用于异常处理、补数后重发版和调试 |
| 2 | `bootstrap` 大量历史数据是否需要并发控制和限速（Tushare QPS）？ | Data | 是，需内建速率控制，第一版按 Tushare 限制做默认 |
| 3 | Research `run` 是否强制要求 snapshot 已 publish？还是允许"开发模式"直接跑？ | Research | 建议严格模式为默认，另提供 `--dev` 跳过检查但标记为不可 publish |
| 4 | Strategy `compile` 是否需要强制要求先跑过 `backtest`？ | Strategy | 建议不强制，但如果未回测过会在 target_portfolio 中标记 `unverified` |
| 5 | Trade `live rebalance` 的二次确认机制具体是什么？CLI 交互式还是 `--confirm` flag？ | Trade | 建议两者都支持：交互式默认，脚本化用 `--confirm` |
| 6 | Pack 的第一版具体内容需要展开吗？还是先把接口定好，内容后补？ | 全域 | 建议先冻结 quality_pack 和 pit_pack 的第一版规则列表，其余可后补 |
| 7 | Preset 第一版需要提供几套？ | 全域 | 建议每个域至少一套默认 preset：`cn_stock_daily_core` / `cn_alpha_basic` / `cn_backtest_default` / `cn_paper_default` |
| 8 | 是否需要 `vortex init` 命令来初始化项目目录结构和默认 profile？ | 全域 | 建议需要，降低首次使用门槛 |
| 9 | run_id 命名规则是什么？ | 全域 | 建议 `{domain}_{date}_{time}_{random4}`，如 `research_20260403_220501_a3f2` |
| 10 | 多策略共用同一数据线时，snapshot 版本如何保证一致性？ | Research → Strategy | 建议 snapshot_id 作为显式参数贯穿全链路，不依赖"最新" |
| 11 | 是否需要一个独立的"因子/特征数据层"作为一等产品对象？ | Research / Data | 当前因子产出只作为 experiment 内的工件存在；如果因子需要长期沉淀、跨实验复用，可能需要独立的 `feature snapshot` 对象和对应的发布/消费链路。建议第一版先观察 Research 域 `feature_set` 是否够用，后续按需升级 |

## 附录 B：典型端到端使用旅程

### B.1 运营者首次建库

```bash
# 1. 查看 profile
vortex profile explain --profile cn_stock_daily

# 2. 首次建库
vortex data bootstrap --profile cn_stock_daily

# 3. 查看状态
vortex data status --profile cn_stock_daily

# 4. 修复失败（如有）
vortex data repair --profile cn_stock_daily

# 5. 发布 snapshot
vortex data publish --profile cn_stock_daily
```

### B.2 研究员日常研究

```bash
# 1. 确认最新 snapshot
vortex data status --profile cn_stock_daily

# 2. 运行研究
vortex research run --profile cn_alpha_basic --snapshot 2026-04-03

# 3. 查看结果
vortex research inspect --run-id research_20260403_220501

# 4. 与上次比较
vortex research compare --left research_20260401_220212 --right research_20260403_220501

# 5. 满意则发布信号
vortex research publish-signal --run-id research_20260403_220501 --tag v1.2
```

### B.3 策略开发者回测与编译

```bash
# 1. 运行回测
vortex strategy backtest --profile dividend_classic --snapshot 2026-04-03 --signal sig_cn_alpha_basic_20260403_220501

# 2. 比较版本
vortex strategy compare --left backtest_20260331_220801 --right backtest_20260403_223118

# 3. 满意则编译组合
vortex strategy compile --profile dividend_classic --snapshot 2026-04-03 --signal sig_cn_alpha_basic_20260403_220501
```

### B.4 交易同学执行调仓

```bash
# 1. 模拟盘验证
vortex trade paper rebalance --profile dividend_live_cn --target tp_dividend_classic_20260403_a

# 2. 查看执行
vortex trade inspect --run-id exec_20260404_091501

# 3. 实盘调仓
vortex trade live rebalance --profile dividend_live_cn --target tp_dividend_classic_20260403_a --confirm

# 4. 日终对账
vortex trade reconcile --profile dividend_live_cn --date 2026-04-04

# 5. 查看状态
vortex trade status --profile dividend_live_cn
```
