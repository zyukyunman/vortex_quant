## 一、数据监控场景——"看不见就是最好的"

### 1.1 场景定位

数据服务对用户最好的状态是**无感知**。

研究员早上打开终端，数据已经是昨天收盘后自动更新的最新版。他不需要知道 Tushare 几点拉的数据、质量检查跑了多少条规则、PIT 对齐花了多少时间。他只需要知道：今天的 snapshot 可用。

只有出了问题，系统才应该主动找他。

### 1.2 日常体验：零操作

每天 22:00 后，后台自动执行完整数据流水线：

```
update → quality → pit → publish
```

成功时，系统不会产生任何通知。研究员第二天使用数据时，snapshot 已经就绪。

如果研究员出于好奇想看一眼，可以用一条命令确认：

```bash
$ vortex data status --profile cn_stock_daily

╭─ cn_stock_daily ─────────────────────────────────────────╮
│  最新 snapshot    2026-04-04 (22:00)     ✅ 已发布       │
│  数据更新         4,892 / 4,892 品种     全部成功        │
│  质量检查         38 条规则              全部通过        │
│  PIT 对齐         财报季延迟修正 3 条    已处理          │
│  下次更新         2026-04-05 22:00       自动            │
╰──────────────────────────────────────────────────────────╯
```

这就够了。不需要更多操作。

### 1.3 异常自动恢复

数据更新失败时，系统不会立即放弃——先自动尝试恢复，恢复不了再通知用户。

#### 自动重试策略

```
首次失败 → 等待 5 分钟 → 第 1 次重试
第 1 次重试失败 → 等待 15 分钟 → 第 2 次重试
第 2 次重试失败 → 等待 30 分钟 → 第 3 次重试
第 3 次重试失败 → 停止 → 推送通知
```

重试规则：
1. **最多重试 3 次**，间隔递增（5min → 15min → 30min）
2. **任一次重试成功则继续正常流程**，不产生通知
3. **3 次全部失败才推送 critical 通知**，附带全部 3 次的错误信息
4. **重试仅针对可恢复错误**（网络超时、数据源临时不可用）；配置错误、认证失败等不可恢复错误直接通知，不重试

重试期间的状态对用户可见：

```bash
$ vortex data status --profile cn_stock_daily

╭─ cn_stock_daily ─────────────────────────────────────────╮
│  最新 snapshot    2026-04-03 (22:00)     ✅ 已发布       │
│  今日更新         🔄 重试中 (第 2 次，下次 22:45)        │
│  失败原因         Tushare daily_basic 返回空数据          │
╰──────────────────────────────────────────────────────────╯
```

#### 重试全部失败后的通知

只有 3 次重试全部失败后，系统才推送通知：

```
🔴 Vortex 数据告警

[cn_stock_daily] 2026-04-04 更新失败（已自动重试 3 次）

问题：Tushare daily_basic 接口返回空数据（疑似数据源维护）
重试记录：
  首次  22:00  失败 - 空响应
  重试#1  22:05  失败 - 空响应
  重试#2  22:20  失败 - 空响应
  重试#3  22:50  失败 - 空响应
影响范围：4,892 品种的基础面数据未更新
当前状态：snapshot 未发布，研究域无法消费今日数据

建议操作：
  1. 确认 Tushare 是否在维护
  2. 恢复后执行：vortex data repair --profile cn_stock_daily
  3. 手动触发重试：vortex data update --profile cn_stock_daily --retry

run_id: data_20260404_220003_b7f1
```

通知的设计原则：
1. **说清楚发生了什么**——不是"任务失败"，而是"哪个接口、返回了什么、重试了几次"
2. **说清楚影响范围**——多少品种受影响、下游研究能不能继续
3. **给出可执行的建议**——不是"请检查"，而是具体该跑什么命令
4. **附带 run_id**——用户可以一键查看完整日志

### 1.4 通知分级

| 级别 | 触发条件 | 推送行为 |
|------|---------|---------|
| 🔴 critical | 数据质量阻断（Fail-Closed 触发）、数据源持续不可用 | 立即推送飞书，附操作建议 |
| 🟡 warning | 部分品种更新失败但不影响 snapshot 发布、质量检查发现可疑但未阻断的指标 | 汇入当日摘要推送 |
| 🟢 info | 每日更新完成、bootstrap 完成 | 仅在用户开启 info 级别通知时推送 |

默认只推送 critical 和 warning。研究员可以在 DataProfile 中配置通知偏好。

### 1.5 与 v1 的关系

v1 冻结了 6 个数据动作（bootstrap / update / backfill / repair / publish / status），本节不改变动作规格，而是补充"正常流程中用户根本不碰这些命令"的产品预期，以及异常时的通知体验。

## 二、因子评测场景——"研究员的工作台"

### 2.1 场景定位

因子评测不是"跑一个脚本看个数字"。研究员的真实需求是：

1. 我想知道这个因子**到底行不行**——不是看一个 IC 数字，是看多个维度、多个周期、多种市场环境下的系统性评测。
2. 我想**随时调出**任何一次评测的完整结果——三个月前跑的那个因子，我现在想看看。
3. 我想**比较**不同因子——谁更好、好在哪、什么环境下更好。
4. 评测结果应该是**好看的**——不是终端里几行数字，而是一份可以发给团队讨论的精美报告。

#### 截面分析——因子评测的核心思想

因子评测的本质是**截面分析**（cross-sectional analysis）：在同一时间点上，比较所有股票的因子值与未来收益的关系。

截面思想贯穿因子评测的每一步：
- **IC 计算**：每个截面（每天）计算因子排名与未来收益排名的相关性，再对所有截面取统计量
- **分组收益**：每个截面按因子值分组，观察组间收益差异是否单调
- **因子分布**：每个截面观察因子值的分布形态，判断是否有极端值或集中度问题
- **因子相关性**：截面上计算两个因子的相关性，判断信息是否重复

截面分析也适用于策略构建（截面选股）、风险分析（截面暴露）、组合构建（截面权重分配）等环节。在本文档中，因子评测、策略回测、风险归因都基于截面思想展开。

### 2.2 因子注册——告诉系统"因子在哪"

在评测之前，系统必须知道因子的数据从哪来。因子本质上是一列按（日期 × 股票）组织的数值，来源有两种：

#### 来源一：代码定义（推荐）

研究员编写一个 Python 文件或 Qlib 表达式，系统运行时从原始数据计算因子值。适合开发中的新因子。

```bash
$ vortex research register \
    --name momentum_20d \
    --source code \
    --path factors/momentum_20d.py

╭─ 因子注册成功 ────────────────────────────────────────────╮
│  名称          momentum_20d                                │
│  来源          代码 (factors/momentum_20d.py)              │
│  输出列        factor_value (float64)                      │
│  依赖数据      close, volume (自动从代码解析)              │
│  状态          ✅ 已注册，可用于评测                       │
╰───────────────────────────────────────────────────────────╯
```

代码文件需要遵循统一接口（具体接口定义见架构设计文档），核心是返回一个 `(date, instrument) → value` 的 DataFrame。

#### 来源二：数据表引用

因子值已经预计算好存在数据库的某张表里（比如外部团队提供的因子、第三方数据源）。用户告诉系统去哪读。

```bash
$ vortex research register \
    --name ext_sentiment \
    --source table \
    --db research_db \
    --table sentiment_scores \
    --column sentiment_score \
    --date-column trade_date \
    --instrument-column stock_code

╭─ 因子注册成功 ────────────────────────────────────────────╮
│  名称          ext_sentiment                               │
│  来源          数据表 (research_db.sentiment_scores)       │
│  值列          sentiment_score (float64)                   │
│  日期列        trade_date                                  │
│  品种列        stock_code                                  │
│  数据范围      2020-01-02 → 2026-04-03 (1,523 交易日)     │
│  覆盖品种      4,892                                       │
│  状态          ✅ 已注册，可用于评测                       │
╰───────────────────────────────────────────────────────────╯
```

#### 注册信息存在哪里

因子注册信息存在**数据库**的因子注册表（factor_registry）中，不是配置文件。

| 字段 | 说明 |
|------|------|
| `name` | 因子名称（唯一标识，用于 `--factor` 参数） |
| `source_type` | `code` 或 `table` |
| `source_path` | 代码文件路径，或 `db.table.column` |
| `date_column` | 数据表来源的日期列名（代码来源不需要） |
| `instrument_column` | 数据表来源的品种列名（代码来源不需要） |
| `registered_at` | 注册时间 |
| `registered_by` | 注册人 |

#### 管理已注册因子

```bash
# 列出所有已注册因子
$ vortex research list-factors

╭─ 已注册因子 ──────────────────────────────────────────────╮
│  名称              来源     类型       最近评测             │
│  momentum_20d      代码     动量型     2026-04-03           │
│  value_bp          代码     价值型     2026-04-01           │
│  quality_roe       代码     质量型     2026-03-28           │
│  ext_sentiment     数据表   另类型     —（未评测）          │
╰───────────────────────────────────────────────────────────╯

# 更新注册信息（比如代码路径变了）
$ vortex research register --name momentum_20d --source code --path factors/momentum_20d_v2.py --update

# 注销因子（不影响历史评测记录，只是不能再发起新评测）
$ vortex research unregister --name old_factor
```

#### 因子值的计算与缓存

注册只是告诉系统"去哪找数据"，**因子值的实际计算发生在首次评测时**：

```
注册 (register)          → 记录来源信息到 factor_registry
                           ↓
首次评测 (run)           → 读取代码/数据表 → 计算因子值 → 缓存到 factor_cache 表
                           ↓
后续评测 (run, 同snapshot) → 直接从 factor_cache 读取，不重复计算
```

缓存失效规则：
- **代码来源**：代码文件的 hash 变了 → 缓存失效，下次评测重算
- **数据表来源**：表数据更新了（通过 snapshot 版本号判断）→ 缓存失效
- **手动清除**：`vortex research clear-cache --factor momentum_20d`

```bash
$ vortex research run --profile cn_alpha_basic --factor momentum_20d --snapshot 2026-04-03

  [1/4] 检查因子缓存...  ✅ 命中缓存 (2026-04-03, code_hash=a3f2c1)
  [2/4] 加载因子值...     5,200 × 1,523 (800 万条)
  [3/4] 执行评测规则...   10 条规则
  [4/4] 生成报告...       tear_sheet.html

# 如果代码改了，缓存自动失效：
$ vortex research run --profile cn_alpha_basic --factor momentum_20d --snapshot 2026-04-03

  [1/4] 检查因子缓存...  ⚠️ 代码已变更 (hash 不匹配)
  [1/4] 重新计算因子值... ██████████ 完成 (12.3s)
  [1/4] 缓存已更新
  ...
```

> **为什么缓存？** 因子计算可能涉及全市场全历史的数据（如 5,000 只股票 × 6 年日频 = 800 万行），每次评测都重算浪费时间。缓存到 DB 后，换不同评测参数（如 IC 周期从 5d 改 20d）时不需要重新计算因子值。

> **设计边界**：注册解决"因子数据从哪来"，缓存解决"因子值不重复计算"，评测规则解决"怎么评"——三者分离。因子名称在 `--factor` 参数和因子档案中全程使用，是因子的唯一用户可见标识。

### 2.3 因子类型分类

不同类型的因子有不同的特征和评测侧重，系统内置以下因子类型分类。

#### 类型何时确定

因子类型在**首次评测时确定**，遵循以下优先级：

1. **用户显式指定**——评测命令中通过 `--factor-type` 参数声明
2. **AI 辅助判定**——用户不确定时，系统引导 AI 分析因子代码和描述，给出分类建议
3. **系统关键词推断**——根据因子名中的关键词匹配（如 `momentum_*` → 动量型，`pb_*` → 价值型）
4. **未能推断**——标记为"未分类"，评测照常执行但不参与同类排名

```bash
# 情况一：自动推断（因子名含 momentum，系统识别为动量型）
$ vortex research run --profile cn_alpha_basic --factor momentum_20d
  → 因子类型：动量型（自动推断）

# 情况二：显式指定（用户自己清楚类型）
$ vortex research run --profile cn_alpha_basic --factor momentum_20d --factor-type technical
  → 因子类型：技术型（用户指定）

# 情况三：不确定类型，请求 AI 辅助判定
$ vortex research run --profile cn_alpha_basic --factor my_custom_factor --factor-type ask
```

#### AI 辅助分类流程

当用户传入 `--factor-type ask` 或系统无法自动推断时，启动 AI 辅助流程。这不是一个独立的 skill，而是评测流程中内置的一段**引导式交互**：

```
╭─ AI 因子分类助手 ──────────────────────────────────────────╮
│                                                             │
│  因子名称：my_custom_factor                                 │
│  因子代码路径：strategies/factors/my_custom_factor.py       │
│                                                             │
│  我已阅读因子代码，分析如下：                               │
│                                                             │
│  📋 因子逻辑摘要：                                          │
│    该因子计算最近 20 日成交量加权平均价(VWAP)与收盘价的      │
│    偏离度，用于衡量机构资金的建仓方向。                      │
│                                                             │
│  🏷️  建议分类：**流动性** (Liquidity)                       │
│                                                             │
│  📝 理由：                                                  │
│    - 核心变量是成交量和价格的关系 → 流动性/微观结构类        │
│    - 非价格动量（不看涨跌方向），非基本面（无财务数据）     │
│    - 类似因子参考：Amihud 非流动性、VPIN                    │
│                                                             │
│  是否采用此分类？ [Y/n/手动选择(s)]                         │
╰─────────────────────────────────────────────────────────────╯
```

用户输入 `s` 可进入手动选择：

```
  请选择因子类型：
    1. 价值型 (Value)
    2. 动量型 (Momentum)
    3. 质量型 (Quality)
    4. 波动型 (Volatility)
    5. 流动性 (Liquidity)    ← AI 建议
    6. 技术型 (Technical)
    7. 另类型 (Alternative)
    8. 复合/ML (Composite)
  输入编号 [5]:
```

> **设计决策：为什么是引导代码而不是独立 skill？**
> - 因子分类只在评测提交这一个节点触发，不是通用能力，不需要独立调用入口
> - 引导逻辑内嵌在 `vortex research run` 的交互流程中，作为提交前的一个确认步骤
> - AI 分析因子代码的 prompt 模板由系统维护，用户不需要知道怎么问——系统帮他问好了

#### 类型存在哪里

因子类型存储在**数据库**中，是因子档案记录的一个字段——不是 Profile 配置项。

> **为什么不放配置？** 因子类型是评测产物，跟着因子走，不跟着运行环境走。同一个 Profile 可以评测不同类型的因子，同一个因子换 Profile 评测类型也不应该变。如果放在配置里，研发容易误实现为"切 Profile 就丢失类型"的行为。
>
> **明确边界**：
> - **落数据库**：factor_type 字段存在因子档案表（factor_archive），随首次评测写入，`retype` 命令直接 UPDATE 该字段
> - **不落配置**：Profile 中不出现 factor_type 字段；`--factor-type` 只是评测命令的输入参数，写入目标是数据库
> - **CLI 参数 vs 持久化**：`--factor-type technical` 的语义是"把这个因子在数据库里标记为技术型"，不是"这次运行用技术型的配置"

每次评测时记录当时的类型，用于同类排名计算。

#### 用户修正类型

如果用户发现某个因子的类型不对（比如自动推断错了），可以随时修正：

```bash
# 查看当前类型
$ vortex research archive --factor momentum_20d
  → 因子类型：动量型（自动推断）

# 修正类型
$ vortex research retype --factor momentum_20d --factor-type technical
  ✅ momentum_20d 因子类型已从"动量型"修正为"技术型"
  📊 历史 12 次评测的同类排名将基于"技术型"重新计算
```

修正后的行为：
- **因子档案中所有历史评测**的同类排名自动按新类型重算
- **已生成的 HTML 报告**不会重新生成（报告中的排名是快照，可在报告页面看到"类型已变更"标记）
- **对比功能**（`research compare`）使用当前类型

#### 类型分类表

| 类型 | 说明 | 典型因子 | 评测侧重 |
|------|------|---------|---------|
| **价值型** (Value) | 衡量股票的"便宜程度" | PE、PB、股息率、EV/EBITDA | IC 稳定性、大盘/小盘差异、Regime 敏感性 |
| **动量型** (Momentum) | 衡量价格或基本面的趋势强度 | 20 日动量、52 周新高、SUE | IC 衰减速度、换手率、反转风险 |
| **质量型** (Quality) | 衡量公司经营质量 | ROE、毛利率稳定性、现金流质量 | 长期有效性（20d IC）、抗衰退能力 |
| **波动型** (Volatility) | 衡量价格或收益的波动特征 | 波动率、特质波动、偏度 | 低波溢价、尾部风险、市场环境依赖 |
| **流动性** (Liquidity) | 衡量交易活跃度和市场摩擦 | 换手率、Amihud 非流动性、买卖价差 | 容量约束、小盘偏差、成本敏感性 |
| **技术型** (Technical) | 基于量价形态的短期信号 | MACD 背离、量价背离、筹码分布 | IC 衰减（1d vs 5d）、交易成本占比、过拟合风险 |
| **另类** (Alternative) | 非传统数据源 | 新闻情绪、分析师一致预期变化、股东行为 | 数据覆盖率、时效性、与传统因子的独立性 |
| **复合/模型** (Composite/ML) | 多因子加权或机器学习输出 | LightGBM 预测、多因子合成信号 | 过拟合指标、样本外衰减、特征重要性 |

因子类型在 `ResearchProfile` 中配置，系统据此自动调整评测的侧重维度和 HTML 报告布局。

### 2.4 评测规则——系统内置，不需要用户定义

因子评测的核心规则由系统内置，研究员不需要自己定义"怎么评"——只需要选择因子、选择 snapshot，系统自动完成全部评测。

#### 内置评测规则清单

| # | 规则 | 计算方式 | 意义 |
|---|------|---------|------|
| 1 | **多周期 IC** | IC_1d、IC_5d、IC_20d：每个截面计算因子值与未来 N 日收益的 Spearman 相关系数 | IC（Information Coefficient，信息系数）：衡量因子对未来收益的预测能力。IC=0.05 意味着因子排名与收益排名有 5% 的相关性，在截面选股中已属有效 |
| 2 | **RankIC** | 与 IC 相同但使用排名值计算，抗极端值干扰 | RankIC 比 IC 更稳健，不受少数极端因子值影响 |
| 3 | **ICIR** | IC 均值 / IC 标准差（类似 Sharpe 的概念） | ICIR（IC Information Ratio）：IC 的夏普比。ICIR > 0.5 说明因子不仅平均有效，而且稳定有效 |
| 4 | **IC 衰减曲线** | IC_1d → IC_5d → IC_10d → IC_20d → IC_60d 的变化趋势 | 衰减快说明因子是短期信号（需要高频调仓），衰减慢说明因子持续性好 |
| 5 | **分位收益** | 按因子值将股票分成 5 组（或 10 组），计算各组的累积收益 | 好因子应该呈现单调性：Top 组收益最高，Bottom 组最低 |
| 6 | **多空收益** | 做多 Top 组 + 做空 Bottom 组的收益 | 最直观的因子盈利能力指标 |
| 7 | **因子覆盖率** | 每期有因子值的股票数 / 全部股票数 | 覆盖率太低（< 70%）的因子不具备实用性 |
| 8 | **因子自相关** | 相邻两期因子排名的 Spearman 相关系数 | 自相关高说明换手低（交易成本低），自相关低说明因子变化快 |
| 9 | **独立性检验** | 扣除市值、行业等已知风格因子后的残差 IC | 判断因子是否提供了市值/行业之外的独立信息 |
| 10 | **截面分布特征** | 每期因子值的偏度、峰度、极端值比例 | 分布异常的因子需要预处理（去极值、标准化）后才可靠 |

> **研究员只需要决定**：用哪个因子、用哪版 snapshot、评测哪几个周期。其余全部由系统自动执行。

### 2.6 因子评测 HTML 报告（Tear Sheet）

HTML 报告是研究员日常工作最重要的产物之一。参考 Alphalens 的 tear sheet 但更丰富，包含 7 个分析页面。每个图表旁附简短指标释义，让第一次使用的研究员也能理解。

> **指标释义原则**：HTML 报告中每个图表的标题下方附一行灰字释义，例如 "IC（信息系数）：因子排名与未来收益排名的截面相关性，越高说明预测能力越强"。释义由系统内置。

#### 页面一：IC 分析

IC 是因子评测的核心——它在每个截面上计算因子值与未来收益的相关性。

| 图表 | 说明 | 怎么看 |
|------|------|--------|
| IC 时序曲线 | IC_1d / IC_5d / IC_20d 三条线，标注均值和置信区间 | 均值为正且置信区间不跨零 → 因子有效 |
| RankIC 时序曲线 | 与 IC 对照，观察排序稳定性 | RankIC 比 IC 更稳健，两者差异大说明有极端值干扰 |
| ICIR 滚动窗口 | 60 日滚动 ICIR，识别因子有效性的时变特征 | ICIR 稳定 > 0.5 → 因子持续有效 |
| IC 衰减曲线（Decay） | 从 1d 到 60d 的 IC 衰减速度 | 衰减慢 → 适合低频调仓；衰减快 → 需高频调仓 |
| IC 分布直方图 | 每日 IC 的分布形态 | 近似正态 → 正常；多峰 → 因子行为不稳定 |

#### 页面二：分组收益

分组收益是截面分析最直观的表达——按因子值分成 N 组，看各组收益是否单调。

| 图表 | 说明 | 怎么看 |
|------|------|--------|
| 分位累积收益曲线 | 5 分位（或 10 分位），各组累积收益走势 | 各组呈扇形展开不交叉 → 单调性好 |
| 分位年化指标表 | 各分位的年化收益、Sharpe、最大回撤、换手率 | Top 组 Sharpe 显著高 → 因子可实操 |
| 多空组合净值曲线 | 做多 Top 组 + 做空 Bottom 组 | 净值持续上升且回撤可控 → 盈利稳定 |
| 多空月度收益热力图 | 按月展示多空收益 | 某些月集中亏损 → 存在季节性风险 |

#### 页面三：截面分布与稳定性

截面质量直接影响因子可靠性——分布异常或覆盖率不稳定的因子，IC 再高也不可信。

| 图表 | 说明 | 怎么看 |
|------|------|--------|
| 截面分布直方图 | 每期截面上因子值的分布形态 | 近似正态 → 好；严重右偏 → 需去极值处理 |
| 因子覆盖率时序 | 每期有因子值的股票数占比 | 持续 > 80% → 因子有实用性 |
| 因子截面自相关 | 相邻两期截面排名的 Spearman 相关 | 高（> 0.8）→ 低换手成本；低 → 变化快 |
| 截面均值/标准差时序 | 截面均值和标准差长期趋势 | 趋势漂移 → 因子定义随时间失效 |

#### 页面四：截面快照（交互式）

这是最具教育意义的页面——让研究员直观理解截面分析到底在看什么。

| 图表 | 说明 | 怎么看 |
|------|------|--------|
| 截面散点图 | 某一天所有股票的因子值(x) vs 未来5日收益(y) | 正相关趋势线 → 因子对这天有效 |
| 截面分位柱状图 | 某一天 5 分位的平均收益 | 单调递增 → 因子在这天的截面有效 |
| 截面回放滑块 | 可交互时间滑块，切换不同截面日期 | 拖动观察不同市场环境下截面形态变化 |

> 通过一个截面的散点图，研究员可以直观理解 IC 的含义——它就是这张散点图的相关系数。

#### 页面五：因子相关性矩阵

| 图表 | 说明 |
|------|------|
| 相关性热力图 | 当前因子与已注册因子库中所有因子的截面相关性 |
| 独立性评分 | 扣除市值、行业等已知风格因子后的残差 IC |
| 增量贡献分析 | 在已有因子组合基础上，加入当前因子后的边际 IC 提升 |

#### 页面六：Regime 分析

Regime（市场状态）划分是一个定性问题。系统提供**规则划分 + AI 辅助**两种方式：

**规则划分（默认）**：基于沪深300 指数的 120 日移动平均线和波动率：
- **牛市**：指数 > MA120 且 MA120 向上
- **熊市**：指数 < MA120 且 MA120 向下
- **震荡市**：MA120 斜率平坦（绝对值 < 阈值）
- **风格切换期**：最近 20 日内发生牛熊/震荡切换

**AI 辅助划分**：用户可通过 `vortex ai ask` 请求 LLM 对特定区间进行定性判断（见 §2.13 AI 辅助判断能力）。

| 图表 | 说明 | 怎么看 |
|------|------|--------|
| Regime 分段表现 | 各 Regime 下的 IC 和多空收益 | 牛市高、熊市低 → 因子有周期性 |
| Regime 转换表现 | 风格切换前后 10 个交易日的因子表现 | 切换期大幅回撤 → 需搭配择时 |
| 波动率分层表现 | VIX（或等效指标）高/低环境下的因子有效性 | 高波有效 → 适合做危机 alpha |

#### 页面七：容量与实用性

| 图表 | 说明 | 怎么看 |
|------|------|--------|
| 按市值分组的因子表现 | 大盘(沪深300) / 中盘(中证500) / 小盘(中证1000) 各组的 IC | 只在小盘有效 → 容量受限 |
| 换手率与交易成本估算 | 按不同调仓频率（日/周/月）估算的双边换手率和手续费 | 年化换手 > 20x → 成本侵蚀严重 |
| 容量上限估算 | 基于成交量约束，估算最大可管理资金规模 | 见下方计算公式 |

**容量估算公式**：

```
容量上限 = 持仓股票池中，每只股票过去 20 日日均成交额 × 参与率上限(默认 10%) × 持仓股票数

举例：
  持仓 30 只股票，平均日均成交额 5,000 万
  容量上限 = 5,000万 × 10% × 30 = 1.5 亿

  如果因子只在小盘股有效（日均成交额 1,000 万）：
  容量上限 = 1,000万 × 10% × 30 = 3,000 万
```

参与率上限 10% 是行业惯例——超过 10% 就会对股价产生明显冲击。用户可在配置中调整。

报告的设计原则：
1. **精美且专业**——排版、配色、字体经过设计，可以直接截图发给团队讨论
2. **交互式**——支持缩放、悬浮提示、切换分位数，不是静态图片
3. **自包含**——一个 HTML 文件包含所有数据和图表，无外部依赖，可离线查看
4. **命名规则**——`tear_sheet.html`，存放在 `research/experiments/{run_id}/` 目录下

```json
      { "participation_rate": 0.05, "capacity": 7500, "impact_cost": 0.1 },

  }

```

### 2.8 并发控制

因子评测和回测都是计算密集型任务。系统内置并发控制，避免资源耗尽：

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `max_concurrent_tasks` | 3 | 同时运行的评测/回测任务数上限 |
| `queue_overflow` | wait | 超过上限时：`wait`（排队）或 `reject`（拒绝） |

```bash
# 同时提交多个因子评测

$ vortex research run --profile cn_alpha_basic --snapshot 2026-04-03 --factor value_bp &
$ vortex research run --profile cn_alpha_basic --snapshot 2026-04-03 --factor quality_roe &
$ vortex research run --profile cn_alpha_basic --snapshot 2026-04-03 --factor volatility_60d &
# 前 3 个立即执行，第 4 个排队等待

$ vortex research status --profile cn_alpha_basic
```

**耗时预估逻辑**：

| 阶段 | 预估方式 |
|------|----------|
| 因子值计算 | 首次：按数据量线性估算（约 1s / 10万行）；后续：跳过（有缓存） |
| 评测规则执行 | 取该因子历史评测中同规模任务的平均耗时 |
| HTML 报告生成 | 固定约 10-15 秒 |
| 首次运行（无历史） | 使用全局平均值 × 数据规模系数，标注"粗略估计" |

进度百分比 = 已完成步骤耗时 / 预估总耗时。预估不准时会动态修正（类似文件拷贝进度条）。

并发上限可在全局配置中调整。首版默认 3，用户根据机器资源自行增减。回测同样受此限制。

### 2.9 因子档案——"每个因子都有履历"

所有评测结果保存在数据库中。研究员随时可以调出任意因子的完整评测历史。

#### 去重机制——同条件不重复存

**核心原则**：同一个因子、同一份数据（snapshot）、同一套评测配置，结果是确定性的——不需要存多次。

系统使用**复合 key** 去重：

```
eval_key = (factor_name, snapshot, market, label_spec, factor_code_hash)
```

| 场景 | eval_key 是否相同 | 行为 |
|------|------------------|------|
| 同因子同 snapshot 再跑一次 | ✅ 相同 | 跳过，返回已有结果 |
| 同因子换 snapshot（新一天的数据） | ❌ 不同 | 新增一条评测记录 |
| 同因子改了代码后再跑 | ❌ 不同（code_hash 变了） | 新增一条，旧记录保留 |
| 同因子换 label_spec（5d → 20d） | ❌ 不同 | 新增一条 |

```bash
$ vortex research run --profile cn_alpha_basic --factor momentum_20d --snapshot 2026-04-03

  ⚠️ 已存在相同条件的评测记录 (research_20260403_220501_a3f2)
  跳过重复评测，直接返回已有结果。
  如需强制重算，使用 --force 参数。
```

这意味着因子档案中记录的"12 次评测"，每一次的 eval_key 都不同——可能是不同的 snapshot（跟踪因子随时间的表现变化）、不同的代码版本（跟踪因子优化迭代），或不同的评测口径。

#### 因子身份标识

因子档案按 `(factor_name, market, label_spec)` 分组汇总统计。同一个因子在不同配置下是不同的档案：

例如：`(momentum_20d, cn_stock, ret_5d)` 和 `(momentum_20d, cn_stock, ret_20d)` 是两个不同的因子档案。

### 2.10 因子定量评分——好坏是比出来的

因子的好坏不应该用绝对阈值判断——不同市场环境、不同因子类型的有效性差异很大。一个 IC=0.04 的因子在低波动市场可能已经很好，在高波动市场可能毫无价值。

Vortex 采用**分位排名 + 历史比较**的定量评分体系，不使用绝对阈值。

#### 评分计算方式

每次评测产出 **5 个维度的分位分数**（0-100），每个分数的含义是"在同类因子历史评测中排在第几分位"：

| 维度 | 计算方式 | 比较基准 |
|------|---------|---------|
| **预测力** | IC_5d 在该因子类型所有历史评测中的分位排名 | 同类型因子历史 IC 分布 |
| **稳定性** | ICIR_5d 在该因子类型所有历史评测中的分位排名 | 同类型因子历史 ICIR 分布 |
| **盈利能力** | 多空年化收益在该因子类型所有历史评测中的分位排名 | 同类型因子历史多空收益分布 |
| **风险控制** | 多空最大回撤（绝对值越小越好）的分位排名 | 同类型因子历史回撤分布 |
| **独立性** | 扣除市值和行业后的残差 IC 分位排名 | 同类型因子历史残差 IC 分布 |

综合分 = 各维度加权平均（默认等权 20% × 5）。

#### 分位排名如何理解

```
综合分 85 → 该因子在同类因子中排名前 15%
综合分 50 → 该因子处于中位水平
综合分 20 → 该因子在同类因子中排名后 20%
```

#### 与历史自身比较

除了和同类因子比，每次评测还会和该因子自身历史评测比较：

```
╭─ 定量评分 ──────────────────────────────────────────────────╮
│  ── 同类排名（vs 动量型因子历史评测 N=247） ──               │
│  预测力         72 分位  (IC_5d=0.068, 超过 72% 同类)       │
│  稳定性         81 分位  (ICIR=0.62, 超过 81% 同类)         │
│  盈利能力       68 分位  (多空年化=14.2%)                    │
│  风险控制       55 分位  (多空回撤=-18.3%)                   │
│  独立性         74 分位  (残差IC=0.041)                      │
│  综合分         70 分位                                      │
├──────────────────────────────────────────────────────────────┤
│  ── 自身趋势（vs 自身最近 10 次评测） ──                     │
│  预测力         ↑ 上升 (+8, 从 64 到 72)                    │

```

#### 冷启动问题

前 30 次因子评测（同类型）积累不够时，分位排名的参考价值有限。系统会标注"样本不足，仅供参考"，同时展示原始指标值供研究员自行判断。

> **设计原则**：分位分数告诉研究员"这个因子在所有你研究过的因子中处于什么位置"，而不是给一个脱离上下文的 A/B/C 标签。好坏是比出来的，不是绝对标准定的。

### 2.11 模型类因子——机器学习工作流

除传统因子（算术公式）外，研究员也可能使用机器学习模型生成因子。模型类因子的工作流与传统因子不同：

#### 工作流程

```
特征准备 → 模型训练 → 模型验证 → 预测生成 → 因子评测
   ↓           ↓          ↓          ↓          ↓
```

模型类因子的特殊关注点：
1. **过拟合监控**——训练集 vs 验证集表现的比值，自动标注风险
2. **特征重要性**——Top N 特征及其贡献度，帮助研究员理解模型在学什么
3. **模型版本管理**——每次训练的模型文件、配置、特征列表都保存，可回溯
4. **预测生成与评测分离**——模型训练生成 pred.pkl，然后走标准因子评测流程

#### 模型训练标准工件

`model_train` workflow 的正式输出工件，与 `factor_eval` 的 `pred.pkl` / `ic_metrics.csv` 等并列：

| 工件 | 说明 |
|------|------|
| `model.pkl` | 训练好的模型文件 |
| `model_report.json` | 训练报告：超参数、训练/验证指标、过拟合指标、耗时 |
| `feature_manifest.json` | 特征列表及版本：特征名、来源、数据类型、缺失率 |
| `validation_metrics.json` | 验证集详细指标：分期 IC、分组收益、学习曲线 |
| `pred.pkl` | 模型预测结果（与传统因子共用格式，可直接进入 `factor_eval`） |

> **衔接规则**：模型训练完成后，`pred.pkl` 可直接作为因子评测的输入。研究员执行 `vortex research run --workflow factor_eval --input {model_run_id}` 即可复用标准评测流程。`inspect` 和 `compare` 命令根据 workflow_type 自动展示不同维度的信息。

### 2.12 训练集与测试集——谁需要、怎么分

#### 传统因子不涉及训练集/测试集

传统因子（价值、动量、质量等）是基于公式计算的，没有"训练"过程。评测时使用全样本数据，不存在过拟合风险——因为没有参数可拟合。

#### 模型因子必须严格区分

模型因子（§2.11）使用机器学习算法从数据中学习规律，**必须**区分训练集和测试集，否则评测结果不可信：

| 数据集 | 用途 | 时间范围（示例） |
|--------|------|-----------------|
| 训练集 (train) | 模型学习参数 | 2015-01-01 → 2022-12-31 |
| 验证集 (validation) | 调超参数、选模型 | 2023-01-01 → 2023-12-31 |
| 测试集 (test) | 最终评估，不能提前看 | 2024-01-01 → 2026-04-03 |

```yaml
# 模型因子的 Profile 配置
model_config:
  train_end: "2022-12-31"
  validation_end: "2023-12-31"
  # test_start 自动为 validation_end + 1
```

系统自动校验：如果模型因子的评测区间包含训练集数据，会在报告中标红 ⚠️ "测试区间与训练集重叠，结果不可信"。

#### 因子评测也存在"间接过拟合"风险

虽然传统因子没有训练过程，但如果研究员反复调整因子公式直到 IC 变好，本质上也是一种过拟合。系统通过因子档案的 `factor_code_hash` 记录每次代码版本，如果同一个因子短期内代码改了很多次且每次 IC 都在涨，报告中会提示"可能存在数据窥探 (data snooping)"。

### 2.13 数据预处理——因子计算前的清洗

因子计算依赖干净的输入数据。系统在因子值计算前自动执行预处理流水线：

```
原始因子值 → 缺失值处理 → 极端值处理 → 标准化 → 可用于评测的因子值
```

| 步骤 | 默认方法 | 说明 |
|------|----------|------|
| 缺失值处理 | 行业中位数填充 | 缺失比例 > 50% 的股票直接排除 |
| 极端值处理 | MAD (中位数绝对偏差) 3 倍截断 | 比 3σ 截断对偏态分布更稳健 |
| 标准化 | 截面 z-score | 每个截面内标准化为均值0、标准差1 |
| 行业/市值中性化 | 可选，默认关闭 | 回归剥离行业和对数市值的影响 |

```bash
# 评测时自动执行预处理（用户无需干预）
$ vortex research run --profile cn_alpha_basic --factor momentum_20d --snapshot 2026-04-03

  [1/5] 加载因子值...      5,200 × 1,523
  [2/5] 数据预处理...
        缺失值处理:  排除 38 只 (缺失率>50%), 填充 12,450 个值
        极端值截断:  处理 2,341 个极端值 (MAD×3)
        截面标准化:  完成
  [3/5] 执行评测规则...
  ...

# 如需自定义预处理参数
$ vortex research run --factor momentum_20d --snapshot 2026-04-03     --winsorize-method sigma --winsorize-n 3     --neutralize industry,log_cap
```

> **用户可覆盖但不需要关心**：预处理参数有合理的默认值，首次使用不需要任何配置。高级用户可以通过 Profile 或命令行参数覆盖。预处理参数也记录在 eval_key 中——不同预处理参数算不同的评测。

### 2.14 AI 辅助判断——定性问题交给大模型

量化研究中存在不少"定量不好确定"的判断，比如：
- 因子该归为什么类型？
- 某个时间段是牛市还是震荡市？
- 因子代码里的逻辑有没有前视偏差？
- 这个异常数据是数据源问题还是真实事件？

系统提供统一的 AI 辅助判断能力，不是独立产品，而是嵌入到各个需要定性判断的流程中。

#### 调用方式

| 嵌入点 | 触发条件 | AI 做什么 |

### 2.15 与 v1 的关系

v1 冻结了 5 个研究动作（run / inspect / compare / publish-signal / status）。本节扩展的内容：

| 增强点 | 涉及的 v1 动作 | 变更类型 |

## 三、策略组合场景——"流水线与灵活组装"

### 3.1 场景定位

策略不是一个"脚本"。研究员真正做的是**组装一条决策流水线**：

1. 从什么范围里选——股池
2. 选什么、怎么打分——信号
3. 买多少——组合构建
4. 什么情况不能买——风控

这 4 个环节各自独立，可以灵活替换。换一个选股逻辑，不需要重写风控；换一个风控规则，不需要重新训练模型。

> **与 Trade 域的边界**：Strategy 域负责"决定买什么、买多少"，输出 `target portfolio`；Trade 域负责"怎么下单、怎么执行"。回测时 Strategy 内部使用成交模拟器（Backtrader），但这是回测基础设施，不等于 Trade 域的 Execution。

### 3.2 策略流水线（4 阶段）

```
┌──────────┐    ┌──────────┐    ┌──────────────────┐    ┌──────────┐
│ Universe │ →  │  Alpha   │ →  │ Portfolio Constr. │ →  │   Risk   │ → target portfolio
│  股池选择 │    │ 信号生成  │    │    组合构建        │    │ 风控约束  │   → Trade 域执行
└──────────┘    └──────────┘    └──────────────────┘    └──────────┘
```

每个阶段的职责和接口：

| 阶段 | 输入 | 输出 | 典型实现 |
|------|------|------|---------|
| Universe | 全市场品种列表 | 候选股票池 | 指数成分、行业过滤、流动性筛选 |
| Alpha | 候选股票池 + snapshot | 每只股票的信号分数 | 因子打分、模型预测、规则筛选 |
| Portfolio Construction | 信号分数 | 目标持仓权重 | 等权、信号加权、风险平价、均值方差 |
| Risk | 目标持仓 | 约束后持仓（即 target portfolio） | 单票上限、行业集中度、回撤止损 |

Risk 阶段的输出即为 `target portfolio`，通过 `compile` 交付给 Trade 域执行。回测时，Strategy 域内部使用 Backtrader 模拟成交，但这只是验证手段，不改变域边界。

**关键设计**：每个阶段可以独立测试。

- 只测 Alpha？看 IC 就行，不用跑全量回测
- 只测 Risk？给定一个持仓，看约束调整了什么
- 测完整流水线？跑 backtest，从头到尾走一遍（Strategy 内部使用 Backtrader 模拟成交）

### 3.3 Alpha 阶段内部组合

Alpha 是最复杂、最灵活的一个阶段。研究员在 Alpha 内部有两种组合方式：

#### 方式一：加权融合

多个因子信号按权重合并为一个综合信号：

```
momentum_20d (权重 0.3)  ─┐
value_bp     (权重 0.3)  ─┼→  加权融合  →  综合信号
quality_roe  (权重 0.2)  ─┤
volatility   (权重 0.2)  ─┘
```

适用场景：多因子选股，每个因子提供独立视角，综合考量。

配置示例：

```yaml
alpha:
  mode: weighted_fusion
  factors:
    - name: momentum_20d
      weight: 0.3
    - name: value_bp
      weight: 0.3
    - name: quality_roe
      weight: 0.2
    - name: volatility
      weight: 0.2
```

#### 方式二：级联

多个子策略按顺序执行，每一步的输出是下一步的输入：

```
全 A 股
  ↓ 指数动量判断 → 当前行情属于"小盘成长"风格
  ↓ 使用小盘成长选股策略 → 筛出 50 只候选
  ↓ 对 50 只候选叠加动量择时 → 最终信号
```

适用场景：多阶段决策，层层筛选，逐步聚焦。

配置示例：

```yaml
alpha:
  mode: cascading
  stages:
    - name: style_rotation
      type: regime_detector
      params:
        indices: [csi300, csi500, csi1000]
        lookback: 20
    - name: stock_selection
      type: factor_score
      params:
        regime_mapping:
          large_value: value_bp
          small_growth: momentum_20d
          balanced: quality_roe
    - name: timing_overlay
      type: momentum_timing
      params:
        entry_threshold: 0.6
        exit_threshold: 0.3
```

两种方式可以嵌套——比如在级联的某一步使用加权融合。由研究员根据策略逻辑自行选择。

### 3.4 一条策略组合的完整旅程

以用户描述的真实场景为例：

> "通过判断多个不同风格的指数动量，预判当下的行情在哪个风格下进行，使用该风格对应的选股策略进行选股，然后在股池里面叠加一个动量策略进行高抛低吸。"

### 3.5 策略阶段独立测试

研究员不一定要跑完整回测才能评估策略。每个阶段可以独立验证：

```bash
# 只测 Alpha 的信号质量（不涉及组合构建和风控）
$ vortex research run --profile style_rotation_alpha --snapshot 2026-04-03 --workflow factor_eval
```

这样做的好处：
1. **快速迭代**——改 Alpha 逻辑时不用等完整回测跑完
2. **问题定位**——回测结果不好，先定位是信号不行还是风控太紧
3. **模块化思维**——鼓励研究员把策略拆成可独立评估的组件

## 四、回测生命周期场景——"不浪费任何一次回测"

### 4.1 场景定位

研究员在开发策略的过程中会跑**大量回测**。当前的问题是：

1. 回测完就忘了——三天前跑的那个参数组合，现在找不到了
2. 不知道哪些回测有价值——100 次回测里可能只有 5 次值得保留
3. 占用空间越来越大——但不敢删，怕删掉有用的

Vortex 的回测生命周期管理目标：**默认全保存，自动评分，智能推荐清理**。

### 4.2 默认全保存

每次回测结果自动保存到数据库，包含：

| 保存内容 | 说明 |
|---------|------|
| 配置快照 | 完整的 StrategyProfile 展开结果，确保可复现 |
| 绩效指标 | 收益率、Sharpe、最大回撤、换手率、成本等全部指标 |
| 资金曲线 | 每日净值时序 |
| 交易记录 | 每笔交易的时间、方向、价格、数量 |
| 流水线状态 | 各阶段的输入输出摘要 |
| 质量评分 | 自动计算的评分（见 4.3） |

研究员不需要主动保存——跑完就存好了。

### 4.3 回测自动评分

每次回测完成后，系统自动从 5 个维度计算质量分数，综合为一个字母评级：

| 维度 | 计算方式 | 权重 |
|------|---------|------|
| **收益风险比** | Sharpe ratio 的统计显著性（bootstrap p-value） | 25% |
| **稳定性** | 滚动 6 个月 Sharpe 的标准差，越小越好 | 25% |
| **回撤控制** | 最大回撤 / 恢复时间的综合评分 | 20% |
| **成本效率** | (收益 - 预估成本) / 换手率 | 15% |
| **过拟合风险** | 训练期 vs 测试期表现差异；参数敏感性 | 15% |

评级标准：

| 评级 | 综合分 | 含义 |
|------|--------|------|
| A | ≥ 85 | 优秀，强烈建议保留并进一步验证 |
| B+ | ≥ 75 | 良好偏优，值得重点关注 |
| B | ≥ 70 | 良好，值得保留观察 |
| B- | ≥ 60 | 尚可，可保留但需谨慎 |
| C | ≥ 50 | 一般，可保留但不建议直接使用 |
| D | ≥ 35 | 较差，建议分析原因后决定 |
| F | < 35 | 无效，建议清理 |

> **回测评级与因子评级共用 A / B+ / B / B- / C / D 等级体系**（回测额外增加 F 级）。

回测完成时直接展示评级。如果当前 Profile 下积累的回测数量超过阈值（默认 50 条），系统在评分后**自动提醒**用户考虑清理：


回测完成时直接展示评级：

### 4.5 双基准对照体系

回测不只是看绝对收益，更要看相对基准的表现。Vortex 支持两种基准模式同时展示：

#### 基准一：指数点位基准

直接使用基准指数的收盘价序列计算净值。简单、无争议，适合快速判断策略跑赢大盘的幅度。

#### 基准二：Alpha 基准（指数成分股仿真）

以基准指数的成分股和权重构建一个"虚拟指数组合"，与策略共用同一套交易规则（考虑涨跌停、停牌、手续费）。这样算出的超额收益才是真正的 Alpha——排除了"策略和指数不可比"的干扰。

| 权重模式 | 说明 | 计算开销 |
|----------|------|----------|
| 季报固定 | 使用指数季报公布的成分股和权重，季报间保持不变 | 低 |
| 动态跟踪 | 回测每个调仓日重新获取成分股和权重，仓位随指数变化 | 高（约增加 30-50% 回测时间） |

```yaml
# 在策略 Profile 中配置
benchmark:
  index: "000300.SH"           # 基准指数代码
  show_price_benchmark: true    # 是否显示指数点位基准
  show_alpha_benchmark: true    # 是否显示 Alpha 基准
  alpha_weight_mode: quarterly  # quarterly（季报固定）或 dynamic（动态跟踪）
```

```
╭─ 双基准对照 ──────────────────────────────────────────────────╮
│  策略年化         28.3%                                       │
│                                                               │
│  vs 沪深300指数   +14.1% (指数年化 14.2%)                     │
│  vs Alpha基准     +10.8% (Alpha基准年化 17.5%)                │
│                                                               │
│  说明：Alpha 基准使用指数成分股季报权重，共用交易规则          │
│       真实 Alpha = 10.8%，指数对比高估了 3.3%                 │
╰──────────────────────────────────────────────────────────────╯
```

> Alpha 基准的意义：指数点位基准忽略了涨跌停、停牌等实际约束，导致超额收益偏高。Alpha 基准通过成分股仿真排除这些干扰，让超额收益更可信。

清理机制的设计原则：
1. **系统只建议，用户做决定**——绝不自动删除任何回测
2. **有 tag 的回测受保护**——研究员标记过的回测不会出现在清理建议里
3. **分级清理**——先清 F 级，再清 D 级，C 及以上默认保留
4. **可恢复**——清理只是标记为 archived，30 天内可恢复

## 五、推送通知场景——"让系统主动找你"

### 5.1 场景定位

量化平台不应该只在用户主动查询时才给信息。很多关键事件——数据异常、回测完成、风控阻断——应该主动推送到用户手上。

当前 Vortex 已经集成了飞书消息通道。本节把通知机制从"能发消息"升级到"知道什么时候该发什么消息"。

### 5.2 通知触发矩阵

| 域 | 触发事件 | 级别 | 默认推送 | 消息内容 |
|---|---------|------|---------|---------|
| Data | 质量检查阻断（Fail-Closed） | 🔴 critical | ✅ | 哪个规则失败、影响品种数、建议操作 |
| Data | 数据源不可用（3 次重试全部失败） | 🔴 critical | ✅ | 数据源名称、错误信息、预计影响 |
| Data | 部分品种更新失败 | 🟡 warning | ✅ | 失败品种列表、原因分类、是否影响 snapshot |
| Data | 每日更新完成 | 🟢 info | ❌ | 品种数、数据条数、耗时 |
| Research | 因子评测完成 | 🟢 info | ❌ | 因子名称、IC、评级、HTML 报告路径 |
| Research | 发现 A 级因子 | 🟡 warning | ✅ | 因子名称、核心指标、建议关注 |
| Strategy | 回测完成 | 🟢 info | ❌ | run_id、评级、核心指标 |
| Strategy | 回测评分 F 级 | 🟡 warning | ✅ | run_id、失败维度、建议 |
| Trade | 风控阻断 | 🔴 critical | ✅ | 阻断规则、涉及订单、等待操作 |
| Trade | 实盘调仓完成 | 🟢 info | ✅ | 成交汇总、偏差率、异常单 |
| Trade | 对账异常 | 🔴 critical | ✅ | 偏差金额、偏差品种、建议操作 |

### 5.3 通知内容规范

每条通知必须包含 4 个要素：


#### 数据从哪来——Watch Provider（关注数据提供者）

用户通过 `vortex watch add --type xxx` 订阅关注时，`--type` 对应的是一个 **Watch Provider**。每个 Provider 负责：从某个域获取数据 → 分析计算 → 格式化为推送消息。这是一种**插件式架构**——系统内置常用 Provider，用户也可以自行扩展。

**内置 Provider 列表**：

| Provider type | 显示名称 | 数据来源 | 快照策略 | 必需参数 |
|---------------|----------|----------|----------|----------|
| `valuation_percentile` | 估值百分位 | Data 域 `valuation` 表（PE/PB/PS 等） | 最新成功快照（前一交易日） | `targets`: 指数代码列表 |
| `strategy_analysis` | 策略调仓分析 | Strategy 域 `signal` 工件（最近一次信号结果） | 读取最近一次已完成的信号工件，不触发新计算 | `profile`: 策略 Profile 名称 |
| `data_status` | 数据健康报告 | Data 域 `run_log` 表 | 最近 24 小时运行记录 | 无（默认监控所有 DataProfile） |
| `factor_monitor` | 因子跟踪 | Research 域 `factor_archive` 表 | 最新评测结果 | `factors`: 因子名列表 |

> **关键设计决策**：`strategy_analysis` 读取的是最近一次已经跑完的信号结果，而不是定时触发一次新的策略计算。原因：策略计算消耗资源较大，不适合作为每日/每周定时推送的触发动作。如果用户需要定时跑信号，应通过运维调度（§六）配置定时策略任务，watch 只负责读取结果并推送。

**Provider 接口契约**：

每个 Provider 实现以下方法：

```python
class WatchProvider:
    """关注数据提供者基类。"""

    provider_type: str          # 唯一标识，如 "valuation_percentile"
    display_name: str           # 展示名称，如 "估值百分位"
    required_params: list[str]  # 必需参数列表
    optional_params: dict       # 可选参数及默认值

    def validate_params(self, params: dict) -> tuple[bool, str]:
        """
        在 watch add 时校验用户参数。
        返回 (是否通过, 错误信息)。
        例如：检查 targets 中的指数代码是否在系统中存在。
        """

    def fetch(self, params: dict, snapshot_date: str) -> dict:
        """
        从对应域获取原始数据。
        snapshot_date 由快照策略决定（通常是最近交易日）。
        返回原始数据字典。
        """

    def analyze(self, raw_data: dict) -> dict:
        """
        分析计算。如：计算百分位排名、对比上期变化等。
        返回分析结果字典。
        """

    def format_message(self, analysis: dict) -> str:
        """
        格式化为推送文本。
        返回适合飞书/Telegram 等渠道的纯文本消息。
        """
```

**执行流程**：

```
调度器(cron) → 读取 watch_subscriptions → 找到到期订阅
  → Provider.validate_params()  # 参数仍有效？
  → Provider.fetch()            # 获取数据
  → Provider.analyze()          # 分析计算
  → Provider.format_message()   # 生成推送文本
  → 通知路由(§5.4)             # 按渠道发送
  → 更新 watch_subscriptions.last_run / last_status
```

**失败处理**：

| 失败阶段 | 处理方式 |
|----------|----------|
| `fetch` 失败（数据未就绪） | 延迟 30 分钟重试，最多 2 次；仍失败则推送降级消息："数据暂未就绪，稍后补发" |
| `analyze` 失败（计算异常） | 记录错误日志，推送降级消息；不影响其他订阅 |
| 通知发送失败 | 复用 §5.4 的多渠道降级机制 |
| 连续 3 天同一订阅失败 | 自动暂停该订阅，推送提醒："您的订阅'估值日报'因连续失败已暂停，请检查" |

**自定义 Provider**：

用户可以注册自己的 Provider，例如监控某个特定指标：

```bash
$ vortex watch register-provider --file my_custom_watch.py
✅ 已注册 Provider: custom_sector_rotation
   参数: targets(必需), lookback_days(可选, 默认60)

$ vortex watch add --name "行业轮动监控"     --type custom_sector_rotation     --targets "申万一级"     --schedule "weekly mon 08:00"     --channel feishu
```

#### 产品归属

定期关注属于**通知域的扩展**（§五），而不是独立域：
- 数据来源：通过 Watch Provider 从各域读取，Provider 负责声明数据源和快照策略
- 计算逻辑：由 Provider 的 `analyze()` 方法实现，不是 watch 模块自己写业务逻辑
- 推送通道：复用 §5.4 的多渠道通知路由
- 调度执行：复用运维调度系统（与数据更新的定时任务同一套机制）

```yaml
# watch 订阅信息存在数据库 watch_subscriptions 表，不是配置文件
watch_subscriptions:
  - name: "估值日报"
    type: valuation_percentile            # 对应内置 Provider
    params: { targets: ["000300.SH", "000905.SH"] }
    schedule: "daily 08:00"
    notification_route: default
    status: active
    last_run: "2026-04-04T08:00:12"       # 最近执行时间
    last_status: success                    # 最近执行状态
    consecutive_failures: 0                 # 连续失败计数
```

## 六、使用流程总览

本节为每个核心场景提供完整的操作流程，既是开发参考也是用户手册。

### 6.1 数据监控——日常无感，异常即知

**典型日常（无需操作）**：

```
06:00  [自动] 定时任务启动 → 拉取前一交易日数据
06:05  [自动] 质量检查 → 通过 → PIT 快照 → 发布
06:06  [自动] 估值分析自动运行 → 结果入库
        研究员起床后看飞书：无消息 = 一切正常
```

**异常日（自动重试 + 通知）**：

```
06:00  [自动] 拉取失败（API 超时）
06:05  [自动] 第 1 次重试（等 5 分钟后）
06:10  [自动] 第 2 次重试（等 15 分钟后）
06:25  [自动] 第 3 次重试（等 30 分钟后）
06:55  [自动] 3 次均失败 → 发送飞书通知
07:30  [手动] 研究员看到通知 → 检查原因 → 手动触发重试
        $ vortex data update --profile cn_stock_daily
```

**主动检查数据状态**：

```bash
$ vortex data status --profile cn_stock_daily
```

### 6.2 因子评测——从提交到查档

**第一步：提交评测**

```bash
$ vortex research run --profile cn_alpha_basic --snapshot 2026-04-03 \
    --workflow factor_eval --factor momentum_20d
```

**第二步：等待执行**（可提交多个任务，受并发控制管理）

```bash
$ vortex research status --profile cn_alpha_basic

```

**第三步：查看结果**

```bash
# 终端格式化打印
$ vortex research show --run-id research_20260403_120530_a1b2

```

**第四步：查阅因子档案**（历史所有评测汇总）

```bash
$ vortex research archive --factor momentum_20d --profile cn_alpha_basic
# 展示第 N 次评测的汇总统计、同类排名、趋势
```

### 6.3 策略组合——从单策略到组合回测

**第一步：开发单阶段策略**

研究员在 `strategies/` 目录下编写策略文件，每个文件实现一个阶段的逻辑。

**第二步：组装流水线**

```yaml
# profiles/style_rotation_momentum.yaml
pipeline:
  universe: "csi_800_liquid"
  alpha:
    mode: cascading
    steps:
      - style_rotation_selector    # 第 1 步：判断当前风格
      - momentum_stock_picker      # 第 2 步：在对应风格股池内动量选股
  portfolio:
    method: equal_weight
    max_positions: 30
  risk:
    max_drawdown: 0.15
    max_sector_exposure: 0.25
```

**第三步：生成信号**

```bash
$ vortex strategy signal --profile style_rotation_momentum --snapshot 2026-04-03
```

**第四步：回测**

```bash
$ vortex strategy backtest --profile style_rotation_momentum --snapshot 2026-04-03 \
    --signal sig_20260403 --start 2020-01-02 --end 2026-04-03
```

**第五步：查看回测报告**

```bash
$ open strategy/runs/backtest_20260403_143022_x7z9/backtest_report.html
```

### 6.4 回测管理——保存、评分、清理

```bash
# 查看所有回测记录
$ vortex strategy list --profile style_rotation_momentum

# 为重要回测打标签
$ vortex strategy tag --run-id backtest_20260403_143022_x7z9 --tag "v1_candidate"

# 智能清理低质量回测
$ vortex strategy cleanup --profile style_rotation_momentum

# 恢复误删的回测
$ vortex strategy restore --run-id backtest_20260301_091500_m3n4
```

```bash
# 测试通知链路
$ vortex notify test --channel feishu --message "Vortex 通知测试"

# 查看通知历史
```
