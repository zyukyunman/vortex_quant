# 配置体验原型 v1

> 本文档只讨论一件事：从用户角度看，profile 到底应该怎么被理解、怎么被选择、怎么被解释，而不是内部类怎么实现。

---

## 一、背景

当前讨论里已经确认：

1. profile 是用户的第一配置入口。
2. 但如果 profile 直接暴露太多技术参数，用户只会觉得它是“一堆看不懂的 YAML”。

所以这里不再从“字段怎么建模”出发，而是从“用户怎么理解 profile”出发。

---

## 二、用户真正想做的不是“写 profile”，而是“选工作模板”

这句话非常关键。

大多数运营者、研究员、策略开发者真正想做的是：

1. 选择一条现成工作线。
2. 看懂它会怎么运行。
3. 只修改极少量差异字段。

他们并不想：

1. 从零定义 provider
2. 从零定义 quality_policy
3. 从零定义 PIT 和发布口径

因此 profile 的产品体验要围绕“选模板 + 看解释 + 改差异”设计，而不是围绕“编辑一份全量配置”设计。

---

## 三、用户视角的三层对象

### 3.1 Profile

用户真正选择的是 `profile`。

它代表：

1. 一条工作线叫什么。
2. 它从什么时候开始。
3. 什么时候自动跑。
4. 跑完是否自动发布。
5. 通知发给谁。

### 3.2 Preset

`preset` 是用户“间接选择”的技术方案。

它代表：

1. 这条数据线默认抓哪些数据。
2. 使用哪个 provider。
3. 默认挂哪些 pack。

用户可以知道它存在，但不应该被迫每天编辑它。

### 3.3 Pack

`pack` 是平台内部规则包。

例如：

1. quality pack
2. PIT pack
3. publish pack
4. storage pack

这层主要服务于平台可替换和团队治理，不是普通运营者的第一编辑面。

---

## 四、一个好的 DataProfile 从用户角度应该长什么样

建议运营者最常见的 DataProfile 长这样：

```yaml
name: cn_stock_daily
type: data
preset: cn_stock_daily_core
history_start: 2018-01-01
schedule: weeknights_2200
publish: research_daily
notify: data_team
```

这份配置的优点在于：

1. 用户一眼就能读懂。
2. 每个字段都和动作有关。
3. 没有暴露 provider、storage、calendar、quality_policy 这类技术细节。

如果一份 profile 需要用户直接维护几十个技术字段，那它的产品体验就已经失败了。

---

## 五、用户应该如何理解每个字段

建议把运营者可见字段固定解释成下表。

### 5.1 最小字段表

1. `name`
含义：这条数据线的名字。
平时是否常改：否。

2. `preset`
含义：选择哪套内置数据方案。
平时是否常改：偶尔。

3. `history_start`
含义：第一次建库时从哪天开始补历史。
平时是否常改：首次常改，之后很少。

4. `schedule`
含义：自动更新的时间策略。
平时是否常改：偶尔。

5. `publish`
含义：更新完成后是否自动做 research snapshot 发布。
平时是否常改：偶尔。

6. `notify`
含义：运行结果通知到谁。
平时是否常改：偶尔。

### 5.2 为什么不建议暴露技术字段

下面这些字段对平台有意义，但对运营者并不是第一层应见内容：

1. `provider`
2. `calendar`
3. `quality_policy`
4. `pit_policy`
5. `snapshot_policy`
6. `storage_backend`

原因不是它们不重要，而是它们更像“系统怎么实现”，不是“用户怎么工作”。

---

## 六、profile 的解释功能应该怎么设计

产品上不应该假设用户靠读 YAML 就能理解一切。

建议固定提供两种解释面：

### 6.1 文件内注释

运营者真正会打开的 profile 文件，必须带中文注释。

注释重点解释：

1. 这个字段做什么。
2. 平时要不要改。
3. 改了会影响什么。

### 6.2 explain / resolve 命令

建议系统提供统一解释命令，例如：

```bash
vortex profile explain --profile cn_stock_daily
vortex profile resolve --profile cn_stock_daily
```

两者分工可以是：

1. `explain`：给人看，解释字段含义、preset 来源、运行动作。
2. `resolve`：给人和机器看，展示展开后的最终配置和运行时解析结果。

---

## 七、从用户体验反推，profile 系统必须满足什么

如果接受上面的体验设计，那么 profile 系统在产品层就必须满足：

1. 用户主要选择 profile，而不是手工拼参数。
2. profile 文件应该短、小、可读。
3. preset 与 pack 需要存在，但默认不直接压给普通用户。
4. profile 必须有解释能力，不应只提供原始 YAML。
5. 运行时 override 必须克制，只用于少量差异。

---

## 八、当前实现映射

### 8.1 当前已有映射

当前仓库里已经有：

1. profile 继承
2. profile resolve
3. 带中文注释的 data profile YAML

### 8.2 仍待形成的产品入口

目前还没有完整形成下面这些稳定能力：

1. 运营者最小 profile 写法
2. preset / pack 独立目录与解释机制
3. `profile explain` 命令
4. 按角色区分的 profile 视图

这部分只是说明落地现状：profile 产品心智已经清楚，但实现还没有完全收敛到“用户只需选模板、看解释、改少量差异”的体验。
