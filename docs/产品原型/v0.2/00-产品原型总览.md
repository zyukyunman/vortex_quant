# 产品原型总览

> 本目录回答的不是“当前仓库里已经有什么命令和脚本”，而是“Vortex 这个产品从用户视角应该长什么样、用户应该怎么理解和使用它”。

---

## 一、为什么要单独做产品原型

当前仓库里已经形成了一批架构设计文档，重点讨论的是：

1. 数据如何分层。
2. profile 如何建模。
3. research / strategy / trade 如何衔接。

这些讨论对实现非常重要，但它们天然偏工程内部视角。

如果没有产品原型层，系统很容易出现两个问题：

1. 架构上很干净，但用户根本不知道自己在操作什么对象、该先做什么动作。
2. 临时原语被误当成最终入口，例如 `data sync --as-of` 这类底层动作直接定义了运营者体验。

所以这里把设计明确拆成两条线：

1. **产品原型**：先定义产品应该是什么，再定义用户如何感知、理解和使用它。
2. **架构设计**：再回答这些产品动作在系统内部如何拆成模块、接口和流程。

这两条线不是谁替代谁，而是互相校正。

---

## 二、产品原型文档到底该写什么

产品原型文档的主线应该固定回答下面 4 类问题：

1. 用户是谁。
2. 用户围绕哪些对象工作。
3. 用户真正需要哪些动作和反馈。
4. 系统交付给用户的结果到底是什么。

这里要特别冻结一条写法原则：

1. **产品原型先写“产品应该怎样”，不先写“当前已经实现了什么”。**
2. **当前实现只能作为“实现映射”或“阶段落地现状”出现，不能反过来定义产品。**
3. **“未来会怎样”也不该成为文档主线；产品原型要直接给出稳定目标形态，而不是写成路线图。**

也就是说，产品原型不是实现清单，也不是 roadmap，而是用户视角下的产品定义。

---

## 三、参考米筐等框架，产品文档应该吸收什么

像 RiceQuant 这类量化平台，用户文档之所以更容易理解，不是因为它们工程更简单，而是因为它们先把“用户正在使用什么能力”讲清楚了。

Vortex 的产品原型建议吸收下面 4 点：

1. **先讲对象，再讲实现。** 例如先讲 snapshot、signal、portfolio、report，而不是先讲 provider、storage、目录结构。
2. **先讲动作语义，再讲底层原语。** 例如先讲 `run / compare / publish / status` 这类稳定动作，而不是先讲脚本和命令组合。
3. **先讲数据口径和可用性，再讲技术细节。** 例如先讲 PIT（Point-In-Time，时点一致）和质量门禁，再讲底层是 Parquet 还是 DuckDB。
4. **先讲交付物，再讲过程。** 用户真正关心的是最后拿到哪版 snapshot、哪份 report、哪组 target portfolio，而不是内部类名。

这也是 Vortex 后续所有产品原型文档都应该统一遵守的表达方式。

## 四、当前产品原型的核心对象

从用户视角，建议把系统收敛到下面几个对象：

1. `profile`：用户选择的一份工作模板（按域分型：DataProfile、ResearchProfile、StrategyProfile、TradeProfile）。
2. `preset`：系统内置的一揽子技术方案。
3. `pack`：更细粒度的规则包，例如质量包、PIT 包、发布包。
4. `snapshot`：某个时点可供研究、回测、执行使用的数据版本。
5. `signal snapshot`：研究层冻结下来、可供策略层直接消费的一版选股/打分结果；不是因子库，也不是最终持仓。
6. `target portfolio`：策略层编译后的目标组合，交给交易层执行。
7. `report`：一次运行的状态、结果和失败原因的统一摘要。

这里最重要的产品边界是：

1. 用户主要接触 `profile`、`snapshot`、`signal snapshot`、`target portfolio`、`report`。
2. `preset` 和 `pack` 更多是平台解释层和装配层，不应该要求普通用户日常手工维护。
3. `signal snapshot` 是 Research → Strategy 的唯一标准交付物；`target portfolio` 是 Strategy → Trade 的唯一标准交付物。

## 五、四类核心用户

### 5.1 数据运营 / 平台维护者

他关心的是：

1. 数据有没有成功更新。
2. 哪个数据集失败了。
3. 是否已经发布新的 research snapshot。
4. 失败后如何续跑。

### 5.2 研究员 / 因子开发者

他关心的是：

1. 使用哪一个 snapshot。
2. 哪些特征和标签可用。
3. 本次实验结果挂在哪个 experiment 下。

### 5.3 策略开发者

他关心的是：

1. 这次回测基于哪版数据和哪版信号。
2. 成本、滑点、调仓口径是什么。
3. 回测结果能否复现。

### 5.4 交易运营 / 执行同学

他关心的是：

1. 今天该执行哪个组合。
2. 组合来自哪个 strategy run。
3. 下单是否成功，对账是否完成。

## 六、产品入口应该长什么样

从产品原型角度，入口不应该是一堆脚本文件名，而应该是少数领域动作。

建议统一入口为：

1. `vortex data ...`
2. `vortex research ...`
3. `vortex strategy ...`
4. `vortex trade ...`
5. `vortex profile ...`

其中，数据域对运营者真正可见的动作建议固定为：

1. `bootstrap`
2. `update`
3. `backfill`
4. `repair`
5. `publish`
6. `status`

这组动作是产品原型层的稳定入口；
而 `sync --as-of` 这类命令更适合作为内部原语或调试入口。

这里还要补一条已经明确下来的产品边界：

1. **Data 域默认不是“运营者每天手工跑命令”，而是“后台服务按调度自动运行”。**
2. `vortex data ...` 这组命令首先是管理入口、调试入口和异常处理入口。
3. 研究员日常面对的不是“某人交付给我的文件”，而是系统已经发布好的数据版本。

---

## 七、产品原型的信息架构

如果未来不仅有 CLI，还会有 Web UI 或 API，那么信息架构建议固定为下面几块：

1. **数据线中心**：查看所有 data line 的状态、最近成功日期、失败任务和可发布状态。
2. **Profile 中心**：查看 profile、preset、pack 的解释与来源关系。
3. **运行中心**：查看最近 runs、阶段进度、失败单元、重试记录。
4. **Snapshot 中心**：查看当前可用 snapshots 及其来源。
5. **通知与审计中心**：查看失败告警、发布记录、对账记录。

对 Data 域来说，这 5 块信息本质上是在支撑一个“自动运行的数据后台服务”：

1. 服务自己按计划跑。
2. 人主要看状态、告警和审计。
3. 研究员直接选择可用数据版本，而不是等待人工交接。

CLI、Web、API 虽然形态不同，但都应围绕这 5 块信息组织。

## 八、产品原型与架构设计的分工

下面这条边界建议长期保持不变。

### 8.1 产品原型负责回答

1. 用户看到什么。
2. 用户做什么动作。
3. 用户需要哪些反馈。
4. 用户拿到什么交付物。

### 8.2 架构设计负责回答

1. 这些动作如何拆成模块与接口。
2. 数据如何流动。
3. 规则如何执行。
4. 如何保证可替换、可恢复、可复现。

如果后续某个设计只停留在架构图里，没有落到用户动作、用户反馈或交付物上，就应该继续补产品原型。

## 九、产品原型文档的推荐结构

为了避免每篇文档又写回“实现说明书”，建议专题文档统一采用下面的结构：

1. 背景与一句话产品定义。
2. 用户真正关心什么。
3. 用户围绕哪些对象工作。
4. 用户真正需要哪些动作。
5. 用户旅程、反馈和标准交付物。
6. 当前实现映射。

其中最后一项“当前实现映射”只负责解释：

1. 这套产品定义现在暂时由哪些原语承载。
2. 哪些地方已经有落点。
3. 哪些地方还没有形成最终入口。

它不负责重新定义产品。

## 十、冻结规则

### 10.1 命名规范

产品原型目录下的文件统一采用以下命名格式：

```
{编号}-{中文短标题}-v{版本号}.md
```

其中：

1. 编号从 `00` 开始，`00` 为总览索引页（不带版本号）。
2. `01`–`05` 为各域体验原型，统一以"体验原型"结尾。
3. `06` 为产品功能冻结文档，是全域规格的 single source of truth。
4. 版本号从 `v1` 开始，每次功能性变更递增。

### 10.2 什么算"冻结"

冻结的含义是：

1. 该版本的产品定义已经过确认，可以作为架构设计和实现的依据。
2. 后续变更必须显式递增版本号（v1 → v2），并在文档头说明变更理由。
3. 冻结后，`06-产品功能冻结文档` 是 single source of truth；各域体验原型负责展开用户旅程细节。

### 10.3 什么情况不需要升版本

1. 修正错别字、格式或断裂链接。
2. 补充"当前实现映射"章节的落地现状。
3. 调整文档内部措辞但不改变产品动作、对象或交付物定义。

### 10.4 什么情况必须升版本

1. 新增或删除核心对象。
2. 新增或删除动作。
3. 修改动作的输入、输出或失败处理规格。
4. 修改交付物 Schema 字段。
5. 修改 Profile 字段冻结表。

## 十一、服务生命周期与交互式初始化

### 11.1 设计原则

**零配置即可运行**：用户拉下工程后，唯一必须提供的外部凭证是数据源 Token（如 `TUSHARE_TOKEN`）。其余所有配置由框架提供合理默认值，并在 `vortex init` 时沉淀到 workspace 中，用户可见、可编辑。

**Provider 驱动数据集**：选择了 provider 就等于选择了它全部的数据采集能力。用户不需要了解 `bars`、`fundamental` 这些内部概念——框架默认拉取 provider 支持的全量数据集。高级用户可以通过 `exclude_datasets` 排除不需要的数据集，或通过 `priority_datasets` 控制初次拉取顺序。

### 11.2 交互式初始化流程

`vortex init` 在终端（TTY）环境下提供交互式配置向导：

```
vortex init [--root <path>]

步骤 1/4: 数据源凭证
  → 检测到 TUSHARE_TOKEN 未设置，请输入 Token: ********
  → Token 已验证 ✅（冒烟测试: trade_cal 可访问）

步骤 2/4: 数据拉取偏好
  → 历史数据起始日？ [2017-01-01]
  
步骤 3/4: 首次数据更新
  → 是否现在开始首次数据更新？ [y/N]
  → 若选“是”，再选择本次先同步的数据集：
    [x] 行情数据 (bars)         ← 最常用，建议优先
    [x] 标的列表 (instruments)   ← 体积小，秒级完成
    [x] 交易日历 (calendar)      ← 体积小，秒级完成
    [ ] 基本面数据 (fundamental)  ← 需较长时间
    [ ] 事件数据 (events)         ← 分红/停牌等
  （其余数据集后续由手动触发或自动调度补齐）

步骤 4/4: 自动调度
  → 每天自动更新数据？ [Y/n]
  → 更新时间？ [18:00]（收盘后 2 小时）
  → Cron 是 5 段格式：分 时 日 月 周
  → 常用 Cron 示例：
    - 0 18 * * 1-5  = 每个交易日 18:00
    - 0 21 * * 1-5  = 每个交易日 21:00
    - 0 6 * * *     = 每天 06:00

初始化完成
  → 工作区: ~/Documents/vortex_workspace
  → 配置已写入 profiles/default.yaml
  → 若步骤 3 选择立即更新，则立刻执行一次受限数据集的 bootstrap
```

**非交互模式**（CI / 脚本场景）：

```bash
vortex init --non-interactive               # 使用全部默认值
vortex init --non-interactive --history-start 20200101  # 覆盖默认值
```

### 11.3 各域配置 Pipeline（渐进式）

交互式配置不是一次性完成的，而是**开发到哪个域、补充哪个域的配置**：

| 域 | 触发时机 | 交互内容 |
|---|---|---|
| Data | `vortex init`（首次） | provider token、优先数据集、history_start、调度时间 |
| Research | 首次 `vortex research run` | 默认市场、标签周期、分组数 |
| Strategy | 首次 `vortex strategy backtest` | 回测基准、成本模型、换仓频率 |
| Trade | 首次 `vortex trade paper-rebalance` | 网关类型、风控规则包 |
| Notification | 首次 `vortex notify test` | 通知通道、webhook、级别过滤 |

每个域的首次使用会检测是否有对应的 profile，若不存在则触发交互式创建。创建后的 YAML 沉淀在 workspace `profiles/` 目录中，用户随时可手动编辑。

### 11.4 服务生命周期

```
vortex server start  →  运行中(accepting)  →  vortex server stop / SIGTERM
                                                     ↓
                                              排空中(draining): 停止接受新任务，等待运行中任务完成
                                                     ↓
                                              已停止(stopped): 资源释放，进程退出

调试模式下可使用 `vortex server start --foreground`，此时允许通过 `Ctrl+C` 直接退出。
```

**安全退出方式**：

| 方式 | 行为 |
|------|------|
| `vortex server stop` | 读取 PID 文件，发送 SIGTERM |
| `Ctrl+C`（SIGINT） | 等同 graceful shutdown |
| `kill <pid>`（SIGTERM） | 等同 graceful shutdown |
| `kill -9 <pid>`（SIGKILL） | 强制退出，运行中任务标记为 interrupted |

**排空超时**：默认等待 60 秒。超时后将运行中任务标记为 `cancelled`，记录断点信息以便后续 `repair`。

### 11.5 任务幂等与互斥

| 场景 | 行为 |
|------|------|
| update 正在运行，再次调用 update | 拒绝，返回"同类任务正在执行" |
| bootstrap 正在运行，调用 update | 拒绝（互斥：同 profile 的数据写入操作不可并行） |
| backfill 日期范围 A 正在运行，backfill 不同日期范围 B | 允许（不冲突） |
| dry-run 模式 | 不创建任务记录，不互斥 |
| 上次任务 failed，再次提交 | 允许 |

互斥粒度：`(domain, profile, resource_key)`，其中 resource_key 由 action + 参数决定。
