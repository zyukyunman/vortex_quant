# 交易执行与风控反馈体验原型 v1

> 本文档从交易执行 / 运营同学视角定义 Vortex 的交易体验。重点不是 Gateway 怎么实现，而是交易同学到底接收什么输入、如何判断能不能下单、执行后看什么反馈，以及风控如何以用户能理解的方式呈现出来。

---

## 一、背景

交易层是最不允许“差不多”的地方。

如果产品原型不清楚，执行层很容易退化成：

1. 手工看一份 target portfolio。
2. 手工跑一个下单脚本。
3. 手工核对成交与持仓。
4. 风控失败只在日志里提示，没人知道是否已经真正阻断。

所以交易产品原型要先明确：

交易同学接收的应该是已经冻结版本的 `target portfolio` 或 `order plan`，而不是研究中间结果；风控反馈应该是结构化结果，而不是模糊日志。

---

## 二、交易同学真正关心什么

交易同学真正关心的是：

1. 今天该执行哪一个组合。
2. 它来自哪个 strategy run。
3. 下单前是否已经通过风控。
4. 订单是否真正提交、成交、撤单或超时。
5. 收盘后是否已经完成对账。
6. 如果失败，到底是哪里失败、能不能重试。

他们通常不想关心：

1. 因子怎么计算。
2. 研究 workflow 如何训练。
3. Qlib 或 Backtrader 内部如何组织。

---

## 三、交易层对用户暴露的核心对象

建议交易同学主要围绕下面 7 个对象工作：

1. `TradeProfile`：交易模板，定义账户、执行模式、通知和默认规则包。
2. `target portfolio`：上游策略层交付的目标组合。
3. `order plan`：执行前冻结的标准订单计划。
4. `execution run`：一次完整执行记录。
5. `execution report`：订单提交、成交、撤单和异常的统一摘要。
6. `reconcile report`：日终对账结果。
7. `risk status`：事前、事中、事后风控反馈摘要。

这里最重要的产品边界是：

1. 交易层输入必须是 `target portfolio` 或 `order plan`。
2. 风控反馈必须以结构化方式回到交易层，而不是藏在内部日志。

---

## 四、交易同学真正需要的动作

建议交易域先固定下面 6 个动作：

1. `paper rebalance`：模拟盘调仓。
2. `live rebalance`：实盘调仓。
3. `inspect`：查看本次执行的计划、成交和异常。
4. `retry`：对失败单元发起重试。
5. `reconcile`：日终对账。
6. `status`：查看账户、最近执行状态和当前阻断项。

这 6 个动作能覆盖交易同学的主要日常工作。

---

## 五、建议的 CLI 语义

建议交易同学看到的入口像这样：

```bash
vortex trade paper rebalance --profile dividend_live_cn --target tp_20260403_a
vortex trade live rebalance --profile dividend_live_cn --target tp_20260403_a
vortex trade inspect --run-id exec_20260404_091501
vortex trade retry --run-id exec_20260404_091501
vortex trade reconcile --profile dividend_live_cn --date 2026-04-04
vortex trade status --profile dividend_live_cn
```

这里的关键不是命令格式，而是交互语义：

1. 交易入口消费的是冻结后的交付对象。
2. 对账、重试和状态必须是标准动作，而不是事后补脚本。

---

## 六、一条交易用户旅程应该怎么走

### 6.1 接收 target portfolio

交易同学先拿到：

1. 今日目标组合
2. 它对应的 strategy run
3. 它对应的数据 snapshot 和 signal snapshot

这一步的目标是确认来源可追溯。

### 6.2 执行前风控校验

系统在真正下单前，应先给出结构化风控结论：

1. 是否通过。
2. 如果未通过，阻断原因是什么。
3. 哪些规则命中。

交易同学不应该自己读日志猜测有没有被阻断。

### 6.3 生成并提交 order plan

系统再把 target portfolio 转成标准 `order plan`，然后提交到模拟盘或实盘网关。

### 6.4 查看执行反馈

交易同学需要看到：

1. 哪些订单已提交。
2. 哪些已成交。
3. 哪些超时或被拒单。
4. 哪些失败可以重试。

### 6.5 日终对账与风险反馈

最后，系统应自动或半自动返回：

1. 对账结果
2. 持仓偏差
3. 成交差异
4. 风控事件摘要

---

## 七、交易同学需要看到什么反馈

一次交易动作至少应返回：

1. `status`
2. `run_id`
3. `profile`
4. `target_portfolio`
5. `order_plan`
6. `pre_trade_result`
7. `artifact_manifest`
8. `blocking_reason`（如果失败）

如果进入执行阶段，还应继续可见：

1. 已提交订单数
2. 已成交订单数
3. 拒单 / 超时订单数
4. 是否需要 retry
5. 是否已完成 reconcile

---

## 八、交易层必须产出哪些标准工件

建议交易层默认产出并展示：

1. `order_plan.json`
2. `execution_report.json`
3. `reconcile_report.json`
4. `daily_risk_snapshot.json`

原因是：

1. 交易域不只需要“执行结果”，还需要“事前是否可下单、日终是否对账完成”的证据链。
2. 风控输出必须进入交易视角，而不是只存在于风控子系统内部。

---

## 九、当前实现映射

### 9.1 当前已有映射

当前仓库里已经有：

1. 回测、执行、风控相关基础代码
2. miniQMT / 对账方向的基础设计

### 9.2 仍待形成的产品入口

目前还没有完整形成下面这些稳定入口：

1. 统一的 `TradeProfile` 用户视图
2. 标准 `target portfolio -> order plan -> execution report -> reconcile report` 链路
3. 结构化风控反馈在交易域的展示方式
4. `retry / reconcile / status` 作为稳定标准动作

这部分只是在说明落地映射：交易域方向已经明确，但产品入口、反馈面和标准工件链路还没有完全收敛。
