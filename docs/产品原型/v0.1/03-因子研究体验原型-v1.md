# 因子研究体验原型 v1

> **⚠️ 本文档内容已合并到 [[用户场景与体验增强设计|07-用户场景与体验增强设计]] §二（因子评测场景）。07-v2 在本文基础上新增了因子注册、截面分析、HTML 报告、AI 辅助、评测去重、数据预处理等完整设计。本文件保留作为 v1 历史参考，后续不再更新。**

> 本文档从研究员 / 因子开发者视角定义 Vortex 的研究体验。重点不是 Qlib 内部类怎么写，而是研究员到底如何开始一次研究、如何判断结果是否可信、以及如何把结果稳定交给下游。

---

## 一、背景

研究域最容易出现的坏味道有两种：

1. 研究员直接连原始数据或随手改路径，导致结果不可复现。
2. 一次实验跑完只有 notebook 输出，没有稳定的实验记录、指标快照和 signal 产物。

所以研究产品原型必须先回答一件事：

研究员不是在“运行脚本”，而是在“基于某个 snapshot 运行一份可复现 workflow，并决定是否生成可下游消费的 signal snapshot”。

### 与 Data 域衍生指标的边界

Research 域做的是“因子分析”——研究员主动发起一次实验，评测因子的预测能力（IC、多空收益等），然后判断是否值得发布 signal。

这和 Data 域自动产出的“衍生分析指标”（估值分位数、市场宽度、资金面等）是两件事：

1. **Data 域衍生指标**：每天自动计算、不需要研究员操作、是数据快照的一部分。回答“市场现在是什么状态”。
2. **Research 域因子分析**：研究员主动触发、需要配置 workflow 和评估口径、产出 experiment 和 signal snapshot。回答“这个因子能不能赚钱”。

Data 域产出的衍生指标可以直接作为 Research 域的输入特征使用。

## 二、研究员真正关心什么

研究员真正想确认的是：

1. 我这次研究到底用的是哪一版 snapshot。
2. 这次 workflow 的参数和口径是什么。
3. IC、RankIC、ICIR、分组收益和多空收益到底怎么样。
4. 这次结果和上次相比是进步还是退步。
5. 这次实验能不能沉淀成稳定的 signal snapshot 给策略层使用。

也就是说，研究员不应该把精力花在：

1. 自己拼 Provider 路径。
2. 自己判断数据是否已经发布完成。
3. 自己手工整理实验参数和产物。

---

## 三、研究域对用户暴露的核心对象

建议研究员主要围绕下面 6 个对象工作：

1. `ResearchProfile`：研究模板，定义 workflow 类型、默认标签、评估口径和产物策略。
2. `snapshot`：本次研究使用的数据版本。
3. `workflow`：这次运行的研究流程定义，例如因子评测、模型训练、信号生成。
4. `experiment`：一次完整运行记录。
5. `signal snapshot`：研究层冻结下来、可供策略层直接消费的一版选股/打分结果。
6. `research report`：把结果、指标、产物位置和失败原因统一收敛的摘要对象。

这里的产品边界非常关键：

1. 研究员选择的是 `ResearchProfile + snapshot`。
2. 系统内部再去装配 Qlib handler、workflow config、recorder 和 artifact 路径。
3. `signal snapshot` 不是因子库，也不是最终持仓；它更像“某一天正式发布的研究信号版本”，里面是给策略层继续处理的股票评分、排序或入选结果。

---

## 四、研究员真正需要的动作

建议研究域先固定下面 5 个主动作：

1. `run`：运行一次研究 workflow。
2. `inspect`：查看某次 experiment 的结果、参数和产物。
3. `compare`：比较两次实验结果。
4. `publish-signal`：把某次实验结果固化为 signal snapshot。
5. `status`：查看最近实验状态和当前可用 signal snapshot。

这 5 个动作比“执行某个脚本”更接近真正的产品入口。

---

## 五、建议的 CLI 语义

建议研究员看到的入口像这样：

```bash
vortex research run --profile cn_alpha_basic --snapshot 2026-04-03
vortex research inspect --run-id research_20260403_220501
vortex research compare --left research_20260401_220212 --right research_20260403_220501
vortex research publish-signal --run-id research_20260403_220501
vortex research status --profile cn_alpha_basic
```

这里真正稳定的不是命令名本身，而是动作语义：

1. `run` 负责生产 experiment。
2. `inspect` 负责解释 experiment。
3. `compare` 负责判断是否值得进入下一步。
4. `publish-signal` 负责把研究结果冻结成策略可消费的一版信号结果。

## 六、一条研究用户旅程应该怎么走

### 6.1 选择 snapshot

研究员先决定：

1. 今天用哪版 snapshot。
2. 这版 snapshot 是否已经通过数据域发布。

研究员不应该自己找物理路径。

### 6.2 选择 ResearchProfile

研究员再选择：

1. 跑因子评测
2. 跑模型训练
3. 跑预测更新
4. 跑信号生成

这一步是“选择研究模板”，不是“手写一大段 workflow 参数”。

### 6.3 运行并记录 experiment

系统应自动记录：

1. `run_id`
2. 使用的 snapshot
3. profile 展开后的配置快照
4. 指标结果
5. 产物位置

### 6.4 比较并决定是否发布 signal

研究员真正要做的决策是：

1. 这次实验是否优于现有版本。
2. 是否值得发布新的 signal snapshot。

所以产品上必须把“比较”作为一等动作，而不是让研究员自己去翻多个目录。

---

## 七、研究员需要看到什么反馈

一次 `research run` 结束后，系统至少应返回：

1. `status`
2. `run_id`
3. `profile`
4. `snapshot`
5. `workflow_type`
6. `artifact_manifest`
7. `key_metrics`
8. `blocking_reason`（如果失败）

其中 `key_metrics` 建议至少包含：

1. IC
2. RankIC
3. ICIR
4. 分组收益
5. 多空收益

原因很直接：这是研究员判断一次实验是否值得继续的第一屏信息。

---

## 八、研究域必须产出哪些标准工件

建议研究域默认产出并展示：

1. `pred.pkl`
2. `ic_metrics.csv`
3. `signal_analysis.csv`
4. `workflow_config_snapshot.yaml`

这样做的意义是：

1. 研究结果不再只是终端日志。
2. 策略层接收到的是稳定工件，而不是研究员临时导出的文件。
3. 后续比较、回溯和上线都有依据。

当前仓库里已经有：

1. 因子评测相关脚本与报告产物

这部分只是在说明落地映射：研究域的产品动作已经定义出来，但当前实现还没有完全收敛成统一入口和统一反馈面。
