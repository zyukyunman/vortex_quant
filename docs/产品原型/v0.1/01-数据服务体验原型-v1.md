# 数据服务体验原型 v1

> **⚠️ 本文档内容已合并到 [[用户场景与体验增强设计|07-用户场景与体验增强设计]] §一（数据监控场景）。07-v2 在本文基础上新增了自动重试策略、通知分级、零操作体验等完整设计。本文件保留作为 v1 历史参考，后续不再更新。**

> 本文档面向“数据运营 / 平台维护者”角色，目标是让你在不理解内部类和模块的前提下，直接开始使用 Vortex 的数据下载功能。

> 说明：本文档同时区分“当前已实现功能”和“下一阶段目标功能设计”，避免把临时原语误读成最终运营入口。

> 本轮讨论补充：Data 功能的默认产品形态不是“运营者每天手工执行下载命令”，而是“后台服务按日自动运行；人在异常、建库和审计时介入”。

---

## 〇、产品定位补充（本轮确认）

先用最直白的话把 Data 域讲清楚。

它首先不是一个“下载脚本集合”，而是一个**后台自动运行的数据服务**。

这条服务每天要自动完成 4 件事：

1. 把该更新的数据更新进来。
2. 把脏数据挡在发布之前。
3. 产出一版研究可直接使用的数据版本。
4. 把今天发生了什么记录清楚。

所以对不同角色，它的意义不一样：

1. 对数据平台来说，它是一个要稳定运行、可追溯、可告警的数据后台。
2. 对研究员来说，它是一个“每天都会自动准备好可用数据版本”的底座。

这里还要澄清“数据落到数据库里面”这句话在当前设计里的含义：

1. 从产品视角，可以把它理解成一套统一的数据底座，也就是“数据库”。
2. 第一版的底层实现是 `Parquet + DuckDB`，重点不是让研究员理解文件结构，而是让系统对外表现成“有统一口径、可查询、可追溯”的数据存储。

也就是说：

1. 运营者不应该每天盯着运行命令。
2. 研究员不应该自己找 Parquet 路径。
3. 大家真正关心的是“今天这版数据有没有自动准备好，以及它能不能放心用”。

## 一、当前可用范围

当前版本已经实现最小可用闭环：

1. 通过 `profile` 选择数据下载配置。
2. 通过命令行触发 Tushare 数据下载。
3. 自动把结果落为 Parquet。
4. 自动生成 `sync_manifest.json`。
5. 自动生成 `data_health_report.json` 作为质量门禁结果。

支持的数据集：见tushare的skill得到

### 1.1 当前实现与目标体验要区分开

它们已经足够验证 profile、下载、质量报告和落盘链路。

但这不等于最终运营体验。

从运营者视角，最终不应该是“手工指定某一天然后执行 sync”，而应该是“选择一条数据线，然后执行完整动作”。

### 1.2 数据服务需要具备的标准动作

建议把 Data 域的标准管理动作固定为下面 6 个动作：

1. `bootstrap`：第一次建库，从历史起点一次性补齐到最新可用交易日。
2. `update`：日常增量更新，从上次成功日期自动跑到最新可用交易日。
3. `backfill`：指定缺口区间补数据。
4. `repair`：只重试失败单元，而不是重跑全部。
5. `publish`：同步完成后执行质量门禁、PIT 对齐、研究视图构建和 snapshot 发布。
6. `status`：查看最近成功日期、最新失败任务、当前可用 snapshot 和整体健康状态。

但这里要强调：

1. **这 6 个动作是系统必须具备的标准能力，不代表需要人每天手工逐个执行。**
2. 日常应该由后台调度自动串起 `update -> quality -> pit -> publish`。
3. 人主要会用到 `status`、`repair`、`backfill`，以及首次建库时的 `bootstrap`。

如果这 6 个动作不先冻结，后面再继续加功能，系统很容易重新退化成“一堆脚本和子命令”。

### 1.3 获取数据不应该按天手工运行

运营者不应该每天手动写一个 `--as-of YYYY-MM-DD`。

更合理的方式是：

1. 第一次运行 `bootstrap`，一次性把历史数据补齐。
2. 之后日常运行 `update`，系统自己决定从哪天增量到哪天。
3. 如果中间某段有缺口，再通过 `backfill` 或 `repair` 处理。

也就是说，`data sync --as-of` 更适合作为：

1. 当前阶段的最小调试原语
2. 平台内部底层能力

而不应该是最终运营主入口。

### 1.4 不应该依赖外围脚本补齐日常使用体验

你不应该在外围再写 shell 脚本来循环跑每天的数据下载。

正确的产品边界应该是：

1. 系统内部提供 `bootstrap/update/backfill/repair/publish/status` 这组标准动作。
2. 调度系统在日常情况下自动调用它们。
3. 运营者主要看状态、告警、报告，并在异常时介入。
4. 研究员直接消费系统已发布的数据版本，而不是等待人工交付文件。

### 1.5 进度、重试、续跑是标准能力，不是后补功能

对数据链路来说，进度、重试和续跑不是锦上添花，而是产品本体的一部分。

建议一开始就按下面几层进度设计：

1. 运行级进度：当前是在 `sync / quality / pit / publish` 哪个阶段。
2. 数据集级进度：当前跑到哪个 dataset。
3. chunk 级进度：对于财务数据，当前完成了多少 `ts_code`、失败了多少、还剩多少。

同时要内建：

1. 网络失败自动重试。
2. 失败单元清单落盘。
3. `repair` 从失败单元续跑，而不是全部重来。

### 1.6 研究员如何使用数据层

从研究员角度，Data 域不应该表现成“某个同学把数据给你了”，而应该表现成“系统里已经有一版可用数据”。

研究员真正需要的是：

1. 看见当前有哪些可用数据版本。
2. 知道每一版数据覆盖哪些市场、频率和数据集。
3. 知道这版数据是否通过质量检查、是否可以直接做研究。
4. 直接通过统一接口读取，而不是自己翻目录找文件。

### 1.7 数据域也负责衍生分析指标的自动生产

数据域不仅负责“原始行情和财务数据的同步落盘”，还负责“在数据更新后自动计算一批衍生分析指标”。

典型的衍生分析指标包括：

1. 估值分析：全市场 PE、PB、PS、股息率的分位数、行业分布、历史百分位。
2. 市场宽度：涨跌家数、创新高/新低比例。
3. 资金面：北向资金净流入排名、融资融券余额变化。

这些指标的特点是：

1. 输入是当天已更新的原始数据，不依赖研究员的实验参数。
2. 产出是结构化的衍生指标，可以直接被研究层当作特征或因子使用。
3. 应该作为 `publish` 流程的一部分自动运行，而不是由研究员手工触发。

也就是说，衍生分析指标是“数据服务自动产出的一部分”，不是“研究实验的产物”。

这里要和 Research 域的因子分析做一个明确区分：

1. **Data 域的衍生指标**：自动产出、每天更新、是数据底座的延伸。研究员直接当特征用，不需要跑实验就能获得。
2. **Research 域的因子分析**：研究员主动发起的实验流程。需要选择 snapshot、配置 workflow、产出 experiment，然后决定是否发布 signal snapshot。

简单说：Data 域回答“市场现在是什么状态”，Research 域回答“这个因子能不能赚钱”。

## 二、前置条件

### 2.1 Python 环境

当前命令基于系统 Python 3.9 运行。

### 2.2 Tushare Token

需要在项目根目录的 `.env` 文件或环境变量中配置：

```env
TUSHARE_TOKEN=你的_token
```

说明：

1. token 不需要写进 profile。
2. profile 只负责业务配置，不负责保存密钥。

## 三、当前内置 profile

当前已经提供：

1. `profiles/data/cn_stock_base.yaml`
2. `profiles/data/cn_stock_daily.yaml`
3. `profiles/data/cn_stock_daily_extended.yaml`

其中：

1. `cn_stock_base` 是基础模板。
2. `cn_stock_daily` 是运营者可直接使用的日频下载配置。
3. `cn_stock_daily_extended` 是扩展下载配置，增加了资金流、涨跌停和财务披露数据。

这里也要补一句：

1. 当前这几份 profile 仍然偏平台模板。
2. 从运营者角度，下一阶段应继续收敛成“选择 preset + 少量业务字段”的最小写法。

## 四、先看 profile 解析结果

建议第一次使用前，先运行：

```bash
/usr/bin/python3 -m vortex profile resolve --profile cn_stock_daily --type data
```

这个命令会告诉你：

1. 最终使用了哪份 profile。
2. 继承链展开后的字段是什么。
3. 数据会写到哪里。

这个动作是只读的，不会下载数据。

---

## 五、执行数据下载

执行命令：

```bash
/usr/bin/python3 -m vortex data sync --profile cn_stock_daily --as-of 2026-04-04
```

说明：

1. `--profile` 指定使用哪份 DataProfile。
2. `--as-of` 是你希望同步的请求日期。
3. 如果 `--as-of` 不是交易日，系统会自动回退到最近一个开市日。

这里要特别说明：

1. 这是当前已实现的最小原语。
2. 它不应该被视为最终运营主入口。
3. 最终运营入口更应该是 `bootstrap`、`update`、`backfill`、`repair`、`publish`、`status`。

例如在当前实现里：

1. 请求日期 `2026-04-04`
2. 实际解析交易日为 `2026-04-03`

## 六、命令执行后会产出什么

### 6.1 数据文件

数据会写入：

```text
data/authoritative/cn_stock_daily/
```

按数据集和 `as_of` 分区保存为 Parquet。

### 6.2 同步清单

同步清单会写入：

```text
data/manifests/cn_stock_daily/as_of=YYYY-MM-DD/sync_manifest.json
```

这份 manifest 会记录：

1. `run_id`
2. `profile`
3. `requested_as_of`
4. `resolved_trade_date`
5. 下载了哪些数据集
6. 每个数据集的行数和文件路径

### 6.3 质量报告

质量报告会写入：

```text
data/manifests/cn_stock_daily/as_of=YYYY-MM-DD/data_health_report.json
```

这份报告会记录：

1. 每个数据集执行了哪些质量规则
2. 每条规则是 `passed`、`failed` 还是 `skipped`
3. 如果失败，具体失败原因是什么

### 6.4 DuckDB catalog

当前也会写一个本地 DuckDB catalog，用来记录同步日志，位置在：

```text
data/manifests/cn_stock_daily/catalog.duckdb
```

## 七、运行时小覆盖

如果你只是临时改一点配置，不需要复制整份 profile，可以使用：

```bash
/usr/bin/python3 -m vortex data sync --profile cn_stock_daily --set datasets=[bar_1d,daily_basic]
```

这表示：

1. 仍然使用 `cn_stock_daily`
2. 但这次只临时同步 `bar_1d` 和 `daily_basic`

这样做不会修改原 profile 文件。

## 八、当前限制

为了优先让运营者跑通最小闭环，当前版本刻意收窄了范围。

当前限制包括：

1. 只实现了 `data sync` 和 `profile resolve` 两个最小命令。
2. 只支持 Tushare。
3. 当前完整质量门禁只覆盖最小运营规则，还没有覆盖更细的财务口径校验与 PIT 对齐。
4. 财务与披露数据采用逐股票串行拉取，在低权限 Tushare 下更稳，但速度会比日频行情慢很多。
5. 还没有实现 `research`、`strategy`、`trade` 三类 profile 的实际运行链路。
6. 还没有实现面向运营者的一次性建库、自动增量、失败续跑和 snapshot 发布主入口。

这些限制是当前阶段的有意取舍，不是最终目标。

同时也说明一件事：

1. 当前实现还偏“手工触发的最小原语”。
2. 但产品方向已经明确收敛为“后台自动服务 + 人工异常处理 + 研究员自助消费”。

## 九、下一阶段要冻结的功能设计

为了避免“想到哪做到哪”，下一阶段建议先冻结这 3 张设计表，再继续实现：

1. 运营者 Data 动作表：`bootstrap/update/backfill/repair/publish/status` 的输入、输出、失败处理和成功条件。
2. 运营者可见 DataProfile 字段表：哪些字段用户能看见、哪些字段常改、哪些字段必须下沉到 preset。
3. preset / pack 分层表：哪些属于用户 profile，哪些属于内部 preset，哪些属于 quality/pit/publish/storage pack。

## 十、建议的实际使用顺序

如果你是第一次使用，建议按下面顺序：

1. 检查 `.env` 是否配置了 `TUSHARE_TOKEN`
2. 运行 `profile resolve` 看最终解析结果
3. 运行 `data sync` 下载当天数据
4. 打开 `sync_manifest.json` 检查是否全部成功
5. 再继续扩展更多数据集或更多 profile
