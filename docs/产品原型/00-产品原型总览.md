# 产品原型总览

> 本目录回答的不是“系统内部怎么实现”，而是“用户到底如何感知、理解和使用这套系统”。

---

## 一、为什么要单独做产品原型

当前仓库里已经形成了一批架构设计文档，重点讨论的是：

1. 数据如何分层
2. profile 如何建模
3. research / strategy / trade 如何衔接

这些讨论对于实现非常重要，但它们天然偏工程内部视角。

如果没有产品原型层，系统很容易出现两个问题：

1. 架构上很干净，但用户根本不知道应该点哪里、填什么、先做什么。
2. 内部原语被误当成最终入口，例如 `data sync --as-of` 这类底层动作直接暴露给运营者。

所以这里把设计明确拆成两条线：

1. **产品原型**：从用户角色、动作、对象、反馈和使用体验出发。
2. **架构设计**：从模块边界、数据流、接口和实现约束出发。

这两条线不是谁替代谁，而是互相校正。

---

## 二、产品原型的目标

产品原型要回答下面这些问题：

1. 用户是谁。
2. 用户真正想完成什么动作。
3. 系统应该暴露哪些对象给用户。
4. 用户应该看到什么反馈、进度和结果。
5. 哪些能力是当前已实现，哪些还是目标设计。

如果一个设计结论只能解释“内部怎么做”，不能解释“用户怎么用”，那它还不算完整。

---

## 三、当前产品原型的核心对象

从用户视角，建议把系统收敛到下面几个对象：

1. `data line`：一条数据线，描述“这批数据怎么被生产出来”。
2. `profile`：用户选择的一份工作模板。
3. `preset`：系统内置的一揽子技术方案。
4. `pack`：更细粒度的规则包，例如质量包、PIT 包、发布包。
5. `snapshot`：某个时点可供研究、回测、执行使用的数据版本。
6. `run`：一次具体执行记录。
7. `report`：一次运行的状态、结果和失败原因。

这里最重要的产品边界是：

1. 用户主要接触 `profile`、`snapshot`、`run`、`report`。
2. `preset` 和 `pack` 更多是平台解释层和装配层，不应该直接要求普通运营者日常手工维护。

## 四、四类核心用户

### 4.1 数据运营 / 平台维护者

他关心的是：

1. 数据有没有成功更新。
2. 哪个数据集失败了。
3. 是否已经发布新的 research snapshot。
4. 失败后如何续跑。

### 4.2 研究员 / 因子开发者

他关心的是：

1. 使用哪一个 snapshot。
2. 哪些特征和标签可用。
3. 本次实验结果挂在哪个 experiment 下。

### 4.3 策略开发者

他关心的是：

1. 这次回测基于哪版数据和哪版信号。
2. 成本、滑点、调仓口径是什么。
3. 回测结果能否复现。

### 4.4 交易运营 / 执行同学

他关心的是：

1. 今天该执行哪个组合。
2. 组合来自哪个 strategy run。
3. 下单是否成功，对账是否完成。

## 五、产品入口应该长什么样

从产品原型角度，入口不应该是一堆脚本文件名，而应该是少数领域动作。

建议统一入口为：

1. `vortex data ...`
2. `vortex research ...`
3. `vortex strategy ...`
4. `vortex trade ...`
5. `vortex profile ...`

其中，数据域对运营者真正可见的动作建议固定为：

1. `bootstrap`
2. `update`
3. `backfill`
4. `repair`
5. `publish`
6. `status`

这组动作是产品原型层的最终入口；
而 `sync --as-of` 这类命令更适合作为内部原语或调试入口。

## 六、产品原型的信息架构

如果未来不仅有 CLI，还会有 Web UI 或 API，那么信息架构建议固定为下面几块：

1. **数据线中心**：查看所有 data line 的状态、最近成功日期、失败任务和可发布状态。
2. **Profile 中心**：查看 profile、preset、pack 的解释与来源关系。
3. **运行中心**：查看最近 runs、阶段进度、失败单元、重试记录。
4. **Snapshot 中心**：查看当前可用 snapshots 及其来源。
5. **通知与审计中心**：查看失败告警、发布记录、对账记录。

CLI、Web、API 虽然形态不同，但都应围绕这 5 块信息组织。

---

## 七、产品原型与架构设计的分工

下面这条边界建议长期保持不变。

### 7.1 产品原型负责回答

1. 用户看到什么。
2. 用户做什么动作。
3. 用户需要哪些反馈。
4. 用户要不要理解某些术语和参数。

### 7.2 架构设计负责回答

1. 这些动作如何拆成模块与接口。
2. 数据如何流动。
3. 规则如何执行。
4. 如何保证可替换、可恢复、可复现。

如果后续某个设计只停留在架构图里，没有落到用户动作或用户反馈上，就应该继续补产品原型。
