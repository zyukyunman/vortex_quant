## 1. 环境准备

### 1.1 系统要求

| 项目 | 要求 |
|------|------|
| Python | 3.11+ |
| 操作系统 | macOS / Linux |
| 包管理 | [uv](https://github.com/astral-sh/uv)（推荐）或 pip |
| 磁盘空间 | ≥ 2GB（全量 A 股日线约 500MB Parquet） |

### 1.2 安装 uv（如未安装）

```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
```

### 1.3 Tushare Token（数据源）

数据采集功能需要 [Tushare Pro](https://tushare.pro) 账号和 Token。

Token 可通过以下方式配置（`vortex init` 交互流程会自动引导）：

```bash
# 方式 1：环境变量（推荐写入 ~/.zshrc 或 ~/.bashrc）
export TUSHARE_TOKEN=你的token

# 可选：声明当前账号积分档位（当前手册默认按 5000 积分建模）
export TUSHARE_POINTS=5000

# 可选：声明已开通的独立权限，逗号分隔
# 例如：news,announcements,research_report,policy,investor_qa,hk_daily,us_daily
export TUSHARE_EXTRA_PERMISSIONS=news,announcements

# 方式 2：vortex init 时交互输入（会自动写入工作区 .env 文件）
```

> **注意**：
> 1. 如果只做框架功能验证（不实际拉取数据），可以跳过 Token 配置。
> 2. 当前代码默认按 **5000 积分档位**控制普通积分接口频次；若你的账号档位不同，请显式设置 `TUSHARE_POINTS`。
> 3. 新闻、公告、港股/美股日线、投资者问答等属于 **独立权限接口**，只有在 `TUSHARE_EXTRA_PERMISSIONS` 中声明后，默认数据集才会纳入这些接口。

## 3. 工作区初始化

Vortex 的所有运行时状态（配置、数据、研究产物）都保存在一个**工作区目录**中。

### 3.1 交互式初始化（默认）

在终端中运行 `vortex init` 会启动 4 步交互向导：

```bash
vortex init
```

向导会依次询问：

1. **TUSHARE_TOKEN**：检测环境变量 → 未检测到则提示输入 → 冒烟测试验证可用性
2. **历史数据起始日**：默认 20170101，可自定义（格式 YYYYMMDD）
3. **首次数据更新**：先问“是否现在开始首次数据更新”。默认否，此时只写入配置，后续由你手动触发或由自动调度更新全量数据。若选择“是”，再选择本次先同步的数据集。  
   数据集选择方式是键盘交互式多选：
   - `↑ / ↓`：移动光标
   - `空格`：勾选或取消当前项
   - `a`：一键全选
   - `n`：一键全不选
   - `d`：恢复默认推荐项
   - `回车`：确认当前选择
   - 列表较长时会自动滚动显示，不会一次性把全部数据集刷满整个终端
4. **自动更新计划**：可选启用每日定时更新，提供常用选项：
   - `1` 每个交易日 18:00（推荐，收盘后）
   - `2` 每个交易日 21:00
   - `3` 每天 06:00（含非交易日）
   - `4` 自定义 Cron 表达式  
   Cron 是 5 段格式：`分 时 日 月 周`，其中 `*` 表示“每”，`1-5` 表示周一到周五。  
   常用 Cron 示例：
   - `0 18 * * 1-5` = 每个交易日 18:00
   - `0 21 * * 1-5` = 每个交易日 21:00
   - `0 6 * * *` = 每天 06:00

完成后自动生成 `profiles/default.yaml` 并写入工作区。
这个文件只沉淀用户真正输入的字段；provider 默认数据集、质量/PIT/发布/存储规则由框架默认值在解析阶段补齐，不会把一堆内部默认项直接展开进 YAML。

> **取消语义**：在 init 任意步骤按 `Ctrl+C`，会直接整体取消本次初始化；若本次初始化新建了工作区，会一并回滚，不留下半成品目录、配置或 bootstrap 中间状态。

### 3.2 非交互模式（CI/脚本）

```bash
vortex init --non-interactive
# 或通过管道输入（自动检测非 TTY 环境）
echo | vortex init
```

非交互模式使用框架默认值，适合 CI/CD 或脚本自动化。

### 3.3 指定自定义工作区路径

```bash
vortex init --root ~/my_workspace
```

### 3.3 工作区目录结构

初始化后的工作区布局：

```
~/Documents/vortex_workspace/
├── profiles/           # Profile 配置文件（YAML）
│   └── default.yaml    # 自动创建的默认 data profile
├── state/              # 控制面状态（SQLite 数据库）
│   └── control.db      # 全局控制数据库
├── resolved/           # 解析后的 profile 快照
├── data/               # 数据面存储（Parquet + DuckDB）
├── research/           # 研究域产物
├── strategy/           # 策略域产物
└── trade/              # 交易域产物
```

> **设计原则**：控制面（`state/`）使用 SQLite，数据面（`data/`）使用 Parquet + DuckDB，二者严格分离。

### 3.4 验证初始化

```bash

# 期望: data  profiles  research  resolved  state  strategy  trade

# 检查默认 profile 是否存在

```

**✅ 验证通过标准**：7 个子目录存在，`default.yaml` 包含 `name: default`、`provider: tushare` 等字段。

## 4. Profile 配置管理

Profile 是 Vortex 的配置单元。
但对 Data 域来说，**日常 `vortex data ...` 默认直接使用工作区的 `profiles/default.yaml`**；只有研究/策略/交易域，或少数高级数据裁剪场景，才需要显式管理多个 profile。

### 4.1 核心概念

| 概念 | 说明 |
|------|------|
| **DataProfile** | 数据域默认配置：通常只关心 provider、history_start，以及少量高级裁剪字段 |
| **继承（extends）** | 子 profile 可继承父 profile 的配置，支持多层继承 |
| **默认值** | 未显式配置的字段自动使用框架默认值 |
| **Override** | CLI 级别可临时覆盖任意字段 |

合并优先级（从低到高）：`框架默认值 < 最远祖先 < ... < 直接父级 < 用户配置 < CLI override`

### 4.2 查看默认 profile 的解析结果

```bash
vortex profile resolve --name default --type data
```

输出示例（JSON 格式，每个字段标注来源）：

```json
{
  "name": {"value": "default", "source": "user"},
  "provider": {"value": "tushare", "source": "default"},
  "datasets": {"value": ["instruments", "calendar", "bars", "fundamental", "events", "valuation", "..."], "source": "default"},
  "exclude_datasets": {"value": [], "source": "default"},
  "priority_datasets": {"value": [], "source": "default"},
  "history_start": {"value": "20170101", "source": "default"}
}
```

### 4.3 查看 profile 的人类可读解释

```bash
vortex profile explain --name default --type data
```

### 4.4 创建自定义 profile

在工作区 `profiles/` 目录下创建 YAML 文件：

```bash
cat > ~/Documents/vortex_workspace/profiles/cn_stock.yaml << 'EOF'
name: cn_stock
provider: tushare
# datasets 默认已展开为 provider 内置的全量 dataset 列表，
# 通常不需要手写；如需裁剪，请用 exclude_datasets
# 如需跳过某些数据集，使用 exclude_datasets:
exclude_datasets:
  - events
history_start: "20200101"
priority_datasets:
  - bars
  - fundamental
EOF
```

验证：

```bash
vortex profile resolve --name cn_stock --type data
```

### 4.5 使用继承

```bash
# 创建子 profile，继承 cn_stock 并覆盖 history_start
cat > ~/Documents/vortex_workspace/profiles/cn_stock_recent.yaml << 'EOF'
name: cn_stock_recent
extends: cn_stock
history_start: "20240101"
EOF

# 查看继承解析结果
vortex profile explain --name cn_stock_recent --type data
# provider 应显示来源 [parent:cn_stock]，history_start 显示来源 [user]
```

**✅ 验证通过标准**：`explain` 正确标注每个字段的来源（default / parent:xxx / user）。

## 5. 数据管理

数据管理覆盖完整的数据生命周期：采集 → 存储 → 质量校验 → PIT 对齐 → 发布。

### 5.1 命令一览

| 命令 | 用途 | 关键参数 |
|------|------|----------|
| `data bootstrap` | 全量初始化 | `--dry-run` |
| `data update` | 增量更新 | `--dry-run` |
| `data backfill` | 补充指定日期范围 | `--start` `--end` |
| `data repair` | 修复失败分区 | `--start` `--end` |
| `data publish` | 发布数据快照 | `--as-of` |
| `data status` | 查看同步状态与进度 | `--watch` |
| `data logs` | 查看后台任务日志 | `--task-id` `--follow` |
| `data cancel` | 取消后台任务 | `--task-id` |

> **运行形态说明**：除 `status`、`logs`、`cancel` 和显式 `--dry-run` 外，Data 写命令默认都会**提交后台任务**并立即返回；如需在当前终端直跑，可追加 `--foreground`。

### 5.2 查看同步状态与进度

```bash
vortex data status

# 持续刷新，直到没有活跃任务
vortex data status --watch
```

首次运行时输出 `ℹ️  无同步记录`，这是正常的。
Data CLI 默认读取当前工作区的 `profiles/default.yaml`，不需要额外传 `--profile`。
若已有后台任务在跑，`status` 会同时展示：

- 活跃任务：`task_id / run_id / pid / log_path`
- 当前阶段：`prepare / fetch / write / quality / publish`
- 当前进度：当前 dataset、dataset 总数、当前 chunk、chunk 总数、累计 `written_rows`
- 最近一次同步结果：用于确认最近成功/失败 run 的落库摘要

### 5.3 查看后台任务日志

```bash
# 默认取唯一活跃任务；没有活跃任务时取最近任务
vortex data logs

# 持续跟随日志输出
vortex data logs --follow

# 指定任务
vortex data logs --task-id <task_id> --follow
```

官方入口优先于手动 `tail -f`，因为它先按控制面任务定位日志，再决定跟哪一个文件。

### 5.4 取消后台任务

```bash
# 取消唯一活跃任务
vortex data cancel

# 或显式指定任务
vortex data cancel --task-id <task_id>
```

取消动作会同时做两件事：

1. 在控制面 `task_queue` 中把任务标记为 `cancelled`
2. 向后台 worker 发送终止信号，让当前抓取/写入循环尽快收敛退出

### 5.5 Tushare 权限与自动跳过策略

当前 Data 默认按 **5000 积分档位**建模：

- 普通积分接口：按 5000 档位推导频次，再保留安全缓冲，不直接顶满官方上限
- 独立权限接口：只有在 `TUSHARE_EXTRA_PERMISSIONS` 中声明后才会进入默认数据集

这意味着像下面这些场景，系统会主动做权限裁剪，而不是跑到一半才报错：

- `news / major_news`：需要 `news`
- `anns_d`：需要 `announcements`
- `research_report`：需要 `research_report`
- `npr`：需要 `policy`
- `irm_qa_sh / irm_qa_sz`：需要 `investor_qa`
- `hk_daily / us_daily`：需要对应跨市场独立权限

如果你当前只有 5000 积分、没有额外独立权限，默认全量数据集会保留 A 股、财务、指数、宏观等普通积分接口，自动排除上述独立权限接口。

### 5.6 失败恢复与跳过策略

Data 任务现在不是“某个接口一报错整条 run 立刻失败”，而是按 dataset 做恢复：

1. 单次 API 调用先做 3 次短重试（网络抖动、限频等）
2. 某个 dataset 仍然失败时，整张表等待 **2 分钟**后再重试
3. 同一 dataset 最多尝试 **3 轮**
4. 仍然失败则记为 **skipped**，继续跑下一个 dataset
5. 全部 dataset 遍历完成后：
   - 若存在 skipped dataset，则本次 run 记为 `partial_success`
   - 不会自动发布 snapshot，避免把缺表快照误当成完整版本

也就是说，**任务会尽量把能拿到的数据先拿完**，而不是被单个接口卡死；但如果某些 dataset 最终被跳过，结果也会明确标成不完整。

### 5.7 全量初始化（bootstrap）

> **前提**：需要配置 `TUSHARE_TOKEN` 环境变量。

```bash
# dry-run 模式：只检查能拉多少数据，不实际写入
vortex data bootstrap --dry-run

# 正式执行（默认只提交后台任务）
vortex data bootstrap

# JSON 格式输出
vortex data bootstrap --format json

# 如需在当前终端前台执行（调试用）
vortex data bootstrap --foreground
```

### 5.8 增量更新（update）

```bash
# 增量更新：从上次同步结束日期 +1 天开始拉取
vortex data update
```

幂等性：重复执行会覆盖写，不产生副作用。

### 5.9 补充数据（backfill）

```bash
# 补充 2025 年 1 月的数据
vortex data backfill --start 20250101 --end 20250131
```

### 5.10 修复失败数据（repair）

```bash
# 修复某个日期范围的数据（当前阶段降级为日期范围重跑）
vortex data repair --start 20250101 --end 20250115
```

> **说明**：后续迭代会支持基于 `--run-id` 自动定位并重跑失败分区。

### 5.11 发布数据快照（publish）

```bash
# 发布当天快照
vortex data publish

# 发布指定日期快照
vortex data publish --as-of 20260401
```

发布前会检查最近一次质量校验是否通过。若质量门禁未通过，发布会被阻断。

## 6. 运行观察与数据库检查

这部分只回答一个问题：**任务到底有没有在跑、跑到了哪里、写进去的数据怎么看。**

### 6.1 先看任务是不是在正常运行

```bash
ROOT=~/Documents/vortex_workspace

# 1) 看任务状态与细粒度进度
vortex data status --root "$ROOT"
vortex data status --root "$ROOT" --watch

# 2) 跟官方日志入口
vortex data logs --root "$ROOT" --follow

# 3) 如需停止，直接取消任务
vortex data cancel --root "$ROOT" --task-id <task_id>

# 4) 看分区数量是否增长（以 bars 为例）
find "$ROOT/data/bars" -maxdepth 1 -type d -name 'date=*' | wc -l
```

观察时建议这样理解：

- `status` 里的 `task_id / run_id / pid / log` 用来看任务是否还活着。
- `status` 里的 `current_stage / current_dataset / current_chunk / written_rows` 用来看任务具体卡在哪个表、哪个分块。
- bootstrap 运行中时，`最近一次同步` 里的 `total_rows` 可能仍然是 `0`；这是因为汇总统计通常在本轮 run 收尾时统一写回。
- 真正能说明任务在推进的信号，是 **PID 仍存活、日志时间持续前进、`data/` 下分区数量继续增长**。
- 日志里偶发 `Connection reset by peer` 这类 Tushare 网络抖动 warning，当前实现会自动重试；短时 warning 不等于任务失败。
- 如果某张表连续失败 3 轮后被跳过，`latest run` 会显示 `partial_success`，并在 `error_message` / `detail` 中给出 skipped dataset 摘要。
- `cancel` 的语义是“请求当前任务尽快收敛退出”，不是回滚已经成功落盘的分区；已写入的数据仍然保留，后续可继续 `repair / backfill / update`。

### 6.2 检查控制面 SQLite：`state/control.db`

`control.db` 负责记录后台任务和运行时状态。

```bash
ROOT=~/Documents/vortex_workspace
export ROOT
python3 - <<'PY'
import os
import sqlite3

root = os.path.expanduser(os.environ['ROOT'])
db = f'{root}/state/control.db'
conn = sqlite3.connect(db)

print('tables =', [r[0] for r in conn.execute(
    "SELECT name FROM sqlite_master WHERE type='table' ORDER BY name"
)])

print('\\nrecent tasks:')
for row in conn.execute("""
    SELECT task_id, domain, action, resource_key, status, run_id, updated_at, progress_json
    FROM task_queue
    ORDER BY updated_at DESC
    LIMIT 10
"""):
    print(row)

conn.close()
PY
```

你通常会重点看：

- `task_queue.status`：`pending / running / success / partial_success / failed / cancelled`
- `resource_key`：幂等键；同类任务重复提交时会复用已有任务
- `run_id`：控制面任务与同步 manifest、日志串起来的主线索
- `progress_json`：当前阶段、当前 dataset、chunk 进度、`written_rows`、`log_path`、`pid`

### 6.3 检查同步 manifest：`state/manifests/<profile>/sync_manifest.db`

`sync_manifest.db` 记录每次同步 run 的结果摘要。

```bash
ROOT=~/Documents/vortex_workspace
export ROOT
python3 - <<'PY'
import os
import sqlite3

root = os.path.expanduser(os.environ['ROOT'])
db = f'{root}/state/manifests/default/sync_manifest.db'
conn = sqlite3.connect(db)

print('latest runs:')
for row in conn.execute("""
    SELECT run_id, action, status, started_at, finished_at, total_rows, error_message
    FROM sync_manifest
    ORDER BY started_at DESC
    LIMIT 10
"""):
    print(row)

conn.close()
PY
```

如果你关心发布后的快照，也可以继续查 `snapshot_descriptors` 表。

### 6.4 检查数据面 Parquet：用 DuckDB 直接查

Vortex 的数据面是 Parquet 文件；DuckDB 在这里主要扮演查询引擎，而不是必须长期存在一个单独的 `catalog.duckdb` 文件。

```bash
ROOT=~/Documents/vortex_workspace

# 看 bars 覆盖范围和总量
uvx duckdb -c "
SELECT
  min(date) AS min_date,
  max(date) AS max_date,
  count(*) AS rows,
  count(DISTINCT symbol) AS symbols
FROM read_parquet('$ROOT/data/bars/date=*/data.parquet', hive_partitioning = true);
"

# 看某个交易日的样例记录
uvx duckdb -c "
SELECT symbol, date, open, high, low, close, volume, amount
FROM read_parquet('$ROOT/data/bars/date=20260407/data.parquet')
ORDER BY symbol
LIMIT 10;
"

# 看当前 instruments universe
uvx duckdb -c "
SELECT count(*) AS symbols
FROM read_parquet('$ROOT/data/instruments/data.parquet');
"
```

### 6.5 和 Tushare 对账时，先统一口径

当前实现里，`instruments/data.parquet` 来自：

- `Tushare stock_basic`
- 且显式使用 `list_status='L'`

也就是说，Vortex 当前默认 universe 是**当前仍在上市的标的集合**。  
因此在核对历史 `bars` 时：

1. **不要**直接拿某个历史交易日的本地 `bars` 去和 Tushare `daily(trade_date=...)` 的“全量历史证券”比较。
2. 要先按 `instruments/data.parquet` 的 symbol 集合过滤 Tushare 返回结果。
3. 过滤后再比较行数与字段值，才是同一口径。

否则，早年交易日里 Tushare 会天然多出已经退市或不在当前 universe 里的标的，行数更大是正常现象，不代表本地写坏了。

## 7. 关键目录与文件

| 路径 | 用途 | 什么时候看 |
|------|------|------------|
| `profiles/default.yaml` | 默认数据配置 | 想确认 provider、history_start、schedule 时 |
| `state/control.db` | 后台任务队列、运行时控制面 | 想看任务状态、幂等复用、失败记录时 |
| `state/manifests/default/sync_manifest.db` | 每次同步 run 的摘要 | 想看最近一次同步是否成功、总行数、错误信息时 |
| `state/logs/*.log` | Server 与 Data 后台任务日志 | `status` 看不出细节时 |
| `data/instruments/data.parquet` | 当前 universe | 做数据对账、理解为何历史行数少于 Tushare 全量时 |
| `data/calendar/data.parquet` | 交易日历 | 看交易日范围、排查日期口径时 |
| `data/bars/date=YYYYMMDD/data.parquet` | 日线行情分区 | 抽样核对、查某日数据是否已落盘时 |

> **原则**：`state/` 是控制面，`data/` 是数据面。  
> 不要把“任务状态”和“行情/财务数据”混到同一个 SQLite 库里理解。

## 8. 常见问题

### Q: `ModuleNotFoundError: No module named 'tushare'`

如果你走的是仓库内运行：

```bash
uv sync
```

如果你走的是全局 `vortex` 命令，不要随手对另一个 `python3` 执行 `pip install tushare`。  
应该把包安装到 **`vortex` 实际使用的解释器** 里；最稳妥的办法就是直接复制报错里给出的那条安装命令。

### Q: `TUSHARE_TOKEN 未配置`

```bash
export TUSHARE_TOKEN=你的token
# 或写入 ~/.zshrc / ~/.bashrc
```

如果你是通过 `vortex init` 输入 token，它会写入工作区 `.env`；后续命令会自动读取。

### Q: `vortex: command not found`

要么先安装全局命令：

```bash
cd ~/Documents/vortex_quant
uv tool install --reinstall .
```

要么直接在仓库里运行：

```bash
uv run vortex --help
```

### Q: `vortex data status` 里显示任务在跑，但 `total_rows=0`

这在 bootstrap 过程中是正常的。  
`total_rows` 属于 run 汇总指标，往往在本轮任务结束时才统一写回。运行中更应该看：

- `pid` 是否还活着
- 日志时间有没有继续前进
- `data/` 分区是否继续增长

### Q: 日志里反复出现 `Connection reset by peer`

这通常是 Tushare 网络抖动或连接被远端重置，当前实现会自动重试。  
如果只是偶发 warning，且日志时间还在推进，一般不用立即介入；如果长时间只有重复重试、分区不再增长，再重点排查网络质量、Token 权限和 Tushare 服务状态。

### Q: `latest run` 显示 `partial_success` 是什么意思？

表示这次任务**没有整体中断**，但至少有一个 dataset 在多轮恢复后仍然失败，最终被标记为 skipped。

这时应这样理解：

- 能成功写入的表已经落盘
- 失败的表不会阻塞后续 dataset
- 本次 run 不会自动发布 snapshot
- 你可以根据 `error_message`、`data logs` 和 `detail.skipped_datasets` 再决定后续是重跑 `update`，还是缩小范围做 `backfill / repair`

### Q: 我只有 5000 积分，为什么某些数据集还是被跳过？

因为 Tushare 有两类权限：

1. **普通积分接口**：靠积分门槛决定能不能访问
2. **独立权限接口**：和积分无关，需要单独开通

像 `news`、`anns_d`、`research_report`、`irm_qa_*`、`hk_daily`、`us_daily` 这类接口，即使你有 5000 积分，也仍然需要在账号侧开通对应权限，并在环境变量里声明到 `TUSHARE_EXTRA_PERMISSIONS`。

### Q: 如何确认本地 `bars` 和 Tushare 是一致的？

先确认你比较的是**同一 universe**。  
当前 Vortex 默认只保留 `instruments/data.parquet` 里的当前上市标的；如果直接拿历史日的本地 `bars` 去对 Tushare 当日全量 `daily`，早年日期通常会天然多出一批已退市标的。

正确做法是：先按 `instruments` 的 symbol 集合过滤 Tushare，再比较行数和字段值。
