## 1. 环境准备

### 1.1 系统要求

| 项目 | 要求 |
|------|------|
| Python | 3.11+ |
| 操作系统 | macOS / Linux |
| 包管理 | [uv](https://github.com/astral-sh/uv)（推荐）或 pip |
| 磁盘空间 | ≥ 2GB（全量 A 股日线约 500MB Parquet） |

### 1.2 安装 uv（如未安装）

```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
```

### 1.3 Tushare Token（数据源）

数据采集功能需要 [Tushare Pro](https://tushare.pro) 账号和 Token。

Token 可通过以下方式配置（`vortex init` 交互流程会自动引导）：

```bash
# 方式 1：环境变量（推荐写入 ~/.zshrc 或 ~/.bashrc）
export TUSHARE_TOKEN=你的token

# 可选：声明当前账号积分档位（当前手册默认按 5000 积分建模）
export TUSHARE_POINTS=5000

# 可选：声明已开通的独立权限，逗号分隔
# 例如：news,announcements,research_report,policy,investor_qa,hk_daily,us_daily
export TUSHARE_EXTRA_PERMISSIONS=news,announcements

# 方式 2：vortex init 时交互输入（会自动写入工作区 .env 文件）
```

> **注意**：
> 1. 如果只做框架功能验证（不实际拉取数据），可以跳过 Token 配置。
> 2. 当前代码默认按 **5000 积分档位**控制普通积分接口频次；若你的账号档位不同，请显式设置 `TUSHARE_POINTS`。
> 3. 新闻、公告、港股/美股日线、投资者问答等属于 **独立权限接口**，只有在 `TUSHARE_EXTRA_PERMISSIONS` 中声明后，默认数据集才会纳入这些接口。

## 3. 工作区初始化

Vortex 的所有运行时状态（配置、数据、研究产物）都保存在一个**工作区目录**中。

### 3.1 交互式初始化（默认）

在终端中运行 `vortex init` 会启动 4 步交互向导：

```bash
vortex init
```

向导会依次询问：

1. **TUSHARE_TOKEN**：检测环境变量 → 未检测到则提示输入 → 冒烟测试验证可用性
2. **历史数据起始日**：默认 20170101，可自定义（格式 YYYYMMDD）
3. **首次数据更新**：先问“是否现在开始首次数据更新”。默认否，此时只写入配置，后续由你手动触发或由自动调度更新全量数据。若选择“是”，再选择本次先同步的数据集。
   数据集选择方式是键盘交互式多选：
   - `↑ / ↓`：移动光标
   - `空格`：勾选或取消当前项
   - `a`：一键全选
   - `n`：一键全不选
   - `d`：恢复默认推荐项
   - `回车`：确认当前选择
   - 列表较长时会自动滚动显示，不会一次性把全部数据集刷满整个终端
4. **自动更新计划**：可选启用每日定时更新，提供常用选项：
   - `1` 每天 18:00（推荐；工作日跑日频/盘中，周末跑低频）
   - `2` 每天 21:00
   - `3` 每天 06:00（含非交易日）
   - `4` 自定义 Cron 表达式
    Cron 是 5 段格式：`分 时 日 月 周`，其中 `*` 表示“每”，`1-5` 表示周一到周五。
    常用 Cron 示例：
    - `0 18 * * *` = 每天 18:00
    - `0 21 * * *` = 每天 21:00
    - `0 6 * * *` = 每天 06:00
   这里只是把调度策略写入 profile；**真正的自动执行由 `vortex server start` 常驻服务负责**。如果你只配置了 `schedule`，但没有启动 server，系统不会自己在后台定时更新。当前实盘环境的默认数据更新职责边界是：`vortex server` 按 `profiles/default.yaml` 的 `schedule: 0 18 * * *` 在每天 18:00 提交 `data update`；策略 `earnings-forecast auto-run` 只在盘前消费最新可见数据生成/执行计划，不应替代晚间 data 服务。
5. **飞书通知配置**（可选）：配置飞书开放平台凭证，使数据更新完成/失败时自动推送消息。逐项填写 App ID、App Secret、默认接收人 ID、接收人 ID 类型。任意一项留空即跳过整个飞书配置。详见 [§7 通知与 AI Agent](#7-通知与-ai-agent)。
6. **AI Agent 配置**（可选）：配置 AI Agent 后端（当前支持 Copilot CLI），使特定事件发生时自动调用 Agent 进行分析或修复。会自动检测 `copilot` 命令是否可用，不可用时提示安装方法。详见 [§7 通知与 AI Agent](#7-通知与-ai-agent)。

完成后自动生成 `profiles/default.yaml` 并写入工作区。
这个文件只沉淀用户真正输入的字段；provider 默认数据集、质量/PIT/发布/存储规则由框架默认值在解析阶段补齐，不会把一堆内部默认项直接展开进 YAML。

> **取消语义**：在 init 任意步骤按 `Ctrl+C`，会直接整体取消本次初始化；若本次初始化新建了工作区，会一并回滚，不留下半成品目录、配置或 bootstrap 中间状态。

### 3.2 非交互模式（CI/脚本）

```bash
vortex init --non-interactive
# 或通过管道输入（自动检测非 TTY 环境）
echo | vortex init
```

非交互模式使用框架默认值，适合 CI/CD 或脚本自动化。

### 3.3 指定自定义工作区路径

```bash
vortex init --root ~/my_workspace
```

### 3.3 工作区目录结构

初始化后的工作区布局：

```
~/Documents/vortex_workspace/
├── profiles/           # Profile 配置文件（YAML）
│   └── default.yaml    # 自动创建的默认 data profile
├── state/              # 控制面状态（SQLite / 状态文件 / 后台日志）
│   ├── control.db      # 全局控制数据库
│   └── logs/           # 后台任务与策略常驻日志
├── resolved/           # 解析后的 profile 快照
├── data/               # 数据面存储（Parquet + DuckDB）
├── research/           # 研究域产物
├── strategy/           # 策略域产物
└── trade/              # 交易域产物
```

> **设计原则**：控制面（`state/`）使用 SQLite，数据面（`data/`）使用 Parquet + DuckDB，二者严格分离。

### 3.4 验证初始化

```bash
# 检查目录是否创建
ls ~/Documents/vortex_workspace/
# 期望: data  profiles  research  resolved  state  strategy  trade

# 检查默认 profile 是否存在
cat ~/Documents/vortex_workspace/profiles/default.yaml
```

### 3.5 启动后台服务与自动调度

```bash
# 前台启动（适合第一次观察日志）
vortex server start --foreground

# 后台启动
vortex server start

# 查看服务状态
vortex server status

# 停止服务
vortex server stop
```

自动调度的真实语义是：

1. `server` 常驻进程每分钟扫描一次 `profiles/*.yaml`
2. 发现某个 data profile 的 `schedule` 到点后，就自动提交一次 `data update`
3. 调度触发和手工执行走的是同一条后台任务链路，所以幂等、日志、`status/logs/cancel` 观察方式完全一致

这也是 Vortex 现在推荐的日常使用方式：**用户只需要把 server 拉起来，让它自己按 profile 里的 schedule 运行**，而不是每天手工拼很多参数。

**✅ 验证通过标准**：7 个子目录存在，`default.yaml` 包含 `name: default`、`provider: tushare` 等字段。

## 4. Profile 配置管理

Profile 是 Vortex 的配置单元。
但对 Data 域来说，**日常 `vortex data ...` 默认直接使用工作区的 `profiles/default.yaml`**；只有研究/策略/交易域，或少数高级数据裁剪场景，才需要显式管理多个 profile。

### 4.1 核心概念

| 概念 | 说明 |
|------|------|
| **DataProfile** | 数据域默认配置：通常只关心 provider、history_start，以及少量高级裁剪字段 |
| **继承（extends）** | 子 profile 可继承父 profile 的配置，支持多层继承 |
| **默认值** | 未显式配置的字段自动使用框架默认值 |
| **Override** | CLI 级别可临时覆盖任意字段 |

合并优先级（从低到高）：`框架默认值 < 最远祖先 < ... < 直接父级 < 用户配置 < CLI override`

### 4.2 查看默认 profile 的解析结果

```bash
vortex profile resolve --name default --type data
```

输出示例（JSON 格式，每个字段标注来源）：

```json
{
  "name": {"value": "default", "source": "user"},
  "provider": {"value": "tushare", "source": "default"},
  "datasets": {"value": ["instruments", "calendar", "bars", "fundamental", "events", "valuation", "..."], "source": "default"},
  "exclude_datasets": {"value": [], "source": "default"},
  "priority_datasets": {"value": [], "source": "default"},
  "history_start": {"value": "20170101", "source": "default"}
}
```

### 4.3 查看 profile 的人类可读解释

```bash
vortex profile explain --name default --type data
```

### 4.4 创建自定义 profile

在工作区 `profiles/` 目录下创建 YAML 文件：

```bash
cat > ~/Documents/vortex_workspace/profiles/cn_stock.yaml << 'EOF'
name: cn_stock
provider: tushare
# datasets 默认已展开为 provider 内置的全量 dataset 列表，
# 通常不需要手写；如需裁剪，请用 exclude_datasets
# 如需跳过某些数据集，使用 exclude_datasets:
exclude_datasets:
  - events
history_start: "20200101"
priority_datasets:
  - bars
  - fundamental
EOF
```

说明：

- `priority_datasets` 只负责手工置顶；其余 dataset 会按系统内置的更新频率顺序自动排队：**日频 → 周频 → 月频 → 季频 → 其他**
- `--frequencies` 是一个**按频率分组筛选 dataset** 的手工入口：例如 `--frequencies daily` 表示“不要一张张手写 bars / valuation / moneyflow 等 dataset 名，而是直接让系统自动挑出所有日频 dataset 来跑”
- 这个参数更适合**调试、补跑、人工干预**；日常自动更新更推荐直接拉起 `server`，让系统按工作日/周末默认分桶去决定本次 `data update` 跑哪些频率

验证：

```bash
vortex profile resolve --name cn_stock --type data
```

### 4.5 使用继承

```bash
# 创建子 profile，继承 cn_stock 并覆盖 history_start
cat > ~/Documents/vortex_workspace/profiles/cn_stock_recent.yaml << 'EOF'
name: cn_stock_recent
extends: cn_stock
history_start: "20240101"
EOF

# 查看继承解析结果
vortex profile explain --name cn_stock_recent --type data
# provider 应显示来源 [parent:cn_stock]，history_start 显示来源 [user]
```

**✅ 验证通过标准**：`explain` 正确标注每个字段的来源（default / parent:xxx / user）。

## 5. 数据管理

数据管理覆盖完整的数据生命周期：采集 → 存储 → 质量校验 → PIT 对齐 → 发布。

### 5.1 命令一览

| 命令 | 用途 | 关键参数 |
|------|------|----------|
| `data bootstrap` | 全量初始化 | `--dry-run` `--frequencies` |
| `data update` | 增量更新 | `--dry-run` `--frequencies` |
| `data backfill` | 补充指定日期范围 | `--start` `--end` `--frequencies` |
| `data repair` | 修复失败分区 | `--start` `--end` `--frequencies` |
| `data publish` | 发布数据快照 | `--as-of` |
| `data status` | 查看同步状态与进度 | `--watch` |
| `data logs` | 查看后台任务日志 | `--task-id` `--follow` |
| `data cancel` | 取消后台任务 | `--task-id` |
| `data inspect` | 抽查某张表的字段、路径和样例数据 | `--dataset` `--filter` `--limit` |

> **运行形态说明**：除 `status`、`logs`、`cancel` 和显式 `--dry-run` 外，Data 写命令默认都会**提交后台任务**并立即返回；如需在当前终端直跑，可追加 `--foreground`。

> **调度口径说明**：如果不显式指定 `priority_datasets`，系统会把剩余 dataset 按更新频率自动排序，默认先跑**日频**，再跑周频、月频、季频和其他低频数据。
>
> **`data update` 默认分桶**：当你没有显式传 `--datasets` 或 `--frequencies` 时，`vortex data update` 会按自然日自动选择频率子集：**工作日只跑 `daily,intraday`，周六周日只跑 `weekly,monthly,quarterly,other`**。这样可以把日更/盘中数据留在交易周内，把低频数据和 `events` 一类慢接口尽量压到周末窗口。
>
> **`--frequencies` 的定位**：它不是日常自动更新必填参数，而是**手工筛批次**的控制杆。你可以把它理解成：
> - `--datasets bars,valuation` = 按表名精确指定
> - `--frequencies daily` = 按更新节奏选择整类 dataset
>
> 如果已经启用了 `server + schedule` 自动更新，通常**不需要每天手工传 `--frequencies`**；只有在你想临时只跑日频、只补周频，或者调试某一类慢接口时，才建议显式使用它。

### 5.2 查看同步状态与进度

```bash
vortex data status

# 持续刷新，直到没有活跃任务
vortex data status --watch
```

首次运行时输出 `ℹ️  无同步记录`，这是正常的。
Data CLI 默认读取当前工作区的 `profiles/default.yaml`，不需要额外传 `--profile`。
若已有后台任务在跑，`status` 会同时展示：

- 活跃任务：`task_id / run_id / pid / log_path`
- 当前阶段：`prepare / fetch / write / quality / publish`
- 当前进度：当前 dataset、dataset 总数、当前 chunk、chunk 总数、累计 `written_rows`
- 最近一次同步结果：用于确认最近成功/失败 run 的落库摘要

### 5.3 查看后台任务日志

```bash
# 默认取唯一活跃任务；没有活跃任务时取最近任务
vortex data logs

# 持续跟随日志输出
vortex data logs --follow

# 指定任务
vortex data logs --task-id <task_id> --follow
```

官方入口优先于手动 `tail -f`，因为它先按控制面任务定位日志，再决定跟哪一个文件。

### 5.4 取消后台任务

```bash
# 取消唯一活跃任务
vortex data cancel

# 或显式指定任务
vortex data cancel --task-id <task_id>
```

取消动作会同时做两件事：

1. 在控制面 `task_queue` 中把任务标记为 `cancelled`
2. 向后台 worker 发送终止信号，让当前抓取/写入循环尽快收敛退出

取消成功后，worker 日志会以 **cancelled** 语义收尾，不再把用户主动取消记成 failed，也不会额外打印失败 traceback。

### 5.5 怎么确认数据抓完没有

用户日常只需要先看这两个信号：

1. `vortex data status` 里 **没有活跃任务**
2. 最近一次 run 的 `status` 是 **`success`**

```bash
vortex data status
```

可以按下面的口径理解输出：

| 状态 | 用户该怎么理解 |
|------|----------------|
| `running` | 还没抓完，继续等 |
| `success` | 这次全量/增量已经完整收尾 |
| `partial_success` | 不是完整成功，至少有部分表被跳过或失败，需要继续排查 |
| `failed` / `cancelled` | 这次没有成功收尾，需要重跑 |

所以对用户来说，**“抓完没有”不是看日志刷没刷，而是看最近一次 run 是否 `success`**。

如果 `vortex data status` 里还显示活跃任务，但额外出现：

- `auto_recovery`
- `next_retry_at`

这表示系统已经识别出本轮是**可恢复错误**，后台 worker 正在等待下一次自动续跑。
这时候用户不需要额外启动脚本；正常情况下等它自动恢复完即可。只有最终仍不是 `success`，或者收到了飞书告警/提醒，才需要人工介入。

### 5.6 抽查某张表：字段、位置、样例数据

现在不需要用户自己进 DuckDB 或 SQLite 才能抽查。官方入口是：

```bash
# 先看当前已经落盘了哪些表
vortex data inspect

# 看 bars 这张表：字段、路径、总行数、样例数据
vortex data inspect --dataset bars

# 抽查某天某只股票
vortex data inspect --dataset bars \
  --filter symbol=000001.SZ \
  --filter date=20260407 \
  --limit 10

# 只看关心的几列
vortex data inspect --dataset fundamental \
  --columns symbol,report_date,total_revenue,n_income_attr_p \
  --filter symbol=600519.SH \
  --limit 5
```

这个命令会直接告诉你：

1. **这张表存在哪里**（`storage_path` / `parquet_glob`）
2. **底层用了哪个 Tushare 接口**（`api`）以及它的**官方详情页地址**（`api_doc_url`）
3. **有哪些字段**（字段名 + 类型 + 中文解释）
4. **总共有多少行**（`total_rows`）
5. **当前筛选条件命中了多少行**（`matching_rows`）
6. **可直接肉眼抽查的样例数据**（`preview`）

对于像 `bars` 这样的常用表，输出里还会额外给出 **表级备注**。
例如会直接提示这是 **A 股不复权日线行情**，并说明 `volume` 是**手**、`amount` 是**千元**，避免用户只看到字段名却不知道口径。

如果你想继续核对字段定义、权限要求或官方示例，也不需要自己再搜接口名。
`inspect` 已经会把底层 Tushare 接口的**官方详情页链接**一并打出来，可以直接在浏览器里继续查，不会再落到 markdown 镜像页。

如果某张表还没有真正落盘，输出里的 `materialized` 会是 `false`。
这比让用户自己去翻 `data/` 目录或手写 DuckDB SQL 更直接。

### 5.7 Tushare 权限与自动跳过策略

当前 Data 默认按 **5000 积分档位**建模：

- 普通积分接口：按 5000 档位推导频次，再保留安全缓冲，不直接顶满官方上限
- 独立权限接口：只有在 `TUSHARE_EXTRA_PERMISSIONS` 中声明后才会进入默认数据集
- 季度财务 / 业绩类表（`income / balancesheet / cashflow / fina_indicator / forecast / express`）：在 5000 分档位下会优先使用对应 `*_vip(period=季度末)` 做**按季度全市场批量抓取**；若积分不足，则自动回退到普通单股票接口

这意味着像下面这些场景，系统会主动做权限裁剪，而不是跑到一半才报错：

- `news / major_news`：需要 `news`
- `anns_d`：需要 `announcements`
- `research_report`：需要 `research_report`
- `npr`：需要 `policy`
- `irm_qa_sh / irm_qa_sz`：需要 `investor_qa`
- `hk_daily / us_daily`：需要对应跨市场独立权限

如果你当前只有 5000 积分、没有额外独立权限，默认全量数据集会保留 A 股、财务、指数、宏观等普通积分接口，自动排除上述独立权限接口。

### 5.8 失败恢复与跳过策略

Data 任务现在有两层恢复，不再需要用户额外跑 monitor 脚本：

1. **run 级自动恢复**：当一次 `bootstrap / update / backfill / repair` 收尾为可恢复的不完整状态时，后台 worker 会按 **5min → 15min → 30min** 再尝试整条 run；重跑时已安全落盘的分区会直接跳过，只补缺口
2. **dataset 级恢复**：单张表内部仍保留短重试和冷却重试，尽量把单表故障收敛在局部

其中 run 级自动恢复遵循下面的用户语义：

- 任意一次自动恢复成功，则任务继续正常完成，不主动打扰用户
- 若最终仍是 `partial_success` 或 `failed`，系统会通过飞书发出 warning / critical 通知，并附带 `run_id` 与排查命令
- 自动恢复期间，`vortex data status` 会显示 `auto_recovery` 与 `next_retry_at`

dataset 级恢复则继续按下面的细则执行：

1. 单次 API 调用先做 3 次短重试（网络抖动、限频等）
2. 某个 dataset 仍然失败时，整张表等待 **2 分钟**后再重试
3. 同一 dataset 最多尝试 **3 轮**
4. 仍然失败则记为 **skipped**，继续跑下一个 dataset
5. 全部 dataset 遍历完成后：
   - 若存在 skipped dataset，则本次 run 记为 `partial_success`
   - 不会自动发布 snapshot，避免把缺表快照误当成完整版本

也就是说，**任务会尽量把能拿到的数据先拿完**，而不是被单个接口卡死；但如果某些 dataset 最终被跳过，结果也会明确标成不完整。

对按季度分区的数据集，控制面现在还会额外登记一层 **coverage 账本**：

- `materialized`：该季度已经真正写出了 Parquet 分区
- `pit_blocked`：抓到了源数据，但在当前 `as_of_end` 下 PIT 对齐后没有可落盘记录
- `source_empty`：本轮明确检查过该季度，但源接口没有返回数据

这层账本的意义是：**同一个 `as_of_end` 下再次续跑时，像最新季度这类“已检查但暂时写不出来”的分区不会再重复抓取**；但如果以后 `as_of_end` 变大，系统仍会重新评估，不会把未来本应补齐的数据永久跳过。

需要注意：只有在写入阶段真正执行分区 upsert 后，某个季度才算“已物化”。也就是说，抓取完成但还没进入落盘阶段之前，Parquet 目录里不会出现对应分区。

对 `events` 这类 **`symbol_once + date` 的稀疏事件表**，系统现在还会登记一层 **exact range coverage**。
也就是说：如果同一个 `history_start ~ as_of_end` 已经完整扫描过一次，下一次再跑相同范围时会直接跳过，不会再把整张事件表按 `ts_code` 全量重扫；但如果你把历史范围从 2024 扩到 2005，系统仍会保守重抓，避免误把更早历史跳过。

### 5.10 增量更新（update）

```bash
# 增量更新：从上次同步结束日期 +1 天开始拉取
vortex data update

# 默认主路径：不传参数，让系统自己判断今天该跑哪一类频率
# - 工作日：daily + intraday
# - 周末：weekly + monthly + quarterly + other

# 手工只更新日频 dataset；适合补跑/调试日频这一整类
vortex data update --frequencies daily
```

这里要特别区分两种用法：

1. **默认自动更新**：不传 `--frequencies`，让系统根据当前日期自动分桶
2. **手工局部更新**：传 `--frequencies daily/weekly/...`，只跑某一类频率

因此，`--frequencies daily` 的目的不是让你每天都手工输入它，而是当你想表达“这次只补日频这一大类，不想手写几十个 dataset 名”时，给你一个更短、更稳定的入口。

幂等性：重复执行会覆盖写，不产生副作用。

实盘环境应优先依赖后台 data 服务自动触发，而不是盘前临时手工触发。检查口径如下：

```bash
# data 调度服务：必须常驻，负责按 profile.schedule 提交 data update
vortex server status --root ~/Documents/vortex_workspace

# data 最近同步：正常情况下 18:00 后应出现当天 update，且最终 success / partial_success / failed 明确收尾
vortex data status --root ~/Documents/vortex_workspace

# 服务健康监控日志：只检查和记录，不主动触发 data update
tail -f ~/Documents/vortex_workspace/state/logs/live-service-health.log
```

如果 `vortex server status` 里 `pid_alive=false`，即使 profile 写了 `schedule`，夜间也不会自动拉数据；这时应先恢复 `vortex server start`，而不是让策略服务承担补数职责。

## 6. 运行观察与数据库检查

这部分只回答一个问题：**任务到底有没有在跑、跑到了哪里、写进去的数据怎么看。**

### 6.1 先看任务是不是在正常运行

```bash
ROOT=~/Documents/vortex_workspace

# 1) 看任务状态与细粒度进度
vortex data status --root "$ROOT"
vortex data status --root "$ROOT" --watch

# 2) 跟官方日志入口
vortex data logs --root "$ROOT" --follow

# 3) 如需停止，直接取消任务
vortex data cancel --root "$ROOT" --task-id <task_id>

# 4) 看分区数量是否增长（以 bars 为例）
find "$ROOT/data/bars" -maxdepth 1 -type d -name 'date=*' | wc -l
```

观察时建议这样理解：

- `status` 里的 `task_id / run_id / pid / log` 用来看任务是否还活着。
- `status` 里的 `current_stage / current_dataset / current_chunk / written_rows` 用来看任务具体卡在哪个表、哪个分块。
- bootstrap 运行中时，`最近一次同步` 里的 `total_rows` 可能仍然是 `0`；这是因为汇总统计通常在本轮 run 收尾时统一写回。
- 真正能说明任务在推进的信号，是 **PID 仍存活、日志时间持续前进、`data/` 下分区数量继续增长**。
- 日志里偶发 `Connection reset by peer` 这类 Tushare 网络抖动 warning，当前实现会自动重试；短时 warning 不等于任务失败。
- 如果某张表连续失败 3 轮后被跳过，`latest run` 会显示 `partial_success`，并在 `error_message` / `detail` 中给出 skipped dataset 摘要。
- `cancel` 的语义是“请求当前任务尽快收敛退出”，不是回滚已经成功落盘的分区；已写入的数据仍然保留，后续可继续 `repair / backfill / update`。

### 6.2 检查控制面 SQLite：`state/control.db`

`control.db` 负责记录后台任务和运行时状态。

```bash
ROOT=~/Documents/vortex_workspace
export ROOT
python3 - <<'PY'
import os
import sqlite3

root = os.path.expanduser(os.environ['ROOT'])
db = f'{root}/state/control.db'
conn = sqlite3.connect(db)

print('tables =', [r[0] for r in conn.execute(
    "SELECT name FROM sqlite_master WHERE type='table' ORDER BY name"
)])

print('\\nrecent tasks:')
for row in conn.execute("""
    SELECT task_id, domain, action, resource_key, status, run_id, updated_at, progress_json
    FROM task_queue
    ORDER BY updated_at DESC
    LIMIT 10
"""):
    print(row)

conn.close()
PY
```

你通常会重点看：

- `task_queue.status`：`pending / running / success / partial_success / failed / cancelled`
- `resource_key`：幂等键；同类任务重复提交时会复用已有任务
- `run_id`：控制面任务与同步 manifest、日志串起来的主线索
- `progress_json`：当前阶段、当前 dataset、chunk 进度、`written_rows`、`log_path`、`pid`

### 6.3 检查同步 manifest：`state/manifests/<profile>/sync_manifest.db`

`sync_manifest.db` 记录每次同步 run 的结果摘要。

```bash
ROOT=~/Documents/vortex_workspace
export ROOT
python3 - <<'PY'
import os
import sqlite3

root = os.path.expanduser(os.environ['ROOT'])
db = f'{root}/state/manifests/default/sync_manifest.db'
conn = sqlite3.connect(db)

print('latest runs:')
for row in conn.execute("""
    SELECT run_id, action, status, started_at, finished_at, total_rows, error_message
    FROM sync_manifest
    ORDER BY started_at DESC
    LIMIT 10
"""):
    print(row)

conn.close()
PY
```

如果你关心发布后的快照，也可以继续查 `snapshot_descriptors` 表。

### 6.4 抽查数据：优先用 `vortex data inspect`

官方推荐的抽查方式不是让用户自己写 DuckDB SQL，而是直接用：

```bash
ROOT=~/Documents/vortex_workspace

# 列出当前已经落盘的表
vortex data inspect --root "$ROOT"

# 看 bars 的字段、路径、字段解释、行数和样例
vortex data inspect --root "$ROOT" --dataset bars

# 抽查某天某只股票
vortex data inspect --root "$ROOT" --dataset bars \
  --filter symbol=000001.SZ \
  --filter date=20260407 \
  --limit 10
```

只有在你确实需要做更复杂的 SQL 对账时，才建议再直接用 DuckDB 去查 Parquet。

### 6.5 和 Tushare 对账时，先统一口径

当前实现里，`instruments/data.parquet` 来自：

- `Tushare stock_basic`
- 且显式使用 `list_status='L'`

也就是说，Vortex 当前默认 universe 是**当前仍在上市的标的集合**。
因此在核对历史 `bars` 时：

1. **不要**直接拿某个历史交易日的本地 `bars` 去和 Tushare `daily(trade_date=...)` 的“全量历史证券”比较。
2. 要先按 `instruments/data.parquet` 的 symbol 集合过滤 Tushare 返回结果。
3. 过滤后再比较行数与字段值，才是同一口径。

否则，早年交易日里 Tushare 会天然多出已经退市或不在当前 universe 里的标的，行数更大是正常现象，不代表本地写坏了。

## 7. 通知与 AI Agent

Vortex 支持在特定事件发生时（数据同步完成/失败、质量校验异常等）通过通知渠道自动发送消息或调用 AI Agent。

### 7.1 通知渠道概览

| 渠道 | 说明 | 配置方式 |
|------|------|----------|
| **飞书（feishu）** | 通过飞书开放平台 API 发送文本消息 | 环境变量 `FEISHU_*` |
| **AI Agent** | 调用 Copilot CLI 等 AI 后端进行自动分析/修复 | 环境变量 `VORTEX_AGENT_*` |

两个渠道都是**默认关闭**的，需要在 `vortex init` 交互流程中配置，或手动设置环境变量。

### 7.2 飞书通知配置

#### 前置条件

1. 在 [飞书开放平台](https://open.feishu.cn) 创建一个应用
2. 在"凭证与基础信息"页面获取 App ID 和 App Secret
3. 确定消息接收人（用户或群组）的 ID

#### 通过 `vortex init` 配置

`vortex init` 的 Step 5 会逐项引导你填写飞书凭证。任意一项留空即跳过整个飞书配置。

#### 手动配置

在工作区 `.env` 文件中添加（或直接 export 环境变量）：

```bash
FEISHU_APP_ID=cli_xxx          # 飞书应用 App ID
FEISHU_APP_SECRET=xxx          # 飞书应用 App Secret（注意保密）
FEISHU_DEFAULT_RECEIVE_ID=xxx  # 默认接收人 ID
FEISHU_DEFAULT_RECEIVE_ID_TYPE=open_id  # ID 类型: open_id / user_id / chat_id / email
```

#### 通知触发时机

| 事件 | 严重程度 | 说明 |
|------|----------|------|
| `data.sync.completed` | info | 数据同步全量完成 |
| `data.sync.partial_failed` | warning | 部分数据集被跳过 |
| `data.sync.failed` | critical | 数据同步整体失败 |
| `data.quality.blocked` | critical | 质量门禁阻断发布 |

严重程度过滤由 profile 中 `notification.level` 控制（默认 warning），critical 事件始终发送。

#### 通知路由

在 profile YAML 中可以精细控制通知路由：

```yaml
notification:
  enabled: true
  channel: feishu        # 默认渠道
  level: warning         # 最低触发级别
  quiet_hours: "23:00-07:00"  # 静默时段（critical 不受限）
  routes:                # 按事件类型路由到不同渠道/接收人
    - type: data_sync_complete
      channel: agent     # 数据完成时调用 Agent
    - type: data_anomaly
      channel: feishu    # 数据异常发飞书
```

### 7.3 AI Agent 配置

#### 前置条件

安装 GitHub Copilot CLI：

```bash
npm install -g @githubnext/github-copilot-cli
```

安装后完成认证：

```bash
copilot auth
```

#### 通过 `vortex init` 配置

`vortex init` 的 Step 6 会：
1. 自动检测 `copilot` 命令是否存在
2. 如果不存在，提示安装方式
3. 引导你设置工作目录范围和推理强度

#### 手动配置

在工作区 `.env` 文件中添加：

```bash
VORTEX_AGENT_ENABLED=true       # 启用 Agent 渠道
VORTEX_AGENT_BACKEND=copilot    # 后端类型（当前仅支持 copilot）
VORTEX_AGENT_SCOPE=/path/to/repo  # Agent 工作目录范围
VORTEX_AGENT_EFFORT=high        # 推理强度: high / medium / low
VORTEX_AGENT_MAX_ATTEMPTS=2     # 同一事件最大调用次数
```

#### 工作原理

Agent 渠道复用了通知管线（NotificationService → Router → Channel）：

1. 事件发生时，NotificationService 构建 NotificationMessage
2. Router 根据 profile 配置决定投递到哪些渠道
3. 如果目标渠道包含 `agent`，AgentChannel 将消息转为 prompt
4. CopilotBackend 通过 `copilot -p "<prompt>" --yolo` 非交互调用

#### 可用性检查

每次调用 Agent 前会检查：
1. `VORTEX_AGENT_ENABLED` 是否为 true
2. `copilot` 命令是否存在于 PATH 中

如果任一条件不满足，该次投递会被记录为 `skipped` 或 `failed`，不会影响其他渠道的正常投递。

### 7.4 验证通知配置

```bash

# 在输出的 notification 段中确认 enabled / channel / level

# 飞书验证：查看 .env 中是否包含 FEISHU_* 变量

```

## 9. 常见问题

### Q: `ModuleNotFoundError: No module named 'tushare'`

```bash
uv sync
```

应该把包安装到 **`vortex` 实际使用的解释器** 里；最稳妥的办法就是直接复制报错里给出的那条安装命令。

### Q: `vortex data status` 里显示任务在跑，但 `total_rows=0`

这在 bootstrap 过程中是正常的。
`total_rows` 属于 run 汇总指标，往往在本轮任务结束时才统一写回。运行中更应该看：

- `pid` 是否还活着
- 日志时间有没有继续前进
- `data/` 分区是否继续增长

### Q: 日志里反复出现 `Connection reset by peer`

这通常是 Tushare 网络抖动或连接被远端重置，当前实现会自动重试。
如果只是偶发 warning，且日志时间还在推进，一般不用立即介入；如果长时间只有重复重试、分区不再增长，再重点排查网络质量、Token 权限和 Tushare 服务状态。

### Q: `latest run` 显示 `partial_success` 是什么意思？

表示这次任务**没有整体中断**，但至少有一个 dataset 在多轮恢复后仍然失败，最终被标记为 skipped。

这时应这样理解：

- 能成功写入的表已经落盘
- 失败的表不会阻塞后续 dataset
- 本次 run 不会自动发布 snapshot
- 你可以根据 `error_message`、`data logs` 和 `detail.skipped_datasets` 再决定后续是重跑 `update`，还是缩小范围做 `backfill / repair`

### Q: 我只有 5000 积分，为什么某些数据集还是被跳过？

因为 Tushare 有两类权限：

1. **普通积分接口**：靠积分门槛决定能不能访问
2. **独立权限接口**：和积分无关，需要单独开通

像 `news`、`anns_d`、`research_report`、`irm_qa_*`、`hk_daily`、`us_daily` 这类接口，即使你有 5000 积分，也仍然需要在账号侧开通对应权限，并在环境变量里声明到 `TUSHARE_EXTRA_PERMISSIONS`。

### Q: 如何确认本地 `bars` 和 Tushare 是一致的？

先确认你比较的是**同一 universe**。
当前 Vortex 默认只保留 `instruments/data.parquet` 里的当前上市标的；如果直接拿历史日的本地 `bars` 去对 Tushare 当日全量 `daily`，早年日期通常会天然多出一批已退市标的。

正确做法是：先按 `instruments` 的 symbol 集合过滤 Tushare，再比较行数和字段值。

   - `/api/trading/orders`
   - `/api/trading/trades`
   - `/api/market/full_tick`
5. 只放行内网访问，启用 token/API key，不暴露公网。

> 建议把“Windows 本地自检 + Mac 远程自检”都跑一遍，再进入策略交接阶段。

### 10.2 Mac 侧可用交互（只读）

#### A) 交易链路状态探测

```bash
vortex trade status \
```

关键字段：

- `qmt_ready=true`：桥接健康、账户读取成功；
- `qmt_blocking_reason`：阻断原因（例如 health 失败、读资产失败）；
- `qmt_cash / qmt_position_count / qmt_order_count / qmt_fill_count`：执行前快照。

#### B) 实时个股行情（Mac 直接查 Windows）

```bash
vortex trade quote \
  --root ~/Documents/vortex_workspace \
  --symbols 000001.SZ,600000.SH \
  --qmt-bridge-url http://<windows-ip>:8000 \
  --qmt-bridge-token <token> \
  --format json
```

返回 `quotes` 字段，包含 `open_price`、`last_price`、`volume`、`amount`、涨跌停/停牌标记，可用于盘中只读监控与执行前校验。

#### C) 业绩预告策略实盘交接包（影子目标 + 账户快照）

```bash
vortex strategy earnings-forecast live-handoff \
  --root ~/Documents/vortex_workspace \
  --start 20260101 \
  --as-of 20260310 \
  --preset stable_100w \
  --qmt-bridge-url http://<windows-ip>:8000 \
  --qmt-bridge-token <token> \
  --qmt-account-id <account_id> \
  --format json
```

该命令会同时产出：

- 影子目标持仓（来自策略信号）
- qmt-bridge 账户快照（资产/持仓/委托/成交）
- `qmt_ready` 与 `blocking_reasons`（是否允许进入下一步）

`live-handoff` 默认按当前 100 万实盘策略运行，即：

```bash
--preset stable_100w
```

`baseline_top110_large` 保留为研究 reference 和大容量回滚/对照版本；`tail_risk_soft_q10_p25` 在 2026-05-17 PIT 修正复跑后未通过 CPCV，只能显式用于 shadow/challenge，不是默认实盘版本。

当前实盘 preset 已内置市值约束：用上一交易日可见的 `valuation.total_mv` 做横截面排名，只保留 A 股总市值前 50% 的股票进入候选池。这样做是为了减少小市值尾部暴露，同时避免盘前使用当天收盘后才知道的市值数据。

生成的 JSON/HTML 可以直接作为“研究 → 执行”的交接件归档到 `strategy/`。

#### D) 开盘前生成当日执行计划（prepare）

如果已经确认执行口径是“**交易日开盘前拉最新可见数据，生成当日执行计划，开盘执行**”，不要在执行时临场重算 alpha，而要先把目标组合冻结下来：

这里强调的是 orchestration（编排层）不再额外叠加一层“固定 T+1”。策略信号自身仍可保留 `delay_days=1`，但执行层应消费开盘前已经冻结好的 `TargetPortfolio`，而不是在开盘前重新漂移信号。

```bash
vortex strategy earnings-forecast prepare-next-session \
  --root ~/Documents/vortex_workspace \
  --start 20260101 \
  --as-of 20260310 \
  --preset stable_100w \
  --qmt-bridge-url http://<windows-ip>:8000 \
  --qmt-bridge-token <token> \
  --qmt-account-id <account_id> \
  --portfolio-notional 1000000 \
  --format json
```

这里的 `--as-of` 是信号可见日，不是执行日。若 `--as-of 20260310`，默认执行日是 20260310 之后的下一交易日；手动指定执行日时也必须满足 `execution_trade_date > signal_as_of`，否则直接报错，避免把当天收盘后才知道的数据用于当天交易。

这个命令会顺序完成：

1. 先跑 `live-handoff`，把策略目标与 bridge 只读快照固化下来；
2. 用 `signal_as_of` 可见的日线、ST、涨跌停、停牌、市值和参考价生成 `TargetPortfolio`；
3. 目标组合写入执行日 `trade_date`，并在 `target_diagnostics` 中同时记录 `signal_as_of` 和 `signal_trade_date`，方便复核目标行来自哪一天。

核心产物位置：

- `trade/targets/<trade_date>/<portfolio_id>.json`：冻结后的 `TargetPortfolio`
- `state/trade/pending_qmt/*.json`：待执行任务

#### E) 通过 qmt-bridge 执行冻结组合（模拟盘）

真正进入模拟盘执行时，消费的是冻结组合，而不是 `shadow-plan` CSV：

```bash
vortex trade qmt rebalance \
  --root ~/Documents/vortex_workspace \
  --target-portfolio ~/Documents/vortex_workspace/trade/targets/20260311/tp_20260311_xxxxx.json \
  --qmt-bridge-url http://<windows-ip>:8000 \
  --qmt-bridge-token <token> \
  --qmt-account-id <account_id> \
  --format json
```

这个命令现在默认只做 QMT dry-run：生成 `order_plan / pre_trade_result / execution_report`，但不会提交委托。若要真实提交，必须额外显式传入：

```bash
  --enable-trading \
  --allowed-account-id <account_id> \
  --confirm-trading CONFIRM_AUTO_TRADING
```

执行语义：

1. 先读当前真实持仓、现金和实时行情；
2. 用“**当前真实持仓 vs 冻结目标组合**”做通用差分调仓；
3. 先卖后买；
4. 缺行情、缺 ST 风险标记、停牌、涨跌停、非整手或风控超限时 fail-closed，不自动下单；
5. 已验证的 QMT 委托状态码会在适配层归一：`54=cancelled`、`56=filled`、`57=rejected`；
6. 把 `order_plan / pre_trade_result / execution_report` 落到 `trade/executions/<exec_id>/`。

### 10.3 推荐日常流程（先模拟再实盘）

1. Windows 开盘前确认 MiniQMT 在线 + bridge health 正常。
2. Mac 执行 `vortex trade status` 与 `vortex trade quote` 做只读联调。
3. 交易日 08:10 后执行 `prepare-next-session`，用执行日前最新已完成交易日作为 `signal_as_of`，生成 `trade_date=today` 的冻结组合和 pending task；手动调用时执行日必须晚于信号日。
4. 当天开盘执行 `vortex trade qmt rebalance`，用冻结组合做 sell-first 差分调仓。
5. 执行后查看 `trade inspect` / `trade reconcile`，确认委托、成交和盘后对账。

如果你希望把这条链路做成常驻后台，而不是每天手工跑两条命令，可以直接启动自动编排服务：

```bash
vortex strategy earnings-forecast auto-run \
  --root ~/Documents/vortex_workspace \
  --start 20260101 \
  --profile default \
  --preset stable_100w \
  --qmt-bridge-url http://<windows-ip>:8000 \
  --qmt-bridge-token <token> \
  --qmt-account-id <account_id> \
  --nav-initial-equity 1000000 \
  --nav-benchmark 000852.SH
```

第一版 `auto-run` 的职责是：

1. 到 `prepare-time`（默认 08:10）后先检查策略关键数据覆盖；若 `bars / valuation / stk_limit / stock_st` 等分区缺失，只对 `forecast、bars、valuation、stk_limit、suspend_d、stock_st、fina_indicator、express` 做定向补数，仍缺数则 fail-closed，不生成交易任务；
2. 到 `execute-time`（默认 09:25）后扫描 `state/trade/pending_qmt/`，执行今天到期的任务；
3. 每轮 tick 同时把摘要写入 `state/strategy/earnings_forecast_auto/status.json`，把结构化事件追加到 `state/logs/strategy-earnings-forecast-auto-run.log`，不再只依赖终端 stdout。

默认不会真实提交委托。要允许 `auto-run` 真实下单，必须同时传 `--enable-trading`、`--allowed-account-id <account_id>`、`--confirm-trading CONFIRM_AUTO_TRADING`，且目标账户在白名单内；否则只生成/审查计划和 dry-run 执行报告。

推荐的后台观察方式：

```bash
# 1) detached / nohup / launchd 拉起常驻进程后，看最近状态
vortex strategy earnings-forecast auto-status --root ~/Documents/vortex_workspace

# 2) 跟稳定日志文件（CLI 封装）
vortex strategy earnings-forecast auto-logs --root ~/Documents/vortex_workspace --follow

# 3) 或直接读状态文件
cat ~/Documents/vortex_workspace/state/strategy/earnings_forecast_auto/status.json
```

`auto-status` 适合回答“最近一轮 tick 成功了吗、现在 PID 还活着吗、最近报错是什么”；`auto-logs` 适合持续观察后台服务是否仍在推进。后续如果主 agent 用 bash detached 方式启动同一条命令，也应优先通过这两个入口和上述状态文件/日志文件排查，而不是依赖启动终端里的瞬时 stdout。

### 10.4 每日自动任务时序与日志巡检

当前实盘观察环境采用“晚间数据服务 + 盘前策略服务 + 健康监控”的职责拆分。核心原则是：正常日频拉数由 `vortex server` 执行，策略服务只做盘前新鲜度校验和必要的定向补数；任何 ST、涨跌停、停牌、市值、行情等关键数据缺失，都不能被解释成“没有风险”。

| 阶段 | 正常行为 | 主要状态/日志 | 异常判断与处理 |
|---|---|---|---|
| 18:00 晚间主数据更新 | `vortex server` 根据 `profiles/default.yaml` 的 `schedule: 0 18 * * *` 提交 `data update` | `vortex server status --root ~/Documents/vortex_workspace`；`vortex data status --root ~/Documents/vortex_workspace --format json`；`state/logs/data-update-*.log` | 如果 `pid_alive=false`，说明后台 data 服务未运行，应先恢复 `vortex server start --root ...`；如果 data run 长时间无成功/失败收尾，先查 data 日志，不要让策略服务替代全量拉数 |
| 20:30/21:00 关键数据复查 | 观察 `bars / valuation / stk_limit / suspend_d / stock_st` 是否覆盖到最新可见交易日；必要时只做关键数据定向补数 | `vortex data status --root ... --format json`；`state/logs/data-update-*.log` | 如果 18:00 多次拿不到关键分区，再考虑把主 schedule 调到 21:00；单日偶发缺数优先定向补，不全量重跑 |
| 08:10 盘前目标生成 | `auto-run` 解析执行日前最新可见信号日，检查关键分区；通过后生成 `trade_date=today` 的 target、质量审查和 pending task；默认策略为 `stable_100w` | `vortex strategy earnings-forecast auto-status --root ...`；`state/strategy/earnings_forecast_auto/status.json`；`state/logs/strategy-earnings-forecast-auto-run.log`；`trade/targets/`；`state/trade/pending_qmt/` | 若提示 `stock_st`、`stk_limit`、`valuation` 等缺分区，策略会定向补数；补后仍缺则停止生成目标，必须人工核查数据源 |

| 执行后对账 | 查看目标、订单、成交、实际持仓和现金差异；说明未成交、废单、现金残差原因 | `trade/executions/<exec_id>/`；`vortex trade inspect`；`vortex trade reconcile` | 如果目标持仓数低于 TopN，先看 task 的 `target_diagnostics`，区分候选不足、ST 剔除、市场权限、最小申报、现金不足 |

#### 10.4.1 查看候选漏斗与市场门控

当目标持仓低于 TopN 时，先看 `target_diagnostics.selection_funnel`，不要直接判断为服务停了或过滤顺序错了。当前 `stable_100w` 的排序前核心门禁已经包含流动性、ST/风险警示、上一交易日总市值前 50%、涨跌停/停牌买入阻断等条件；`selected_position_count` 低于 TopN 通常表示核心门禁之后合格信号不足，或者后续权限、整手、最低金额等执行规则导致无法补满。

```bash
python3 - <<'PY'
import json
from pathlib import Path

root = Path.home() / "Documents/vortex_workspace"
target = sorted((root / "trade/targets").glob("*/*.json"))[-1]
payload = json.loads(target.read_text())
diagnostics = payload.get("metadata", {}).get("target_diagnostics", {})

print("target =", target)
print("\n## selection_funnel")
print(json.dumps(diagnostics.get("selection_funnel", {}), ensure_ascii=False, indent=2))
print("\n## market_gate")
print(json.dumps(diagnostics.get("market_gate", {}), ensure_ascii=False, indent=2))
print("\n## shortfall_reason")
print(diagnostics.get("shortfall_reason"))
PY
```

候选漏斗建议按下面顺序解释：

| 字段 | 含义 |
|---|---|
| `raw_signal_count` | 当日业绩预告信号数量。 |
| `positive_signal_count` | 正向业绩预告信号数量。 |
| `after_liquidity_count` | 流动性过滤后数量。 |
| `after_st_filter_count` | ST/风险警示过滤后数量。 |
| `after_market_cap_top50_count` | A 股总市值前 50% 过滤后数量。 |
| `after_open_block_count` | 涨停、停牌等开盘买入阻断后数量。 |
| `after_quality_block_count` | 持仓质量审查 blocked 后数量。 |
| `after_permission_count` | 账户市场权限过滤后数量。 |
| `executable_candidate_count` | 资金、整手、科创板最低 200 股等执行规则后仍可买数量。 |
| `selected_position_count` | 最终冻结目标持仓数量。 |

市场门控看 `target_diagnostics.market_gate`。如果 `risk_on=false`，当天应解释为指数下行或确认不足导致空仓/不新增，而不是策略服务异常；重点字段是 `benchmark`、`benchmark_momentum`、`benchmark_above_support`、`risk_on_confirmations` 和 `required_confirmations`。

#### 10.4.2 调仓惯性保留

实盘冻结目标中有一层 `LivePositionInertiaConfig`，只处理“当前已持有且仍在新目标池”的股票，目的是避免赢家上涨后被机械卖回 5%。

规则如下：

1. 先按 base 策略生成基础目标股数，单票基础目标仍来自 `max_weight=5%`、整手和最低金额约束。
2. 若当前持仓股数低于基础目标股数，按正常目标补仓。
3. 若当前持仓股数高于基础目标股数，只对超额股数做动量保留：`目标股数 = 基础目标股数 + round_lot((当前股数 - 基础目标股数) * 动量保留比例)`。
4. 动量保留比例使用交易前可见日线计算：5 日收益横截面分位、20 日收益横截面分位、20 日下行波动反向分位，默认权重为 `0.4 / 0.4 / 0.2`。
5. 缺少动量数据时默认保留比例为 0，不保护超额股数；这样不会因为数据缺失而把上涨持仓永久锁住。
6. 股票退出目标池、触发 ST/quality blocked/market permission 风险、停牌或涨跌停执行阻断时，不受惯性保护，仍按原风控和执行规则处理。

### 10.5 账户子账本策略净值台账

净值统计属于 [[交易域设计说明书|交易域]] 的账户级能力，不绑定某一个策略。当前默认口径是**账户子账本**：一个 QMT 账户只跑一个自动策略，但账户里可能有超过策略名义本金的闲置现金。首条快照会锁定 `external_cash_offset = account_total_asset - initial_equity`，后续用 `strategy_equity = account_total_asset - external_cash_offset` 计算策略权益和净值。只要其余钱没有被动用，服务短暂停止后重启就延续同一个账户和策略绑定，不新开净值曲线。

因此 `status.summary.latest_total_asset` 表示策略子账本权益，`status.summary.latest_account_total_asset` 才是 QMT 全账户总资产；`status.binding.external_cash_offset` 是被冻结在净值口径外的非策略资金。只有显式传 `--reset`、更换账户、重设初始资金或指定新 `run_id` 时，才视为新净值序列。

`earnings-forecast auto-run` 在成功执行 QMT 调仓后，会用只读资产接口自动记录当日净值快照。下面的 `snapshot` 命令用于手工补记、调试或独立巡检：

```bash
vortex trade nav snapshot \
  --root ~/Documents/vortex_workspace \
  --strategy-name earnings_forecast_auto \
  --strategy-version stable_100w \
  --initial-equity 1000000 \
  --benchmark 000852.SH \
  --qmt-bridge-url http://<windows-ip>:8000 \
  --qmt-bridge-token <token> \
  --qmt-account-id <account_id> \
  --format json
```

查看净值、基准和窗口超额：

```bash
vortex trade nav status \
  --root ~/Documents/vortex_workspace \
  --strategy-name earnings_forecast_auto \
  --strategy-version stable_100w \
  --qmt-account-id <account_id> \
  --format json
```

默认 `benchmark=000852.SH`，即中证 1000；如果当前策略要对标沪深 300 或中证 500，可以在 `snapshot` 时改成对应指数代码。台账文件位置如下：

```text
~/Documents/vortex_workspace/state/nav/<run_id>.json   # 策略账户绑定
~/Documents/vortex_workspace/trade/nav/<run_id>.csv    # 每日净值 ledger
```

`status` 会输出成立以来、近 1 周、1 月、3 月、6 月、1 年的策略收益、基准收益、超额收益和最大回撤。第一版不自动做出入金份额法；如果账户总资产出现无法由盈亏解释的跳变，或有人动用了子账本之外的资金，应在巡检中标记为疑似现金流或外部持仓漂移，再人工确认是否需要开启新 run 或补现金流账本。
