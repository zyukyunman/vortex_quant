# 数据域设计说明书 v1

> 本文档是 Vortex 的第一份实现级设计文档，目标是把“可替换数据源 + 可替换存储 + 严格阻断”落成可编码接口。

---

## 一、输入约束（已确认）

1. 市场不设上限：有数据就可研究。
2. 频率双轨：日频 + 分钟频。
3. 可研究快照时间：默认 22:00（可配置）。
4. 财务严格 PIT。
5. Tushare 路线默认全A。
6. 质量失败阻断流程。

---

## 二、设计目标

1. 数据链路可替换：Provider 插拔。
2. 存储链路可替换：Backend 插拔。
3. 研究与执行共享一份权威数据快照。
4. 任意失败可追溯（run_id + manifest + report）。

这一节想解决的真实问题是：

1. 数据工程不能只为 Qlib 服务，还要为排错、回填、审计、执行、对账服务。
2. 研究系统不能和某一个数据源、某一个数据库、某一个市场写死。
3. 第一版必须先把流程正确性做出来，而不是先做最复杂的基础设施。

---

## 三、术语解释

### 3.1 Provider

Provider 指“数据提供方适配器”。

例如：

1. `TushareProvider`
2. `RQDataProvider`
3. `PolygonProvider`

它只负责“从外部拿数据”，不负责研究逻辑，不负责快照发布。

### 3.2 StorageBackend

StorageBackend 指“底层存储后端”。

例如：

1. `ParquetDuckDBBackend`
2. `PostgresBackend`
3. `ClickHouseBackend`

它只负责“如何存、如何读、如何做快照视图”，不负责外部 API 取数。

### 3.3 权威快照

权威快照是某个时间点系统认定“可供研究/回测/执行使用”的统一数据版本。

它解决的是：

1. 今天研究用哪一版数据
2. 回测复现用哪一版数据
3. 实盘调仓依据哪一版数据

### 3.4 PIT

PIT 是 Point-In-Time。

意思不是“财务值是多少”，而是“在当时那个时间点，市场是否已经看得到这个值”。

例如：

1. 某公司 2025 年年报利润很好
2. 但如果年报在 2026-03-28 才公告
3. 那么 2026-03-27 的策略不允许使用这份年报数据

### 3.5 ODS / DWD / DWS / ADS

这些不是为了显得复杂，而是为了解决不同层的职责问题：

1. ODS：保存原始取数结果，便于排错和重跑
2. DWD：做字段统一、代码统一、时间统一
3. DWS：做研究可直接使用的数据视图
4. ADS：做具体应用场景快照，比如研究快照、执行快照

## 四、组件边界

### 4.1 DataProvider 抽象

```python
class DataProvider:
    name: str
    supported_markets: list[str]
    supported_freqs: list[str]

    def fetch_instruments(self, market: str): ...
    def fetch_calendar(self, market: str, start: str, end: str): ...
    def fetch_bars(self, market: str, symbols: list[str], freq: str, start: str, end: str): ...
    def fetch_fundamental(self, market: str, symbols: list[str], fields: list[str], start: str, end: str): ...
    def fetch_events(self, market: str, symbols: list[str], start: str, end: str): ...
```

默认实现：TushareProvider。

接口语义：

1. `fetch_instruments`：获取某市场标的主数据，不带研究逻辑。
2. `fetch_calendar`：获取该市场交易日历，是所有后续校验的时间轴基准。
3. `fetch_bars`：获取行情数据，必须显式给出市场、标的、频率、时间范围。
4. `fetch_fundamental`：获取财务、估值、指标等结构化基本面。
5. `fetch_events`：获取公告、停牌、ST、成分变更等事件型数据。

最小返回契约：

1. 所有接口默认返回 DataFrame，而不是业务对象。
2. 行情类数据至少统一到 `market/symbol/trade_time/freq` 这组主键语义。
3. 基本面数据至少统一到 `market/symbol/report_period/ann_date` 这组主键语义。
4. 事件类数据至少统一到 `market/symbol/event_time/event_type` 这组主键语义。
5. 原始源字段可以保留，但标准字段必须存在，后续模块只依赖标准字段。

### 4.2 StorageBackend 抽象

```python
class StorageBackend:
    name: str

    def upsert(self, dataset: str, df, partition: dict): ...
    def read(self, dataset: str, filters: dict): ...
    def exists(self, dataset: str, partition: dict) -> bool: ...
    def list_partitions(self, dataset: str): ...
    def snapshot(self, as_of: str) -> dict: ...
```

默认实现：ParquetDuckDBBackend。

接口语义：

1. `upsert`：按分区写入或覆盖指定数据集。
2. `read`：按过滤条件读出数据集。
3. `exists`：判断某分区是否已经存在，用于幂等调度。
4. `list_partitions`：用于检查落地完整性。
5. `snapshot`：生成某个时点的统一读取视图。

最小返回契约：

1. `upsert` 必须是幂等的，重复执行不能制造重复记录。
2. `read` 必须支持按市场、标的、时间范围、频率过滤。
3. `snapshot` 返回的不是裸路径集合，而是“可被研究/执行模块直接消费的快照描述对象”。
4. 快照描述对象至少要包含 `as_of`、版本号、数据集清单、物理位置、校验摘要。

### 4.3 QualityGate 抽象

```python
class QualityGate:
    def run(self, dataset: str, df, rules: dict) -> dict: ...
```

默认策略：Fail-Closed（任何关键项失败即阻断）。

这里“阻断”不是报个 warning，而是：

1. 不发布 snapshot
2. 不进入研究流程
3. 不允许向执行链路继续传递

### 4.4 PITAligner 抽象

```python
class PITAligner:
    def align(self, financial_df, disclosure_df, calendar_df): ...
```

它的职责是把“财务记录日期”和“市场可见日期”区分开。

输出要求：

1. 对每一条财务记录给出 `effective_from`。
2. 在 `effective_from` 之前，不允许该记录进入研究特征表。
3. 若公告日期缺失或冲突，默认进入阻断或人工复核，而不是静默放行。

## 五、第一版为什么不是直接上 Qlib 格式

这个问题必须单独讲清楚。

### 5.1 结论

第一版不建议把 Qlib 格式作为系统底层唯一数据库。

建议：

1. 权威层：Parquet + DuckDB
2. 研究层：Qlib bin

### 5.2 原因

Qlib 格式的优势在“研究读写性能”和“表达式因子计算”，不是“做全系统底座”。

它不适合作为唯一底座的原因：

1. 对公告、财务、事件、元数据、审计信息支持不自然。
2. 不适合作为排错和回填的原始依据。
3. 不适合直接承担多市场异构数据治理。
4. 不适合直接承担 SQL 检查和工程侧巡检。

### 5.3 那为什么不是直接上 Postgres/ClickHouse

也不是不能上，而是第一版没有必要先把问题做重。

第一版最重要的是：

1. 数据流程正确
2. PIT 正确
3. 快照可复现
4. 研究链路可跑通

Parquet + DuckDB 更适合第一阶段，因为：

1. 开发门槛低
2. 列式分析性能好
3. 与 Pandas/Polars/Qlib 生态兼容好
4. 后续迁移成本可控

## 六、为什么要分权威层和研究层

如果不分层，会出现两个问题：

1. 研究为了追求速度，牺牲了排错与治理能力
2. 底座为了追求通用性，研究读取又变慢

所以分层的目的是：

1. 权威层解决“存得准、查得清、可追溯”
2. 研究层解决“算得快、读得快、适合 Qlib”

这不是重复存储，而是职责分离。

## 八、数据分层

1. ODS：原始响应落地（保留源字段）。
2. DWD：标准化字段（symbol/time/freq/market/adjusted_price）。
3. DWS：研究视图（标签、特征、PIT 对齐财务）。
4. ADS：应用快照（研究快照、执行快照、对账快照）。

解释：

1. ODS 是“原样保留”，方便追源头。
2. DWD 是“统一口径”，方便后续所有模块共用。
3. DWS 是“研究友好”，方便因子和模型使用。
4. ADS 是“场景就绪”，方便研究快照和执行快照直接消费。

## 九、每日调度（默认）

1. 18:30 - 20:00：增量同步（日频、分钟频、财务、资金流）。
2. 20:00 - 21:00：PIT 对齐与标准化。
3. 21:00 - 21:40：质量校验（关键项阻断）。
4. 21:40 - 22:00：Qlib 增量构建。
5. 22:00：发布 research snapshot。

说明：22:00 仅为默认值，配置项可调整。

这个顺序不能反：

1. 先同步
2. 再校验
3. 再构建研究视图
4. 最后发布快照

否则就会出现“快照已发布，但质量未验证”的工程事故。

## 十、阻断规则（第一版）

以下任一失败即阻断：

1. 交易日历缺失当日记录。
2. 核心行情字段缺失超过阈值。
3. 复权因子异常（非正值或突变异常）。
4. PIT 对齐失败（公告日后可见性校验未通过）。
5. 快照构建不完整（关键数据集未落地）。

这里要强调：

1. 阻断并不意味着系统崩溃
2. 阻断意味着“停在安全边界，不往下游传播脏数据”

## 十三、v1 交付边界

v1 只承诺：

1. TushareProvider + ParquetDuckDBBackend 跑通。
2. 全A + 日频/分钟频 + 严格 PIT + 严格阻断。

## 十四、可直接编码的目录结构

这一节的目标不是画一个“好看但没法写”的分层图，而是给出第一版可以直接建目录、建文件、补类的最小结构。

### 14.1 为什么建议使用 `vortex/data/`

如果按重写思路推进，数据域建议提升为一级包 `vortex/data/`，而不是继续埋在 `vortex/core/data/`。

原因是：

1. 数据已经不是某个内部工具模块，而是研究、执行、风控、运维共同依赖的一级领域。
2. 后续还会有 `vortex/research/`、`vortex/execution/`、`vortex/risk/` 等并列域，平级更清晰。
3. 这样更容易把“领域接口”和“基础设施实现”分开，而不是所有内容都堆进一个 `core`。

### 14.2 第一版建议目录树

### 14.3 每个目录负责什么

1. `contracts/`：只放抽象协议，不放实现细节。
2. `models/`：只放值对象（value object）和 manifest/report，不放外部 I/O。
3. `registry/`：负责注册与解析 Provider、Storage、DatasetSpec。
4. `providers/`：不同数据源适配器实现。
5. `storage/`：不同存储后端实现。
6. `normalize/`：统一字段、代码、时间口径。
7. `quality/`：质量规则与阻断门禁。
8. `pit/`：PIT 对齐逻辑。
9. `repository/`：把 `StorageBackend` 包装成“按数据集读写”的领域读写层。
10. `services/`：串联多个组件，完成一段完整业务动作。
11. `orchestration/`：负责编排整条每日数据流水线。

## 十五、核心类清单

这一节给的不是泛泛而谈的“可能有这些类”，而是第一版建议直接落文件的类。

### 15.1 基础异常与公共类型

文件建议：`vortex/data/exceptions.py`

1. `DataContractError`：Provider 或 Normalizer 没有满足标准字段契约时抛出。
2. `DataQualityBlockedError`：质量门禁阻断时抛出。
3. `PITAlignmentError`：PIT 对齐无法完成时抛出。
4. `SnapshotPublishError`：快照发布失败时抛出。
5. `RegistryNotFoundError`：注册表中找不到 provider/storage/spec 时抛出。

文件建议：`vortex/data/types.py`

1. `MarketName`
2. `FreqName`
3. `DatasetName`
4. `PartitionDict`

这些类型不复杂，但能让方法签名更清楚。

### 15.2 契约层

文件建议：`vortex/data/contracts/provider.py`

1. `DataProvider`

职责：

1. 只负责向外部数据源请求数据。
2. 返回标准字段可映射的 DataFrame。
3. 不负责编排、落库、快照、Qlib 构建。

文件建议：`vortex/data/contracts/storage.py`

1. `StorageBackend`

职责：

1. 只负责数据落地、读取、分区管理、快照读取基础能力。
2. 不负责数据质量判断。

文件建议：`vortex/data/contracts/quality.py`

1. `QualityRule`
2. `QualityGate`

职责：

1. `QualityRule` 负责单条规则。
2. `QualityGate` 负责把多条规则汇总为统一阻断结果。

文件建议：`vortex/data/contracts/pit.py`

1. `PITAligner`

职责：

1. 接收财务表与披露日历。
2. 输出附带 `effective_from` 的可研究数据。

### 15.3 值对象与清单对象

文件建议：`vortex/data/models/dataset.py`

1. `DatasetSpec`
2. `DatasetPartition`

`DatasetSpec` 负责描述数据集静态定义：

1. 数据集名
2. 所属层级（ODS/DWD/DWS/ADS）
3. 默认 provider
4. 主键字段
5. 分区字段
6. 是否需要 PIT
7. 质量规则列表

`DatasetPartition` 负责描述一次物理或逻辑分区：

1. market
2. freq
3. trade_date / year / month
4. 其他 profile 维度

文件建议：`vortex/data/models/task.py`

1. `SyncTask`
2. `SyncPlan`
3. `SyncResult`

职责：

1. `SyncTask`：描述一条可执行同步任务。
2. `SyncPlan`：描述某次运行要做哪些任务。
3. `SyncResult`：描述某条任务的结果、产物、阻断状态。

文件建议：`vortex/data/models/manifest.py`

1. `SyncManifest`
2. `SnapshotManifest`
3. `QlibBuildManifest`

职责：

1. 统一描述产物，而不是散落写 JSON。
2. 后续序列化到 JSON 只是输出形式，不是领域建模本身。

文件建议：`vortex/data/models/report.py`

1. `QualityReport`
2. `PITAlignmentReport`

职责：

1. 把阻断原因结构化。
2. 让研究、运维、调度三方看到的是同一份报告对象。

### 15.4 注册层

文件建议：`vortex/data/registry/provider_registry.py`

1. `ProviderRegistry`

职责：

1. 按名字注册和解析 `DataProvider`。
2. 屏蔽上层对具体 provider 类名的硬编码。

文件建议：`vortex/data/registry/storage_registry.py`

1. `StorageRegistry`

职责：

1. 按名字注册和解析 `StorageBackend`。

文件建议：`vortex/data/registry/dataset_catalog.py`

1. `DatasetCatalog`

职责：

1. 集中维护所有 `DatasetSpec`。
2. 决定某数据集该用哪个 provider、哪些质量规则、是否需要 PIT。

### 15.5 实现层

文件建议：`vortex/data/providers/tushare.py`

1. `TushareProvider`

职责：

1. 把 Tushare API 转成 `DataProvider` 契约。
2. 处理分页、限频、字段映射、接口差异。

文件建议：`vortex/data/storage/parquet_duckdb.py`

1. `ParquetDuckDBBackend`

职责：

1. 负责 Parquet 分区写入。
2. 负责 DuckDB SQL 查询与快照视图生成。

### 15.6 规范化与质量层

文件建议：`vortex/data/normalize/schema.py`

1. `SchemaNormalizer`

职责：

1. 把 provider 返回字段转换为标准字段名。
2. 校验标准字段是否齐全。

文件建议：`vortex/data/normalize/symbol.py`

1. `SymbolNormalizer`

职责：

1. 统一股票、指数、基金、港股、美股代码口径。

文件建议：`vortex/data/normalize/time.py`

1. `TimeNormalizer`

职责：

1. 统一时区、交易时间戳、日期字段。

文件建议：`vortex/data/quality/rules.py`

1. `RequiredColumnRule`
2. `PrimaryKeyUniqueRule`
3. `NullRatioRule`
4. `TradingCalendarConsistencyRule`
5. `AdjFactorSanityRule`

文件建议：`vortex/data/quality/gate.py`

1. `DefaultQualityGate`

职责：

1. 依次执行规则。
2. 汇总为 `QualityReport`。
3. 在 fail-closed 模式下抛出或返回阻断状态。

文件建议：`vortex/data/pit/financial.py`

1. `FinancialPITAligner`

职责：

1. 根据 `ann_date`、`disclosure_calendar`、`trading_calendar` 计算 `effective_from`。
2. 输出对齐后的财务表和 `PITAlignmentReport`。

### 15.7 Repository 与 Service 层

文件建议：`vortex/data/repository/dataset_repository.py`

1. `DatasetRepository`

职责：

1. 使用 `StorageBackend` 读写具体数据集。
2. 对上层隐藏 Parquet 路径、DuckDB SQL、分区细节。

文件建议：`vortex/data/repository/manifest_repository.py`

1. `ManifestRepository`

职责：

1. 统一保存和读取 `SyncManifest`、`SnapshotManifest`、`QlibBuildManifest`。

文件建议：`vortex/data/services/sync_service.py`

1. `DatasetSyncService`

职责：

1. 执行单个 `SyncTask`。
2. 串联 provider -> normalize -> quality -> PIT -> repository。
3. 生成 `SyncResult` 和 `SyncManifest`。

文件建议：`vortex/data/services/qlib_service.py`

1. `QlibViewBuilder`

职责：

1. 从权威层读取数据。
2. 构建或增量更新 Qlib 研究视图。
3. 输出 `QlibBuildManifest`。

文件建议：`vortex/data/services/snapshot_service.py`

1. `SnapshotPublisher`
2. `SnapshotResolver`

职责：

1. `SnapshotPublisher` 负责把本次可用数据集、Qlib 视图、校验摘要打包成 `SnapshotManifest`。
2. `SnapshotResolver` 负责给研究、回测、执行模块解析某个 `as_of` 对应的快照入口。

### 15.8 编排层

文件建议：`vortex/data/orchestration/plan_builder.py`

1. `SyncPlanBuilder`

职责：

1. 根据 profile、日期、市场、频率生成 `SyncPlan`。
2. 决定这次要同步哪些数据集和哪些分区。

文件建议：`vortex/data/orchestration/workflow.py`

1. `DailyDataWorkflow`

职责：

1. 跑完每日数据主流程。
2. 顺序调用 `SyncPlanBuilder`、`DatasetSyncService`、`QlibViewBuilder`、`SnapshotPublisher`。

## 十六、关键类的最小方法签名建议

下面这些签名不是最终定稿，但已经足够作为第一版骨架直接开写。

```python
from dataclasses import dataclass


@dataclass(frozen=True)
class DatasetSpec:
    name: str
    layer: str
    provider: str
    primary_keys: tuple[str, ...]
    partition_keys: tuple[str, ...]
    requires_pit: bool
    quality_rules: tuple[str, ...]


@dataclass(frozen=True)
class DatasetPartition:
    values: dict[str, str]


@dataclass(frozen=True)
class SyncTask:
    dataset: str
    partition: DatasetPartition
    start: str
    end: str
    market: str
    freq: str | None = None


@dataclass(frozen=True)
class SyncPlan:
    run_id: str
    profile: str
    tasks: list[SyncTask]


class DatasetSyncService:
    def run(self, task: SyncTask) -> "SyncResult": ...


class QlibViewBuilder:
    def build(self, as_of: str) -> "QlibBuildManifest": ...


class SnapshotPublisher:
    def publish(self, as_of: str) -> "SnapshotManifest": ...


class DailyDataWorkflow:
    def run(self, as_of: str, profile: str) -> "SnapshotManifest": ...
```

这里的重点不是字段多少，而是边界清楚：

1. `SyncTask` 是执行单元。
2. `DatasetSyncService` 是单任务执行器。
3. `DailyDataWorkflow` 是整日编排器。

## 十七、这些类之间到底怎么衔接

这一节回答你最关心的问题：类不是平铺罗列，而是怎样真正串起来。

### 17.1 启动阶段

启动时完成三件事：

1. `ProviderRegistry` 注册 `TushareProvider`。
2. `StorageRegistry` 注册 `ParquetDuckDBBackend`。
3. `DatasetCatalog` 注册所有 `DatasetSpec`。

这一步完成后，系统才知道：

1. 某个数据集该由谁取
2. 取完往哪里存
3. 要过哪些规则
4. 是否需要 PIT

### 17.2 单个数据集同步链路

以 `daily_basic` 为例，一条 `SyncTask` 的调用顺序应当是：

1. `DailyDataWorkflow` 从 `SyncPlan` 取出一条 `SyncTask`。
2. `DatasetSyncService` 根据 `task.dataset` 向 `DatasetCatalog` 取出 `DatasetSpec`。
3. `DatasetSyncService` 根据 `spec.provider` 向 `ProviderRegistry` 解析出 `TushareProvider`。
4. `TushareProvider` 拉取原始 DataFrame。
5. `SchemaNormalizer`、`SymbolNormalizer`、`TimeNormalizer` 依次把原始数据转成标准口径。
6. `DefaultQualityGate` 根据 `spec.quality_rules` 运行质量规则，输出 `QualityReport`。
7. 如果数据集需要 PIT，则 `FinancialPITAligner` 执行对齐并输出 `PITAlignmentReport`。
8. `DatasetRepository` 调用 `ParquetDuckDBBackend` 把结果写入权威层。
9. `ManifestRepository` 写出本任务 `SyncManifest`。
10. `DatasetSyncService` 返回 `SyncResult` 给 `DailyDataWorkflow`。

这里最关键的边界是：

1. `Provider` 不知道 Parquet 怎么写。
2. `StorageBackend` 不知道 Tushare 怎么取。
3. `QualityGate` 不知道任务怎么编排。
4. `Workflow` 不直接操作 DataFrame 细节。

### 17.3 每日全量编排链路

每日主流程建议固定为：

1. `DailyDataWorkflow.run(as_of, profile)` 启动。
2. `SyncPlanBuilder.build(as_of, profile)` 生成 `SyncPlan`。
3. `DailyDataWorkflow` 遍历 `plan.tasks`，逐条调用 `DatasetSyncService.run(task)`。
4. 只要任何关键 `SyncResult` 为 blocked，`DailyDataWorkflow` 直接停止后续发布。
5. 全部关键任务成功后，调用 `QlibViewBuilder.build(as_of)`。
6. `QlibViewBuilder` 从 `DatasetRepository` 读取权威层数据并生成 Qlib 视图。
7. `SnapshotPublisher.publish(as_of)` 汇总 `SyncManifest`、`QualityReport`、`PITAlignmentReport`、`QlibBuildManifest`，生成 `SnapshotManifest`。
8. `ManifestRepository` 持久化 `SnapshotManifest`。

这条链保证的是：

1. 没有通过质量门禁的数据，进不了快照。
2. 没有完成 PIT 的财务数据，进不了研究层。
3. 没有完成 Qlib 构建的日期，不会被标成“可研究”。

### 17.4 研究与执行如何接这条链

研究侧和执行侧都不应该自己拼路径、自己猜版本。

它们统一这样接：

1. 研究模块向 `SnapshotResolver` 请求某个 `as_of` 的研究快照。
2. `SnapshotResolver` 返回 `SnapshotManifest` 中声明的权威层入口和 Qlib 入口。
3. Qlib workflow 使用研究入口。
4. 回测执行或组合编译使用权威层或执行快照入口。

这样做的目的，是把“数据版本选择权”从研究脚本里拿掉，收回到数据域本身。

### 17.5 一句话概括每层依赖方向

依赖方向建议固定为：

1. `workflow` 依赖 `services`
2. `services` 依赖 `repository`、`registry`、`contracts`、`normalize`、`quality`、`pit`
3. `repository` 依赖 `storage`
4. `providers` 和 `storage` 实现 `contracts`
5. `models` 被所有上层读取，但自己不依赖外部 I/O

这个依赖方向不能反过来，否则很快就会重新变成一团耦合代码。

### 17.6 构造依赖关系应该怎么写

如果进入编码阶段，建议直接用构造注入（constructor injection），不要在方法里临时 new 对象。

一个清晰的装配关系应该像这样：

```python
class DatasetSyncService:
    def __init__(
        self,
        dataset_catalog,
        provider_registry,
        dataset_repository,
        manifest_repository,
        schema_normalizer,
        symbol_normalizer,
        time_normalizer,
        quality_gate,
        pit_aligner,
    ) -> None: ...


class QlibViewBuilder:
    def __init__(self, dataset_repository, manifest_repository) -> None: ...


class SnapshotPublisher:
    def __init__(self, manifest_repository) -> None: ...


class DailyDataWorkflow:
    def __init__(
        self,
        plan_builder,
        sync_service,
        qlib_builder,
        snapshot_publisher,
    ) -> None: ...
```

这样装配以后，依赖关系会非常明确：

1. `DailyDataWorkflow` 不知道 Tushare 细节，只知道调用 `sync_service`。
2. `DatasetSyncService` 不知道 DuckDB 细节，只知道调用 `dataset_repository`。
3. `DatasetRepository` 不知道业务规则，只知道调用 `StorageBackend`。
4. `QlibViewBuilder` 不知道外部 provider，只从权威层读数据。

这也是后续单元测试最省力的写法，因为每个依赖都可以单独替换成 fake 或 stub。

## 十八、第一版建议的最小落地顺序

如果下一步开始写代码，不建议同时铺满所有文件，建议按下面顺序逐层落：

1. 先写 `models/`、`contracts/`、`exceptions.py`，把边界定死。
2. 再写 `registry/` 和 `DatasetCatalog`，把数据集定义集中起来。
3. 再写 `TushareProvider` 和 `ParquetDuckDBBackend`。
4. 再写 `SchemaNormalizer`、`DefaultQualityGate`、`FinancialPITAligner`。
5. 再写 `DatasetRepository` 和 `DatasetSyncService`。
6. 最后写 `QlibViewBuilder`、`SnapshotPublisher`、`DailyDataWorkflow`。

这样推进的好处是：

1. 每一步都能单测。
2. 每一步都能验证边界是否合理。
3. 不会一上来就写一个 500 行的大 syncer 把所有职责揉在一起。
