## 一、设计目标与产品映射

### 2.1 核心设计目标

| # | 目标 | 含义 |
|---|------|------|
| D1 | **数据链路可替换** | Provider（数据源适配器）和 StorageBackend（存储后端）通过协议接入，内部逻辑不绑定具体实现 |
| D2 | **Fail-Closed 质量门禁** | 关键校验失败即阻断，不发布快照、不流入下游 |
| D3 | **严格 PIT**（Point-In-Time，时点对齐） | 财务数据必须在公告日之后才对研究/回测可见，杜绝未来信息泄露 |
| D4 | **快照即版本** | 跨域传递以不可变快照为单位，禁止 latest 等可变引用 |
| D5 | **任意失败可追溯** | 每次运行产出 run_id + sync_manifest + report，支持审计与复现 |

这些目标直接承接 00 §2.1 架构原则 P1–P7。

### 2.2 CLI 动作映射

产品冻结文档定义了 Data 域 6 个 CLI 动作（命令行操作），涵盖全量初始化、增量更新、数据修复、快照发布等场景。下表说明每个动作在架构层面由哪些组件承接：

| CLI 动作 | 入口参数 | 主要组件调用链 | 产出 |
|----------|---------|---------------|------|
| `bootstrap` | `--profile [--dry-run]` | CLI 提交后台任务 → ProfileService → SyncPlanBuilder（全量）→ DatasetSyncService → QualityGate → PITAligner → DerivedMetricsBuilder → QlibViewBuilder → SnapshotPublisher | task_id + run_id；完成后产出 snapshot_descriptor |
| `update` | `--profile [--dry-run]` | CLI 提交后台任务；其余同 bootstrap，但 SyncPlanBuilder 仅生成增量任务 | task_id + run_id；完成后产出 snapshot_descriptor |
| `backfill` | `--profile --start --end` | CLI 提交后台任务 → SyncPlanBuilder（指定范围）→ DatasetSyncService；不触发 publish | task_id + run_id；完成后更新 sync_manifest |
| `repair` | `--profile [--run-id]` | CLI 提交后台任务 → 读取 failed_units → 重跑失败分区 → QualityGate → PITAligner | task_id + run_id；完成后更新 sync_manifest |
| `publish` | `--profile [--as-of] [--dry-run] [--force]` | CLI 提交后台任务 → QualityGate（再验证）→ DerivedMetricsBuilder → QlibViewBuilder → SnapshotPublisher | task_id + run_id；完成后产出 snapshot_descriptor |
| `status` | `--profile [--format] [--watch]` | 读取控制面 `task_queue` + 最近 run 的 manifest/report；`--watch` 周期性刷新直到没有活跃任务 | 终端摘要或 JSON |
| `logs` | `--profile [--task-id] [--lines] [--follow]` | 先从控制面解析目标任务，再读取 `log_path`；`--follow` 持续跟随日志流 | 日志尾部或实时日志流 |
| `cancel` | `--profile [--task-id]` | 在控制面把任务标记为 `cancelled`，并向 worker PID 发送终止信号 | 取消确认摘要 |

> 参数名与退出码规范见 00 §18；run_id 格式见 02。

> **运行语义约束**：除了显式 `--foreground` 或 `--dry-run`，Data 域写命令默认只负责提交任务，不在当前终端前台阻塞执行完整流水线。

> **恢复语义约束**：dataset 不是“一次失败整条 run 立刻中断”的原子单元。当前实现采用 **调用级短重试 + dataset 级冷却重试 + 最终 skip** 的恢复模型：单次 API 调用先做有限重试；某个 dataset 连续失败后等待 2 分钟再重试整张表；达到上限后将该 dataset 标记为 skipped，并继续处理后续 dataset。若最终存在 skipped dataset，则 run 状态记为 `partial_success`，且不自动发布 snapshot。

### 2.3 四级进度模型

每次 Data 动作的执行粒度分为四级，支持断点续跑和进度追踪：

1. **run**：一次完整 CLI 调用，对应唯一 run_id。
2. **stage**：run 内的用户可见阶段（`prepare → fetch → write → quality → publish`）。
3. **dataset**：stage 内按数据集维度拆分（如 `bars`、`income`、`valuation`）。
4. **chunk**：dataset 内按分区维度拆分（如按交易日、报告期、symbol 批次）。

之所以对外只暴露 5 个 stage，而不是把 normalize / PIT / derived 全都拆成更细的终端状态，是因为产品目标是先回答“现在跑到哪张表、哪一段、写了多少”，而不是让用户面对一串难以解释的内部子步骤。PIT 对齐、标准化和衍生指标仍然存在，但在控制面上被收敛进稳定的 `write / quality / publish` 阶段。

控制面中的 `progress_json` 负责记录这些字段；`status` 直接展示 stage / dataset / chunk / written_rows，`logs` 用于补充更细的 provider 级执行细节，`cancel` 则依赖同一份控制面状态对当前 worker 发出终止请求。

此外，dataset 进入执行队列前还要经过一层 **权限可访问性判断**。以 Tushare 为例，控制面不会把“当前账号无权访问”的独立权限接口误当成普通失败重试，而是直接标记为 skipped，并把原因写入 run 摘要，避免整条数据链路被不可访问接口长期拖住。

### 2.4 抓取调度与接口配额预算

前面的恢复语义解决了"失败后不要整条 run 立刻中断"；但对 Tushare 这类**按接口限频**的数据源，还需要再补一层"**抓取调度语义**"，否则系统即使能断点续跑，也可能在单一接口上长时间连续打满配额。

这里要区分两个概念：

1. **分区粒度**：数据按什么维度切块，例如 `trade_date`、`report_date`、`symbol × quarter`。
2. **全局请求预算**：当前账号在单位时间内最多还能发多少次请求。

对当前默认的 5000 积分档位，普通积分接口上限应理解为**全局共享的 500 次/分钟**，不是"每个 API 各有 500 次/分钟"。因此产品层正确的调度语义不是"在不同 API 之间轮转以叠加吞吐"，而是：

- 把预算抽象成 **global request budget**，而不是写死成"450 天"；
- 对 500 次/分钟档位，默认可收敛为 **450 requests / minute** 的安全阈值；
- 在全局预算约束下，优先把请求花在**仍然缺失的 chunks** 上，而不是任何已经落地或已经判定覆盖的分区。

这意味着 SyncPlanBuilder 输出的关键不只是"本次跑哪些 dataset"，而是"本次每个 dataset 还有哪些 **missing chunks**，这些 chunks 是否值得在当前全局预算下继续抓取"。调度器的首要目标是**跳过已覆盖分区、减少无效请求**，而不是为了切换 API 而切换 API。

对 Tushare 来说，结合官方文档 `doc_id=230`，建议按接口类型区分默认抓取方式：

| 接口类型 | 推荐 chunk | 默认策略 |
|----------|------------|----------|
| 全市场日频时间序列（如 `daily` / `daily_basic` / `moneyflow`） | `trade_date` | **优先按交易日遍历**，不按 `ts_code` 全历史循环；先检查缺失交易日分区，再只抓缺口 |
| 财务季频（如 `income` / `balancesheet` / `cashflow` / `fina_indicator` / `forecast` / `express`） | `report_date × symbol batch` | 先检查缺失季度分区；5000 分档位优先走 `*_vip(period=季度末)` 全市场季度批量抓取，否则再回退到普通单股票接口 |
| 参考表 / 名录表（如 `stock_basic` / `index_basic` / `index_classify`） | 整表或 market 维度 | 若本地已有有效缓存，默认复用；只有显式 refresh 才强制重抓 |
| 日期范围型语料/宏观接口（如 `news` / `anns_d` / `cn_cpi`） | 年 / 月窗口 | 先按窗口检查已有分区，再抓缺失窗口；不能安全判定完整性的接口允许保守重抓 |

因此，你提的思路里真正应该保留的是**"先检查缺口，再决定是否发请求"**，而不是"450 条后切下一个接口"这个细节：

 - 对 `trade_date` 型接口，1 request ≈ 1 个交易日，所以全局预算会自然表现成"今天还能抓多少个缺失交易日"；
 - 对季频/符号型接口，1 request 可能只是 1 个 `symbol × quarter`，此时预算消耗的始终是"请求次数"，不是"天数"；
- 在全局预算共享的前提下，真正能节省时间的是**减少重复请求**，而不是假设切换 API 就能拿到新的独立 500/min 窗口。

对按季度分区的 dataset，Data 域现在不再只看“Parquet 分区目录是否存在”，而是同时维护控制面的 `dataset_partition_coverage` 账本。  
这张账本会为每个 `dataset + partition_key + partition_value + as_of_end` 记录 `materialized / pit_blocked / source_empty` 等状态，用来表达“这个季度在当前时点已经检查过，但未必已经物化”。  
这样同一 `as_of_end` 下的续跑不会反复重抓 `pit_blocked` 或 `source_empty` 的季度；而当 `as_of_end` 向后推进时，系统仍会重新评估这些季度，保证可补齐的数据不会被永久跳过。

对于 `events` 这类 `symbol_once + date` 的稀疏事件表，Data 域还需要维护 **exact range coverage**：  
因为这类表无法像 `trade_day_all` 一样提前枚举完整目标日期，缺少某个 `date=` 分区并不等于“那一天一定缺数据”，也可能只是当天根本没有事件。  
因此系统只在“同一 `history_start ~ as_of_end` 范围已经完整扫描过”的前提下跳过再次抓取；如果范围扩大，则仍按保守策略重新扫描。

这类稀疏事件表还有一个额外约束：**分区日期字段必须由 dataset contract 显式声明候选顺序，不能由 provider 看到哪个列名存在就直接猜。**  
以 `events=dividend` 为例，当前约定的回退顺序是 `ex_date → record_date → pay_date → ann_date → end_date`。这样即使某些记录尚未填好 `ex_date`，系统也会按契约继续寻找可用事件日期，而不是因为空值或脏值在标准化阶段把整张表打回重试。

日志也必须显式体现这一层调度决策，例如：

```text
dataset=bars dedup: partition_key=date target=545 existing=398 missing=147
global_budget effective_rpm=400 api=daily decision=continue
dataset=fundamental dedup: partition_key=end_date target=84 existing=60 covered=1 missing=23
```

没有这层日志，用户只能看到"现在在跑哪张表"，却看不到系统是否真的在执行"跳过已覆盖区间 + 全局限流"这两个关键动作。

## 三、对外消费契约（域 API 边界）

Data 域作为数据链路的源头，必须明确：**对外提供什么、别人怎么消费、哪些是冻结边界**。

| 消费入口 | 消费者 | 访问方式 | 不可变性 |
|----------|--------|---------|---------|
| `snapshot_descriptor` | Research / Strategy | 通过 SnapshotResolver 按 as_of 解析，获取权威层路径和 Qlib 视图入口 | 同 as_of 可重发布（生成新 revision_id，旧 revision 保留为 superseded），下游 run 通过 lineage 中的 `snapshot_revision_id` 精确追溯（详见 00 §5.2） |
| Qlib 研究视图 | Research | 目录路径（publish 生成） | 随 snapshot 重发布而更新 |
| CalendarService | 所有域 | 共享契约层 API（详见 02） | Data 获取并缓存，其他域只读 |
| `data_health_report` / `pit_report` | 审计 / 运维 | 文件读取 | 只读 |
| `data.*` 事件 | Notification 域 | 事件发射（emit），不影响数据流 | — |

**冻结规则**：

1. 下游域只能消费已发布的快照，不能读取同步中间态。
2. 下游域不直接调用 DataProvider 或 StorageBackend——数据获取是 Data 域的专属职责。
3. snapshot_descriptor 的契约结构属于 Data 域，变更需评估下游影响。

## 四、DataProvider 协议

DataProvider 是"数据提供方适配器"的抽象协议。它只负责从外部数据源获取原始数据，不负责存储、校验、快照。

### 4.1 能力语义

| 能力 | 语义 | 说明 |
|------|------|------|
| 获取标的主数据 | 返回某市场全部可交易标的列表 | 含代码、名称、上市/退市日期等元信息 |
| 获取交易日历 | 返回指定市场和时间范围的交易日序列 | 是所有后续校验的时间轴基准 |
| 获取行情数据 | 返回 OHLCV（开盘价/最高价/最低价/收盘价/成交量）等时间序列 | 须显式指定市场、标的、频率、时间范围 |
| 获取基本面数据 | 返回财务报表、估值指标等 | 须携带报告期和公告日，供 PIT 对齐使用 |
| 获取事件数据 | 返回公告、停复牌、ST、成分变更等 | 时间戳为事件发生时间 |

> 具体方法签名见 00 §10.1 概要；完整参数与返回类型见 06。

### 4.2 契约不变量

1. 所有能力返回表格型结果（DataFrame 语义），不返回业务对象。
2. 行情数据必须可映射到 canonical 主键（规范化的统一标识组合）：`market / symbol / trade_time / freq`。
3. 基本面数据必须可映射到：`market / symbol / report_period / ann_date`——缺少 `ann_date` 的记录必须标记，不能静默丢弃。
4. 事件数据必须可映射到：`market / symbol / event_time / event_type`。
5. Provider 的返回结果必须支持幂等——相同参数多次调用返回相同数据。

默认实现：TushareProvider（全 A 日频 + 分钟频）。

## 五、StorageBackend 协议

StorageBackend 是"底层存储后端"的抽象协议。它只负责数据的物理落地、读取、分区管理和快照视图生成。

### 5.1 能力语义

| 能力 | 语义 | 关键约束 |
|------|------|---------|
| 写入 | 按分区写入或覆盖指定数据集 | 必须幂等，重复执行不产生重复记录 |
| 读取 | 按过滤条件读出数据集 | 支持按市场、标的、时间范围、频率过滤 |
| 存在性检查 | 判断某分区是否已落地 | 用于幂等调度与增量判断 |
| 分区列表 | 列出数据集已有分区 | 用于完整性检查 |
| 快照视图 | 生成某时点的统一读取描述 | 返回 snapshot_descriptor，包含 as_of、数据集清单、物理位置、校验摘要 |

> 具体方法签名见 06。

这里要特别强调：**存在性检查不是存储层的“可有可无辅助能力”，而是 SyncPlanBuilder / DatasetSyncService 必须真正消费的闭环输入**。也就是说，Data worker 在调用 Provider 之前，必须先基于 `list_partitions` / 读取能力判断：

1. 当前请求范围内哪些分区已经存在；
2. 哪些分区仍然缺失，需要真正发起外部抓取；
3. 当前这个 dataset 是“全部跳过”“只抓缺口”还是“无法安全判定，只能按请求范围重抓”。

只有把这一步放到抓取前，"写入幂等" 才不会退化成"每次都重新打 API，只是落盘时覆盖写"。此外，worker 日志必须显式输出这一步的判断结果（如 target / existing / missing 分区数量与最终决策），否则用户无法确认系统是否真的走了幂等/恢复语义。

### 5.2 选型：Parquet + DuckDB

第一版权威层采用 Parquet + DuckDB，理由简述（完整讨论见 00 §9.2）：

1. Qlib bin 适合研究读取，不适合数据治理（回填、修复、审计）。
2. Parquet 是中性列式格式，任何工具都能读取。
3. DuckDB 零部署、直查 Parquet、SQL 能力完备。
4. 后续可通过实现新的 StorageBackend 迁移到 PostgreSQL 或 ClickHouse。

权威层与研究层分层的核心逻辑：权威层解决"存得准、查得清、可追溯"；研究层（Qlib 视图）解决"算得快、读得快"。两者不是重复存储，而是职责分离。

## 六、数据生命周期

### 6.1 端到端流水线

数据从外部采集到对外发布经历七个阶段，严格顺序执行，后一阶段的输入来自前一阶段的输出：

```
采集(fetch) → 标准化(normalize) → 质量校验(quality) → PIT 对齐(pit)
    → 衍生指标(derived) → Qlib 视图构建(build) → 快照发布(publish)
```

每个阶段对应四级进度模型中的一个 phase，不可跳过。

### 6.2 数据分层

| 层 | 职责 | 写入时机 | 消费者 |
|----|------|---------|--------|
| ODS（操作数据层） | 保存 Provider 原始响应，保留源字段 | fetch 阶段 | 排错、回填、审计 |
| DWD（明细数据层） | 统一字段、代码、时间口径 | normalize 阶段 | 质量校验、PIT 对齐 |
| DWS（汇总数据层） | 研究可用视图、PIT 对齐后的财务表、衍生指标 | quality + pit + derived 阶段 | 研究、Qlib 构建 |
| ADS（应用数据层） | 场景快照（研究快照、执行快照） | publish 阶段 | Research / Strategy / Trade |

**不可逆写回规则**：下游层的加工结果不得回写上游层。ODS 和 DWD 是权威层，只有 fetch 和 normalize 阶段有权写入。

### 6.3 默认每日调度

| 时段 | 阶段 | 说明 |
|------|------|------|
| 18:30–20:00 | 增量同步 | 日频行情、分钟频行情、财务、资金流 |
| 20:00–21:00 | PIT 对齐与标准化 | 含衍生指标计算 |
| 21:00–21:40 | 质量校验 | 关键项阻断 |
| 21:40–22:00 | Qlib 视图构建 + 快照发布 | 22:00 为默认快照截止时间（可配置） |

此顺序不可变：先同步 → 再校验 → 再构建 → 最后发布。否则会出现"快照已发布但质量未验证"的工程事故。

## 七、质量门禁

### 7.1 Fail-Closed 原则

质量门禁采用 Fail-Closed 策略（承接 00 §2.1 原则 P2）：任何关键规则失败即阻断，不发布快照、不进入研究流程、不允许向执行链路传递。

"阻断"不是报 warning，而是：

1. 不发布 snapshot_descriptor。
2. 将 run 状态标记为 blocked。
3. 写出 `data_health_report`，记录所有规则的检查结果。
4. 发射 `data.quality.blocked` 事件（强制通知）。

### 7.2 规则分类

质量规则分为三个级别：

| 级别 | 语义 | 失败后果 | 示例 |
|------|------|---------|------|
| **关键（critical）** | 数据完整性或正确性的底线 | 阻断流程 | 交易日历缺失当日记录；核心行情字段缺失超阈值；复权因子异常；PIT 校验未通过 |
| **警告（warning）** | 数据质量可疑但不致命 | 记录到 report，不阻断 | 个别品种缺失但不影响整体；字段分布偏移 |
| **信息（info）** | 统计性指标 | 仅记录 | 数据条数、覆盖率、延迟统计 |

### 7.3 quality_pack 概念

quality_pack 是一组预定义的质量规则集合。

当前产品口径里，它首先是 **Data 域内置的框架默认规则**，而不是 init 过程要求普通用户显式填写的配置项。
也就是说：

- 普通用户初始化 workspace 时，只需要关心 `provider`、`history_start`、是否启用自动更新等少量输入；
- `quality_pack` / `pit_pack` / `publish_pack` / `storage_pack` 由框架按默认能力补齐；
- 只有在后续确实出现多套规则并存的高级场景，才需要把这些 pack 作为显式 override 暴露出来。

不同 Profile 可使用不同的 quality_pack。例如：

- `cn_stock_daily`：使用完整的 A 股日频规则包（日历一致性、行情完整性、复权因子、PIT）。
- `cn_stock_1min`：使用分钟频规则包（含集合竞价窗口校验）。
- 自定义 Profile：可组合内置规则或扩展新规则。

quality_pack 的具体规则清单和配置格式详见 06；但对默认 Data 工作流而言，这些规则应保持“框架内置、用户少感知”的产品形态。

## 八、数据清洗与视图预处理

> 本章对应产品 §2.13"数据预处理"。注意：这里的"清洗"是面向 DWS 研究友好视图的预处理，**不是质量门禁的一部分**，也**不改变 ODS/DWD 权威层数据**。

### 8.1 预处理流水线

```
DWD 标准化数据 → 缺失值处理 → 极端值处理 → 标准化 → DWS 研究视图
```

| 步骤 | 默认方法 | 说明 |
|------|----------|------|
| 缺失值处理 | 行业中位数填充 | 缺失比例 > 50% 的标的标记排除 |
| 极端值处理 | MAD（中位数绝对偏差）3 倍截断 | 比 3σ 截断对偏态分布更稳健 |
| 标准化 | 截面 z-score | 每个截面内标准化为均值 0、标准差 1 |
| 行业/市值中性化 | 可选，默认关闭 | 回归剥离行业和对数市值影响 |

### 8.2 设计约束

1. 预处理参数有合理默认值，用户首次使用无需配置。
2. 高级用户可通过 Profile 或命令行参数覆盖（如 `--winsorize-method sigma`）。
3. 预处理参数记录在 run lineage 中——不同预处理参数视为不同评测。
4. 预处理结果写入 DWS 层，不得回写 ODS/DWD。

## 九、PIT 对齐

### 9.1 核心规则

PIT（Point-In-Time）对齐解决的核心问题：区分"财务值本身的日期"和"市场可以看到这个值的日期"。

**严格 PIT 规则**（承接 00 §2.1 原则 P4）：

1. 对每条财务记录计算 `effective_from`——该记录对研究/回测可见的最早日期。
2. 在 `effective_from` 之前，该记录不允许进入 DWS/ADS 层。
3. 策略在回测中也必须遵守此约束，不能使用未来信息。

### 9.2 公告日判定逻辑

财务数据的 `effective_from` 取决于公告日（ann_date）：

| 情况 | 判定 | 处理 |
|------|------|------|
| ann_date 存在且合法 | effective_from = 公告日次一交易日 | 正常写入 DWS |
| ann_date 缺失 | 无法确定可见时间 | **阻断**，写入 pit_report，不静默放行 |
| ann_date 与报告期冲突 | 公告日早于报告期截止日 | **阻断**，标记为数据源异常 |
| 同一标的同一报告期多条记录 | 以最新 ann_date 为准 | 记录覆盖逻辑到 pit_report |

### 9.3 产出

PITAligner 输出：

1. 附带 `effective_from` 的可研究财务表（写入 DWS）。
2. `pit_report`：记录对齐结果、阻断原因、覆盖记录。

## 十、快照发布与修正策略

### 10.1 snapshot_descriptor 契约

snapshot_descriptor 是 Data 域对外的核心产出物，描述"某个时点可供研究/回测/执行使用的统一数据版本"。

**必须包含的信息**：

1. `as_of`：快照对应的数据截止日期。
2. `snapshot_id`：全局唯一标识（格式见 02）。
3. 数据集清单：本次快照包含哪些数据集，每个数据集的覆盖范围。
4. 物理位置：权威层路径入口和 Qlib 视图路径入口。
5. 校验摘要：关键质量指标和 PIT 对齐状态。
6. lineage：run_id、profile、创建时间、vortex_version、config_hash。

> 字段级定义详见 06。

### 10.2 覆盖与不可变性

承接 00 §5.2 的快照版本语义（逻辑快照 + 物理修订模型）：

1. **同一 as_of 可重新 publish**：生成新的 `snapshot_id`（即新的 revision_id）。旧 revision 标记为 `superseded` 但**物理文件保留**，不删除。
2. **下游引用精确到 revision**：下游域在 lineage 中记录 `snapshot_revision_id`，指向具体的 revision。旧 revision 的数据始终可访问，确保历史 run 可严格复现。
3. **latest_revision_for(as_of)**：SnapshotResolver 默认返回最新 revision。显式指定 `--snapshot <revision_id>` 可锁定特定版本。
4. **清理机制**：`vortex data gc` 扫描全域 lineage，只有无下游引用的 superseded revision 才允许物理删除。

### 10.3 发布流程

publish 动作的内部流程：

1. QualityGate 对权威层数据执行再验证（不信任上次 update 的结果，因为中间可能发生了 backfill）。
2. DerivedMetricsBuilder 计算衍生指标（见第十一章）。
3. QlibViewBuilder 从权威层构建或增量更新 Qlib 研究视图。
4. SnapshotPublisher 汇总所有产出，写出 snapshot_descriptor。
5. 发射 `data.publish.completed` 事件。

### 10.4 迟到数据与修正策略

| 场景 | 处理方式 | 对已发布快照的影响 |
|------|---------|-------------------|
| 迟到数据（数据源延迟推送） | 下次 update 自动补入 | 已发布快照不受影响；新数据进入下一次快照 |
| 数据修正（数据源修改历史值） | 手动 backfill + 重新 publish | 生成新 revision_id；旧 revision 保留为 superseded，下游旧 run lineage 指向旧 revision 不变 |
| 财务数据修正稿 | backfill + PIT 重新对齐 + publish | 修正后的 ann_date 重新计算 effective_from |

## 十一、衍生指标计算

Data 域不仅采集原始数据，还负责计算一组**通用衍生指标**，供研究域和 Notification Watch Provider 消费。

### 11.1 指标分类

| 类别 | 指标 | 数据来源 | 计算频率 | 主要消费者 |
|------|------|---------|---------|-----------|
| **估值百分位** | PE/PB/PS 百分位排名（滚动历史窗口） | daily_basic + 历史估值 | 日频 | Watch Provider（估值日报）、Research |
| **市场广度** | 上涨/下跌比率、新高/新低数量、涨停/跌停统计 | bar_1d | 日频 | Research（择时因子）、Watch Provider |
| **流动性指标** | Amihud 非流动性、换手率分布、日均成交额 | bar_1d + daily_basic | 日频 | Research（流动性因子）、Strategy（容量约束） |

### 11.2 计算时机

衍生指标由 DerivedMetricsBuilder 在 publish 流程中计算（见 §10.3 第 2 步）。计算结果写入 DWS 层的 `derived/` 目录，随 snapshot 一起发布。

### 11.3 设计约束

1. 衍生指标是确定性计算：相同输入 + 相同参数 = 相同输出。
2. 计算参数（如百分位滚动窗口长度）通过 Profile 配置。
3. 衍生指标不进入质量门禁的阻断判定——门禁只管原始数据质量。
4. 衍生指标的口径定义和计算公式详见 06。

## 十二、日历服务

### 12.1 所有权

交易日历由 Data 域通过 DataProvider 获取并缓存。Data 域是日历数据的唯一 owner。

### 12.2 跨域消费

其他域通过共享契约层的 CalendarService 访问日历（详见 02）：

- **Research**：确定因子计算的时间范围和截面日期。
- **Strategy**：确定调仓日、回测时间轴。
- **Trade**：确定交易窗口、判断是否交易日。

### 12.3 设计要点

1. 默认 A 股日历（来自 Tushare `trade_cal` 接口）。
2. 多市场通过 Profile 配置扩展，每个市场独立日历。
3. 日历缓存在本地，避免每次查询都请求 Provider。
4. 日历数据参与质量门禁：缺失当日交易日记录属于关键规则失败（见 §7.2）。
5. 日历过期或不可用作为一类独立的故障模式（见 §16.5）。

## 十三、重试与故障恢复

### 13.1 重试策略

对齐产品 §1.3"异常自动恢复"的设计：

```
首次失败 → 等待 5 分钟  → 第 1 次重试
第 1 次失败 → 等待 15 分钟 → 第 2 次重试
第 2 次失败 → 等待 30 分钟 → 第 3 次重试
第 3 次失败 → 停止 → 发射 data.sync.failed 事件（强制通知）
```

**规则**：

1. 最多重试 3 次，间隔递增（指数退避）。
2. 任一次重试成功则继续正常流程，不产生通知。
3. 3 次全部失败才触发 critical 通知，附带全部错误信息。
4. **仅针对可恢复错误重试**：网络超时、数据源临时不可用、空响应。
5. **不可恢复错误不重试**：配置错误、认证失败、Provider 未注册——直接通知。

> **当前实现映射（Phase 1）**：run 级自动恢复由 Data 后台 worker 内建执行，而不是依赖外部巡检脚本。worker 在单次 run 收尾为 `partial_success / failed` 后，会先按错误类型判断是否属于可恢复错误；若是，则按 `5min → 15min → 30min` 发起新的 attempt run。由于分区 coverage 与已落盘数据都会保留，后续 attempt 只会补未完成缺口，不会把已经安全落盘的分区整表重抓。

### 13.2 failed_units

当 run 中某些 chunk 失败时，ProgressTracker 自动写出 `failed_units.json`，记录每个失败单元的 dataset、partition 和错误类型。

> failed_units.json 的结构定义见 02（共享产物规范）。

### 13.3 repair 机制

`repair` 命令读取 failed_units，仅重跑失败分区，避免全量重算：

1. 定位失败单元：通过 `--run-id` 参数或自动选取最近一次失败的 run。
2. 重跑失败分区：对每个失败 chunk 重新执行 fetch → normalize → quality → pit。
3. 更新进度：成功的 chunk 标记为 done，仍然失败的 chunk 保留在新的 failed_units 中。
4. repair 成功后不自动 publish——用户需显式执行 `publish` 确认。

repair 的 run 关联 `parent_run_id`，形成恢复链，便于审计。

## 十四、通知集成

Data 域通过事件发射（emit）与 Notification 域集成。Data 域只负责"在正确的时机发出事件"，路由、渠道、投递由 Notification 域处理。

> **当前实现映射（Phase 1）**：最小通知链路已经落到主代码：Data worker 在 run 收敛后发出事件，NotificationService 负责写入控制面 `notification_log`，并通过 Feishu Bot 发送消息。外部 `feishuDirect` 诊断工具与运行时共用同一组 `FEISHU_*` 环境变量，但它不是产品主链路的一部分。

### 14.1 事件触发矩阵

| 事件 | 触发时机 | 通知级别 | 说明 |
|------|---------|---------|------|
| `data.sync.completed` | update / bootstrap 全部成功 | info（默认静默） | 附带品种数、数据条数、耗时 |
| `data.sync.failed` | 3 次重试全部失败 | **critical（强制通知）** | 附带全部重试记录、影响范围、建议操作 |
| `data.quality.blocked` | 质量门禁阻断 | **critical（强制通知）** | 附带失败规则、影响品种数、建议操作 |
| `data.publish.completed` | 快照发布成功 | info（默认静默） | 附带 snapshot_id、数据集清单 |
| `data.sync.partial_failed` | 部分 dataset 失败，run 收尾为 partial_success | warning | 附带失败 dataset 列表、原因分类、是否仍需人工介入 |

### 14.2 通知级别语义

承接 00 §7.2：

- **critical**：强制推送，不可关闭。系统有问题必须知道。
- **warning**：默认推送，用户可关闭。需要关注但不紧急。
- **info**：默认静默，用户可开启。一切正常时无需打扰。

### 14.3 术语统一

统一使用 `notification` 字段名。DataProfile 中的 `notification: dict` 配置块详见 02 Profile 规范。

## 十六、域故障模式

本节列出 Data 域的主要故障模式，包括触发条件、系统行为、用户操作和通知级别。

### 16.1 Provider 超时 / 不可用

| 项 | 说明 |
|----|------|
| **触发** | DataProvider 请求超时、返回空数据、HTTP 错误、认证失败 |
| **系统行为** | 可恢复错误（超时、空响应）→ 3 次指数退避重试；不可恢复错误（认证、配置）→ 直接记录失败 |
| **失败范围** | chunk 级——仅受影响的数据集分区标记为 failed |
| **用户操作** | 确认数据源状态 → `repair` 重跑失败分区，或 `update --retry` |
| **通知级别** | 重试全部失败 → critical；部分品种失败 → warning |

### 16.2 质量阻断

| 项 | 说明 |
|----|------|
| **触发** | 关键质量规则失败（日历缺失、行情字段大面积缺失、复权因子异常） |
| **系统行为** | Fail-Closed：停止后续 publish，写 data_health_report，发射 `data.quality.blocked` |
| **失败范围** | run 级——整个 publish 被阻断（即使只有一个数据集触发关键失败） |
| **用户操作** | 查看 health_report 定位原因 → `backfill` 补数据或修正配置 → 重新 `publish` |
| **通知级别** | critical（强制通知） |

### 16.3 PIT 对齐失败

| 项 | 说明 |
|----|------|
| **触发** | 财务数据缺少 ann_date、ann_date 与报告期冲突、披露日历不完整 |
| **系统行为** | 受影响的财务记录被阻断（不进入 DWS），写 pit_report |
| **失败范围** | dataset 级——仅 PIT 相关数据集受影响；行情数据集不受影响 |
| **用户操作** | 查看 pit_report → 确认数据源是否有修正 → `backfill` 补充披露日历 → 重新 `publish` |
| **通知级别** | 部分品种 → warning；大面积（>10%）→ critical |

### 16.4 存储满 / 写入失败

| 项 | 说明 |
|----|------|
| **触发** | 磁盘空间不足、文件系统权限错误、Parquet 写入异常 |
| **系统行为** | chunk 写入失败 → 记录到 failed_units；bootstrap 前应执行磁盘空间预检查 |
| **失败范围** | chunk 级 |
| **用户操作** | 清理磁盘空间或修复权限 → `repair` |
| **通知级别** | critical |

### 16.5 日历不可用 / 过期

| 项 | 说明 |
|----|------|
| **触发** | Provider 无法返回交易日历、日历缓存过期且无法刷新 |
| **系统行为** | 日历是所有后续校验的时间轴基准——日历缺失导致整个 run 无法启动 |
| **失败范围** | run 级——整个 run 被阻断 |
| **用户操作** | 确认 Provider 日历接口状态 → 手动刷新日历缓存或切换 Provider |
| **通知级别** | critical |

## 附录 A：可替换设计

Data 域的两个核心接入点通过协议解耦，支持替换：

### A.1 替换数据源

1. 实现新的 DataProvider（满足 §4 契约不变量）。
2. 在 provider_registry 注册。
3. Profile 中切换 provider 名称。

**兼容性约束**：新 Provider 必须返回可映射到 canonical 主键的数据，PIT 语义不变，并能接入默认质量规则链路；不应要求最终用户为接入新 Provider 额外理解或手填 quality_pack。

### A.2 替换存储后端

1. 实现新的 StorageBackend（满足 §5 能力语义）。
2. 在 storage_registry 注册。
3. 适配 snapshot 读取路径。

**兼容性约束**：upsert 幂等、snapshot_descriptor 契约不变、Fail-Closed 语义不变。
