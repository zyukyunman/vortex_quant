> **版本**：v1.0

## 1. 目标

从零开始、按域分阶段实现 Vortex 量化研究平台的**全部框架功能**。产品设计（docs/产品原型/v0.2/）和软件架构设计（docs/架构设计/00-08）中提到的每一项能力都必须落地。

### 1.1 Scope 边界

**在 scope 内的框架基础设施**（虽是具体实现，但属于框架而非用户策略实例）：

| 组件 | 说明 |
|------|------|
| `TushareProvider` | 数据适配器，唯一内置的 DataProvider 实现 |
| `PaperGateway` | 模拟交易网关 |
| 内置 preset / pack | 配置预设包（quality / pit / publish / storage） |
| HTML 报告模板 | 因子评测报告 7 页 + 回测报告 6 页 |
| 内置 Watch Provider | data_health / valuation_percentile / strategy_analysis / factor_monitor |

**不在 scope 内**：

- 具体因子实现（momentum、fcf_yield、low_vol 等）——用户通过 `vortex research register` 注册自己的因子
- 具体策略流水线实例——用户通过 StrategyProfile 配置自己的管线
- MiniQMT 真实网关（仅留 Gateway Protocol + placeholder）
- AI 辅助功能（因子自动分类、regime 标注——标记为 optional/deferred）

## 2. 开发节奏

采用渐进式闭环策略：每完成一个步骤即可形成可用闭环，逐步扩大能力范围。

```
Step 1: Phase 0 + 1A  ──→  平台骨架 + 数据采集可用
Step 2: Phase 1B + 2   ──→  数据发布 + 因子研究闭环
Step 3: Phase 5        ──→  通知接入，形成 数据→研究→通知 闭环
Step 4: Phase 3A       ──→  策略核心（管线 + 回测 + 评分）
Step 5: Phase 3B       ──→  策略报告 + 生命周期管理
Step 6: Phase 4        ──→  交易域（模拟/实盘），最终闭环
```

**全程贯穿**：
- **Phase 6（集成测试 + 用户手册）**：每个步骤完成后更新用户手册对应章节，补充端到端测试
- **Phase 5（通知）**：Step 3 搭建后，后续每个域的事件通知持续接入

### 2.1 步骤依赖图

```
Step 1: [Phase 0] → [Phase 1A]
           ↓
Step 2: [Phase 1B] → [Phase 2]
           ↓
Step 3: [Phase 5A] → [Phase 5B]     ← 数据→研究→通知 闭环
           ↓
Step 4: [Phase 3A]                   ← 策略核心
           ↓
Step 5: [Phase 3B]                   ← 策略报告 + 生命周期
           ↓
Step 6: [Phase 4]                    ← 交易域，全平台闭环
```

## 3. Phase 0：平台运行时与控制面

**目标**：搭建所有域共用的基础设施，使后续每个域可以独立开发并接入。

### 3.1 项目骨架

- `pyproject.toml` 依赖确认 + `vortex/` 包初始化
- 模块布局：`vortex/{config,data,research,strategy,trade,notification,runtime,shared}/`
- 日志框架（structured logging, `--verbose` flag）
- 错误码体系（`{DOMAIN}_{CATEGORY}_{DETAIL}` 格式，参见 06 §1.5）
- 确定性约束：
  - 排序规则：date ASC, symbol ASC
  - 精度约定：参见 06 §1.2 精度表
  - 时区：Asia/Shanghai (UTC+8)，参见 06 §1.3

### 3.2 工作区与状态管理

- `vortex init` 命令：创建 workspace 目录结构（参见 06 §8.5 / 00 §6.3）
- 控制面 SQLite 初始化（`state/control.db`）：所有域共用 DDL 基表
- 数据面目录结构：`data/` `research/` `strategy/` `trade/` 分离（参见 00 §6.2 控制面/数据面分离）
- `run_id` 生成规则：`{domain}_{date}_{time}_{random4}`
- Lineage 基础字段：`parent_run_id` / `created_at` / `vortex_version`

### 3.3 配置/Profile 系统

- Profile 基础模型：`BaseProfile` + 4 种子类（DataProfile / ResearchProfile / StrategyProfile / TradeProfile）
- 五阶段管线：Store → Loader → Merger → Validator → Resolver（参见 02 §三-七）
- Preset 系统：内置技术方案包（参见 02 §三.2）
- Pack 系统：可插拔规则包（quality / pit / publish / storage，参见 02 §三.3）
- Profile 继承：`extends` 字段（参见 02 §八.1）
- CLI 命令：`vortex profile explain` / `vortex profile resolve`
- ResolvedProfile 输出：`.vortex/resolved/{name}.resolved.yaml` + 来源标注

### 3.4 服务运行时

- Server 进程：`vortex server start` / `stop` / `status`
- CLI dispatcher → Server RPC 调用链（参见 00 §4 服务化架构）
- 全局任务队列 + 并发控制（`max_concurrent_tasks`，参见 07 §2.8）
- 任务状态机：`pending → running → success / failed / cancelled`
- 任务进度追踪：run / stage / dataset / chunk 四级（参见 06 §3.8）
- 任务取消协议：cancel 信号 + graceful shutdown

### 3.5 事件总线与调度

- 事件 schema：`event_type` / `domain` / `level` / `payload` / `timestamp`
- 事件发布/订阅机制（进程内 bus）
- 调度器框架（cron-like schedule 配置，支持 data auto-run）
- 审计日志基表

### 3.6 跨域交换对象

- `SnapshotDescriptor`（Data → Research/Strategy/Trade，参见 06 §7.1）
- `SignalSnapshotDescriptor`（Research → Strategy，参见 06 §7.2）
- `TargetPortfolio`（Strategy → Trade，参见 06 §7.3）
- `ExecutionReport`（Trade → Audit/Notification）
- 共享工具：交易日历服务、rounding helpers、timezone helpers

## 4. Phase 1A：数据采集与存储层

**目标**：实现数据从外部源到本地存储的完整链路。

### 4.1 Provider 抽象 + TushareProvider

- `DataProvider` Protocol（参见 01 §2 / 06 §3）：
  - `fetch_instruments()` / `fetch_calendar()` / `fetch_bars()` / `fetch_fundamental()` / `fetch_events()`
  - 返回排序约束：date ASC, symbol ASC
- `TushareProvider` 实现：行情 / 财务 / 事件 / 日历
- 分段拉取 + 限流 + 重试（指数退避 5m → 15m → 30m，参见 07 §1.3）

### 4.2 存储层（ODS → DWD → DWS → ADS）

- `StorageBackend` Protocol（参见 06 §3）：`upsert()` / `read()` / `list_partitions()` / `snapshot()`
- Parquet 分区存储（按日期分区）
- DuckDB catalog 维护（查询加速层）
- 四层存储契约（参见 01 §3）：
  - **ODS**（原始层）：Provider 返回的原始数据保留
  - **DWD**（清洗层）：标准化字段、去重、类型转换
  - **DWS**（研究层）：PIT 对齐、衍生指标
  - **ADS**（发布层）：快照冻结、对外交换

### 4.3 数据管线

- 管线进度追踪：复用 Phase 0 四级进度（run / stage / dataset / chunk）

### 4.4 质量门禁

- 质量规则引擎（critical / warning 分级，参见 01 §4）
- 内置规则：

| 规则 | 级别 | 说明 |

## 6. Phase 2：研究域

**目标**：实现因子注册、评测引擎、分类型评测、HTML 报告、信号发布的完整研究闭环。

### 6.1 因子注册与管理

- `FactorRegistry`：register / unregister / retype / list-factors（参见 03 §2）
- `Factor` Protocol：`name` / `category` / `direction` / `compute(data) -> Series`
- 8 种因子类型：value / momentum / quality / volatility / liquidity / technical / alternative / composite
- 因子缓存 + `code_hash` 自动失效
- `factor_archive`：评测历史（append-only）
- 去重机制：`eval_key = (factor_name, snapshot, market, label_spec, code_hash)`

### 6.2 评测引擎（10 维通用指标 L1）

参见 03 §4.1-4.6 / 06 §4.3：

| 维度 | 指标 |
|------|------|
| 预测力 | IC / RankIC / ICIR |
| 信号持续性 | IC 衰减曲线（1d / 5d / 20d） |
| 盈利性 | 分组收益（5组/10组）/ 多空年化收益 |
| 覆盖率 | 因子覆盖率 |
| 稳定性 | 因子自相关 |
| 独立性 | 残差 IC（剔除已知因子后） |
| 分布 | 截面分布统计 |

### 6.3 分类型评测（L2-L4）

参见 03 §4.8 / 06 §4.4：

- **L2 CategoryMetric**：按因子类型有条件计算（10 个 category metric）
- **L3 ScoringPolicy**：百分位排名 + 加权评分（5 维度 × 8 类型权重表）
- **L4 EvalEmphasis**：每种因子类型的展示侧重配置（sidebar / highlighted / hints / panels）
- 评分等级：A ≥ 85 / B+ ≥ 75 / B ≥ 70 / B- ≥ 60 / C ≥ 50 / D ≥ 35 / F < 35

### 6.4 HTML 因子评测报告（7 页）

对齐产品原型 `factor_tear_sheet.html`，payload 遵循 06 §4.8，渲染契约遵循 06 §4.9：

| 页面 | 内容 |
|------|------|
| IC 分析 | IC 时序 + 衰减曲线 + 累计 IC |
| 分组收益 | 分组净值 + 多空 + 年度分层 |
| 截面分布 | 稳定性热力图 + 分布变化 |
| 截面快照 | 时间滑块回放 + 散点图 + 回归线 |
| 相关性矩阵 | 因子间相关 + 增量 IC + 市值相关 |
| Regime 分析 | 分市场环境 IC + 切换期表现 |
| 容量分析 | 分市值 IC + 换手/成本 + 敏感度 |

### 6.5 信号发布

- `publish-signal`：实验 → 不可变信号快照（参见 03 §4.6）
- `signal_id` 格式：`{factor_family}_{variant}_{version}`
- **不可变规则**：发布后永远不可修改/覆盖，修复必须新建 signal_id
- **硬约束**：model_train 不能直接 publish-signal，必须经过 factor_eval

### 6.6 模型训练工作流

- Qlib 适配层（ModelTrainer，参见 03 §3）
- 训练/验证分离 + 过拟合检测
- 特征重要性报告
- 输出 `pred.pkl` → 走 factor_eval 管线

### 6.7 研究域 CLI 完整命令

```

vortex research unregister     --name <name>
vortex research retype         --factor <name> --factor-type <type>
vortex research list-factors   [--status active|inactive]
vortex research archive        --factor <name> [--limit N]
vortex research run            --profile <name> --snapshot <id> --factor <name>
vortex research inspect        --run-id <id>
```

## 8. Phase 3A：策略核心

### 8.1 四阶段管线

- 阶段独立测试（`--stage` + `--dry-run`）

### 8.2 回测引擎

- `BacktestEngine` 适配层（可插拔后端，参见 06 §5.3）
- 成本模型：`CnStockDefaultCost`（3bps 佣金 + 1‰ 印花税）
- 换仓频率配置（daily / weekly / monthly）
- 换手率计算（年化双边）

### 8.3 多维度回测

参见 04 §5 / 07 §4.4：

| 维度 | 目的 | 关键参数 |

| Walk-Forward | 过拟合检验 | `train=252d`, `test=63d`, `step=63d` |
| Regime | 市场环境拆分 | MA120 判断牛/熊/震荡 |

### 8.4 双基准系统

### 8.5 质量评分

| 过拟合风险 | 15% | Full vs Random vs WF degradation |

等级：A ≥ 85 / B+ ≥ 75 / B ≥ 70 / B- ≥ 60 / C ≥ 50 / D ≥ 35 / F < 35

## 10. Phase 4：交易域

**目标**：实现模拟/实盘交易的完整链路，这是全平台最终闭环。

### 10.1 网关抽象

- `Gateway` Protocol（参见 05 §3 / 06 §6）：
  - `connect()` / `submit_order()` / `cancel_order()` / `query_positions()` / `query_orders()` / `on_fill()`
- `PaperGateway` 实现：FIFO 撮合、涨跌停限制、停牌拒绝
- `MiniQMTGateway` placeholder（仅接口定义）

### 10.2 订单生命周期

- 状态机：`created → submitted → [partial_fill / filled / rejected / timeout / cancelled]`
- 终态：filled / rejected / timeout / cancelled
- `OrderIntent` 控制面持久化（先写 intent 再提交）
- `client_order_id`：`{exec_id}_{seq}` 确定性生成
- 幂等性：canonical hash 指纹冲突检测

### 10.3 执行引擎

- 订单计划生成（目标组合 diff 当前持仓）
- 提交 → 监控 → 成交回报处理
- 超时/拒绝处理

### 10.4 三层风控

参见 05 §4：

| 层 | 触发时机 | 规则示例 | 处置 |
|----|----------|----------|------|
| Pre-Trade | 提交前 | 集中度/涨跌停/ST | **Fail-Closed** |
| Intra-Trade | 每笔成交 | 滑点异常/成交量异常 | 取消剩余 |
| Post-Trade | 日终 | 持仓偏差/价格偏差 | 日志 + 通知 |

- 风控规则 YAML 配置（`RiskRule` Protocol）

### 10.5 对账

- 三组比对（参见 05 §5）：
  1. actual（网关）vs target（组合）
  2. target vs fills（成交日志）
  3. fills vs actual（一致性）
- 阈值：< 1% 记录 / 1-3% 告警 / > 3% 异常

### 10.6 交易域 CLI 完整命令

```

vortex trade live-rebalance   --profile <name> --target <portfolio_id> --confirm
vortex trade inspect          --run-id <id>
vortex trade retry            --run-id <id>
vortex trade reconcile        --profile <name> --date <date>
vortex trade cancel           --run-id <id>
vortex trade status           --profile <name>
```
