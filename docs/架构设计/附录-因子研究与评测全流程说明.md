# 附录 - 因子研究与评测全流程说明

> **定位**：本文档解释“一个因子从被提出，到得到是否值得纳入策略的结论”为止的完整研究流程。  
> **范围**：只讨论因子构建、预处理、评测、归档与准入，不讨论数据抓取和数据发布。

---

## 1. 为什么要单独把这条链路讲清楚

很多量化系统容易把“因子研究”压缩成一句话：**算 IC、跑分组、看报告**。  
但真正可靠的因子研究，至少要回答下面几类问题：

1. 这个因子到底是怎么构建出来的？
2. 进入评测前，是否做了正确的预处理？
3. 它对哪个 horizon（预测周期）真正有效？
4. 统计显著能不能转成交易可用？
5. 它在不同市场环境下是否稳定？
6. 它是不是只是在重复已有因子的信息？

如果这些问题没有被系统化回答，最终很容易把“看起来有收益的列”误当成“可上线的因子”。

---

## 2. 因子研究栈的分层理解

下面这张表把你提到的几类工具放到 Research 体系里的正确位置。

| 层 | 典型工具 | 在 Vortex 中的定位 |
|----|---------|-------------------|
| **因子构建层（Feature Engineering）** | `TA-Lib Python`、pandas 因子生态（如 `pandas-ta / pandas-factor` 类库） | 提供技术指标和便捷算子，但不直接等于产品主契约 |
| **因子评测层（Factor Evaluation）** | `Alphalens` | 提供 IC、分组收益、换手、tear sheet 的方法参考与口径对照 |
| **风险指标层（Risk Metrics）** | `Empyrical` | 提供 Sharpe、drawdown、alpha/beta 等成熟风险指标原语 |
| **一体化工作流层（Workflow / Data / Model / Backtest）** | `Qlib` | 提供 data handler、workflow、model、signal analysis、backtest 的一体化平台能力 |
| **组合报告层（Portfolio Analytics）** | `Pyfolio` | 更适合组合/回测收益分析，不应作为 Research 第一期核心依赖 |

这背后的设计取舍是：

- **TA-Lib / pandas 因子生态**负责“怎么构造因子”
- **Alphalens**负责“怎么评价一个因子像不像 alpha”
- **Empyrical**负责“怎么衡量收益风险质量”
- **Qlib**负责“怎么把数据、模型、评测、回测串成 workflow”

所以 Vortex 不能只要其中一个，而是要把这些层拼成一条完整链路。

---

## 3. 全流程总览

```
已发布 snapshot / 已注册字段 / 公式 / 代码
  → 因子定义与注册
  → 因子构建（feature engineering）
  → 研究样本过滤
  → 因子预处理
  → forward return / label 对齐
  → 单因子有效性评估
  → 分层回测
  → 风险调整收益分析
  → 稳定性与鲁棒性检验
  → 正交性 / 独立性检验
  → 准入、归档、发布
```

每一段都在回答不同问题：

| 阶段 | 核心问题 |
|------|---------|
| 因子构建 | 这列因子值到底怎么来的？ |
| 预处理 | 输入是否可比较？ |
| 单因子有效性 | 它有没有预测力？ |
| 分层回测 | 它有没有交易价值？ |
| 风险调整收益 | 收益质量是否足够好？ |
| 稳定性 / 鲁棒性 | 它是不是只在某一段时间碰巧有效？ |
| 正交性 | 它是不是只是重复已有因子？ |
| 准入 / 归档 | 它是否值得进入后续策略环节？ |

---

## 4. 阶段一：先定义研究问题，再定义因子

真正的因子研究不应从“先写 RSI”开始，而应先定义：

1. **这个因子想解释什么现象**  
   比如估值修复、趋势延续、反转、流动性约束、聪明钱、质量溢价。
2. **它的预测周期是什么**  
   是 1d、5d、20d 还是 120d？
3. **它的调仓周期是什么**  
   这是月度、季度还是半年策略？
4. **它预计属于什么因子类型**  
   价值、动量、质量、技术、另类、复合？

这里最容易混淆的一点是：

- **预测周期 ≠ 调仓周期**

一个因子可能对未来 5 个交易日有效，但你未必应该把它直接拿去支撑半年调仓策略。  
所以后续评测一定要做**多周期**，而不是只看一个 horizon。

---

## 5. 阶段二：因子构建（Feature Engineering）

### 5.1 三类因子来源

Vortex 里应支持三类来源：

| 来源 | 说明 | 适合场景 |
|------|------|---------|
| 字段因子 | 直接引用某个 dataset / column | 已经有现成字段，只需评估其 alpha 能力 |
| 公式因子 | 用 operator 词表表达字段变换 | 大多数标准化研究入口 |
| 代码因子 | 用 Python 写完整逻辑 | 复杂、实验性或多表拼装因子 |

### 5.2 因子构建层真正做什么

这一层其实可以比“产品化能力”更简单：  
**它不需要单独暴露成一个产品动作，也不需要独立 CLI。**  
对用户来说，真正暴露的仍然只有：

- `register`：记录因子定义
- `run`：在运行评测时自动触发构建

也就是说，因子构建是 `vortex research run` 内部的**第一个执行阶段**，不是新的产品入口。

最小可行的构建方式应收敛成三种：

| 方式 | 什么时候用 | 说明 |
|------|-----------|------|
| **Data API 读取** | 单表字段提取、少量字段组合 | 默认优先，最简单、最可控 |
| **SQL 读取** | 多表 join、聚合、窗口函数、筛选 | 对复杂拼接更直接 |
| **Python 计算** | 公式难以表达的自定义逻辑 | 作为扩展口，而不是默认主路 |

典型构建动作仍然是这些：

- 时序变换：`lag / delta / ts_mean / ts_std / ts_rank`
- 截面变换：`rank / zscore / winsorize`
- 技术指标：`rsi / macd / atr / bbands / obv`
- 组合算子：`log / clip / add / sub / mul / div / where`

但这些动作的执行方式，不必都包装成产品级调用；更合理的是：

1. **先用 Data API 或 SQL 把基础数据读出来**
2. **再在 Research 内部做公式执行或 Python 计算**
3. **产出原始因子值 raw_factor**

### 5.3 推荐实现分层与代码落位

这层的关键不是“暴露多少命令”，而是**内部代码分层要清楚**。建议按下面的结构落地：

| 层 | 建议实现位置 | 作用 |
|----|-------------|------|
| 运行编排层 | `vortex/research/service.py` | `run` 动作入口，编排构建 → 预处理 → 评测 |
| 构建编排层 | `vortex/research/build/service.py` | 决定本次构建走 API、SQL 还是 Python |
| Data API 读取层 | `vortex/research/build/data_api.py` | 按 snapshot / dataset / column 读取研究数据 |
| SQL 读取层 | `vortex/research/build/sql_reader.py` | 在 DuckDB / 研究视图上执行 SQL，做 join/聚合 |
| 公式执行层 | `vortex/research/build/formula_engine.py` | 解析公式，调 operator backend 计算 |
| Python 因子加载层 | `vortex/research/build/python_loader.py` | 加载用户代码因子，执行 `compute()` |
| 构建产物层 | `vortex/research/build/manifest.py` | 生成 `factor_build_manifest.json` |
| 预处理层 | `vortex/research/preprocess/pipeline.py` | 样本过滤、去极值、标准化、中性化 |

推荐调用链：

```
vortex research run
  → ResearchService.run_factor_eval(...)
  → FactorBuildService.build(...)
  → SnapshotResolver.resolve(snapshot_id)
  → 选择读取方式：
       - DataAPIReader.read(dataset, columns)
       - SQLReader.query(sql)
       - PythonFactorLoader.compute(...)
  → FormulaEngine / Python loader 产出 raw_factor
  → PreprocessPipeline.run(raw_factor, context)
  → EvalKernel.evaluate(processed_factor, labels)
```

这里的重点是：

- **构建层是研究域内部实现，不单独对用户暴露**
- **Data API / SQL 是默认取数手段**
- **Python 是扩展口，不是主路径**

### 5.4 构建阶段必须沉淀的元数据

每次构建后的 manifest 至少应记录：

- 因子来源类型
- 依赖字段 / 数据集
- 公式文本或代码 hash
- operator backend
- 参数和 lookback window
- 原始覆盖率与缺失比例
- 因子方向（高值更好 / 低值更好 / 未知）

没有这份 manifest，后面就谈不上可复现、可审计。

---

## 6. 阶段三：研究样本过滤

因子预处理之前，必须先把**研究样本（research sample）**定义清楚。  
这一步不是数据抓取，也不是作弊，而是保证横截面比较的前提一致。

默认建议至少做：

1. 剔除 ST/PT、退市整理等异常状态样本
2. 剔除上市未满约 1 年的样本
3. 剔除停牌、无法买入或没有未来收益标签的样本
4. 仅保留本次 snapshot / market / universe 内的样本

这一步回答的是：

- 今天这个截面里，哪些股票本来就不该拿来比较？

---

## 7. 阶段四：因子预处理（Factor Preprocessing）

### 7.1 为什么预处理是核心能力

因子评测的对象不是“任意一列数”，而是**可比较的因子截面**。  
预处理的目标不是“把结果调好看”，而是让输入在统计上更稳健、在横截面上更可比。

### 7.2 推荐默认顺序

```
原始因子值
  → 缺失值处理
  → 去极值（MAD 默认，3σ 可选）
  → 初次标准化（z-score）
  → 行业中性化（默认开启）
  → 市值中性化（可选）
  → 最终标准化（z-score）
  → 可用于评测的 processed factor
```

### 7.3 关键步骤说明

| 步骤 | 推荐做法 | 为什么 |
|------|---------|-------|
| 去极值 | **MAD** 默认，`3σ` 作为可选 | 偏态分布下更稳健 |
| 标准化 | 截面 `z-score` | 让不同量纲可比较 |
| 行业中性化 | 回归残差，**默认开启** | 行业暴露是最常见的伪 alpha 来源 |
| 市值中性化 | 可选 | 不是所有因子都该去掉市值暴露，尤其 `size` 因子本身不能默认做这个步骤 |

### 7.4 一个重要边界

- **行业中性化**更接近通用默认项
- **市值中性化**更接近研究选项

也就是说：

- 行业中性化回答“这个因子是不是只是行业轮动”
- 市值中性化回答“这个因子是不是只是大/小盘暴露”

后者很重要，但不应该无脑强制到所有因子上。

---

## 8. 阶段五：forward return / label 对齐

进入正式评测前，系统还要把因子值和未来收益标签对齐。  
这一步要严格避免前视偏差（look-ahead bias）。

### 8.1 多周期 horizon

对同一个因子，至少应支持多组 forward returns：

- 短线：`1d / 5d / 10d / 20d`
- 中线：`20d / 40d / 60d`
- 长线：`60d / 120d / 250d`

### 8.2 horizon 选择建议

| 场景 | 推荐 horizon |
|------|-------------|
| 高频 / 微观结构 / 情绪 | `1d, 5d, 10d, 20d` |
| 月度调仓 | `5d, 20d, 40d` |
| 季度调仓 | `20d, 60d, 120d` |
| 半年调仓 | `20d, 60d, 120d, 250d` |
| 年度 / 超低换手 | `60d, 120d, 250d` |

---

## 9. 阶段六：单因子有效性评估

### 9.1 第一层：多周期 RankIC

单因子评测的第一层，是看它对未来收益是否存在稳定的排序能力。  
默认以 **Spearman RankIC** 为核心：

- `IC_1d`
- `IC_5d`
- `IC_20d`
- 按需要扩展到 `IC_60d / IC_120d / IC_250d`

为什么优先用 Spearman：

- 对极端值更稳健
- 更符合“排序型选股因子”的本质
- 不要求线性，只要求单调

### 9.2 每个 horizon 至少要输出的统计量

- `IC_mean`
- `IC_std`
- `ICIR`
- `positive_rate`
- `t-stat / 显著性`（作为辅助统计量）
- `IC decay`（衰减曲线）

### 9.3 这一步回答什么

- 哪个 horizon 最强？
- 因子方向是什么？
- 这个因子的预测力是短命还是持久？

---

## 10. 阶段七：分层回测（Stratified Backtest）

只看 IC 还不够，因为**统计相关性不一定等于交易可用性**。  
所以第二层必须做分层回测：

1. 按因子值做 5 组或 10 组
2. 观察分组收益是否单调
3. 观察 top 组和 bottom 组能否形成可解释的收益差

至少要输出：

- 分组累计收益
- 分组年化收益
- 多空收益
- 单调性得分
- 换手率
- 简单成本估算

如果一个因子：

- IC 看起来不错
- 但分组收益不单调
- 或多空收益为负

那它通常不应被视为可用因子。

---

## 11. 阶段八：风险调整收益

分层回测之后，还要回答一个问题：  
**这些收益是“好收益”吗，还是高波动、深回撤硬换来的？**

这一步建议使用 `Empyrical` 一类轻量风险指标库，输出：

- Sharpe ratio
- Information ratio
- Max drawdown
- Alpha / Beta
- 下行风险相关指标（按需要扩展）

这一层的作用是把“有收益”进一步分成：

- 收益质量高
- 收益质量一般
- 收益质量差但看起来暴利

---

## 12. 阶段九：稳定性与鲁棒性

一个只在单一时期、单一市场风格、单一参数下有效的因子，往往不值得进入策略。

### 12.1 必看的稳定性问题

1. **子区间稳定性**：不同年份、牛熊震荡是否都有效？
2. **容量稳定性**：大盘/小盘、流动性分层后是否仍有效？
3. **衰减稳定性**：从 1d 到 20d，再到 120d 是否平滑衰减，而不是突然失效？
4. **覆盖稳定性**：覆盖率是否随着时间明显下滑？
5. **参数鲁棒性**：窗口从 20 改 18 或 22，结论是否完全翻转？
6. **样本外衰减**：对复合 / ML 因子，样本外表现是否明显塌缩？

### 12.2 这一步的判断标准

鲁棒性不是要求“所有环境都最强”，而是要求：

- 逻辑可解释
- 失效方式可理解
- 不依赖单一偶然区间

---

## 13. 阶段十：正交性 / 独立性

这是很多系统容易漏掉，但实际非常关键的一步。

### 13.1 为什么不能只看 IC

一个新因子就算有正 IC，也可能只是：

- 换了一种写法的市值因子
- 换了一种包装的动量因子
- 某个已有核心因子的高相关复制品

所以**正交性不是“锦上添花”，而是准入前的必要检查**。

### 13.2 至少要做三件事

| 检查项 | 说明 |
|--------|------|
| 原始相关矩阵 | 看它与核心因子库（value / momentum / volatility / size 等）的横截面相关性 |
| 残差 IC | 回归剥离行业、市值、核心因子后，再看 residual factor 的 IC |
| 增量贡献 | 放进已有因子组合后，看是否提升 IC、分组收益或策略表现 |

### 13.3 一个重要边界

**正交化 ≠ 默认预处理。**

默认预处理是为了让输入可比较；  
正交化分析是为了判断“它有没有新信息”。

只有当研究员明确要构造残差因子时，系统才应把 orthogonalization 本身写成构建步骤。

---

## 14. 阶段十一：准入、归档与发布

评测的终点不是“生成一份 HTML”，而是得出一个明确的研究结论。

### 14.1 一个因子至少应满足的大方向

1. 在目标 horizon 上有稳定正向（或稳定反向）的 IC
2. 分层收益具有可解释的单调性
3. 风险调整后的收益质量不过差
4. 在关键子区间 / Regime 下不过度脆弱
5. 能提供相对独立的信息
6. 换手、容量、成本没有明显不可实施问题

### 14.2 归档时至少要保存

- `factor_build_manifest`
- `preprocess_report`
- 多 horizon IC 表
- 分组收益 / 多空结果
- 风险指标摘要
- 稳定性 / 鲁棒性摘要
- 正交性 / 增量贡献摘要
- 结论与评级

---

## 15. Vortex v1 建议默认口径

| 项 | 建议默认值 |
|----|-----------|
| 因子来源 | 字段 / 公式 / 代码三类都支持 |
| 去极值 | MAD × 3 |
| 标准化 | 截面 z-score |
| 行业中性化 | 默认开启 |
| 市值中性化 | 可选，不作为统一强制默认 |
| 短线因子 horizon | `1d, 5d, 10d, 20d` |
| 常规日频因子 horizon | `5d, 20d, 60d` |
| 长线因子 horizon | `20d, 60d, 120d, 250d` |
| 分层回测 | 默认 5 分组 + 多空 |
| 风险指标 | Sharpe / IR / MDD / alpha-beta |
| 正交性 | 相关矩阵 + 残差 IC + 增量贡献，默认都输出 |
