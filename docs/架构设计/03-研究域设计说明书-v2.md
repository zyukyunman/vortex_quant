## 一、设计目标、范围与文档边界

### 1.1 研究域在全链路中的位置

研究域（Research）负责因子研发、模型训练和信号生成，是连接数据与策略的中间环节：

```
Data 域 snapshot publish (22:00)
  → Research 域 workflow run（因子评测 / 模型训练）
    → signal snapshot publish
      → Strategy 域消费
```

### 1.2 三个设计目标

1. **可复现**（Reproducibility）：同一 profile + 同一 snapshot + 同一 code_hash，任何时候重跑，指标和工件一致。确定性保证包括 seed（随机数种子）固定：涉及随机性的过程（模型训练等），`seed = hash(profile + snapshot_id + workflow_type)`，写入 lineage（详见 00-v2 §12.1）。
2. **可追溯**（Traceability）：每次 experiment 保留完整的配置快照、数据版本、指标和工件，事后可审计。血统（lineage）贯穿全链路。
3. **可交付**（Deliverability）：研究成果以 signal snapshot（信号快照）形式冻结，供下游 Strategy 域消费。signal 一旦发布严格不可变。

### 1.3 CLI 动作清单

Research 域提供 10 个 CLI 动作：

| 动作 | 一句话 |
|------|--------|
| `register` | 注册因子来源（代码或数据表） |
| `unregister` | 注销因子（历史记录保留） |
| `retype` | 修正因子类型分类 |
| `list-factors` | 列出所有已注册因子 |
| `archive` | 查看因子历史评测档案 |
| `run` | 运行研究 workflow（factor_eval / model_train） |
| `inspect` | 查看单次 experiment 详情 |
| `compare` | 并排对比因子或实验 |
| `publish-signal` | 冻结为 signal snapshot |
| `status` | 查看研究线状态与任务队列 |

## 二、域 API 契约总览

本章定义每个动作的架构级契约。函数签名和字段级 schema 详见 06；Profile 字段定义详见 02。

### 2.1 register

| 项 | 说明 |
|----|------|
| 输入（代码因子） | `--name`、`--description`、`--source code`、`--path`（Python 文件路径或 Qlib 表达式） |
| 输入（表因子） | `--name`、`--description`、`--source table`、`--db`、`--table`、`--column`、`--date-column`、`--instrument-column` |
| 输入（公式因子） | `--name`、`--description`、`--source formula`、`--dataset`、`--formula`（基于 Vortex operator 词表） |
| 前置条件 | 因子名未被占用（或传 `--update` 覆盖） |
| 输出 | factor_registry 新增一条记录；CLI 打印注册摘要（来源、输出列、依赖数据、覆盖范围） |
| 工件 | 无文件工件；写入 factor_registry 数据库表 |
| 幂等 | `--update` 覆盖已有注册信息；无 `--update` 时同名注册报错 |
| 阻断 | 代码文件不存在、数据表连接失败、公式解析失败、必填参数缺失 → 注册失败 |

> **默认交互**：`vortex research register` 不带长参数时进入引导式向导。长参数形式保留给脚本、批量注册和 CI 使用；交互式向导是面向研究员的默认入口。

### 2.2 unregister

| 项 | 说明 |
|----|------|
| 输入 | `--name` |
| 前置条件 | 因子名已注册且状态为 active |
| 输出 | factor_registry 标记为 inactive；历史评测记录和缓存不删除 |
| 幂等 | 重复注销同一因子 → 无副作用 |
| 阻断 | 因子名不存在 → 报错 |

### 2.3 retype

| 项 | 说明 |
|----|------|
| 输入 | `--factor`、`--factor-type`（合法枚举：value / momentum / quality / volatility / liquidity / technical / alternative / composite） |
| 前置条件 | 因子已存在于 factor_archive |
| 输出 | factor_archive 中该因子的 factor_type 字段更新；历史评测的同类排名按新类型重算 |
| 幂等 | 设为与当前相同的类型 → 无副作用 |
| 阻断 | 因子不存在、类型值不在合法枚举内 → 报错 |

### 2.4 list-factors

| 项 | 说明 |
|----|------|
| 输入 | 可选 `--status active\|inactive\|all`（默认 active） |
| 输出 | 已注册因子列表：名称、来源类型、因子类型、最近评测时间 |
| 工件 | 无 |

### 2.5 archive

| 项 | 说明 |
|----|------|
| 输入 | `--factor`；可选 `--limit`（默认 20） |
| 输出 | 因子档案：评测次数、IC 均值/最优/最差（含 run_id）、综合分趋势、最近 N 次评测摘要 |
| 阻断 | 因子未注册 → 报错；无评测记录 → 返回空档案 |

### 2.6 run（factor_eval 工作流）

run 是研究域的核心动作，执行因子评测或模型评估。每次 run 由一组关键参数唯一标识（称为 eval_key，即评测去重键），相同 eval_key 的重复提交直接返回已有结果。

| 项 | 说明 |
|----|------|
| 输入模式一（因子） | `--profile`、`--snapshot`、`--factor`；可选 `--factor-type`、`--window auto(default)` / `--start` / `--end`、`--force`、预处理参数覆盖 |
| 输入模式二（模型输出） | `--profile`、`--snapshot`、`--workflow factor_eval`、`--input {model_run_id}`（读取 model_train 的 pred.pkl 作为因子输入） |
| 前置条件 | snapshot 已 publish 且校验通过；因子已注册或 model_run_id 存在且 status=success；并发队列未满 |
| 输出 | experiment_record、因子构建摘要、预处理摘要、评测指标、HTML Tear Sheet、归档到 factor_archive |
| 工件 | `experiments/{run_id}/` 下：factor_build_manifest.json、preprocess_report.json、ic_metrics.csv、pred.pkl、signal_analysis.csv、tear_sheet.html、research_report.json；`state/runs/{run_id}/` 下：experiment_record.json、workflow_config_snapshot.yaml |
| 去重 | 因子模式：eval_key = (factor_name, snapshot, market, label_spec, code_hash)。模型输出模式：eval_key = (model_run_id, snapshot, market, label_spec)。相同 eval_key 跳过（返回已有结果）；`--force` 强制重算 |
| 幂等 | 同 eval_key 返回相同 run_id |
| 阻断 | snapshot 不可用 → fail-closed（阻断）；Qlib 执行异常 → 记录 blocking_reason；队列满且 overflow=reject → 拒绝 |

> **评测窗口默认规则**：`run` 的默认窗口是 `auto`，而不是“全历史无上限”。系统根据 `factor_type` 和目标 horizon 选择默认样本长度：短线 / 技术 / 流动性倾向最近 36 个月，常规日频因子倾向最近 60 个月，长线 / 基本面因子倾向最近 96 个月。用户可显式传 `--start` / `--end` 或 `--window` 覆盖。

### 2.7 run（model_train 工作流）

| 项 | 说明 |
|----|------|
| 输入 | `--profile`、`--snapshot`、`--workflow model_train` |
| 前置条件 | snapshot 已 publish；Profile 中配置了 model_config（train/validation/test 分割） |
| 输出 | model artifact、pred.pkl、训练报告 |
| 工件 | `experiments/{run_id}/` 下：model.pkl、model_report.json、feature_manifest.json、validation_metrics.json、pred.pkl |
| 关键不变量 | **model_train 只产出模型与预测，不能直接 publish-signal**。pred.pkl 必须经过 factor_eval 验证后才可发布 |
| 阻断 | 训练/测试区间重叠 → 泄露检查阻断；数据不足 → 阻断 |

### 2.8 inspect

| 项 | 说明 |
|----|------|
| 输入 | `--run-id` |
| 输出 | 完整元数据、指标、工件清单、血统信息。根据 workflow_type（factor_eval / model_train）自动展示不同维度 |
| 工件 | 无（只读操作） |
| 阻断 | run_id 不存在 → 报错 |

### 2.9 compare

| 项 | 说明 |
|----|------|
| 输入 | `--run-id left,right`（两次实验对比）或 `--factors a,b,c`（多因子档案对比） |
| 前置条件 | 实验模式：两个 run_id 均存在且 status=success。因子模式：因子均有评测记录 |
| 输出 | 配置 diff、指标对齐、delta 计算、相关性矩阵 |
| 工件 | 可选生成 `compare_{timestamp}/factor_compare.html` |
| 阻断 | run_id 不存在或 status 非 success → 报错 |

### 2.10 publish-signal

| 项 | 说明 |
|----|------|
| 输入 | `--run-id`；可选 `--tag`（人类标注，一经绑定不可重指向） |
| 前置条件 | experiment 状态为 success；**workflow_type 必须为 factor_eval**（model_train 不能直接发布） |
| 输出 | signal_snapshot_descriptor.json、冻结工件 |
| 工件 | `signals/{signal_id}/` 下：signal_snapshot_descriptor.json、pred.pkl、ic_metrics.csv |
| 不可变性 | **signal 一旦发布，内容严格不可变、不可覆盖**（对齐 00-v2 §5.3）。如需修正，必须重新 run + 重新 publish，生成新 signal_id |
| 幂等 | 同一 run_id 重复提交 publish 请求：如已存在对应 signal，返回已有 signal_id，不创建新的。这是"请求幂等"（返回已有结果）而非"内容覆盖" |
| 阻断 | experiment 非 success 状态 → 阻断；workflow_type 为 model_train → 阻断；tag 已被其他 signal 占用 → 阻断 |

### 2.11 status

| 项 | 说明 |
|----|------|
| 输入 | 可选 `--profile` |
| 输出 | 最近 experiments 概览、已发布 signals、当前任务队列状态（运行中 N/M + 进度、等待中 + 预估启动时间） |
| 工件 | 无（只读操作） |

## 三、核心对象、存储所有权与生命周期

### 3.1 对象清单

| 对象 | 权威存储 | 可变性 | 域可见性 | 生命周期 |
|------|---------|--------|---------|---------|
| **factor_registry** | 控制面数据库 | 可变（register / unregister / update） | 域内 | 注册 → 活跃（active）→ 注销（inactive） |
| **factor_cache** | 数据面（factor_cache/） | 按 code_hash + snapshot 键控；code_hash 变更自动失效 | 域内 | 写入 → 命中 → 失效 → 重算 |
| **factor_archive** | 控制面数据库 | 追加写入（每次评测追加一条记录） | 域内 | 随评测累积，不主动删除 |
| **experiment_record** | 控制面（state/runs/{run_id}/） | 创建后仅状态字段可变（running → success / failed） | 域内 | running → success / failed；failed 不删除 |
| **model_artifact** | 数据面（experiments/{run_id}/） | 不可变 | 域内 | 随 experiment 生命周期 |
| **signal_snapshot** | 数据面（signals/{signal_id}/） | **严格不可变** | **跨域可见**（Strategy 消费） | 发布后永久保留 |

### 3.2 存储布局

控制面与数据面分离（对齐 00-v2 §6）：experiment_record 等元数据存控制面（state/），大工件存数据面（research/）。

```
state/runs/{run_id}/                        # 控制面：运行状态与元数据
├── experiment_record.json                  #   状态、血统、指标摘要、工件索引
└── workflow_config_snapshot.yaml            #   配置快照（可复现）

research/                                    # 数据面：研究产出
├── factor_cache/                           #   因子值缓存
│   └── {factor_name}/{snapshot}_{code_hash}.parquet
├── experiments/{run_id}/                   #   工件目录
│   ├── factor_build_manifest.json          #   因子构建 manifest（字段、公式、operator、backend）
│   ├── preprocess_report.json              #   预处理口径与样本过滤摘要
│   ├── pred.pkl                            #   预测结果
│   ├── ic_metrics.csv                      #   IC/RankIC/ICIR 时序
│   ├── signal_analysis.csv                 #   分组收益、多空收益
│   ├── research_report.json                #   汇总报告
│   ├── tear_sheet.html                     #   HTML 交互报告
│   ├── model.pkl                           #   模型文件（model_train 专有）
│   ├── model_report.json                   #   训练报告（model_train 专有）
│   └── feature_manifest.json               #   特征清单（model_train 专有）
└── signals/{signal_id}/                    #   跨域交付（严格不可变）
    ├── signal_snapshot_descriptor.json      #   信号快照描述符
    ├── pred.pkl                            #   冻结预测结果
    └── ic_metrics.csv                      #   冻结评测指标
```

### 3.3 run_id、eval_key 与血统

研究域使用三类关键标识符：run_id 标识每次执行，eval_key 用于去重检测，signal_id 标识发布给下游的信号。

**run_id 生成规则**：`{domain}_{date}_{time}_{random4}`，例如 `research_20260403_220501_a3f2`。人类可读性优先于唯一性，4 位随机十六进制足以避免同秒冲突。

**eval_key**（因子评测去重键）：
- 因子模式：`(factor_name, snapshot, market, label_spec, code_hash)`
- 模型输出模式：`(model_run_id, snapshot, market, label_spec)`

相同 eval_key 的重复提交返回已有结果，不创建新 run。

**signal_id 生成规则**：`sig_{profile}_{date}_{time}`，从 source_run_id 中提取日期时间。同一 run_id 的 signal_id 是确定的，支持请求幂等。

**血统必填字段**（对齐 00-v2 §5.1）：

```json
{
  "lineage": {
    "run_id": "research_20260403_220501_a3f2",
    "workflow_type": "factor_eval",
    "profile": "cn_alpha_basic",
    "snapshot_id": "snap_20260403_220000_b2c3",
    "factor_name": "momentum_20d",
    "factor_code_hash": "sha256:a3f2c1...",
    "parent_run_id": null,
    "created_at": "2026-04-03T22:05:01+08:00",
    "vortex_version": "0.2.0",
    "config_hash": "sha256:abc123...",
    "seed": "sha256:hash(cn_alpha_basic+snap_20260403+factor_eval)"
  }
}
```

model_train 的血统扩展字段：feature_set、train_split、val_split、test_split、model_type、hyperparams。

### 3.4 工件清单契约

每次 experiment 完成后，experiment_record.json 中记录完整的工件清单（artifact manifest）。逻辑名是固定的（pred、ic_metrics、tear_sheet、model 等），物理路径由 artifact_root + 相对路径组合。上层代码不需要硬编码文件名。

### 3.5 并发任务管理与状态模型

研究域工作负载（因子评测、模型训练）是计算密集型任务。系统内置并发控制：

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| max_concurrent_tasks | 3 | 同时运行的评测/训练任务数上限 |
| queue_overflow | wait | 超过上限时：wait（排队等待）或 reject（拒绝） |

**任务状态模型**：

```
提交 → [去重检查]
         ├── 命中 eval_key → 返回已有结果（不入队、不消耗资源）
         └── 未命中 → [并发检查]
                       ├── 有空位 → running
                       └── 无空位 → queued（排队等待）/ rejected（overflow=reject 时）

running → success / failed
queued  → running（前序任务完成后自动启动）
```

**耗时预估**（供 status 展示）：

| 阶段 | 预估方式 |
|------|----------|
| 因子值计算 | 首次：按数据量线性估算（约 1s / 10 万行）；后续：跳过（有缓存） |
| 评测规则执行 | 取该因子历史评测中同规模任务的平均耗时 |
| HTML 报告生成 | 固定约 10–15 秒 |
| 首次运行（无历史） | 全局平均值 × 数据规模系数，标注"粗略估计" |

进度百分比 = 已完成步骤耗时 / 预估总耗时。预估不准时动态修正。

## 四、因子注册与评测架构

### 4.1 因子注册子系统

#### 三种因子来源

| 来源 | 说明 | 缓存失效条件 |
|------|------|-------------|
| **代码因子**（Code Factor） | Python 文件或 Qlib 表达式，运行时从原始数据计算因子值 | 代码文件 hash 变更 → 缓存失效 |
| **表因子**（Table Factor） | 外部数据库预计算因子值，通过 db.table.column 引用 | snapshot 版本号变更 → 缓存失效 |
| **公式因子**（Formula Factor） | 基于 operator 词表的表达式，由 Research 执行引擎编译成 DAG 并计算 | 公式文本、依赖字段或 operator backend 版本变更 → 缓存失效 |

#### factor_registry 概念模型

factor_registry 是控制面数据库中的注册表，记录"因子数据从哪来"：

| 字段 | 说明 |
|------|------|
| name | 因子名称（唯一标识，用于 --factor 参数） |
| description | 用户用自然语言写下的因子定义、业务假设和用途说明 |
| source_type | code / table / formula |
| source_path | 代码文件路径（code）、db.table.column（table）或公式引用（formula） |
| source_dataset | 表因子 / 公式因子依赖的主要 dataset，用于可追溯展示 |
| date_column | 表因子的日期列名（代码因子不需要） |
| instrument_column | 表因子的品种列名（代码因子不需要） |
| operator_backend | native / talib / pandas_ta / mixed（仅公式因子需要） |
| status | active / inactive |
| registered_at | 注册时间 |

**设计决策**：注册信息存数据库而非 Profile YAML。因子注册是全局共享的持久化状态（多个 Profile 可引用同一因子），不跟运行环境走。

#### register 的引导式交互

Research 的默认产品入口不是一长串参数，而是一条收敛后的交互路径：

1. 选择来源类型：字段 / 公式 / 代码
2. 若是字段或公式，先做字段发现与口径确认
3. 输入 `name`
4. 输入 `description`
5. 预览依赖字段、输出列和覆盖范围
6. AI 或规则给出 `factor_type` 建议，由用户确认
7. 默认进入 `window=auto` 的首次评测

这里的 `description` 是契约字段，不是 UI 装饰。它会被 AI 分类、因子档案检索、compare 展示和用户手册引用。

#### 因子值缓存机制

注册只记录来源，**因子值的实际计算发生在首次评测时**：

```
register → 写入 factor_registry（来源信息）
              ↓
首次 run   → 读取代码/数据表 → 计算因子值 → 写入 factor_cache
              ↓
后续 run   → 检查 code_hash + snapshot → 命中缓存则直接读取
```

缓存键：`(factor_name, snapshot, code_hash)`。任一维度变化 → 缓存失效，下次评测重算。

**为什么缓存**：全市场全历史因子计算（5,000 标的 × 6 年日频 ≈ 800 万行）耗时显著。缓存后切换评测参数（如 IC 周期从 5d 改 20d）不需重算因子值。缓存是加速手段而非权威数据，损坏时自动重算，不影响正确性。

#### 工具生态分层

Research 不采用“全部纯手写”或“完全托管给第三方库”这两种极端方式，而采用**Vortex 自控契约 + Python 生态适配层**：

| 层 | 归属 | 角色 |
|----|------|------|
| OperatorRegistry | Vortex 自研 | 定义统一 operator 名称、参数、返回 schema 和版本 |
| NativeOps | Vortex 自研 | 基础时序 / 截面算子，如 `lag`、`ts_mean`、`rank`、`neutralize` |
| TALibAdapter | 第三方适配 | 提供 RSI、MACD、ATR、BBANDS、OBV 等成熟技术指标 |
| PandasTAAdapter | 第三方适配（可选） | 作为长尾技术指标兼容层，不作为核心契约真相源 |
| EvalKernel | Vortex 自研 | 负责 IC、RankIC、分组收益、中性化、换手、HTML payload 产出 |
| EmpyricalAdapter | 第三方适配 | 负责 Sharpe、drawdown、alpha/beta 等标准风险指标原语 |
| ReportAdapter | 可选适配 | 可输出与 Alphalens / Pyfolio 风格接近的结果，但不把其 schema 设为主契约 |

**设计决策**：

1. `TA-Lib Python` 适合作为技术指标后端，因为函数稳定、覆盖广、输入输出边界清晰。
2. `Empyrical` 适合作为风险指标原语层，因为它轻量、职责单一。
3. `Alphalens` 更适合作为方法论参考和可选比对输出，不应直接成为主评测内核；Vortex 需要自己控制 A 股口径、artifact schema 和可复现性。
4. `Pyfolio` 更偏组合收益与持仓分析，不应作为 Research 第一阶段核心依赖；如需引入，也应放在 Strategy / Portfolio 分析适配层。

### 4.2 因子类型分类

不同类型因子的评测侧重不同，系统内置 8 种因子类型：价值型（Value）、动量型（Momentum）、质量型（Quality）、波动型（Volatility）、流动性（Liquidity）、技术型（Technical）、另类（Alternative）、复合/模型（Composite/ML）。

#### 分类确定优先级

因子类型在**首次评测时确定**，遵循以下优先级：

1. **用户显式指定**——`--factor-type` 参数
2. **AI 辅助判定**——`--factor-type ask` 触发 LLM 读取因子描述、字段预览、公式或代码片段并建议分类
3. **系统关键词推断**——因子名关键词匹配（momentum_* → 动量型，pb_* → 价值型）
4. **无法推断**——标记为"未分类"，评测照常执行但不参与同类排名

#### factor_type 的唯一真相源

**factor_type 存数据库（factor_archive），不存 Profile。** Profile 中不出现 factor_type 字段；`--factor-type` 是评测命令的输入参数，写入目标是数据库。这确保同一因子换 Profile 评测时类型不变。`retype` 命令直接 UPDATE 数据库字段。

> 产品文档 §2.3 末尾提及"因子类型在 ResearchProfile 中配置"为措辞偏差，以本文档"存数据库"为准。

#### AI 辅助分类

AI 辅助分类不是独立模块，而是 `run` 流程中的一段引导式交互。LLM 优先读取因子 `description`，再结合字段预览、公式文本或代码片段给出分类建议和理由，用户确认或手动选择。AI 的分析过程不持久化——最终选定的因子类型写入数据库，但 AI 建议本身不保存，因此它仅作为决策辅助工具，不是数据源。

LLM 后端通过全局配置切换（local / API），支持 fallback。AI 不可用时降级为关键词推断，不阻断评测。详见产品文档 §2.14。

### 4.3 评测规则

系统内置 10 维评测规则，研究员无需自定义"怎么评"：

| # | 规则 | 核心含义 |
|---|------|---------|
| 1 | **多周期 IC** | IC_1d / IC_5d / IC_20d：因子排名与未来收益排名的截面 Spearman 相关性 |
| 2 | **RankIC** | 基于排名值的 IC，抗极端值干扰 |
| 3 | **ICIR** | IC 均值 / IC 标准差（IC 的"夏普比"），衡量稳定性 |
| 4 | **IC 衰减曲线** | 1d→5d→10d→20d→60d 的衰减速度，判断因子持续性 |
| 5 | **分位收益** | 按因子值分 5/10 组，观察累积收益单调性 |
| 6 | **多空收益** | 做多 Top 组 + 做空 Bottom 组的年化收益 |
| 7 | **因子覆盖率** | 每期有因子值的股票占比（< 70% 不具实用性） |
| 8 | **因子自相关** | 相邻两期排名的 Spearman 相关，衡量换手成本 |
| 9 | **正交性 / 独立性检验** | 与基准因子集合的相关矩阵、残差 IC、增量贡献，判断是否提供新信息 |
| 10 | **截面分布特征** | 偏度、峰度、极端值比例，判断是否需要预处理 |

### 4.4 评测去重

同一 eval_key 的重复提交自动跳过，返回已有结果。`--force` 强制重算。

| 场景 | eval_key 是否相同 | 行为 |
|------|------------------|------|
| 同因子同 snapshot 再跑一次 | 相同 | 跳过，返回已有结果 |
| 同因子换 snapshot | 不同 | 新增评测记录 |
| 同因子改了代码 | 不同（code_hash 变了） | 新增记录，旧记录保留 |
| 同因子换 label_spec（5d → 20d） | 不同 | 新增记录 |

### 4.5 因子档案与定量评分

每次评测自动归档到 factor_archive，按 `(factor_name, market, label_spec)` 分组汇总。

**定量评分**采用分位排名机制——不使用绝对阈值，而是在同类型因子历史评测中计算百分位：

| 维度 | 指标 | 比较基准 |
|------|------|---------|
| 预测力 | IC_5d | 同类型因子历史 IC 分布 |
| 稳定性 | ICIR_5d | 同类型因子历史 ICIR 分布 |
| 盈利能力 | 多空年化收益 | 同类型因子历史多空收益分布 |
| 风险控制 | 多空最大回撤 | 同类型因子历史回撤分布 |
| 独立性 | 残差 IC | 同类型因子历史残差 IC 分布 |

综合分 = 五维等权平均（默认各 20%）。

**冷启动**：同类型评测不足 30 次时标注"样本不足，仅供参考"，同时展示原始指标值。

**趋势分析**：因子档案记录每次评测的综合分变化趋势。同一因子短期内 code_hash 多次变更且 IC 持续上升时，系统提示"可能存在数据窥探（data snooping）"。

### 4.6 因子构建与预处理管道

Research 评测不是“拿到一列数就直接算 IC”。在进入 L1-L4 评测之前，系统必须先完成两段工作：

1. **因子构建（feature engineering）**：把字段、公式或代码解析成原始因子值
2. **因子预处理（factor preprocessing）**：把原始因子值清洗成可比较、可评测的截面

完整口径、方法背景和准入逻辑见《附录-因子研究与评测全流程说明》；本节只定义架构真相源。

```
字段 / 公式 / 代码
  → FactorBuilder（字段提取 / 公式 DAG / Python Loader）
  → 原始因子值 raw_factor
  → 研究样本过滤
  → 缺失值处理
  → 极端值处理
  → 初次标准化
  → 行业中性化（默认开启）
  → 市值中性化（可选）
  → 最终标准化
  → 可用于评测的因子值 processed_factor
  → forward returns / label 对齐
  → L1 / L2 / L3 / L4 评测
```

#### 因子构建层（Feature Engineering）

因子构建层的职责不是“评判因子好不好”，而是把研究员的表达稳定地转成一列 `(date, instrument) → value` 的原始因子值。

> **边界声明**：构建层不提供独立 CLI 动作，也不是独立产品入口；它是 `vortex research run` 内部的首个执行阶段。

| 组件 | 角色 | 边界 |
|------|------|------|
| `FactorBuildService` | 编排本次构建流程，决定走 API / SQL / Python 哪条路径 | 只在 `run` 内部调用，不对外单独暴露 |
| `DataAPIReader` | 按 snapshot / dataset / column 读取研究数据 | 默认优先，适合字段直取与少量字段组合 |
| `SQLReader` | 在 DuckDB / 研究视图上执行 SQL | 适合多表 join、聚合、窗口函数 |
| `FormulaEngine` | 把公式因子解析成 operator 表达式并执行 | 使用 reader 产出的中间数据，不直接负责取数 |
| `PythonFactorLoader` | 加载 Python 因子定义并执行 `compute()` | 作为复杂逻辑的扩展口 |
| `OperatorBackend` | `native / talib / pandas_compat` 等内部后端 | 不把第三方函数名直接暴露给产品层 |

推荐调用链：

```
CLI run
  → ResearchService
  → FactorBuildService
  → SnapshotResolver
  → DataAPIReader / SQLReader / PythonFactorLoader
  → FormulaEngine（如适用）
  → raw_factor
  → PreprocessPipeline
  → EvalKernel
```

每次 `run` 都必须产出 `factor_build_manifest.json`，至少记录：

- 来源类型（字段 / 公式 / 代码）
- 依赖字段与数据集
- 公式文本或代码 hash
- operator backend、参数和 lookback window
- 原始覆盖率与空值比例

#### 研究样本过滤

预处理前先做**研究样本过滤**，其目标不是“美化结果”，而是先保证输入横截面是可交易、可比较的：

| 步骤 | 默认规则 | 目的 |
|------|----------|------|
| 股票状态过滤 | 剔除 ST/PT、退市整理期样本 | 避免异常交易状态污染评测 |
| 上市时长过滤 | 剔除上市未满约 1 年（默认 250 交易日）的样本 | 避免新股制度与流动性噪声 |
| 可交易性过滤 | 剔除停牌、无法成交、无未来收益标签的样本 | 保证 forward return 有意义 |
| 研究 Universe 对齐 | 只保留本次 snapshot / market / universe 内样本 | 保证可复现和横截面一致 |

#### 因子预处理契约

| 步骤 | 默认方法 | 说明 |
|------|----------|------|
| 缺失值处理 | 行业中位数填充；横截面覆盖率过低时整期标记低置信度 | 先保留可用样本，再显式暴露覆盖问题 |
| 极端值处理 | MAD（中位数绝对偏差）× 3 截断；`3σ` 作为可选覆盖参数 | MAD 对偏态分布更稳健 |
| 初次标准化 | 截面 z-score | 把不同量纲转成可比较分数 |
| 行业中性化 | **默认开启**，行业虚拟变量回归取残差 | 行业暴露是多数截面因子的基本控制项 |
| 市值中性化 | 可选，默认 `auto` | `size` 因子默认不做市值中性化；其他类型可由 Profile/CLI 开启 |
| 最终标准化 | 对中性化后的残差再次做截面 z-score | 保证最终输入可比较 |

#### 正交化与独立性边界

“正交化（orthogonalization）”不是默认预处理的强制步骤，而是**评测中的独立性分析能力**：

- 系统始终输出与核心因子集合的相关矩阵
- 系统始终输出残差 IC 和增量贡献
- 只有在研究员明确要求构造 residual factor（残差因子）时，才把正交化作为额外的构建步骤写入 `factor_build_manifest`

也就是说：

- **预处理**解决“输入是否可比较”
- **正交性分析**解决“它是否提供新增信息”

两者相关，但不是一回事。

预处理参数、样本过滤规则和构建 backend 都记录在 `eval_key` 与 `preprocess_report.json` 中——不同口径视为不同评测。

### 4.7 HTML Tear Sheet 架构

Tear Sheet 是评测的核心产物之一——7 页交互式分析报告，参考 Alphalens 但更丰富。

#### 7 页内容定义

| 页面 | 核心内容 | 关键图表 |
|------|---------|---------|
| **一：IC 分析** | 因子预测能力 | IC 时序曲线、RankIC、ICIR 滚动、IC 衰减、IC 分布直方图 |
| **二：分组收益** | 截面选股能力 | 分位累积收益、分位年化指标表、多空净值、月度收益热力图 |
| **三：截面分布** | 因子质量 | 截面分布直方图、覆盖率时序、自相关、均值/标准差趋势 |
| **四：截面快照** | 交互式教育 | 散点图（因子值 vs 收益）、分位柱状图、时间回放滑块 |
| **五：相关性矩阵** | 因子独立性 | 相关性热力图、独立性评分、增量贡献分析 |
| **六：Regime 分析** | 市场环境依赖 | 牛/熊/震荡分段表现、风格切换表现、波动率分层 |
| **七：容量与实用性** | 可操作性 | 市值分组表现、换手率与成本估算、容量上限 |

#### Regime（市场状态）划分规则

默认基于沪深 300 指数的 120 日移动平均线：
- **牛市**：指数 > MA120 且 MA120 向上
- **熊市**：指数 < MA120 且 MA120 向下
- **震荡市**：MA120 斜率平坦（绝对值 < 阈值）
- **风格切换期**：最近 20 日内发生切换

用户可通过 AI 辅助（`vortex ai ask`）对特定区间进行定性判断。

#### 数据嵌入方式

所有数据内嵌在 HTML 的 `<script id="vortex-data" type="application/json">` 标签中，使用 Plotly.js 渲染。内嵌 JSON == `research_report.json` 的 canonical schema（二者内容相同，不允许 HTML 端额外计算）。报告自包含（单一 HTML 文件，无外部依赖，可离线查看）。5/10 分组两份数据同时产出，前端切换时无需重新计算。完整 payload 字段定义见 06 §4.8。共享渲染契约见 06 §4.9。

#### 交互能力

| 交互 | 行为 |
|------|------|
| 页面切换 | 顶部导航标签切换 7 页 |
| 深色/浅色模式 | 图表配色同步刷新 |
| 图表缩放 | 框选放大 / 双击还原 / 滚轮缩放 |
| 悬浮提示 | 精确数值和日期 |
| 截面回放 | 时间滑块 + 自动播放（默认 500ms/帧），散点图和分位图实时刷新 |
| 分组数切换 | 5 分组 / 10 分组切换 |
| IC 周期切换 | 图例点击切换 1d/5d/20d |
| 图表全屏 / 导出 PNG | 每个图表卡片右上角按钮 |

### 4.8 分类型评测侧重与动态报告架构

产品文档 §2.3 要求"系统据此自动调整评测的侧重维度和 HTML 报告布局"。本节定义这一能力的架构设计——如何让不同因子类型的评测在**计算、评分、展示**三个层面都产生差异化行为。

#### 设计动机

当前 10 维评测规则（§4.3）对所有因子一视同仁。但不同类型因子的关键问题截然不同：

- 价值型因子的核心关切是 **IC 稳定性和大盘/小盘差异**——IC 绝对值通常不高，但胜在持续稳定
- 动量型因子则需要关注 **IC 衰减速度和反转风险**——短期有效但可能突然反转
- 复合/ML 因子的首要问题是 **过拟合和样本外衰减**——训练集表现好不代表实盘有效

如果所有类型用相同指标、相同权重、相同报告布局，研究员无法快速聚焦到自己类型因子真正需要关注的维度。

#### 四层架构模型

```
┌─────────────────────────────────────────────────────┐
│  L4: EvalEmphasis（展示层）                          │
│  侧栏排序 · 高亮标记 · 上下文提示 · 补充面板选择    │
│  纯展示逻辑，不产生新计算                            │
├─────────────────────────────────────────────────────┤
│  L3: ScoringPolicy（评分层）                         │
│  同类型百分位归一化 → 维度加权 → 综合评级            │
│  per-type 权重 + 冷启动处理 + 版本快照               │
├─────────────────────────────────────────────────────┤
│  L2: CategoryMetrics（分类指标层）                    │
│  仅对适用类型计算的补充分析指标                       │
│  反转频率 · 大小盘IC差异 · OOS衰减 · 覆盖时效       │
├─────────────────────────────────────────────────────┤
│  L1: UniversalMetrics（通用指标层）                   │
│  10 维评测规则，所有因子统一计算                      │
│  IC/RankIC/ICIR/衰减/分位收益/多空/覆盖率/自相关/... │
└─────────────────────────────────────────────────────┘
```

**数据流方向**：L1→L2→L3→L4，每层输入来自前一层输出。

```
factor + data ──→ [L1: 通用评测] ──→ universal_result
                                          │
factor_type ──→ [L2: 分类指标] ──→ category_result（仅适用类型执行）
                                          │
(universal + category) ──→ [L3: 评分策略] ──→ percentile_scores + grade
                                                    │
all results ──→ [L4: 展示侧重] ──→ HTML Tear Sheet（动态布局）
```

#### L1: UniversalMetrics（通用指标层）

即 §4.3 定义的 10 维评测规则。所有因子统一计算，结果存入规范化的 `EvalResult` 结构。

**关键约束**：所有参与评分（L3）的基础指标必须在 L1 产出。L1 产出的字段集合是**规范化的评测契约**，payload builder 和报告层只做格式化和选择，不做评分相关的计算。

L1 中需要补充的**展示辅助指标**（不参与评分，但所有因子都计算）：

| 指标 | 计算方式 | 用途 |
|------|---------|------|
| ic_positive_ratio | count(IC_t > 0) / count(IC_t) | IC 分析页侧栏展示 |
| max_consecutive_neg_ic | IC_t < 0 的最长连续天数 | IC 分析页侧栏展示 |
| long_short_max_drawdown | 多空净值序列的最大回撤 | 分组收益页侧栏 + **参与评分** |
| monotonicity_score | abs(Spearman(分组序号, 分组均收益)) | 分组收益页侧栏展示 |
| snap_slope, snap_r2 | 截面 OLS 回归斜率和 R² | 截面快照页侧栏展示 |

> 标记为"参与评分"的指标（如 long_short_max_drawdown）在 L3 中作为评分维度"风险控制"的输入，因此必须在 L1 产出，不能下沉到报告层。

#### L2: CategoryMetrics（分类指标层）

**仅对适用的因子类型执行计算**的补充分析指标。与 L1 的区别：L1 对所有因子无差别执行；L2 根据因子类型有条件执行。

每个 CategoryMetric 声明自己适用的因子类型集合。评测引擎在 L1 完成后检查当前因子类型，执行匹配的 L2 指标。

**内置 CategoryMetric 清单**：

| metric_id | 适用类型 | 计算方式 | 意义 |
|-----------|---------|---------|------|
| cap_group_ic_diff | 价值型、质量型 | 大盘 IC 均值 - 小盘 IC 均值 | 判断因子是否只在小盘股有效 |
| reversal_frequency | 动量型、技术型 | IC 符号变化频率（月度窗口） | 反转风险——频率越高，动量策略越危险 |
| ic_decay_ratio_1d_5d | 动量型、技术型 | IC_1d / IC_5d | 短期衰减速度——比值越大，因子越短命 |
| recession_ic | 质量型 | 熊市期间 IC 均值 | 抗衰退能力——质量因子应在熊市仍有效 |
| low_vol_premium | 波动型 | 低波组超额收益（年化） | 低波溢价是否存在 |
| tail_risk_ic | 波动型 | 极端下跌日（<-2σ）的 IC | 尾部风险环境下因子表现 |
| coverage_timeliness | 另类 | 覆盖率随时间的衰减斜率 | 数据时效性——另类数据常随时间覆盖率下降 |
| traditional_independence | 另类 | 与传统因子（PB/MOM/VOL）的平均相关系数 | 是否提供独立于传统因子的新信息 |
| oos_decay_ratio | 复合/ML | 样本外 IC / 样本内 IC | 过拟合程度——比值越接近 1 越好 |
| feature_importance | 复合/ML | 特征贡献度排名（依赖因子实例提供） | 黑盒模型的可解释性 |

**CategoryMetric 契约**：

```
CategoryMetric:
  metric_id: str                   # 唯一标识
  applicable_types: frozenset[str] # 适用的因子类型集合
  output_schema: dict              # 输出字段的类型定义
  version: str                     # 指标计算版本号
  compute(universal_result, factor_data, market_data) → typed_dict
```

**关键约束**：
- CategoryMetric 的输出**不参与 L3 评分**（v1 阶段），仅用于报告展示和研究员参考
- 未来版本可将成熟的 CategoryMetric 提升为评分维度，但需要同步更新 ScoringPolicy 版本
- CategoryMetric 返回类型化字典（非 Any/dict），每个 metric_id 有明确的输出 schema
- 计算失败时返回 None，报告层显示"数据不足"占位，不阻断评测流程

#### L3: ScoringPolicy（评分策略层）

§4.5 定义了五维评分和分位排名机制。本层的职责是**按因子类型配置化**这一评分过程。

**评分三步流水线**：

```
Step 1: 提取原始指标
  从 L1 universal_result 中读取 5 个评分维度的原始值：
  预测力 → IC_5d
  稳定性 → ICIR_5d
  盈利能力 → 多空年化收益
  风险控制 → 多空最大回撤（绝对值越小越好，取反后排名）
  独立性 → 残差 IC

Step 2: 同类型百分位归一化
  对每个维度，在**同类型因子的历史评测分布**中计算百分位排名：
  percentile_score = rank_in_same_type / total_same_type × 100
  
  冷启动处理：同类型评测不足 30 次 → 标注"样本不足，仅供参考"，
  同时展示原始指标值，综合分仍然计算但附带置信度警告。

Step 3: 加权求综合分
  composite = Σ(weight_i × percentile_score_i)
  综合分映射到评级：A(≥85) / B+(≥75) / B(≥60) / B-(≥45) / C(≥30) / D(<30)
```

**per-type 评分权重**：

不同类型因子的评分维度权重不同——反映该类型因子的核心关切。权重总和 = 100%。

| 类型 | 预测力 | 稳定性 | 盈利能力 | 风险控制 | 独立性 | 设计理由 |
|------|--------|--------|---------|---------|--------|---------|
| **默认** | 20% | 20% | 20% | 20% | 20% | 无类型信息时的基线 |
| **价值型** | 15% | 30% | 15% | 20% | 20% | 价值因子 IC 低但稳定，稳定性最重要 |
| **动量型** | 25% | 15% | 20% | 25% | 15% | 预测力和风险控制并重——动量可能反转 |
| **质量型** | 15% | 25% | 15% | 20% | 25% | 长期稳定 + 独立于市值行业 |
| **波动型** | 20% | 20% | 20% | 25% | 15% | 低波策略的风险控制更关键 |
| **流动性** | 20% | 20% | 15% | 20% | 25% | 独立性重要——流动性因子常与市值高相关 |
| **技术型** | 30% | 10% | 20% | 25% | 15% | 短期信号预测力优先，风险控制防过拟合 |
| **另类** | 20% | 20% | 15% | 15% | 30% | 独立性是另类数据的核心价值 |
| **复合/ML** | 20% | 15% | 25% | 20% | 20% | 盈利能力优先，稳定性要求低（已组合化） |

**版本化与可复现**：

每次评测运行持久化以下版本信息到运行工件（run artifact）和 HTML payload meta：

| 字段 | 含义 | 示例 |
|------|------|------|
| evaluated_type | 本次评测使用的因子类型 | "momentum" |
| current_type | 因子库中该因子的当前类型（报告打开时查询） | "momentum" |
| type_changed | evaluated_type != current_type | false |
| scoring_policy_version | 评分策略版本号 | "v1.0" |
| emphasis_version | 展示侧重配置版本号 | "v1.0" |
| report_schema_version | payload 结构版本号 | "2" |

> **为什么持久化 evaluated_type？** 因子类型可能被 `retype` 修改（§4.2），但已生成的报告应保留评测时的类型快照。报告页头同时展示 `evaluated_type` 和 `current_type`，若不一致则标注"类型已变更"。

> **可选页（nullable section）**：某些分析页面在数据不足时可以为 `null`（如因子库仅 1 支因子时 `correlation` 页无意义）。此时 HTML 展示"数据不足，无法展示"占位卡片。7 页导航始终存在，不因数据缺失而隐藏——保持结构一致性。具体哪些页面/字段支持 null 见 06 §4.8 payload schema。

#### L4: EvalEmphasis（展示侧重层）

**纯展示配置，不产生任何新计算。** 每种因子类型对应一份 EvalEmphasis 配置，决定报告的布局差异。

**EvalEmphasis 配置模型**：

```
EvalEmphasis:
  factor_type: str              # 因子类型标识
  display_name: str             # 中文展示名
  sidebar_metrics:              # 每页侧栏指标排序（有序列表）
    page_id → [metric_id, ...]  # 排在前面的指标更醒目
  highlighted: set[str]         # 高亮指标集合（视觉强调）
  hints: dict[str, str]         # 上下文提示 metric_id → 中文提示文本
  supplementary_panels: list[str]  # 补充分析面板 ID
```

**EvalEmphasis 存储方式**：代码常量（Python 中的 frozen dataclass 注册表），不存数据库、不存配置文件。理由：这是系统的领域知识，由开发者维护，变更频率低且需要代码审查。通过 `emphasis_version` 确保可追溯。

**示例——动量型因子的 EvalEmphasis**：

```
EvalEmphasis(factor_type="momentum"):
  sidebar_metrics:
    ic:          [ic_5d, ic_1d, icir, ic_decay_ratio_1d_5d*, ic_positive_ratio]
    group:       [ls_annual_return, ls_sharpe, ls_max_drawdown, monotonicity]
    snapshot:    [snap_ic, snap_slope, snap_r2, snap_count]
    correlation: [independence_score, mktcap_correlation, top_correlated]
    regime:      [transition_ic*, bull_ic, bear_ic, regime_share_pct]
    capacity:    [daily_turnover*, annual_cost, capacity_estimate]
    
  highlighted: {ic_decay_ratio_1d_5d, transition_ic, reversal_frequency}
  
  hints:
    ic_decay_ratio_1d_5d: "动量因子的核心关切：IC衰减越快，所需调仓频率越高，成本越大"
    transition_ic: "风格切换期是动量策略最危险的时候——动量崩溃常发生在此"
    reversal_frequency: "IC符号频繁变化意味着高反转风险，需配合止损或信号确认"
    
  supplementary_panels: [reversal_frequency, ic_decay_ratio_1d_5d]
```

（`*` 标记的指标来自 L2 CategoryMetrics）

#### 8 类因子的展示侧重摘要

| 类型 | 高亮指标 | 补充面板 | 上下文提示核心信息 |
|------|---------|---------|-------------------|
| **价值型** | ICIR、bear_ic、mktcap_correlation | cap_group_ic_diff | "价值因子 IC 低但稳定是正常的""关注大盘有效性——小盘偏便宜不代表有 alpha" |

| **质量型** | ICIR、20d_IC、independence | recession_ic | "质量因子的价值在于长期稳定""熊市仍有效才是真质量" |
| **波动型** | bear_ic、ls_max_drawdown | low_vol_premium、tail_risk_ic | "低波溢价是否存在""尾部风险环境下的表现" |
| **流动性** | capacity_estimate、mktcap_correlation | cap_group_ic_diff | "容量约束是流动性因子的天花板""与市值的相关性必须检查" |
| **技术型** | ic_decay_ratio、annual_cost | reversal_frequency、ic_decay_ratio_1d_5d | "短期信号衰减快是预期内的""交易成本可能吃掉全部 alpha" |
| **另类** | coverage、independence | coverage_timeliness、traditional_independence | "数据覆盖率随时间衰减了吗""与传统因子独立才有价值" |
| **复合/ML** | ls_sharpe、ICIR | oos_decay_ratio、feature_importance | "样本外衰减是过拟合的最佳检测""特征重要性帮助理解黑盒" |

#### 动态 Tear Sheet 组合规则

报告生成的组合逻辑：

```
for each page in 7_pages:
  1. 渲染通用区域（主图表/主表格）—— 所有类型完全相同
  2. 渲染侧栏指标卡片 —— 按 EvalEmphasis.sidebar_metrics[page] 排序
     · 高亮指标使用强调样式（colored badge + 粗体）
     · 非高亮指标使用标准样式
  3. 渲染上下文提示 —— 高亮指标附带 tooltip（来自 hints）
  4. 渲染补充分析面板 —— 仅当 supplementary_panels 中有该页相关的面板时
     · 补充面板出现在侧栏末尾或页面底部
     · 数据来自 L2 CategoryMetrics 的输出
     · 若 L2 计算返回 None → 显示"数据不足，无法展示"占位
```

**7 页始终存在，不因类型而增减。** 变化仅在于：侧栏指标的排序和高亮、tooltip 提示的内容、补充面板的有无。

#### 扩展性

添加一个新的因子类型需要：

| 步骤 | 改动位置 | 必须/可选 |
|------|---------|----------|
| 1. 定义 EvalEmphasis 配置 | emphasis 注册表 | 必须 |
| 2. 定义 ScoringPolicy 权重 | scoring 注册表 | 必须 |
| 3. 实现 CategoryMetric | category_metrics 模块 | 可选 |
| 4. 更新版本号 | emphasis_version, scoring_policy_version | 必须 |

**不需要**改动：eval_engine（L1）、payload builder 基础逻辑、HTML 模板基础结构。

> **设计原则**：通用评测 = 不变的基座；分类差异 = 可插拔的配置 + 有条件的计算。新增因子类型是**扩展**而非**修改**，符合 OCP（开闭原则）。

### 4.9 执行引擎边界（Qlib 适配层）

#### 为什么需要适配层

直接在业务代码中调用 Qlib API 会导致三个问题：版本耦合（Qlib API 随版本变化）、测试困难（需要真实数据目录）、配置散落（provider_uri 等需从多处汇聚）。适配层的原则是：**业务代码只通过适配层与 Qlib 交互，不直接 import qlib**。

#### 适配层职责

| 组件 | 职责 |
|------|------|
| QlibHandler | 封装 Qlib 底层 API（init、dataset、workflow、recorder），隔离版本变化 |
| QlibWorkflowAdapter | 将研究域运行上下文翻译为 Qlib 可执行配置，驱动运行，提取结果 |
| WorkflowTemplateRegistry | 内置 workflow 模板（如 qlib_alpha_daily），运行时渲染为完整配置并快照持久化 |
| Experiment Namespace | 按 profile 隔离 Qlib experiment namespace，避免 recorder 混杂 |

#### 可替换性

Qlib 在 00-v2 中定位为研究执行引擎，不是系统数据库（Parquet + DuckDB 才是权威底座）。适配层通过接口隔离确保：如果未来 Qlib 不再维护或需求超出，可替换为其他引擎，上层业务代码无需变更。输入契约是 ResearchRunContext，输出契约是标准化的指标 dict + 工件路径 dict。

## 五、模型训练与评测衔接

### 5.1 model_train 工作流

模型训练（model_train）与因子评测（factor_eval）是两种不同的 workflow，共享 experiment 基础设施但有不同的执行流程和输出：

| 对比项 | factor_eval | model_train |
|--------|------------|-------------|
| 输入 | 单因子值 + 评测参数 | 多特征 + 超参数配置 |
| 核心过程 | 截面分析（IC、分组、分布） | 机器学习训练 + 验证 |
| 产出 | 评测指标 + HTML Tear Sheet | 模型文件 + pred.pkl + 训练报告 |
| 是否可 publish-signal | ✅ 是 | ❌ 否（必须先经 factor_eval） |
| 数据分割 | 全样本（无训练过程） | train / validation / test 严格分割 |

### 5.2 数据分割与泄露检查

模型因子必须严格区分三个数据集：

| 数据集 | 用途 | 约束 |
|--------|------|------|
| 训练集（train） | 模型学习参数 | 时间最早 |
| 验证集（validation） | 调超参数、选模型 | 紧接训练集 |
| 测试集（test） | 最终评估 | 不能提前看 |

分割边界在 Profile 的 model_config 中配置（train_end、validation_end），test_start 自动为 validation_end + 1。

**泄露检查**（Leakage Check）：系统在训练前自动校验——如果训练区间与测试区间有交叉，阻断执行。如果模型因子的 factor_eval 评测区间包含训练集数据，报告中标红"测试区间与训练集重叠，结果不可信"。

### 5.3 模型训练到信号发布的边界

这是研究域最关键的不变量之一：

```
model_train → pred.pkl → factor_eval（验证） → publish-signal
```

**不允许的路径**：

```
model_train → publish-signal    ← 禁止！
```

理由：模型预测必须经过标准化因子评测流程（10 维评测规则、Tear Sheet、归档），才有资格发布为下游交易链路的输入。训练集上的结果不得直接成为交易信号。

衔接方式：`vortex research run --workflow factor_eval --input {model_run_id}` 将 model_train 的 pred.pkl 作为因子输入，走标准评测流程。

### 5.4 模型训练的特殊关注点

1. **过拟合监控**——训练集 vs 验证集 IC 比值，< 0.6 标注为严重过拟合
2. **特征重要性**——Top N 特征及其贡献度，帮助研究员理解模型学习方向
3. **模型版本管理**——每次训练的模型文件、配置、特征列表都保存在 experiment 工件中，可通过 inspect 查看

## 六、信号发布与跨域交付契约

### 6.1 publish-signal 的冻结语义

publish-signal 是整条链路中最关键的冻结点（对齐 00-v2 §4.2）：

```
Data snapshot（不可变）
  → Research 因子计算 / 模型训练 + 评测
    → publish-signal
      → signal_snapshot（严格不可变）
        → Strategy 只能引用已发布的 signal_snapshot
```

**不可变性级别**：signal_snapshot 是系统中不可变性要求最高的工件。一旦发布：
- 内容不可修改、不可覆盖
- 如需修正，必须重新 run + 重新 publish，生成新 signal_id
- 旧 signal 永久保留，可追溯

**幂等语义**：同一 run_id 重复发起 publish 请求——如果已存在对应 signal，返回已有 signal_id。这是"请求幂等"（返回已有结果），不是"内容覆盖"。

### 6.2 signal_snapshot_descriptor 契约

signal_snapshot_descriptor.json 是 Research 域向 Strategy 域交付的唯一出口：

```json
{
  "signal_id": "sig_cn_alpha_basic_20260403_220501",
  "source_run_id": "research_20260403_220501_a3f2",
  "source_snapshot_id": "snap_cn_stock_daily_20260403",
  "workflow_type": "factor_eval",
  "profile": "cn_alpha_basic",
  "factor_name": "momentum_20d",
  "tag": "v1.2",
  "key_metrics": {
    "IC_1d": 0.042, "IC_5d": 0.068,
    "ICIR_5d": 0.62, "long_short_annual": 0.142
  },
  "artifacts": {
    "pred": "pred.pkl",
    "ic_metrics": "ic_metrics.csv"
  },
  "publish_time": "2026-04-03T22:15:32+08:00",
  "status": "published"
}
```

**Strategy 域消费方式**：通过 signal_id 定位 descriptor，读取 pred.pkl 路径和 key_metrics。**跨域引用必须使用 signal_id**（不可变标识），不允许通过 profile + tag 等可变别名（对齐 00-v2 §2.1 P5"快照即版本"原则）。tag 仅用于人类标注和 CLI 检索辅助，一经绑定不可重指向，(profile, tag) 全局唯一。

单向依赖：Strategy → Research（通过文件协议），Research 不依赖 Strategy。

### 6.3 与其他域的集成

#### 与 Data 域

研究域不直接接触原始数据，通过两个接口消费 Data 域产出：
1. **SnapshotResolver**：查询 snapshot 状态与 Qlib 视图路径。snapshot 未 publish 或校验失败时触发阻断（fail-closed）。
2. **Qlib 视图路径**：Data 域 QlibViewBuilder 构建的 Qlib 可读格式目录，研究域只需传给执行引擎。

**边界**：Data 域负责衍生分析指标（估值分位数、市场宽度等）的产出与质量控制；Research 域可将其作为特征输入，但不负责其定时调度和发布。Research 不反向回写 Data 域。

#### 与 Profile 域

Profile 解析分两阶段：
1. **通用解析**（config 域）：ProfileService.prepare() 处理 extends、defaults、override，产出 ResolvedProfile
2. **领域解析**（研究域）：ResearchProfileResolver 提取研究域特有字段（Qlib 数据目录、workflow 模板、experiment namespace），查询 snapshot 状态，构建 ResearchRunContext

Config 域保持领域无关，研究域特殊逻辑不泄漏到 config 域。

## 八、域故障模式与恢复策略

### 8.1 故障清单

| # | 故障模式 | 触发条件 | 影响范围 | 严重级别 |
|---|---------|---------|---------|---------|
| F1 | **Snapshot 不可用** | Data 域未 publish 或校验失败 | 整次 run 不启动 | 阻断 |
| F2 | **Qlib 执行异常** | Qlib 内部错误、内存不足、数据格式不匹配 | 当前 experiment 失败 | 阻断 |
| F3 | **因子数据不足** | 覆盖率过低或时间序列过短 | 评测结果不可靠 | 警告 |
| F4 | **缓存损坏** | factor_cache 文件损坏或被意外删除 | 需要重算因子值 | 可恢复 |
| F5 | **eval_key 冲突** | 极罕见的 hash 碰撞 | 去重误判 | 可恢复（--force） |
| F6 | **并发队列满** | 超过 max_concurrent_tasks 且配置为 reject | 任务被拒绝 | 可重试 |
| F7 | **模型训练失败** | 特征全为空、超参不合理、内存不足 | 当前 experiment 失败 | 阻断 |
| F8 | **泄露检查失败** | train/test 区间重叠 | 训练不启动 | 阻断 |
| F9 | **AI 辅助不可用** | LLM 后端连接失败或超时 | 因子分类降级为关键词推断 | 降级（非阻断） |
| F10 | **HTML 生成失败** | Plotly 渲染异常或数据格式错误 | 评测完成但无 Tear Sheet | 警告 |

### 8.2 恢复策略

**F1 Snapshot 不可用**：fail-closed 设计——没有合格数据就不跑研究，不允许降级为"用上次的数据凑合"。用户需先确保 Data 域 publish 成功。

**F2 Qlib 执行异常**：ResearchService 捕获异常 → ExperimentTracker.record_failure(run_id, error) → 状态更新为 failed，记录 blocking_reason。失败的 experiment 不删除，可通过 inspect 查看。失败的 experiment 不能 publish-signal。

**F3 因子数据不足**：评测照常执行但在报告中标注警告。覆盖率 < 70% 的因子在 Tear Sheet 第三页（截面分布）高亮提醒。

**F4 缓存损坏**：下次评测时 cache miss → 自动重算。用户也可通过运维命令手动清除特定因子的缓存后重跑（具体命令见 06）。缓存是加速手段而非权威数据，损坏不影响正确性。

**F7 模型训练失败**：同 F2 处理。额外记录特征缺失率、内存使用峰值等诊断信息到 model_report.json。

**F8 泄露检查失败**：训练前阻断，不消耗计算资源。CLI 输出重叠区间的具体日期范围，引导用户修正 model_config。

**F9 AI 辅助不可用**：降级处理——因子分类回退到关键词推断；如果也无法推断，标记为"未分类"。AI 不可用不阻断评测流程。

**F10 HTML 生成失败**：评测指标和工件照常保存，仅 Tear Sheet 缺失。CLI 输出警告并建议重新生成。experiment 状态仍为 success（Tear Sheet 是附加产物，不是核心成果）。

### 8.3 Fail-Closed 边界

以下场景触发 fail-closed（阻断），不允许跳过或静默降级：

1. Snapshot 不可用（F1）
2. Qlib 执行异常导致无法产出任何指标（F2）
3. 泄露检查发现 train/test 重叠（F8）
4. publish-signal 时 experiment 非 success 或非 factor_eval workflow

以下场景允许降级：

1. AI 辅助不可用 → 关键词推断或"未分类"（F9）
2. HTML 生成失败 → 指标和工件照常保存（F10）
3. 因子数据不足 → 标注警告，研究员自行判断（F3）
