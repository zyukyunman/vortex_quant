# 12 - 研究协作与产物治理

> **定位**：本文档回答一个工程治理问题：当 AI 或研究员在 Vortex 上持续挖掘因子、策略和研究报告时，哪些东西应该进入核心仓库，哪些东西应该留在研究工作区，哪些东西需要经过评审后才能产品化。
>
> **结论**：Vortex 不应该把每一次研究探索都直接变成核心代码，也不应该只留下聊天结论。正确做法是：**核心产品和研究探索分轨运行，通过标准 artifact 和晋升机制连接**。

---

## 1. 背景：为什么需要这份治理规则

业绩预告漂移策略研究证明：研究员式探索可以快速发现有效 alpha，也能产出启动月份、持仓安全性、仓位模式和容量分析。但这个过程也暴露出软件迭代风险：

1. HTML 报告、网格 JSON、持仓 CSV 和测试总结很容易大量涌入仓库。
2. 临时脚本如果直接进入产品代码，会让核心架构被研究过程带跑。
3. 研究员需要自由试错，但核心产品需要稳定边界、统一 schema 和可复现能力。
4. 用户真正需要的是“研究结论能复盘、有效策略能上线、数据底座长期可靠”，而不是仓库里堆满每次实验的中间产物。

因此，Vortex 需要把“研究自由度”和“工程可维护性”分开治理。

---

## 2. 轨道划分：一个产品，两个工作轨道

Vortex 的长期形态应按两个轨道协作：

| 轨道 | 目标 | 主要产物 | 稳定性要求 |
|---|---|---|---|
| **核心产品轨道** | 维护数据、回测、报告、实盘/模拟盘能力 | Data 服务、Research/Strategy 内核、CLI/API 入口、标准报告、正式策略实现 | 高，必须测试和审计 |
| **研究探索轨道** | 快速试因子、试策略、做复盘、发现有效方法 | 临时脚本、实验网格、研究 HTML、持仓分析、失败记录 | 中低，允许试错 |

两个轨道不是互相隔离，而是通过固定出口连接：

```text
研究探索
  → 生成标准 artifact
  → 形成策略卡/测试报告
  → 通过防过拟合与可交易审查
  → 抽取可复用能力
  → 晋升到核心产品轨道
```

### 2.1 两速流程：研究要快，晋升要严

研究轨道不等于每个想法都必须走完整产品流程。真正可持续的方式是“两速流程”：

| 阶段 | 目标 | 允许的快法 | 不可突破的底线 |
|---|---|---|---|
| **Research Spike**（快速探索） | 判断一个想法有没有 alpha | 可以直接读本地数据、写临时 runner、跑小网格、生成临时 HTML/JSON | 不用未来函数，不伪装成正式策略 |
| **Candidate Hardening**（候选加固） | 判断是否值得拥有 | 保留快速 runner，但补因子评测、walk-forward、成本、容量、可交易性和持仓复盘 | 必须有标准 artifact 和失败原因 |
| **Product Promotion**（产品晋升） | 进入 Strategy / 模拟盘 | 重构为 Research signal snapshot + Strategy consumer，补测试和文档 | 必须符合 Data → Research → Strategy → Trade |

这意味着：

1. 早期研究可以快，不必每个假设都先注册因子、发布 signal、再回测。
2. 只要结果进入 `candidate`，就必须补可信度审查，不能继续只靠临时脚本。
3. 只有进入 `promoted` 或 `production-ready`，才强制走完整产品化链路。

业绩预告漂移策略当前应按 `candidate` 处理：研究 runner 可以保留，但必须补 `forecast_surprise` 因子评测、`stk_limit/suspend_d` 精确可交易复核和 artifact 治理；只有这些过关后，才晋升为正式 Strategy 消费的 signal snapshot。

## 3. 是否需要拆成两个仓库

### 3.1 当前阶段：不急着拆仓库

短期不建议立刻拆成两个 Git 仓库。原因是：

1. 当前 Research/Strategy 仍在快速补基础设施，过早拆仓会增加路径、依赖和文档同步成本。
2. 许多研究能力还没有稳定边界，暂时放在同一仓库更方便抽象成通用模块。
3. 现在更重要的是先建立“什么能进核心代码、什么只能当 artifact”的规则。

### 3.2 先用目录边界模拟双仓

当前推荐结构：

```text
vortex_quant/
  vortex/                 # 核心产品代码：可复用、可测试、可被 CLI/API 调用
  tests/                  # 核心能力测试
  docs/架构设计/           # 产品与架构治理
  docs/策略研究/           # 精简后的策略卡、最终测试报告、少量代表性 HTML
  .github/skills/         # 可复用研究方法论和领域经验

外部 workspace 或未来 vortex_research/
  experiments/            # 临时研究脚本
  reports/                # 大量实验 HTML/JSON
  holdings/               # 每日持仓和归因大表
  grids/                  # 参数网格、压力测试结果
```

### 3.3 什么时候真的拆仓

满足以下任意条件时，应考虑新增 `vortex_research` 仓库或独立研究工作区：

1. 单个策略产生大量中间 HTML/JSON/CSV，导致核心仓库难以浏览。
2. 研究脚本依赖实验性第三方库，不适合放进核心产品依赖。
3. 研究 artifact 体积明显变大，影响 Git 性能。
4. 研究员需要保留大量失败实验，但核心仓库只需要最终结论。
5. 多个研究方向并行，临时脚本命名、参数和数据路径开始互相干扰。

拆仓后，核心仓库只消费研究仓发布出来的标准 artifact 和已晋升模块。

## 4. 产物分类：什么应该沉淀成代码，什么应该沉淀成结论

研究产物按四类处理：

| 产物类型 | 示例 | 归属 | 是否进入核心仓库 |
|---|---|---|---|
| **一次性探索** | 某次参数网格脚本、临时 SQL、临时图表 | 研究工作区 | 默认不进 |
| **标准 artifact** | `backtest_report.json`、`daily_holdings.csv`、`tear_sheet.html` | workspace 或策略研究目录 | 只保留代表性最终版 |
| **可复用工具** | 启动月份分析、持仓安全性分析、容量压力测试 | `vortex/research` 或 `vortex/strategy` | 进，但必须泛化和测试 |
| **成熟策略** | 业绩预告漂移策略配置、标准 runner、正式报告 schema | `vortex/strategy` + docs | 进，必须可复现 |

判断是否沉淀成代码，用以下标准：

1. 是否会在两个以上策略中复用。
2. 是否有稳定输入输出 schema。
3. 是否能写自动化测试。
4. 是否不依赖某次实验的硬编码路径。
5. 是否能被 CLI/API 或自动化任务调用。

如果答案大多是否定，它就是研究 artifact，不是核心代码。

---

## 5. 晋升机制：从研究想法到正式策略

每个策略或因子至少经过五个状态：

| 状态 | 说明 | 允许的产物 | 下一步条件 |
|---|---|---|---|
| `idea` | 只有假设，还没验证 | 研究笔记 | 明确数据字段、可见时间和交易口径 |
| `experiment` | Research Spike，正在探索 | 临时脚本、网格结果、初版 HTML | 证明不是未来函数，计入成本 |
| `candidate` | 值得继续 | 标准 JSON/HTML、持仓 CSV、测试报告 | walk-forward、成本、容量、可交易性过关 |
| `promoted` | 晋升为正式策略候选 | 核心策略模块、配置、测试、文档 | CLI/API 可运行，可复现 |
| `production-ready` | 可模拟盘/实盘观察 | target portfolio、风控配置、审计记录 | Trade 域风控与执行审查通过 |

禁止从 `experiment` 直接跳到 `production-ready`。高收益回测只能说明“值得审查”，不能说明“可以上线”。

---

## 6. 文档策略：少量正式文档 + 完整 artifact 链接

核心仓库不应该为每次实验都新增一篇长 MD。推荐文档结构：

| 文档 | 目的 | 内容边界 |
|---|---|---|
| 策略卡 | 给人快速理解策略 | 策略逻辑、数据口径、适用场景、风险、当前评级 |
| 测试报告 | 给未来复盘 | 有效路径、无效路径、防过拟合、容量、可交易性、上线建议 |
| 方法论 skill | 给 AI/研究员复用 | 开发流程、踩坑、检查清单、标准输出 |
| artifact 索引 | 给用户和自动化流程打开报告 | 链接最终 HTML/JSON/CSV，不堆聊天式总结 |

研究笔记可以详细，但进入核心仓库前必须压缩成“结论 + 证据 + 边界 + 下一步”。

---

## 7. HTML 报告治理

HTML 是用户体验的一部分，但不能成为唯一事实来源。

### 7.1 标准规则

1. 每个 HTML 报告背后必须有同名或可追溯的 JSON 数据。
2. 自动化流程首先读取 JSON schema，再决定如何展示或比较结果。
3. HTML 可以直接打开，但不能只靠 HTML 解析结果。
4. 临时 HTML 默认放 workspace；核心仓库只保留代表性最终版和产品演示需要的版本。
5. 重要报告必须能追溯到数据 snapshot、配置、代码版本、成本假设和运行时间。

### 7.2 对业绩预告策略的归位

本轮业绩预告策略中的产物建议如下：

| 当前产物 | 处理意见 | 原因 |
|---|---|---|
| `event_backtest.py` 仓位模式 | 保留核心代码 | 稀疏事件策略通用 |
| `earnings_forecast_drift.py` | 保留核心策略模块 | 已形成正式候选策略 |
| 启动月份分析 | 保留并逐步泛化 | 可复用于其他季节性/窗口策略 |
| 持仓安全性 HTML | 保留最终版 | 用户需要看“到底买了什么” |
| 多个参数网格 JSON/HTML | 后续迁移到 artifact 区 | 中间产物不应长期污染核心仓库 |
| 研究总结 MD | 保留但压缩为策略卡 + 测试报告 | 方便人读和未来复盘 |

## 8. 协作协议：用户、研究员、工程助手如何配合

### 8.1 用户/产品负责人负责

用户不需要写因子代码，但需要定义投资与产品边界：

1. 资金规模和容量要求。
2. 是否允许空仓、满仓、现金留存。
3. 是否允许小市值、科创板、创业板、ST、北交所。
4. 单票/行业/风格集中度偏好。
5. 策略上线门槛：收益、回撤、Calmar、胜率、换手、可解释性。
6. HTML 报告和标准 JSON 中必须能看到哪些问题的答案。

### 8.2 研究员负责

研究员的核心职责是发现和验证，而不是维护产品边界：

1. 提出因子或策略假设。
2. 做 PIT、安全交易口径、成本和容量验证。
3. 记录有效方法和无效方法。
4. 输出标准 artifact。
5. 把可复用分析能力提议为工程模块。

### 8.3 工程助手负责

工程助手负责把成熟研究产品化：

1. 抽象稳定 schema。
2. 把重复脚本改成通用函数、服务或 API。
3. 给核心模块补测试。
4. 把 HTML/JSON 报告接入标准 artifact 索引。
5. 保证 Data → Research → Strategy → Trade 的边界不被临时研究打破。

---

## 9. 对报告与 artifact 索引的影响

当前暂缓浏览器交互界面，报告与 artifact 索引仍应支持三类信息视图：

| 视图 | 面向对象 | 核心问题 |
|---|---|---|
| 研究档案 | 实验和候选策略 | 这个想法试过什么？哪些有效/无效？ |
| 策略卡 | 已晋升策略 | 规则是什么？风险是什么？现在能不能上线观察？ |
| 工件浏览器 | JSON/HTML/CSV artifact | 净值、持仓、交易、归因、容量在哪里看？ |

报告索引只能把 `candidate` 及以上状态展示为“策略候选”；`experiment` 状态必须标注为“研究中”，避免用户误以为已经可实盘。

---

## 10. 当前执行规则

从本文档生效后，Vortex 的研究与开发遵循以下默认规则：

1. **Research Spike 可以快**：允许直接 runner、临时脚本和小网格，但只能产出 `experiment`。
2. **Candidate 必须加固**：补因子评测、walk-forward、成本、容量、可交易性和持仓复盘。
3. **Promoted 才产品化**：只有晋升时才强制 Research signal snapshot + Strategy consumer。
4. **核心代码只接收可复用能力和成熟候选策略**。
5. **大量实验 artifact 默认外置到 workspace**。
6. **核心仓库内的策略研究文档必须精简、可复盘、可链接 artifact**。
7. **每个值得保留的 HTML 都必须有 JSON 事实来源**。
8. **研究员可以自由探索，但不能绕过晋升机制进入实盘/正式策略**。

一句话原则：

> **研究可以快，晋升必须严；早期用 runner 发现 alpha，产品化时再补齐 signal、报告、测试和审计。**
