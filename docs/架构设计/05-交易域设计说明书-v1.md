# 交易域设计说明书 v1

> 把 target portfolio（目标组合）安全、可追溯地送进模拟盘或实盘，并在日终完成对账闭环。

---

## 一、设计目标与产品映射

本文档对应产品冻结 §六「交易执行与风控」，落地以下 6 个 CLI 动作：

| 动作 | 一句话 |
|------|--------|
| `paper rebalance` | 模拟盘调仓：生成 order plan → 事前风控 → 提交 → 监听回报 |
| `live rebalance` | 实盘调仓：同 paper 流程，但网关为 miniQMT 实盘、风控更严、强制审计 |
| `inspect` | 查看某次执行的全部工件 |
| `retry` | 对失败订单重新执行（重新过事前风控） |
| `reconcile` | 日终对账：网关持仓 vs 目标组合 vs 成交记录 → 偏差报告 |
| `status` | 查看当前交易线状态（网关连接、未完成订单、最近执行摘要） |

核心目标有三个：

1. **安全**：三层风控 + Fail-Closed，任何异常都不放行。
2. **可追溯**：每次执行产出结构化工件（order plan / execution report / reconcile report），可回溯、可审计。
3. **可替换**：Gateway 抽象层让 PaperGateway 和 MiniQMTGateway 共享同一套上层逻辑；未来接入新券商只需实现 Gateway 接口。

## 二、模块结构

```text
vortex/trade/
    __init__.py              # 域入口，注册 CLI 命令组
    types.py                 # 领域值对象：OrderSide, OrderStatus, FillRecord 等
    exceptions.py            # 交易域专用异常
    profile.py               # TradeProfileResolver：加载 + resolve + 校验
    plan.py                  # OrderPlanGenerator：target → order plan
    engine.py                # TradeEngine：编排 plan → risk → submit → track
    tracker.py               # ExecutionTracker：订单状态机 + 成交归集
    reconcile.py             # ReconcileEngine：日终对账
    notify.py                # 通知适配（飞书 / webhook）
    gateway/
        __init__.py          # Gateway 抽象基类
        paper.py             # PaperGateway：内存撮合
        miniqmt.py           # MiniQMTGateway：miniQMT 协议适配
    risk/
        __init__.py          # RiskEngine + RiskRule 基类
        pre_trade.py         # 事前规则实现
        intra_trade.py       # 事中规则实现
        post_trade.py        # 事后规则实现
        packs/               # risk_pack YAML 定义目录
    artifacts.py             # 工件读写：plan / report / reconcile 持久化
    cli.py                   # Click 命令定义
```

每个模块一句话：

- `profile.py`：薄适配层，委托 `ProfileService.prepare()` 获取通用 `ResolvedProfile`，提取交易特有字段构造 `ResolvedTradeProfile`。
- `plan.py`：读取 target portfolio，获取当前持仓，计算调仓差异，输出 `OrderPlan`。
- `engine.py`：整条链路的编排者——调用 plan → risk → gateway → tracker，是 `paper rebalance` / `live rebalance` 的入口。
- `tracker.py`：维护每张订单的状态机，归集成交回报，判定是否需要 retry。
- `reconcile.py`：日终拉取网关实际持仓与成交，与目标组合做差异比对。
- `gateway/`：统一交易通道抽象；paper 用内存撮合，live 走 miniQMT 协议。
- `risk/`：三层风控引擎，规则可插拔，按 risk_pack 配置加载。
- `artifacts.py`：统一工件序列化/反序列化，保证目录结构一致。

### 3.2 Gateway 接口

```python
from abc import ABC, abstractmethod
from vortex.trade.types import Order, FillRecord, PositionSnapshot

class Gateway(ABC):
    """统一交易通道抽象，灵感来自 vn.py 的 BaseGateway。"""

    name: str  # 网关标识，如 "paper" / "miniqmt"

    @abstractmethod
    def connect(self, config: dict) -> None:
        """建立连接。失败抛 GatewayConnectionError。"""
        ...

    @abstractmethod
    def submit_order(self, order: Order) -> str:
        """提交订单，返回网关侧 order_id。"""
        ...

    @abstractmethod
    def cancel_order(self, order_id: str) -> bool:
        """撤单，返回是否成功。"""
        ...

    @abstractmethod
    def query_positions(self) -> list[PositionSnapshot]:
        """查询当前持仓快照。"""
        ...

    @abstractmethod
    def query_fills(self, order_id: str) -> list[FillRecord]:
        """查询指定订单的成交回报。"""
        ...

    @abstractmethod
    def disconnect(self) -> None: ...
```

为什么要做 Gateway 抽象：模拟盘和实盘的上层逻辑（plan → risk → submit → track → reconcile）完全一致，区别仅在"订单实际去了哪里"。抽象层让我们能在开发阶段用 PaperGateway 跑通全链路，上线时只切一个配置字段 `gateway: miniqmt`。

## 四、数据流

### 4.1 `vortex trade paper rebalance` 完整流程

```text
用户: vortex trade paper rebalance --profile dividend_paper --target tp_latest

  TradeProfileResolver.resolve()     → ResolvedTradeProfile
          │
  Gateway.connect()                  → 连接网关（失败则阻断）
          │
  OrderPlanGenerator.generate()      → 读 target + 当前持仓 → OrderPlan
          │
  RiskEngine.check_pre_trade()       → 通过 / 不通过
          │
     ┌────┴────┐
  通过│     不通过│→ 阻断，写 pre_trade_result，不提交
     │
  Gateway.submit_order()             → 逐笔提交
          │
  ExecutionTracker.on_fill()         → 归集回报 + 事中风控
          │
  输出: order_plan.json / pre_trade_result.json / execution_report.json
```

关键决策：

1. **先连网关再做 plan**：如果网关不可用，提前失败，避免白算一轮 plan。
2. **风控不通过 = 零提交**：事前风控是原子判定，任意规则不通过则整个 plan 不提交。
3. **部分失败记录 retry**：某些订单被拒或超时时，ExecutionTracker 标记 `needs_retry`，用户后续可执行 `vortex trade retry --run-id <exec_id>`。

### 4.2 `vortex trade reconcile` 完整流程

```text
用户: vortex trade reconcile --profile dividend_paper --date 2026-04-04

  Gateway.query_positions() ─┐
  加载 target_portfolio     ─┼→ ReconcileEngine.reconcile() → 逐标的比对
  加载 execution_report     ─┘
          │
  RiskEngine.check_post_trade()
          │
     ┌────┴────┐
  正常│     异常│→ 标记 abnormal + 通知 + 审计
     │
  输出: reconcile_report.json
```

## 五、与其他域的集成

### 5.1 与 Strategy 域

交易域消费 Strategy 域产出的 **target portfolio**（目标组合）。

- target portfolio 存放于 Strategy 域的工件目录，格式为 `{symbol: target_weight}` 或 `{symbol: target_shares}`。
- `--target tp_latest` 表示取最近一次策略运行的输出；也可指定 `--target <tp-ref>`。
- 交易域只读不写 target portfolio，单向依赖。

### 5.2 与 Data 域

交易域需要 Data 域提供：

1. **交易日历**（Trading Calendar）：判断当天是否交易日、当前是否在 `trading_window` 内。
2. **最新行情**：限价单需要参考价（最新价、涨跌停价），PaperGateway 撮合也需要行情。
3. **标的状态**：ST 标记、停牌状态——这些信息同时被事前风控规则使用。

### 5.3 与 Profile 域

交易域通过 `TradeProfileResolver` 调用 Profile 域的通用解析链路：

```text
ProfileService.prepare() → ResolvedProfile → 提取交易字段 → 补网关/账户派生字段 → 校验
```

关键派生字段如 `resolved_gateway_endpoint`、`resolved_account_credentials_ref` 由 resolve 阶段自动填充，用户不可见。

## 六、Gateway 适配层设计

### 6.1 为什么需要 Gateway 抽象

原因有三个：

1. **模拟盘与实盘逻辑统一**：从 plan 到 risk 到 track 的整条链路不因网关不同而分叉，减少维护和测试成本。
2. **可测试**：单元测试和集成测试使用 PaperGateway，不依赖外部券商环境。
3. **可扩展**：未来接入其他券商（如 CTP、恒生 UFT）只需新增一个 Gateway 实现。

### 6.2 Gateway 接口定义

见 §3.2。核心方法：`connect`, `submit_order`, `cancel_order`, `query_positions`, `query_fills`, `disconnect`。

设计约束：

- 所有方法同步返回（v1 不引入异步事件循环，降低复杂度）。
- 成交回报通过 `query_fills()` 轮询获取（v1 不做 push 回调，MiniQMT 也支持轮询模式）。
- 连接失败必须抛异常，不允许静默降级。

### 6.3 PaperGateway 内存撮合逻辑

撮合规则：market 单以最新价即时全额成交；limit 买单最新价 ≤ 限价时成交，limit 卖单最新价 ≥ 限价时成交，否则挂起；挂起订单在 trading_window 结束时标记 timeout。行情来源为 Data 域最新快照价格。

为什么 v1 不做部分成交模拟：模拟盘核心价值是验证全链路流程（plan → risk → submit → track → reconcile），而非精确模拟市场微观结构。简单撮合已足够，避免过早复杂化。

### 6.4 MiniQMTGateway 协议适配

通过 xtquant SDK 与 miniQMT 客户端通信。核心适配：`connect()` → `xtquant.connect(endpoint)`；`submit_order()` → 转为 xtquant 委托参数异步提交；`query_positions()` / `query_fills()` → 查询并转换为内部类型。

边界条件：miniQMT 客户端必须已启动且处于登录状态，否则 `connect()` 失败。v1 不做自动重连，断开直接抛异常。

### 6.5 未来扩展

新增网关只需要：

1. 实现 `Gateway` 接口的全部方法。
2. 在 `gateway/__init__.py` 注册网关名到类的映射。
3. 在 TradeProfile preset 中增加对应的 gateway 配置项。

## 七、风控引擎设计

### 7.1 三层模型总览

| 层级 | 触发时机 | 输入 | 阻断行为 |
|------|---------|------|---------|
| 事前（Pre-Trade） | order plan 生成后、提交前 | `OrderPlan` | 不通过 → 不提交任何订单，输出结构化原因 |
| 事中（Intra-Trade） | 每笔成交回报到达时 | `FillRecord` | 触发 → 撤回未成交订单 / 暂停执行 |
| 事后（Post-Trade） | 日终对账完成后 | `ReconcileReport` | 偏差超限 → 标记异常、通知、审计日志 |

### 7.2 RiskRule 接口

```python
from abc import ABC, abstractmethod
from enum import Enum

class RiskLevel(Enum):
    PRE_TRADE = "pre_trade"
    INTRA_TRADE = "intra_trade"
    POST_TRADE = "post_trade"

class RiskResult:
    passed: bool
    rule_name: str
    reason: str | None    # 不通过时的结构化原因
    details: dict | None  # 可选的详细数据

class RiskRule(ABC):
    """可插拔风控规则基类。"""
    name: str
    level: RiskLevel

    @abstractmethod
    def evaluate(self, context: "RiskContext") -> RiskResult:
        """评估规则。context 按 level 分型，确保类型安全。"""
        ...


@dataclass(frozen=True)
class PreTradeContext:
    plan: "OrderPlan"
    profile: "ResolvedTradeProfile"

@dataclass(frozen=True)
class IntraTradeContext:
    fill: "FillRecord"
    tracker: "ExecutionTracker"

@dataclass(frozen=True)
class PostTradeContext:
    reconcile: "ReconcileReport"

RiskContext = PreTradeContext | IntraTradeContext | PostTradeContext
```

### 7.3 risk_pack 加载机制

`risk_pack` 是一组规则的命名集合，定义在 YAML 中：

```yaml
# risk/packs/cn_execution_v1.yaml
name: cn_execution_v1
description: A 股执行风控规则包 v1

pre_trade:
  - rule: single_stock_concentration
    params: { max_weight: 0.10 }
  - rule: limit_up_down_check
  - rule: st_check
  - rule: suspension_check

intra_trade:
  - rule: slippage_limit
    params: { max_slippage_bps: 50 }
  - rule: abnormal_fill_check

post_trade:
  - rule: position_deviation
    params: { max_deviation_pct: 2.0 }
  - rule: fill_price_deviation
    params: { max_price_deviation_bps: 30 }
```

`RiskEngine` 初始化时根据 `profile.risk_pack` 名称加载对应 YAML，实例化规则对象列表。

### 7.4 事前规则示例

| 规则 | 检查逻辑 | 阻断条件 |
|------|---------|---------|
| `single_stock_concentration` | 单票目标权重 / 总组合 | 超过 `max_weight` |
| `limit_up_down_check` | 查标的当日是否涨跌停 | 涨停时禁止买入，跌停时禁止卖出 |
| `st_check` | 查标的是否 ST / *ST | ST 标的禁止买入（可配置是否允许卖出） |
| `suspension_check` | 查标的是否停牌 | 停牌标的不可交易 |

### 7.5 事中规则示例

| 规则 | 检查逻辑 | 阻断条件 |
|------|---------|---------|
| `slippage_limit` | 成交价与委托价偏差（bps） | 超过 `max_slippage_bps` → 撤回剩余未成交 |
| `abnormal_fill_check` | 成交量 / 委托量异常检测 | 异常 → 暂停后续提交 |

### 7.6 事后规则示例

| 规则 | 检查逻辑 | 阻断条件 |
|------|---------|---------|
| `position_deviation` | 实际持仓 vs 目标组合偏差 | 偏差超 `max_deviation_pct` |
| `fill_price_deviation` | 成交均价 vs 委托限价偏差 | 偏差超 `max_price_deviation_bps` |

### 7.7 Fail-Closed 原则

这是风控引擎最重要的设计决策：

1. **规则执行异常（抛异常、超时）视为不通过**，不允许因为规则本身出错而放行订单。
2. **规则缺失视为不通过**：如果 risk_pack 配置引用了未注册的规则名，初始化阶段直接报错，不会进入"无规则 = 全通过"的状态。
3. **风控结果必须持久化**：不论通过与否，`pre_trade_result.json` 都会写入工件目录，供审计追溯。

## 八、订单状态机

### 8.1 订单生命周期

```text
                       submit_order()
    ┌─────────┐      ────────────────>     ┌───────────┐
    │ pending  │                            │ submitted │
    └─────────┘                            └─────┬─────┘
                                                 │
                          ┌──────────────────────┼──────────────────┐
                          │                      │                  │
                          ▼                      ▼                  ▼
                   ┌─────────────┐      ┌──────────────┐    ┌───────────┐
                   │ partial_fill│      │   rejected   │    │  timeout  │
                   └──────┬──────┘      └──────────────┘    └───────────┘
                          │
                          ▼
                   ┌─────────────┐
                   │   filled    │
                   └─────────────┘

    任何状态均可 ──────> cancelled（用户主动撤单或事中风控触发）
```

### 8.2 状态定义

| 状态 | 含义 |
|------|------|
| `pending` | 订单已生成，等待提交 |
| `submitted` | 已提交至网关，等待成交回报 |
| `partial_fill` | 部分成交 |
| `filled` | 全部成交 |
| `rejected` | 被网关或交易所拒绝 |
| `timeout` | 在 trading_window 内未全部成交 |
| `cancelled` | 被撤单（用户主动或事中风控触发） |

### 8.3 retry 逻辑

`vortex trade retry --run-id <exec_id>` 的执行流程：

1. 加载指定 `exec_id` 的 `execution_report.json`。
2. 筛选 `needs_retry == true` 的订单（状态为 `rejected` 或 `timeout`）。
3. 用这些订单构建一个新的 `OrderPlan`。
4. **重新执行事前风控**——不跳过，因为市场状态可能已变化（例如标的已涨停）。
5. 通过后重新提交，生成新的 `exec_id` 和执行报告。
6. 新执行报告中记录 `parent_exec_id` 以建立追溯链。

## 九、对账引擎设计

### 9.1 对账数据源

对账需要三个数据源的交叉验证：

| 数据源 | 来源 | 用途 |
|--------|------|------|
| 目标组合（target portfolio） | Strategy 域工件 | 期望持仓 |
| 成交记录（execution report） | 当日执行工件 | 实际成交了什么 |
| 网关持仓（gateway positions） | Gateway.query_positions() | 券商侧实际持仓 |

### 9.2 偏差计算方法

逐标的计算三组差异：

1. **目标 vs 网关持仓**：`abs(actual_weight - target_weight)`，反映组合偏离程度。
2. **目标 vs 成交**：理论上成交后应接近目标，有偏差说明部分订单未成交或被拒。
3. **成交 vs 网关持仓**：不一致说明可能存在网关同步延迟或外部干预。

### 9.3 阈值判定与异常通知

```python
# reconcile_policy 中配置
reconcile_policy:
  enabled: true
  run_after_close: true
  thresholds:
    max_position_deviation_pct: 2.0   # 单标的持仓偏差上限
    max_total_deviation_pct: 5.0      # 组合总偏差上限
```

- 超过阈值 → `ReconcileReport.status = "abnormal"`。
- 触发事后风控规则检查。
- 通过通知渠道（飞书 / webhook）发送告警。
- 写入审计日志。

## 十、审计与通知

### 10.1 live rebalance 强制审计

`live rebalance` 与 `paper rebalance` 的流程差异：

| 差异项 | paper | live |
|--------|-------|------|
| 网关 | PaperGateway | MiniQMTGateway |
| 风控严格度 | risk_pack 配置决定 | 必须配置 risk_pack，不可为空 |
| 二次确认 | 不需要 | 交互式确认或 `--confirm` 跳过 |
| 审计日志 | 可选 | 强制写入 |
| 通知 | 可选 | 强制发送 |

### 10.2 二次确认机制

交互模式下，live rebalance 在提交前展示执行摘要（profile、account、gateway、订单数、买卖金额），要求用户输入 `y` 确认。`--confirm` 标志跳过交互确认，适用于自动化调度场景。即便使用 `--confirm`，审计日志仍记录"由 --confirm 标志自动确认"。

### 10.3 通知策略

通知通过 `notify.py` 统一发送，支持飞书机器人 webhook 和自定义 webhook（POST JSON），v1 暂不支持邮件。触发场景：live rebalance 完成、对账异常偏差、事中风控撤单。

## 十二、失败处理

交易域的失败处理遵循 **Fail-Closed** 总原则——宁可不执行，不可错执行。

| 场景 | 处理方式 |
|------|---------|
| 风控不通过 | 不提交任何订单，写 `pre_trade_result.json` 记录原因 |
| 部分订单失败 | ExecutionTracker 标记 `needs_retry`，用户可 `retry` |
| 风控规则本身异常 | 视为不通过（Fail-Closed） |
| 网关不可用 | `connect()` 抛异常，上层阻断整个执行 |
| 网关连接中断（执行中） | 停止提交后续订单，已提交的等待回报或超时 |
| 对账偏差超限 | 标记 `status: deviation_exceeded`，通知 + 审计 |
| target portfolio 不存在 | 启动阶段即报错，不进入 plan 生成 |
| TradeProfile 无效 | resolve 阶段报错，不进入执行 |

为什么不做自动补单：v1 阶段，自动补偿行为可能在异常市场条件下放大风险。失败后由人工判断是否 retry 更安全。自动化场景下可通过调度系统触发 retry，但仍会重新过风控。

## 十三、第一版实现范围

### MVP 边界

**v1 做**：

1. `paper rebalance` 全链路（plan → pre-trade risk → submit → track → report）。
2. `live rebalance` 全链路（含二次确认、审计日志、通知）。
3. `inspect` / `retry` / `status` 基础实现。
4. `reconcile` 全链路（网关持仓 vs target vs 成交）。
5. PaperGateway：简单内存撮合（market 单即时成交、limit 单价格比较）。
6. MiniQMTGateway：通过 xtquant 接入 miniQMT。
7. 事前风控：单票集中度、涨跌停、ST、停牌 4 条规则。
8. 事后风控：持仓偏差超限。
9. risk_pack YAML 配置加载。
10. 飞书通知。
11. 结构化工件输出（order_plan / execution_report / reconcile_report）。

**v1 不做**：

1. 异步事件驱动（v1 用同步轮询，降低复杂度）。
2. PaperGateway 部分成交模拟。
3. 自动重连（网关断开直接失败）。
4. 自动补单 / 自动 retry。
5. 多账户并行执行。
6. 实时盈亏监控面板。
7. CTP / 恒生等其他券商网关。

### 交付顺序建议

```text
Phase 1: 骨架
  types.py + exceptions.py + Gateway 接口 + PaperGateway

Phase 2: 核心链路
  OrderPlanGenerator + RiskEngine(pre-trade) + TradeEngine + ExecutionTracker

Phase 3: 工件与 CLI
  artifacts.py + cli.py（paper rebalance / inspect / status）

Phase 4: 实盘
  MiniQMTGateway + live rebalance + 二次确认 + 审计

Phase 5: 对账与事后风控
  ReconcileEngine + post-trade rules + 通知

Phase 6: 补全
  retry + 事中风控 + risk_pack 扩展
```
