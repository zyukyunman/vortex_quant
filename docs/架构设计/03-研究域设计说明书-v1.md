# 研究域设计说明书 v1

> 本文档把"基于 Qlib 的因子评测 / 模型训练 / 信号发布"落成可编码的接口与数据流，回答研究域"如何实现"。

---

## 一、设计目标与产品映射

本设计对应产品规格：06-产品功能冻结文档 §四 研究域。

研究域在 Vortex 日常流水线中的位置是：

```
Data 域 snapshot publish (22:00)
  → Research 域 workflow run
    → signal snapshot publish
      → Strategy 域消费
```

这条链路要求研究域做到 3 件事：

1. **可复现**：同一个 profile + 同一个 snapshot，任何时候重跑，指标和工件一致。
2. **可追溯**：每次 experiment 都保留完整的配置快照、数据版本、指标和工件，事后可审计。
3. **可交付**：研究成果以 signal snapshot（信号快照）形式冻结，供下游 Strategy 域消费，而不是散落在某个目录下的 pkl 文件。

### 5 个动作

| 动作 | 一句话 | 核心设计关注点 |
|------|--------|---------------|
| `run` | 运行研究 workflow | Qlib 适配、experiment 追踪、工件管理 |
| `inspect` | 查看单次 experiment | 元数据读取、工件索引 |
| `compare` | 并排对比两次实验 | 配置 diff、指标对齐 |
| `publish-signal` | 冻结为 signal snapshot | 幂等发布、descriptor 生成 |
| `status` | 查看研究线状态 | experiment 列表、signal 索引 |

```
│   └── resolver.py          # ResearchProfileResolver：profile → 可执行配置
├── qlib_adapter/
│   ├── __init__.py
│   ├── handler.py           # QlibHandler：dataset init / workflow run / recorder read
│   ├── workflow_template.py # 内置 workflow 模板注册与渲染
│   └── namespace.py         # experiment namespace 隔离逻辑
├── tracker/
│   ├── __init__.py
│   └── experiment_tracker.py # ExperimentTracker：run_id、指标、工件、config 快照
├── compare/
│   └── experiment_compare.py # ExperimentCompare：双实验对比
├── publisher/
│   └── signal_publisher.py  # SignalPublisher：experiment → signal snapshot
├── services/
│   └── research_service.py  # ResearchService：5 个动作的业务编排
└── report/
    └── report_builder.py    # ResearchReportBuilder：汇总指标 → research_report.json
```

每个目录一句话说明：

- `models/`：纯数据对象，不做 I/O。
- `profile/`：ResearchProfile → 运行时可执行配置。
- `qlib_adapter/`：封装 Qlib，隔离其 API 变化对上层的影响。
- `tracker/`：experiment 生命周期——创建、记录指标、管理工件。
- `compare/`：两次 experiment 的配置 diff 与指标对比。
- `publisher/`：experiment → signal snapshot 冻结发布。
- `services/`：面向 CLI 的业务编排层。
- `report/`：汇总指标 → `research_report.json`。

## 三、核心接口与类

### 3.1 ResearchProfileResolver

职责：接收 `ResolvedProfile`（由 config 域已完成的 profile 解析链产出），进一步派生研究域运行时需要的字段——Qlib 数据目录、workflow 模板引用、experiment namespace。

为什么单独存在：config 域的 `ProfileResolver` 负责通用解析（extends、defaults、override），但研究域有自己的派生逻辑（比如根据 `data_profile` 找到 snapshot 的 Qlib 视图路径）。这部分知识不应泄漏到 config 域。

```python
@dataclass(frozen=True)
class ResearchRunContext:
    """研究 workflow 运行所需的完整上下文。"""
    profile_name: str
    snapshot_id: str
    qlib_uri: str                    # Qlib 数据视图路径
    feature_set: str
    label_spec: dict[str, Any]
    workflow_template: str           # workflow 模板名
    experiment_namespace: str
    signal_output: dict[str, Any]
    extra_overrides: dict[str, Any]  # --set 传入的覆盖


class ResearchProfileResolver:
    def __init__(self, snapshot_resolver: "SnapshotResolver") -> None: ...

    def resolve(
        self,
        resolved_profile: "ResolvedProfile",
        snapshot_ref: str,
    ) -> ResearchRunContext:
        """
        从 ResolvedProfile 派生研究运行上下文。
        snapshot_ref 可以是具体 id 或别名（latest, latest_success）。
        """
        ...
```

### 3.2 QlibWorkflowAdapter

职责：把 Vortex 的 `ResearchRunContext` 翻译成 Qlib 能执行的 workflow 配置，驱动 Qlib 运行，并从 Qlib recorder 中提取结果。

为什么不直接调用 Qlib：见第六节。

```python
@dataclass(frozen=True)
class QlibWorkflowResult:
    """Qlib workflow 运行结果的标准化表示。"""
    recorder_id: str
    metrics: dict[str, float]       # IC, RankIC, ICIR 等
    artifact_paths: dict[str, str]  # pred.pkl, ic_metrics.csv 等的物理路径
    raw_config: dict[str, Any]      # 实际使用的 Qlib 配置快照


class QlibWorkflowAdapter:
    def __init__(self, handler: "QlibHandler") -> None: ...

    def run(self, context: ResearchRunContext) -> QlibWorkflowResult:
        """
        1. 根据 context 渲染 workflow 模板。
        2. 调用 QlibHandler 初始化 dataset 并运行 workflow。
        3. 从 recorder 提取指标和工件路径。
        """
        ...
```

### 3.3 ExperimentTracker

职责：experiment 全生命周期管理——生成 run_id、持久化配置快照、记录指标、索引工件。

```python
    def record_result(
        self,
        run_id: str,
        workflow_result: QlibWorkflowResult,

        """把 workflow 结果（指标、工件路径）写入 experiment 记录。"""
        ...

    def get(self, run_id: str) -> "ExperimentRecord":
        """读取单次 experiment 的完整记录（inspect 动作）。"""

    def list_by_profile(
```

### 3.4 SignalPublisher

职责：把一次成功的 experiment 冻结为 signal snapshot，生成 `signal_snapshot_descriptor.json`。

关键设计决策：publish 幂等——同一 run_id 重复 publish 覆盖已有 signal snapshot 而非报错。

```python
class SignalPublisher:
    def __init__(self, artifact_root: Path, tracker: ExperimentTracker) -> None: ...

    def publish(
        self,
        run_id: str,
        tag: str | None = None,
    ) -> "SignalSnapshotDescriptor":
        """
        1. 从 tracker 读取 experiment 记录。
        2. 验证 experiment 状态为 success。
        3. 复制关键工件到 signals/{signal_id}/ 目录。
        4. 生成 signal_snapshot_descriptor.json。
        5. 幂等：同一 run_id 重复 publish 覆盖已有 descriptor。
        """
        ...

    def list_signals(
        self,
        profile_name: str | None = None,
    ) -> list["SignalSnapshotDescriptor"]:
        """列出已发布的 signal snapshots。"""
        ...
```

## 五、与其他域的集成

### 5.1 与 Data 域

研究域不直接接触原始数据，也不拼物理路径。它通过两个接口消费 Data 域产出：

1. **SnapshotResolver**：查询 `snapshot_ref` 对应的 `SnapshotManifest`，拿到 Qlib 视图路径（`qlib_uri`）。snapshot 未 publish 或校验失败时，触发研究域阻断。
2. **Qlib 视图路径**：Data 域 `QlibViewBuilder` 构建的 Qlib 可读格式目录，研究域只需传给 `QlibHandler.init_dataset`。

#### 与 Data 域“衍生分析指标”的边界

为了避免 Data 与 Research 职责混淆，研究域实现需遵守以下边界：

1. Data 域在 `publish` 阶段自动产出衍生分析指标（如估值分位数、市场宽度、资金面指标），并写入 `data/derived/{profile}/as_of={date}/`。
2. Research 域可以把这些衍生指标作为 `feature_set` 的输入特征，但不负责它们的定时调度、质量门禁和发布时点控制。
3. Research 域产出的是 experiment 与 signal snapshot，不反向回写 Data 域的权威 snapshot。

### 5.2 与 Strategy 域

Strategy 域消费研究域产出的方式是读取 signal snapshot：

1. 通过 `signal_id` 或 `profile + tag` 定位 `signal_snapshot_descriptor.json`。
2. descriptor 声明了 `pred.pkl` 路径和 `key_metrics`，Strategy 域直接使用。

单向依赖：Strategy → Research（通过文件协议），Research 不依赖 Strategy。

### 5.3 与 Profile 域

研究域 profile 解析分两阶段：

1. **通用解析**（config 域）：`ProfileService.prepare()` 处理 extends、defaults、override、validator，产出 `ResolvedProfile`。
2. **领域解析**（研究域）：`ResearchProfileResolver.resolve()` 提取研究域特有字段，查询 snapshot 状态，构建 `ResearchRunContext`。

config 域保持领域无关，研究域特殊逻辑不泄漏到 config 域。

## 六、Qlib 适配层设计

### 6.1 为什么需要适配层

直接在业务代码中调用 `qlib.init()`、`qlib.workflow` 会导致 3 个问题：

1. **耦合 Qlib 版本细节**：Qlib API 随版本变化（DataHandlerLP 参数、recorder 字段名等），散布在多个文件中会导致升级困难。
2. **测试困难**：Qlib 初始化需要真实数据目录，不做适配层无法在无数据环境中单测业务逻辑。
3. **配置散落**：provider_uri、dataset config、experiment name 需从多处汇聚，直接调用会让汇聚逻辑散落各处。

适配层的原则是：**业务代码只看到 `QlibHandler` 和 `QlibWorkflowAdapter`，不直接 import qlib**。

### 6.2 QlibHandler 封装

`QlibHandler` 是对 Qlib 底层 API 的薄封装，提供 3 个能力：

```python
class QlibHandler:
    """
    封装 Qlib 底层 API。
    业务代码不应直接 import qlib，统一通过此类访问。
    """

    def init_qlib(self, qlib_uri: str, region: str = "cn") -> None:
        """初始化 Qlib 运行环境（provider_uri 等）。"""
        ...

    def init_dataset(
        self,
        feature_set: str,
        label_spec: dict[str, Any],
        segments: dict[str, tuple[str, str]],
    ) -> Any:
        """
        构建 Qlib DataHandler + Dataset。
        feature_set 映射到预定义的特征配置（如 alpha158_cn_v1）。
        返回 Qlib Dataset 对象。
        """
        ...

    def run_workflow(self, workflow_config: dict[str, Any], experiment_name: str) -> str:
        """运行 Qlib workflow，返回 recorder_id。"""
        ...

    def read_recorder(self, experiment_name: str, recorder_id: str) -> tuple[dict[str, float], dict[str, str]]:
        """从 recorder 提取指标和工件路径，返回 (metrics_dict, artifact_paths_dict)。"""
        ...
```

### 6.3 Workflow 模板机制

Qlib workflow 配置是一个嵌套 dict（对应 YAML），字段多且和具体任务强相关。直接让用户写完整 workflow 配置不现实。

解决方案是 **workflow 模板**：平台内置标准模板（如 `qlib_alpha_daily`），`ResearchProfile.qlib_workflow` 指定模板名，`QlibWorkflowAdapter` 在运行时用 `ResearchRunContext` 渲染模板。渲染后的完整配置持久化为 `workflow_config_snapshot.yaml`，确保可复现。

```python
class WorkflowTemplateRegistry:
    """注册和渲染 workflow 模板。"""

    def register(self, name: str, factory: Callable[..., dict]) -> None: ...

    def render(
        self,
        template_name: str,
        context: ResearchRunContext,
    ) -> dict[str, Any]:
        """
        渲染模板，返回完整的 Qlib workflow 配置 dict。
        如果模板不存在则抛出 WorkflowTemplateNotFoundError。
        """
        ...
```

### 6.4 Experiment Namespace 隔离

Qlib 的 experiment 系统用 experiment name 作为隔离维度。不同 profile 的实验必须使用不同的 namespace，否则 recorder 会混在一起。

规则：experiment name = `ResearchProfile.experiment_namespace`（如 `cn_alpha_daily`）。同一 namespace 下的 recorder 按 run_id 区分。

这样做的好处是：`list_by_profile` 只需要查询对应 namespace 下的 recorder，而不需要全局扫描。

## 七、run_id 与工件管理

### 7.1 run_id 生成规则

格式：`{domain}_{date}_{time}_{random4}`

示例：`research_20260403_220501_a3f2`

各段含义：

- `domain`：固定 `research`，用于区分数据域的 `data_` 前缀。
- `date`：运行日期 `YYYYMMDD`。
- `time`：运行时间 `HHMMSS`。
- `random4`：4 位随机十六进制，避免同秒冲突。

为什么不用 UUID：run_id 出现在目录名、日志、CLI 输出中，人类可读性比唯一性更重要。4 位随机足以避免同秒冲突。

### 7.2 工件目录结构

```
research/
├── experiments/{run_id}/
│   ├── experiment_record.json          # 元数据
│   ├── workflow_config_snapshot.yaml   # Qlib 配置快照
│   ├── pred.pkl                       # 预测结果
│   ├── ic_metrics.csv                 # IC/RankIC/ICIR 时序
│   ├── signal_analysis.csv            # 分组收益、多空收益
│   └── research_report.json           # 汇总报告
└── signals/{signal_id}/
    ├── signal_snapshot_descriptor.json # 信号快照描述符
    ├── pred.pkl                       # 从 experiment 复制
    └── ic_metrics.csv                 # 从 experiment 复制
```

### 7.3 工件清单 manifest

每次 experiment 完成后，`ExperimentTracker` 会在 `experiment_record.json` 中记录完整的工件清单：

```python
@dataclass(frozen=True)
class ResearchArtifactManifest:
    run_id: str
    artifact_root: str                     # 工件根目录
    artifacts: dict[str, str]              # 逻辑名 → 相对路径
    created_at: str                        # ISO 时间戳
```

逻辑名是固定的（`pred`, `ic_metrics`, `signal_analysis`, `workflow_config`, `research_report`），物理路径由 `artifact_root + 相对路径` 组合。这样上层代码不需要硬编码文件名。

## 八、失败处理与幂等性

### 8.1 Snapshot 不可用 → 阻断

`ResearchProfileResolver.resolve()` 在查询 snapshot 时，如果 `SnapshotResolver` 返回 snapshot 不存在或状态非 `published`，直接抛出 `SnapshotNotAvailableError`，整个 run 不启动。

这是 fail-closed 设计：没有合格数据就不跑研究，不能降级为"用上次的数据凑合"。

### 8.2 Workflow 异常 → 记录 blocking_reason

`QlibWorkflowAdapter.run()` 可能因为多种原因失败（Qlib 内部异常、内存不足、数据格式不匹配）。

处理逻辑：

1. `ResearchService` 捕获异常，调用 `ExperimentTracker.record_failure(run_id, error)` 把状态更新为 `failed`，记录 `blocking_reason`。
2. 失败的 experiment 不会被删除，仍可通过 `inspect` 查看。
3. 失败的 experiment 不能 `publish-signal`。

```python
@dataclass
class ExperimentRecord:
    run_id: str
    profile_name: str
    snapshot_id: str
    status: str                     # running / success / failed
    config_snapshot: dict[str, Any]
    key_metrics: dict[str, float] | None
    artifact_manifest: ResearchArtifactManifest | None
    blocking_reason: str | None     # 失败原因
    created_at: str
    finished_at: str | None
```

### 8.3 publish-signal 幂等

同一 `run_id` 重复 `publish-signal`：覆盖 `signal_snapshot_descriptor.json` 和工件文件，`publish_time` 更新为当前时间，不创建新 `signal_id`。

### 8.4 signal_id 生成规则

格式：`sig_{profile}_{date}_{time}`

从 `source_run_id` 中提取 date 和 time 部分，拼上 profile name。这保证同一 run_id 的 signal_id 是确定的，支持幂等覆盖。

## 九、第一版实现范围

### 9.1 第一版实现

1. **`run` 动作**：完整实现，包括 profile 解析、Qlib 适配、experiment 追踪、工件管理、report 生成。
2. **`inspect` 动作**：读取 `experiment_record.json` 并格式化输出。
3. **`publish-signal` 动作**：工件复制和 descriptor 生成。研究域向 Strategy 域交付的唯一出口。
4. **`status` 动作**：列出最近 experiments 和已发布 signals，基于文件系统扫描。
5. **单一 workflow 模板**：先实现 `qlib_alpha_daily`（日频因子评测）。
6. **构造注入**：所有 service 依赖通过 `__init__` 注入，和数据域保持一致。

### 9.2 暂不实现

1. **`compare` 动作**：依赖两次 experiment 都成功完成，先把单次 run 跑通。排入第二版。
2. **分钟频 workflow 模板**：`qlib_alpha_minute` 需要分钟级数据支持，Qlib 分钟频适配有额外复杂度。
3. **自动触发**：第一版 `run` 手动执行，不接 ops 域调度。
4. **experiment 清理/归档**：第一版不做过期清理。
5. **多模型对比 workflow**：第一版只支持单模型 workflow。

### 9.3 MVP 判定标准

以下流程能端到端跑通，即为 MVP 达标：

```bash
# 1. 假设 Data 域已发布 snapshot
vortex research run --profile cn_alpha_daily --snapshot snap_cn_stock_daily_20260403
# → 输出 run_id, IC, RankIC, ICIR

# 2. 查看这次 experiment
vortex research inspect --run-id research_20260403_220501_a3f2
# → 输出完整元数据和工件清单

# 3. 冻结为 signal snapshot
vortex research publish-signal --run-id research_20260403_220501_a3f2 --tag v1.0
# → 输出 signal_id，descriptor 文件写入磁盘

# 4. 查看研究线状态
vortex research status --profile cn_alpha_daily
# → 输出最近 experiments 和 signals 概览
```

### 9.4 建议的最小落地顺序

和数据域一样，按下面顺序逐层推进：

1. 先写 `models/`、`exceptions.py`：定死数据对象。
2. 再写 `qlib_adapter/handler.py`：验证 Qlib 能初始化、跑 workflow、读 recorder（技术风险最高）。
3. 再写 `qlib_adapter/workflow_template.py`：`qlib_alpha_daily` 模板。
4. 再写 `tracker/experiment_tracker.py`：run_id 生成、experiment_record 读写。
5. 再写 `profile/resolver.py`：连接 config 域和研究域。
6. 再写 `publisher/signal_publisher.py` 和 `report/report_builder.py`。
7. 最后写 `services/research_service.py`：串联上述模块对接 CLI。

这样推进的好处是：

1. 每一步都能单测。
2. 技术风险最高的 Qlib 适配最先验证。
3. 不会一上来就写一个大 service 把所有职责揉在一起。
