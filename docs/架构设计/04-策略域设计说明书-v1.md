# 策略域设计说明书 v1

> 本文档把"信号进来、组合出去、回测可验证"落成可编码接口，回答策略域（Strategy Domain）如何实现。

---

## 一、设计目标与产品映射

本文档对应产品冻结 §五（策略开发与回测体验），实现 5 个 CLI 动作：

| 动作 | 一句话 | 关键输入 | 关键输出 |
|------|--------|----------|----------|
| `backtest` | 运行一次回测 | `--profile` `--snapshot` `--signal` | `backtest_report.json`, `equity_curve.csv`, `trade_log.csv` |
| `inspect` | 查看指定回测记录 | `--run-id` | 终端格式化展示 |
| `compare` | 并排比较两次回测 | `--left` `--right` | 差异表 + 关键指标对比 |
| `compile` | 信号 → 目标持仓 | `--profile` `--snapshot` `--signal` | `target_portfolio.json`, `rebalance_diff.csv`, `order_plan_preview.csv` |
| `status` | 策略线概览 | `--profile` | 最近 run、当前 portfolio、异常汇总 |

核心设计目标有 4 条：

1. **信号标准化消费**：策略域只接受已 publish 的 signal snapshot，不接受裸文件或临时 DataFrame。
2. **回测引擎可替换**：第一版用 Backtrader，但上层不写死 Backtrader API。
3. **compile 与 backtest 分离**：backtest 是验证工具，compile 才是产出交付物（target portfolio）的动作。
4. **全链路可追溯**：每次执行产出 run_id + manifest + 工件，与数据域、研究域的追溯体系一致。

这一节想解决的真实问题是：

1. 策略层不能只是一个回测脚本，它必须对上游信号做准入检查、对下游交易提供标准化交付物。
2. 回测和生产组合编译必须走同一套 profile 解析与约束逻辑，否则回测结论在实盘无法复现。
3. 第一版的重点是流程正确性，不是策略丰富度。

## 三、核心接口与类

### 3.1 StrategyProfileResolver

> **重要**：策略域不应自行加载 YAML 和 preset。所有 profile 解析统一委托给配置域的 `ProfileService.prepare()`（见 02-配置模型说明书 §十一），策略域只消费返回的 `ResolvedProfile`。
>
> `StrategyProfileResolver` 是策略域内的薄适配层，负责从通用 `ResolvedProfile` 提取策略特有字段并构造 `ResolvedStrategyProfile`。

```python
class StrategyProfileResolver:
    """从配置域获取 ResolvedProfile，提取策略特有字段。

    调用链：
    1. 委托 ProfileService.prepare(name, 'strategy', command_scope, ...) 获取 ResolvedProfile
    2. 从 ResolvedProfile 提取 engine/cost_model/slippage/risk_pack 等策略字段
    3. 校验策略特有约束（如 engine 是否支持、risk_pack 规则是否合法）
    4. 返回 ResolvedStrategyProfile
    """

    def __init__(self, profile_service: "ProfileService") -> None: ...

    def resolve(
        self,
        profile_name: str,
        command_scope: str,
        snapshot_ref: str | None = None,
        overrides: dict[str, str] | None = None,
    ) -> "ResolvedStrategyProfile":
        """返回完全展开的策略配置。

        Raises:
            ProfileNotFoundError: profile 文件不存在
            ProfileValidationError: 必填字段缺失或类型错误
        """
        ...
```

`ResolvedStrategyProfile` 包含用户层字段和 preset 层字段的完整合并结果，是后续所有组件的唯一配置输入。

### 3.2 BacktestEngine（协议）与 BacktraderAdapter

```python
from typing import Protocol

class BacktestEngine(Protocol):
    """回测引擎协议。第一版实现为 BacktraderAdapter，未来可替换为 vnpy/zipline。"""

    def run(
        self,
        profile: ResolvedStrategyProfile,
        feed: DataFeed,
        signal: SignalSnapshot,
        cost_model: CostModel,
        slippage_model: SlippageModel,
    ) -> BacktestReport:
        """执行回测，返回标准化报告。"""
        ...
```

为什么需要 Protocol 而不是直接写 Backtrader：

1. Backtrader 的 API 风格（Cerebro/Strategy 子类/Analyzer）是它自己的约定，不应该泄漏到策略域上层。
2. 未来如果换 vnpy 回测或自研引擎，只需要新增一个 adapter，不动 runner 和 CLI。

```python
class BacktraderAdapter:
    """Backtrader 回测引擎适配器。

    职责：
    1. 把 ResolvedStrategyProfile 翻译成 Cerebro 配置
    2. 把 DataFeed（Parquet/Qlib）翻译成 bt.feeds
    3. 把 SignalSnapshot 注入 Strategy.next() 逻辑
    4. 把 Broker commission/slippage 设置为 profile 要求的成本模型
    5. 挂载标准 Analyzer，回测结束后提取统一指标
    """

    def __init__(self, engine_config: dict | None = None):
        self._engine_config = engine_config or {}

    def run(
        self,
        profile: ResolvedStrategyProfile,
        feed: DataFeed,
        signal: SignalSnapshot,
        cost_model: CostModel,
        slippage_model: SlippageModel,
    ) -> BacktestReport:
        ...
```

### 3.3 CostModel / SlippageModel

```python
class CostModel(Protocol):
    """交易成本模型。返回单笔交易的佣金 + 税费。"""

    name: str

    def calculate(
        self, symbol: str, direction: str, price: float, volume: int
    ) -> float:
        """返回该笔交易的总成本（元）。"""
        ...


class SlippageModel(Protocol):
    """滑点模型。返回实际成交价与理论价的偏离。"""

    name: str

    def apply(
        self, symbol: str, direction: str, price: float, volume: int
    ) -> float:
        """返回滑点调整后的实际成交价。"""
        ...
```

第一版提供两个内置实现：

1. `CnStockDefaultCost`：佣金万三 + 卖出印花税千一。
2. `FixedBpsSlippage`：固定基点滑点（默认 5bps）。

这两个模型通过 profile 的 `cost_model` 和 `slippage_model` 字段选择，由 preset 提供默认值。

## 五、与其他域的集成

### 5.1 与 Research 域：signal snapshot 消费

策略域是研究域的下游消费者。

集成契约：

1. 研究域通过 `vortex research publish-signal` 显式发布 signal snapshot 到 `research/signals/{signal_id}/`。`publish-signal` 是信号从实验到可消费状态的唯一发布动作。
2. signal snapshot 必须包含描述文件 `signal_snapshot_descriptor.json`，至少声明 `profile`、`source_run_id`、`source_snapshot_id`、`publish_time`、`artifacts`、`key_metrics`、`status=published`。
3. 策略域在 backtest/compile 前校验 signal snapshot 的 `status`。如果不是 `published`，阻断并写入 `blocking_reason`。
4. 策略域只读取 signal snapshot 的标准字段（symbol、score、rank），不依赖研究域内部的因子中间表。

为什么不让策略域直接读研究的中间文件：

1. 耦合到研究内部结构，一旦研究重构因子管线，策略域会跟着碎。
2. signal snapshot 是显式发布的契约边界，有版本、有状态、有元数据。

### 5.2 与 Data 域：snapshot 消费

策略域通过 snapshot 获取回测所需的行情和基准数据。

1. `FeedBuilder` 接受 snapshot_id，通过数据域的 `SnapshotService` 获取物理路径。
2. 回测引擎只读取已 publish 的 snapshot，不直接读 ODS/DWD 层。
3. 如果 snapshot 不存在或未 publish，BacktestRunner 阻断。

### 5.3 与 Trade 域：target portfolio 交付

策略域是交易域的上游供给者。

1. `compile` 产出的 `target_portfolio.json` 写入 `strategy/portfolios/{portfolio_id}/`。
2. 交易域通过 `portfolio_id` 发现并消费 target portfolio。
3. target portfolio 的 `status` 字段标识其生命周期：`frozen`（可消费）、`draft`（未完成）、`superseded`（已被新版替代）。
4. 策略域不直接调用交易域 API，两个域通过工件目录解耦。

为什么用文件级解耦而不是内存调用：

1. 策略域和交易域的执行时机不同（compile 可能在盘前，trade 在盘中）。
2. 文件是天然的审计记录，可追溯、可复现、可人工检查。
3. 未来如果部署分离，文件可以替换为对象存储，接口不变。

### 5.4 与 Profile 域：StrategyProfile resolve

1. `StrategyProfileResolver` 复用配置模型说明书（02）定义的四层合并逻辑。
2. StrategyProfile 声明 `research_profile` 字段，建立策略→研究→数据的 profile 链。
3. 系统在 resolve 时校验引用的 research_profile 存在且合法。

## 六、Backtrader 适配层设计

### 6.1 为什么需要适配层

Backtrader 是成熟的回测引擎，但它的 API 有自己的约定：

1. 数据必须包装成 `bt.feeds` 对象。
2. 策略必须继承 `bt.Strategy`，在 `next()` 里写交易逻辑。
3. 成本通过 `Broker.setcommission()` 设置。
4. 指标通过 `Analyzer` 提取。

如果策略域直接暴露这些 API，整个上层就绑死在 Backtrader 上。

适配层的职责是：**把 Vortex 内部标准对象翻译成 Backtrader 能理解的格式，把 Backtrader 的输出翻译回 Vortex 标准格式。**

### 6.3 Strategy 生命周期映射

Vortex 的策略 API 参考 RiceQuant 风格（`init / before_trading / handle_bar / after_trading`），需要映射到 Backtrader 的 `__init__ / prenext / next / stop`。

```text
RiceQuant 风格            Backtrader 映射
─────────────────         ─────────────────
init()                →   __init__()
before_trading()      →   prenext() 首次 / next() 每日开盘前逻辑
handle_bar()          →   next()
after_trading()       →   notify_order() + notify_trade()
```

适配层在 `BacktraderAdapter` 内部创建一个 `bt.Strategy` 子类，该子类的 `next()` 方法：

1. 读取当天日期对应的 signal snapshot 行（symbol → score → rank）。
2. 调用 `order_target_percent()` 按 signal 调整仓位。
3. 这里的核心思想是 RiceQuant 的 `order_target_portfolio`：不发增量订单，而是声明目标仓位，由引擎计算差额。

## 七、PortfolioCompiler 设计

### 7.1 信号消费方式

PortfolioCompiler 消费 signal snapshot 的标准流程：

```text
signal_snapshot
  │  每个标的有 score（浮点数，越大越好）
  ▼
[1] 排名（rank）
  │  按 score 降序排列
  ▼
[2] 筛选（top_n）
  │  取前 N 名（N 由 portfolio_constraints.max_names 决定）
  ▼
[3] 赋权（weight_scheme）
  │  equal_weight / market_cap_weight / score_proportional
  ▼
[4] 约束裁剪（constraints）
  │  逐条应用，超限则裁剪并重新归一化
  ▼
[5] 输出 holdings
```

### 7.2 约束应用

约束规则定义在 `portfolio_constraints` 中，由 `ConstraintEngine` 按顺序应用：

```python
class ConstraintEngine:
    """按声明顺序应用组合约束。

    支持的约束类型（第一版）：
    - max_weight:     单只标的最大权重（如 5%）
    - min_positions:  最少持仓数量（如 15 只）
    - max_positions:  最多持仓数量（如 30 只）
    - sector_limit:   单行业最大权重（如 20%，需要行业分类数据）

    每条约束独立检查，如果某条约束修改了权重，后续约束基于修改后的权重继续检查。
    所有约束应用完毕后做一次归一化，确保权重之和为 1.0。
    """

    def apply(
        self,
        holdings: list[HoldingDraft],
        constraints: list[Constraint],
    ) -> list[HoldingDraft]:
        ...
```

为什么约束按顺序而不是同时应用：

1. 约束之间可能冲突（max_weight 裁剪后可能违反 min_positions）。
2. 顺序应用让行为可预测、可调试。
3. 如果出现无法同时满足的约束组合，最后一条约束优先，并在 manifest 中记录 warning。

### 7.4 幂等性保证

compile 的幂等性通过以下机制保证：

1. **portfolio_id 确定性生成**：`tp_{profile}_{snapshot_date}_{signal_id_suffix}`，相同输入产生相同 id。
2. **写入前检查**：如果 `strategy/portfolios/{portfolio_id}/` 已存在且内容一致，直接返回已有结果。
3. **不依赖外部状态**：compile 只读取 signal snapshot 和 profile，不读取交易系统的实时状态。

## 九、失败处理

策略域遵循全系统的 Fail-Closed 原则：宁可阻断不执行，不可带着问题执行。

### 9.1 前置条件不满足 → 阻断

| 场景 | 检查点 | 行为 |
|------|--------|------|
| snapshot 不存在或未 publish | BacktestRunner / CompileRunner 开始前 | 阻断，返回 `blocking_reason: "snapshot not published"` |
| signal snapshot 不存在或未 publish | BacktestRunner / CompileRunner 开始前 | 阻断，返回 `blocking_reason: "signal not published"` |
| profile 不合法（字段缺失/引用不存在） | StrategyProfileResolver | 抛出 `ProfileValidationError` |

### 9.2 回测执行异常 → blocking_reason

回测过程中的异常（Backtrader 内部错误、Feed 数据缺失、内存不足等）：

1. 捕获异常，不让 Backtrader 的堆栈直接暴露给用户。
2. 写入 manifest：`status: "failed"`, `blocking_reason: "backtest engine error: {msg}"`。
3. 已产出的部分工件保留（便于排查），但 manifest 标记为 failed。
4. CLI 返回非零退出码 + JSON 错误响应。

### 9.3 compile 幂等 → 安全重跑

compile 的幂等性意味着：

1. 重复执行不产生副作用。
2. 如果上一次 compile 中途失败（manifest 标记 failed），重跑会重新执行并覆盖失败记录。
3. 如果上一次 compile 成功，重跑直接返回已有结果。

## 十、第一版实现范围

### 10.1 MVP 包含

1. **backtest 动作**：单策略单周期回测，Backtrader 引擎，A 股日频。
2. **compile 动作**：signal → target portfolio，等权方案 + max_weight / min_positions 约束。
3. **inspect 动作**：加载 run_id 对应的 manifest 和 report，终端格式化输出。
4. **status 动作**：扫描 `strategy/runs/` 和 `strategy/portfolios/`，汇总最近执行。
5. **StrategyProfile**：支持 `dividend_classic` preset，YAML 配置 + `--set` 覆盖。
6. **CnStockDefaultCost** + **FixedBpsSlippage**：内置成本模型。
7. **工件持久化**：JSON + CSV，目录结构如本文第八节。

### 10.2 MVP 不包含

1. **compare 动作**：第二版实现。
2. **多引擎支持**：Protocol 接口已定义，但第一版只实现 BacktraderAdapter。
3. **sector_limit 约束**：需要行业分类数据支持，等数据域补齐后实现。
4. **分钟频回测**：Feed 构建和性能优化复杂度较高，延后。
5. **实时信号消费**：第一版只支持批量 signal snapshot，不支持流式信号。
6. **多 benchmark 对比**：第一版只支持单 benchmark。
