# 共享契约与配置模型说明书 v2

> **定位**：本文档回答两个核心问题：
> 1. Vortex 对外让用户配置什么——Profile 是什么、怎么组织、怎么解析。
> 2. 各业务域共享哪些不可变契约——ID、时间、审计、确定性、错误、工件格式、血统。
>
> 架构总览见 [[软件架构总览]]；代码级签名与 DDL 详见 [[代码细节设计说明书]]。

---

# Part A：配置模型

---

## 一、背景

在 Vortex 的设计里，使用者视角收敛到 4 个核心对象：`profile`、`snapshot`、`workflow`、`report`。如果不把 `profile` 讲清楚，系统仍然会退化成"看起来有统一命令，实际还是一堆脚本参数"。

本章解决三个问题：

1. `profile` 到底承载哪些用户可见参数。
2. `profile` 和数据版本、研究流程、策略回测、交易执行是什么关系。
3. `profile` 是不是应该做成一个超级大 YAML。

结论很明确：

1. `profile` 必须是用户的**第一配置对象**。
2. `profile` 不能是一个巨大的全局配置文件。
3. `profile` 应该是"**统一命名、分域建模、稳定对外**"的配置契约。

### 本文档的边界

> 为避免 02 和 06 的内容边界模糊，明确划分如下。

| 02 定义 | 02 不定义（归属 06 或各域文档） |
|---------|-------------------------------|
| 共享不变量与语义契约 | 函数签名、类接口 |
| Profile 分型、字段类别、生命周期约束 | DDL 字段清单、JSON Schema 细节 |
| 跨域工件的必需字段类别与 owner/consumer | 状态机细节、实现级时序图 |
| ID / 时间 / 审计 / 确定性的架构级约束 | ID 编码细则（前缀字典、随机段长度）、错误码枚举 |
| 契约版本化原则 | 具体存储形态（文件、SQLite 或 JSONL） |
| 审计事件的语义契约 | 审计日志的物理存储格式与路径 |

## 二、Profile 核心理念

### 2.1 Profile 是对外配置契约，不是内部设置转储

用户看到的 `profile`，本质上表达：

1. 我在哪个**业务上下文**里工作。
2. 这个上下文默认使用什么**口径**。
3. 这次动作允许我改哪些参数，不允许我改哪些参数。

所以 `profile` **不应该**直接暴露：Parquet 物理路径、DuckDB 连接细节、provider token 原文（如 Tushare API 密钥）、内部缓存目录、内部类名。这些属于内部解析结果，不属于用户配置契约。

### 2.2 统一命名，但按领域分型

从 CLI 看，用户永远写 `--profile xxx`，这是统一体验。但从配置模型看，不应该只有一个"大一统 Profile"。按领域分成 4 类：

| Profile 类型 | 关注点 |
|-------------|--------|
| **DataProfile** | 数据口径和快照产出方式——provider、market、freq、quality |
| **ResearchProfile** | 研究怎么吃快照、怎么产出信号——snapshot、feature、label、workflow |
| **StrategyProfile** | 信号如何编译成可回测、可执行的目标组合——signal、portfolio、cost、slippage |
| **TradeProfile** | 目标组合如何送进模拟盘或实盘——account、gateway、retry、reconcile |

如果把所有字段塞进一个 profile，使用者看到一堆与自己无关的字段，最后还是不敢改。

### 2.3 Profile 管"环境与口径"，Snapshot 管"版本固定"

这两个对象不能混：

- `profile` 说明"**应该怎样工作**"——长期稳定配置。
- `snapshot` 说明"**这次实际用了哪一版数据**"——当天生成的具体版本。

例如：`cn_alpha_daily` 是一个 ResearchProfile；`2026-04-04` 是一个 research snapshot。

### 2.4 使用者脑海里的正确模型

下图展示四类 Profile 如何串联成端到端的数据流——每一步的产出是下一步的输入：

```text
DataProfile
  → 生成 Snapshot
  → 被 ResearchProfile 消费
  → 产出 SignalSnapshot
  → 被 StrategyProfile 消费
  → 产出 TargetPortfolio
  → 被 TradeProfile 消费
  → 产出 ExecutionReport / ReconciliationReport
```

这比"一个 profile 管全世界"更符合真实工作分工。

## 三、配置合并模型

### 3.1 设计哲学：框架给默认，用户只写差异

好的配置系统不是"允许填很多东西"，而是"框架已经有一套合理的默认配置，用户只需要写自己想改的部分"。

核心原则：
- 框架自带**完整的默认配置**（default profile），覆盖所有字段，保证开箱即用。
- 用户配置文件**只写差异**——你不需要知道有多少字段，只需要写"我和默认值不一样的地方"。
- 系统启动时自动合并：**用户配置覆盖默认配置**，相同字段以用户为准，用户没写的字段用默认值。
- 合并后输出一份 **`.resolved.yaml`**（dump 文件），让用户能看到最终实际运行的完整配置。

### 3.2 两层合并模型

```text
┌─────────────────────────────────────────────────┐
│  框架默认层（Default Profile）                     │
│  框架内置的完整默认配置，覆盖所有字段                 │
│  放在 defaults/ 目录，按 profile 类型分类            │
│  例如 defaults/data/cn_stock.yaml                  │
└───────────────────────┬─────────────────────────┘
                        │ 用户配置覆盖（深度合并）
┌───────────────────────▼─────────────────────────┐
│  用户层（User Profile YAML）                      │
│  用户只写想改的字段，通常只有几行                     │
│  放在 profiles/ 目录                               │
│  例如 profiles/data/my_daily.yaml                  │
└───────────────────────┬─────────────────────────┘
                        │ CLI 临时覆盖（可选）
                        │ --set key=value
                        ▼
              最终合并结果（ResolvedProfile）
              输出为 .resolved.yaml 供用户检查
```

**配置优先级**（从低到高依次覆盖）：

| 优先级 | 层 | 来源 | 解决的问题 |
|--------|------|------|-----------|
| 1（最低） | 框架默认层 | 框架内置的完整默认配置 | 开箱即用，不需要用户从零写起 |
| 2 | 用户配置层 | 用户 profile YAML 文件 | 用户只写差异，覆盖默认值 |
| 3（最高） | 运行时覆盖层 | CLI `--set key=value` | 今天这次只改一点点 |

### 3.3 合并规则

1. **深度合并**：嵌套对象逐层合并，不是整个替换。例如用户只写了 `cost_model.commission_bps: 5`，框架默认的 `cost_model.tax_bps: 10` 仍然保留。
2. **列表替换**：列表类型字段（如 `datasets`）整个替换，不做元素级合并——因为列表的顺序和完整性通常有语义。
3. **null 显式清除**：用户写 `field: null` 表示显式清除该字段（而非使用默认值）。
4. **类型校验**：合并后立即校验字段类型，用户不能把一个数值字段覆盖为字符串。

### 3.4 `.resolved.yaml` dump 文件

每次 run 开始前，系统将最终合并结果输出为 `.resolved.yaml`，放在 run 工件目录下：

```bash
# 查看最终配置（不执行任何业务）
vortex profile resolve --profile my_daily

# 实际运行时也会自动生成
# → runs/{run_id}/.resolved.yaml
```

`.resolved.yaml` 包含：
- 所有字段的最终值（来自默认 or 用户 or CLI override）
- 每个字段标注来源（`# [default]`、`# [user]`、`# [override]`）
- 别名解析结果（如 `snapshot: latest_success → 2026-04-04`）
- `config_hash`（用于可复现性校验）

这样用户遇到"为什么跑出来的结果不对"时，直接看 `.resolved.yaml` 就能确认配置是否符合预期。

### 3.5 字段分类

所有 profile 字段分为 3 类：

| 类别 | 含义 | 示例 |
|------|------|------|
| **必选字段** | 用户必须声明，框架默认不提供 | `name`、`type` |
| **可选字段** | 有默认值，用户想改才写 | `datasets`、`universe`、`benchmark`、质量阈值 |
| **禁止直填** | 只能由系统派生，用户不应碰 | 物理路径、凭证、内部 backend 类名 |

### 3.6 建议目录结构

```text
defaults/                    # 框架内置默认配置（随框架发布）
  data/
    cn_stock.yaml            # A 股日频数据默认配置
  research/
    cn_alpha.yaml            # A 股因子研究默认配置
  strategy/
    cn_equity.yaml           # A 股策略默认配置
  trade/
    cn_equity.yaml           # A 股交易默认配置

profiles/                    # 用户配置（只写差异）
  data/
    my_daily.yaml            # 几行：name + type + 想改的字段
  research/
    my_alpha.yaml
  strategy/
    dividend_classic.yaml
  trade/
    dividend_paper.yaml
```

### 3.7 所有 Profile 共享的基础字段

不管哪一类 profile，都保留一组统一基础字段：

1. `name`：profile 唯一名称（必选）。
2. `type`：profile 类型，取值为 `data / research / strategy / trade`（必选）。
3. `base`：继承哪个默认配置（可选，默认按 type 匹配框架默认配置）。
4. `description`：用途说明。
5. `owner`：责任人或团队。
6. `tags`：检索标签，例如 `cn`、`daily`、`paper`。
7. `enabled`：是否启用。

## 四、Profile 解析流程

### 4.1 标准解析链

```text
CLI/API
  │  --profile, --snapshot, --set
  ▼
ProfileStore ── 根据名字和类型找到用户 YAML 文件
  │
  ▼
DefaultLoader ── 加载对应类型的框架默认配置
  │
  ▼
ProfileMerger ── 深度合并（默认 ← 用户 ← CLI override）
  │
  ▼
ProfileValidator ── 字段校验 + 跨字段语义校验 + 命令空间匹配
  │
  ▼
SnapshotSelector ── 解析 latest_success 等别名为具体 snapshot
  │
  ▼
ProfileResolver ── 产出 ResolvedProfile（冻结）+ 写 .resolved.yaml
  │
  ▼
业务模块 ── 只消费 ResolvedProfile，不再关心合并细节
```

> 以上组件名（ProfileStore、DefaultLoader 等）是参考实现术语，描述职责分工。具体类名和接口签名由 06 定义。

### 4.2 解析冻结规则

**这是架构级约束**：

1. **冻结时机**：run 开始前，所有 alias（如 `latest_success`）、override、extends 必须完成解析，产出不可变的 ResolvedProfile。
2. **冻结后不可变**：run 进行中禁止动态修改配置。如需调整，必须终止当前 run 并重新发起。
3. **冻结结果入审计**：ResolvedProfile 的完整快照（含 resolved snapshot_ref、所有 override 值）必须写入血统记录和审计日志。
4. **config_hash 覆盖范围**：`config_hash` 必须覆盖 ResolvedProfile 的全部运行时有效字段，包括 override 值。
5. **别名解析结果透明**：系统在输出中必须打印最终解析结果，例如 `snapshot=latest_success → resolved_snapshot=2026-04-04`。

### 4.3 关键边界

- 业务模块不应该自己读 YAML。
- 业务模块不应该自己解析 `extends`。
- 业务模块不应该自己猜 `latest_success` 对应哪一天。

## 五、各类 Profile 用户可见字段

### 5.1 DataProfile

**框架默认配置**（`defaults/data/cn_stock.yaml`，随框架发布）：

```yaml
# 框架默认 DataProfile —— A 股日频全量数据
type: data
market: cn_stock
universe: all_a
freqs: [1d]
provider: tushare
datasets: all                     # all = Tushare 全量接口
timezone: Asia/Shanghai
calendar: xshg
history_start: 2015-01-01
schedule: weeknights_2200

quality:
  mode: fail_closed               # 质量不通过就阻断
  rules: [required_columns, primary_key_unique, calendar_consistency, adj_factor_sanity]

pit:
  mode: strict
  ann_date_field: ann_date
  disclosure_dataset: disclosure_date

publish:
  publish_time: "22:00"
  selector: latest_success
  retain_days: 365

notify: default
```

**用户配置**（`profiles/data/my_daily.yaml`，只写差异）：

```yaml
name: my_daily
type: data
history_start: 2018-01-01         # 只想要 2018 年以后的数据
freqs: [1d, 1m]                   # 加上分钟频
notify: data_team                 # 通知发给我的团队
```

合并后系统自动生成 `.resolved.yaml`，包含框架默认 + 用户覆盖的全部字段，每个字段标注来源。

**不应暴露**：Parquet 根目录、DuckDB 连接细节、provider token 原文、内部缓存目录。

### 5.2 ResearchProfile

**用户配置**（`profiles/research/my_alpha.yaml`）：

```yaml
name: my_alpha
type: research
data_profile: my_daily            # 引用我的数据配置
label_spec:
  name: next_5d_return
  horizon: 5
  transform: rank
signal_output:
  topk: 100
```

框架默认值已包含：`market: cn_stock`、`universe: all_a`、`freq: 1d`、`feature_set: alpha158_cn_v1`、`qlib_workflow: qlib_alpha_daily` 等。用户不写就用默认值。

**不应暴露**：Qlib 物理目录、MLflow URI、实验工件物理路径、中间缓存路径。

### 5.3 StrategyProfile

**用户配置**（`profiles/strategy/dividend_classic.yaml`）：

```yaml
name: dividend_classic
type: strategy
research_profile: my_alpha
benchmark: 000300.SH

rebalance_rule:
  frequency: monthly
  trading_day: 1

portfolio_constraints:
  max_names: 30
  max_weight_per_name: 0.08
```

框架默认值已包含：`cost_model`（佣金/税费）、`slippage_model`、`risk_pack`、`backtest_engine: backtrader`、`report_template` 等。大多数用户不需要改这些。

**不应暴露**：Backtrader adapter 类路径、内部 broker stub 实现名、低层订单模拟对象细节。

### 5.4 TradeProfile

**用户配置**（`profiles/trade/dividend_paper.yaml`）：

```yaml
name: dividend_paper
type: trade
strategy_profile: dividend_classic
mode: paper
gateway: miniqmt
account: cn_equity_paper_01
```

框架默认值已包含：`trading_window`（09:30-14:55）、`order_policy`（limit + twap_like）、`retry_policy`（max_retries: 2）、`risk_pack`、`reconcile_policy`（收盘后自动对账）等。

**不应暴露**：miniQMT 主机地址细节、认证密钥原文、网关 session 内部状态、RPC 连接对象。

### 5.5 最小 Profile 可以短到什么程度

如果合并模型设计正确，大多数 profile 只有 2-5 行：

```yaml
# 数据：用默认配置，只改起始日期
name: my_data
type: data
history_start: 2020-01-01
```

```yaml
# 策略：只改基准
name: smallcap_strategy
type: strategy
research_profile: my_alpha
benchmark: 000905.SH
```

```yaml
# 交易：只选账户和模式
name: my_live
type: trade
strategy_profile: dividend_classic
mode: live
account: cn_equity_live_02
```

如果用户仍然普遍需要手写 20+ 个字段，说明框架默认配置设计得不够好。

## 六、运行时覆盖

### 6.1 CLI 使用方式

大多数命令只需要 `--profile`。次常用场景加上 `--snapshot` 或 `--as-of`。极少数场景允许 `--set key=value` 小覆盖：

```bash
vortex data sync --profile cn_stock_daily
vortex research run --profile cn_alpha_daily --snapshot latest
vortex strategy backtest --profile dividend_classic --set benchmark=000905.SH
```

设计重点不是让 override 很强，而是让 override **很克制**。

### 6.2 架构约束

1. **命令空间限定类型**：`vortex data *` 只接受 DataProfile，`vortex research *` 只接受 ResearchProfile，依此类推。类型不匹配时入口层立即报错。
2. **override 只在当次 run 生效**，不持久化到 YAML 文件。
3. **override 必须进入 config_hash**：运行时覆盖的值参与 config_hash 计算，确保可复现。
4. **别名必须解析后透明展示**：`latest_success` → `2026-04-04`，解析结果打印到输出并记入血统。

### 6.3 解析预览与 dump

系统提供解析预览命令，让用户先看到最终合并配置再决定是否执行：

```bash
vortex profile resolve --profile my_daily
```

输出展示：
- 每个字段的最终值
- 字段来源标注（`[default]` / `[user]` / `[override]`）
- 别名解析结果（`snapshot: latest_success → snap_20260404_220000_b2c3`）
- config_hash

实际运行时，`.resolved.yaml` 自动写入 run 工件目录，用于事后审计和问题排查。

## 七、Profile 验证规则

Profile 验证分三个层次：

### 7.1 字段级校验

- 必选字段不为空（`name`、`type`）。
- 字段类型正确（`freqs` 是列表、`commission_bps` 是数值）。
- 枚举值合法（`type` ∈ `{data, research, strategy, trade}`，`mode` ∈ `{paper, live}`）。

### 7.2 跨字段语义校验

- `research_profile` 或 `signal_source` 引用的对象必须存在。
- `risk_pack` 引用的 pack 必须已注册。
- `live` 模式的 TradeProfile 必须配置非空 `risk_pack`。
- 用户引用的 `base` profile 必须存在且类型匹配。

### 7.3 命令空间匹配校验

- `vortex data *` 命令传入非 DataProfile 时立即拒绝。
- 校验在解析链的早期执行，不等运行到一半才失败。

# Part B：跨域共享契约

> Part A 定义了"用户输入什么"（配置模型）。Part B 定义各业务域协作时必须共同遵守的不变量——ID 如何编码、时间如何定义、工件如何标记血统等。
>
> Part B 定义的契约由共享契约层（Shared Contract Layer）承载。共享契约层不包含业务逻辑，只提供跨域协调所需的标识、时间、审计、确定性、错误和工件格式约定。所有业务域依赖共享契约层，共享契约层不依赖任何业务域。

---

## 八、ID 生成规范

### 8.1 设计目标

所有系统生成的标识符必须满足：

1. **全局唯一**：在整个 Vortex 实例生命周期内不冲突。
2. **可排序**：从 ID 可推断大致生成时间。
3. **自描述**：从 ID 可判断所属域或工件类型。
4. **无外部依赖**：不依赖中心化发号器（单机系统）。

### 8.2 ID 语义组成

各类 ID 的语义结构：

| ID 类型 | 语义组成 | 用途 | 示例（参考格式） |
|---------|---------|------|----------------|
| `run_id` | **运行类型前缀** + 日期 + 时间 + 随机段 | 标识一次完整的业务运行 | `backtest_20260403_223118_c3d4` |
| `snapshot_id` | 前缀 + 日期 + 时间 + 随机段 | 标识一次数据快照发布 | `snap_20260403_220000_b2c3` |
| `signal_id` | 前缀 + 日期 + 时间 + 随机段 | 标识一次信号发布（严格不可变） | `sig_20260403_223000_c3d4` |
| `order_id` | 执行 ID + 序号 | 标识一笔订单（执行域内部） | `exec_20260403_0930_d4e5_001` |

**关于 `run_id` 前缀**：前缀反映的是**运行类型（run type）或工作流类型（workflow type）**，而非简单等同于域名称。例如同属 Research 域的 `factor_eval` 和 `model_train` 应使用不同前缀，因为它们是语义不同的工作流。前缀字典（哪些前缀合法、如何与域关联）由 06 定义。

> **架构约束**：ID 必须包含时间信息（可排序）和类型前缀（自描述）。随机段的长度和编码方式由 06 根据并发需求确定，但必须满足实际并发场景下的唯一性要求。

### 8.3 ID 不可变性

- `run_id` 在 run 创建时生成，整个生命周期不变。
- `signal_id` 一经发布不可修改、不可覆盖。如需修正，必须重新 publish 生成新 signal_id。

### 8.4 `snapshot_id` 与 `revision_id` 的语义

Data 快照采用**逻辑快照 + 物理修订**双层标识，需要区分以下层面：

1. **逻辑快照（as_of）**：代表某个交易日的数据切面。`snapshot_id` 格式为 `snap_{as_of}_{publish_timestamp}_{hash_4}`，每次 publish 生成唯一值。

2. **物理修订（revision_id）**：同一 `as_of` 可以多次 publish，每次生成不同的 `revision_id`（即每次的 `snapshot_id` 本身就是唯一的 revision 标识）。系统维护 `latest_revision_for(as_of)` 映射，指向最新发布的版本。

3. **下游引用规则**：下游域（Research/Strategy/Trade）在血统中记录 `snapshot_revision_id`（即具体的 `snapshot_id`），而非仅记录 `as_of`。旧 revision 的物理文件保留（标记 `superseded`），确保旧 run 可严格复现。新 run 默认使用 `latest_revision_for(as_of)`。

4. **与 signal_id 的区别**：signal_snapshot 是比 data snapshot 更严格的不可变——publish-signal 后内容不可修改、不可重发布。这是因为 signal 是交易链路的直接输入，覆盖风险更高。

> **清理机制**：旧 revision 默认保留。用户可通过 `vortex data gc` 清理无下游引用的 superseded revision。

## 九、时间与日历契约

### 9.1 核心时间概念

| 概念 | 定义 | 示例 |
|------|------|------|
| **交易日**（Trading Day） | 由交易日历定义的可交易日期 | 2026-04-03 |
| **快照截止时间**（Snapshot Cutoff） | 数据快照的截止时刻，默认 22:00 | 22:00 后的数据归入次日快照 |
| **PIT 可见日**（Point-in-Time Visible Date） | 数据可被研究/回测看到的最早日期 | 财报公告日（ann_date） |
| **调仓日**（Rebalance Day） | 策略规定的调仓日期 | 每月第一个交易日 |
| **交易窗口**（Trading Window） | 订单允许提交的时段 | 09:30-15:00 |

### 9.2 时区约定

**架构约束**：

1. 默认时区为 `Asia/Shanghai`。
2. 所有时间戳必须带时区信息（ISO 8601 格式，如 `2026-04-03T22:00:00+08:00`）。
3. 时区必须配置化，为多市场预留扩展能力。
4. 系统内部传参和存储统一使用带时区的时间戳，不依赖运行环境的本地时区设置。

### 9.3 交易日历服务

- 交易日历由 Data 域获取并缓存（来自 DataProvider）。
- 其他域通过共享契约层的日历服务（参考实现：CalendarService）访问，不直接调用 DataProvider。
- 默认 A 股日历（SSE / SZSE）；多市场通过配置扩展。
- 日历服务提供的能力：查询是否交易日、获取区间内交易日列表、获取前/后 N 个交易日。

### 9.4 PIT（Point-in-Time）可见日规则

**这是架构级不变量**（对应 00 原则 P4：严格 PIT）：

1. 任何数据只有在**公告日（ann_date）之后**才对研究/回测可见。
2. 绝不允许未来信息泄露。
3. PIT 对齐失败时**阻断流程**（Fail-Closed，即失败即阻断），不允许跳过或静默降级。

### 9.5 迟到数据与修正

1. **迟到数据**：下次 update 自动补入；已发布快照不受影响。
2. **数据修正**：手动 backfill + 重新 publish 新快照。旧快照数据文件可能被覆盖，但旧 run 血统中的引用保持可追溯。

## 十、审计契约

### 10.1 审计事件模型

审计系统记录所有需要事后追溯的操作。审计记录的存储形态（追加日志文件、SQLite 表或 JSONL）由 06 确定，但必须满足以下架构约束。

**审计事件必须包含的字段**：

| 字段 | 含义 |
|------|------|
| 时间戳 | ISO 8601 带时区 |
| 域 | 发生操作的业务域 |
| 动作 | 操作类型（如 submit_order、config_change、manual_confirm） |
| 操作人 | 发起操作的用户或调度器标识 |
| run_id | 关联的运行标识 |
| 结果 | 操作结果（success / failed / blocked） |
| 详情 | 操作相关的补充信息（可选） |

### 10.2 必须审计的操作

以下操作**必须写入审计记录**，不可配置跳过：

| 类别 | 具体操作 |
|------|---------|
| **实盘交易** | 订单提交、订单成交、订单取消、风控阻断 |
| **配置变更** | Profile 创建/修改/删除、pack 变更 |
| **手动确认** | `--confirm` 绕过确认、手动 retry、手动 repair |
| **数据发布** | snapshot publish、signal publish |
| **关键状态转换** | run 状态变为 failed / blocked |

### 10.3 不可删除性

**架构约束**：审计记录是**追加不可删除（append-only）**的。系统不提供删除审计记录的 API 或命令。归档策略允许将旧审计记录迁移到冷存储，但不允许丢弃。

## 十一、确定性契约

> 对应 00 原则 P1（确定性优先）：相同输入 + 相同环境 = 相同输出。

### 11.1 威胁与防御

| 威胁 | 来源 | 架构级防御 |
|------|------|-----------|
| 浮点精度漂移 | 不同 CPU/库版本 | 排序用 rank（整数），关键指标定义精度容忍度 |
| 随机性 | 回测抽样、模型训练 | 种子固定规则（见 §11.2） |
| 排序不稳定 | 相同分值排序不确定 | 所有排序必须定义二级键（如 `score desc, symbol asc`） |
| 并发竞争 | 多 run 同时执行 | 每个 run 独立 workdir，缓存按 `code_hash + snapshot` 隔离 |
| 时间依赖 | `datetime.now()` | 时间显式传参（见 §11.3） |
| 依赖版本漂移 | pip 更新 | lockfile 固定；run 记录 `vortex_version` |
| 数据修订 | 数据源修正历史 | 快照不可变；修正生成新快照 |

### 11.2 种子规则

**架构约束**：任何涉及随机性的操作，种子必须可确定地派生：

```text
seed = hash(profile_name + snapshot_id)
```

- 不允许使用不固定种子的随机操作进入正式 run（探索研究除外）。
- 种子值记入血统（lineage），确保可复现。

### 11.3 时间显式传参

**架构约束**：

1. **禁止在业务逻辑中隐式获取当前时间**（如 `datetime.now()`、`time.time()`）。
2. 所有需要"当前时间"的逻辑，必须通过显式参数（如 `as_of`、`run_time`）传入。
3. 时间注入点应集中在 run 的入口层，不散落在业务模块内部。

### 11.4 环境契约

run 开始前必须满足以下环境条件：

1. Python >= 3.10。
2. 依赖版本与 lockfile 一致。
3. 时区正确配置（`Asia/Shanghai` 或 profile 指定值）。
4. 磁盘空间足够（bootstrap 前检查）。

### 11.5 可复现性验证

- **golden master（黄金基准）**：固定输入 + 固定种子，输出与基准逐字节比对。
- **差分测试**：同一输入运行两次，结果必须完全一致。
- 详见 07 测试方案。

## 十二、错误模型

### 12.1 错误层次

所有 Vortex 内部错误继承自统一基类 `VortexError`，按业务域分层管理，确保上层（CLI、调度器）能快速定位问题来源并采取相应恢复措施：

```text
VortexError（基类）
├── ConfigError            配置解析/验证失败
├── DataError              数据域
│   ├── ProviderError        数据源获取失败
│   ├── QualityError         质量校验不通过
│   └── PITError             PIT 对齐失败
├── ResearchError          研究域
│   ├── FactorError          因子计算失败
│   └── ModelError           模型训练失败
├── StrategyError          策略域
│   ├── BacktestError        回测执行失败
│   └── CompileError         组合编译失败
├── TradeError             交易域
│   ├── RiskBlockError       风控阻断
│   ├── GatewayError         网关通信失败
│   └── ReconcileError       对账异常
└── NotificationError      通知投递失败（不阻断主流程）
```

> 具体错误码枚举和子类细节由 06 定义。

### 12.2 错误处理策略

| 错误类型 | 处理方式 | 是否阻断 | 通知级别 |
|---------|---------|---------|---------|
| QualityError | 记录 + 阻断 + 写 health_report | **阻断** | error |
| PITError | 记录 + 阻断 + 写 pit_report | **阻断** | error |
| ProviderError（网络） | 重试 → 记录失败单元 | 部分阻断 | warn |
| RiskBlockError | 记录 + 阻断整批 + pre_trade_result | **阻断** | error |
| GatewayError | 重试 + 记录 + 阻断 | **阻断** | error |
| NotificationError | 记录，不阻断主流程 | 不阻断 | 仅日志 |

**Fail-Closed 原则**（对应 00 原则 P2）：质量、PIT、风控校验失败时阻断流程，不允许跳过或静默降级。

### 12.3 错误输出契约

所有错误必须产出结构化信息，至少包含以下字段：

```json
{
  "error_type": "QualityError",
  "domain": "data",
  "run_id": "sync_20260404_220000_a1b2",
  "message": "critical rule 'not_empty' failed on dataset 'daily_bars'",
  "blocking": true,
  "details": {},
  "timestamp": "2026-04-04T22:01:30+08:00"
}
```

错误信息必须包含足够上下文（run_id、domain、blocking 状态），使得下游系统和运维人员可以据此判断影响范围和恢复措施。

## 十三、跨域工件 schema 契约

### 13.1 设计原则

域间数据传递以**不可变工件（Artifact）**为单位（对应 00 原则 P5：快照即版本）。禁止通过内存引用或共享数据库连接传递。

### 13.2 工件所有权

每份工件有且仅有一个 owner（产出方），其他域只能只读消费。**schema 定义权**决定了谁有权变更工件的字段结构：

| 工件 | Schema 定义方 | Owner（产出方） | Consumer（消费方） | 传递方式 |
|------|-------------|----------------|-------------------|---------|
| `snapshot_descriptor` | 共享契约层 + Data 域 | Data | Research, Strategy | 不可变路径引用 |
| `signal_snapshot_descriptor` | 共享契约层 + Research 域 | Research | Strategy | 不可变路径引用 |
| `target_portfolio` | 共享契约层 + Strategy 域 | Strategy | Trade | JSON 文件 |
| `execution_report` | 共享契约层 + Trade 域 | Trade | Notification, 审计 | JSON 文件 |
| `reconcile_report` | 共享契约层 + Trade 域 | Trade | Notification, 审计 | JSON 文件 |
| `ResolvedProfile` | 共享契约层 | 共享契约层 | 所有域 | 内存对象（run 内） |

**所有权规则**：

1. **共享契约层定义基础字段类别**（血统、schema_version 等所有工件共有的部分）。
2. **Owner 域定义域特有字段类别**（如 Data 定义 snapshot_descriptor 中的数据范围和质量摘要字段）。
3. **Consumer 只读**：消费方不得修改工件内容，不得在自己的代码中假设未在契约中声明的字段。
4. **字段变更需双方协调**：Owner 域变更自身工件的 schema 时，必须通知所有 Consumer 域并更新 schema_version。

### 13.3 各工件必需字段类别

以下描述每种工件在架构层面**必须包含的字段类别**。具体字段名和 JSON Schema 由 06 定义。

> **通用必需字段**：所有跨域工件都必须包含 `schema_version`（契约版本）和 `lineage`（血统基础字段）。以下各工件表格中不再重复列出这两项。

#### snapshot_descriptor（数据快照描述符）

数据域 publish 时生成，描述一次快照发布的完整元信息。

| 字段类别 | 含义 |
|---------|------|
| 快照标识 | snapshot_id（同时作为 revision_id）、as_of 日期、revision_status（current / superseded） |
| 数据范围 | 市场、标的范围、频率、包含的数据集列表 |
| 数据质量摘要 | 质量校验结果、通过的规则数 / 总规则数 |
| 产出信息 | 产出时间、关联的 data profile、run_id |
| 存储位置 | 各数据集的引用路径 |
| 血统 | lineage 基础字段（见 §十四） |

#### signal_snapshot_descriptor（信号快照描述符）

研究域 publish-signal 时生成。**严格不可变**——一经发布不可修改。

| 字段类别 | 含义 |
|---------|------|
| 信号标识 | signal_id |
| 来源快照 | 依赖的 snapshot_id |
| 信号内容摘要 | 标的数量、信号值范围、生成方法标识 |
| 产出信息 | 产出时间、关联的 research profile、run_id |
| 存储位置 | 信号数据的引用路径 |
| 血统 | lineage 基础字段 + 研究域扩展字段 |

#### target_portfolio（目标组合）

策略域 compile 时生成，描述调仓目标。

| 字段类别 | 含义 |
|---------|------|
| 组合标识 | portfolio_id、版本 |
| 来源信号 | 依赖的 signal_id |
| 持仓明细 | 目标权重、标的列表、目标现金比例 |
| 约束摘要 | 使用的组合约束和风控规则 |
| 产出信息 | 产出时间、关联的 strategy profile、run_id |
| 血统 | lineage 基础字段 + 策略域扩展字段 |

#### execution_report（执行报告）

交易域执行完成后生成。**执行后不可修改**。

| 字段类别 | 含义 |
|---------|------|
| 执行标识 | execution_id、关联的 target_portfolio_id |
| 订单汇总 | 订单数量、成交数量、拒绝数量 |
| 执行明细 | 每笔订单的状态、成交价格、成交数量 |
| 风控结果 | 事前/事中风控的通过/阻断记录 |
| 产出信息 | 执行时间、关联的 trade profile、run_id |
| 血统 | lineage 基础字段 + 交易域扩展字段 |

#### reconcile_report（对账报告）

交易域收盘后对账生成。

| 字段类别 | 含义 |
|---------|------|
| 对账标识 | reconcile_id、关联的 execution_id |
| 比对结果 | 网关持仓 vs 本地记录的差异 |
| 偏差明细 | 差异标的、差异数量、差异原因分类 |
| 结论 | 是否一致、是否需要人工介入 |
| 产出信息 | 对账时间、run_id |
| 血统 | lineage 基础字段 |

### 13.4 契约版本化规则

**架构约束**：

1. 每个跨域工件必须携带 `schema_version` 字段，标识其遵循的 schema 版本。
2. schema 变更时遵循语义化版本（Semantic Versioning）：
   - **新增可选字段**：minor 版本升级，下游向后兼容。
   - **删除字段或变更语义**：major 版本升级，下游必须适配。
3. 具体版本号管理和兼容性实现由 06 定义。

**兼容性落地机制**：

4. **兼容责任方规则**：reader（消费方）必须容忍 minor 版本差异——遇到未知字段时忽略（不报错），遇到缺失的可选字段时使用默认值。writer（生产方）升级 schema_version 时必须在 CHANGELOG 中记录变更。
5. **历史版本支持范围**：系统至少支持读取 **当前 major 版本和前一个 major 版本** 的工件。例如当前 schema_version = "2.x" 时，仍能读取 "1.x" 的工件。超出范围的旧版本需通过 `vortex migrate` 升级。
6. **升级失败行为**：如果读取到不兼容的 schema_version（major 版本跨度超出支持范围），**Fail-Closed 阻断当前操作**，输出明确错误信息（包含当前版本、目标版本、建议的迁移命令），不做静默降级。

**Profile 配置的版本化**：

7. `.resolved.yaml` 必须包含 `profile_schema_version` 字段，记录解析该 profile 使用的 schema 版本。当框架升级导致 profile schema 变更时，`vortex profile explain` 会提示需要执行 `vortex config migrate`。
8. 框架默认配置文件本身必须包含 `schema_version` 注释或字段，用于检测用户配置与框架默认之间的版本差异。

### 13.5 不可变性级别

| 工件 | 不可变性级别 | 违反处理 |
|------|------------|---------|
| snapshot_descriptor | 可重发布（生成新 revision_id，旧 revision 保留为 superseded） | 下游 run 通过 lineage 中的 `snapshot_revision_id` 精确追溯 |
| signal_snapshot_descriptor | **严格不可变** | 如需修正，生成新 signal_id |
| target_portfolio | 编译后不可修改 | 需重新编译生成新版本 |
| execution_report | 执行后不可修改 | 追加 reconcile_report 记录偏差 |

## 十四、血统（Lineage）契约

### 14.1 设计目标

对应 00 原则 P6（血统可追溯）：任何产出物必须能追溯到——使用了哪个快照、哪个配置版本、哪次运行。

### 14.2 基础字段

所有工件的血统记录**必须包含**以下基础字段：

| 字段 | 含义 |
|------|------|
| `run_id` | 产出本工件的运行标识 |
| `profile` | 使用的 profile 名称 |
| `snapshot_revision_id` | 依赖的数据快照修订标识（即具体的 snapshot_id，确保可精确复现） |
| `parent_run_id` | 父 run 标识（repair/retry 场景；首次运行为空） |
| `created_at` | 产出时间（ISO 8601 带时区） |
| `vortex_version` | Vortex 系统版本号 |
| `config_hash` | ResolvedProfile 的哈希值（含 override） |

**示例**（参考格式）：

```json
{
  "lineage": {
```

### 14.3 工作流扩展字段

各域根据自身复现需求，在基础字段之上追加扩展字段：

| 域 / 工作流 | 扩展字段类别 | 用途 |
|-------------|------------|------|
| Research / factor_eval | 因子名称、因子代码哈希、标签规格、预处理配置 | 因子复现 |
| Research / model_train | 特征集、训练/验证/测试划分、模型类型、超参数 | 模型复现 + 泄露审计 |
| Strategy / backtest | signal_id、回测模式、种子值、基准、费用模型、调仓频率 | 回测复现 |
| Trade / execution | target_portfolio_id、网关类型、风控规则包、订单 ID 前缀 | 执行审计 |

### 14.4 血统链完整性

端到端的血统链确保从最终交易结果可以逆向追溯到原始数据：

```text
execution_report
  → target_portfolio (via target_portfolio_id)
    → signal_snapshot (via signal_id)
      → data_snapshot (via snapshot_id)
        → 原始数据源 + 同步记录
```

**架构约束**：任何打断此链条的操作（如引用未发布的中间产物）都是禁止的。
