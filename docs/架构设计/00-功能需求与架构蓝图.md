# Vortex Quant 架构蓝图（确认版）

> 主线固定：自建数据（Tushare） -> 因子研究（Qlib） -> 策略编写（参考 RiceQuant + vn.py） -> 策略回测（Backtrader） -> 模拟盘与实盘（miniQMT）。

---

## 一、已确认输入（2026-04-04）

1. 市场范围：不预设市场边界；只要有可用数据，就允许研究与策略开发（A股/美股/其他）。
2. 频率范围：日频与分钟频都要支持。
3. 数据可研究快照时间：默认 22:00，必须配置化可调整。
4. 财务口径：严格 PIT（仅公告后可见）。
5. 默认股票池：默认全A。
6. 质量策略：阻断流程（Fail-Closed），先保证流程正确再谈容错。
7. 核心工程要求：数据接口与存储后端必须可替换。

### 2.1 RiceQuant

2. 目标组合优先：order_target_portfolio 思想。

3. App 化拆分：策略、回测、风控、数据、组合管理独立演进。

1. 策略层采用 RiceQuant 风格统一 API。
2. 执行层采用 vn.py 风格事件驱动与网关抽象。
3. 研究中枢采用 Qlib；执行精确验证采用 Backtrader；生产执行采用 miniQMT。

## 三、总架构（按主线重排）

```text
Pluggable Data Providers
   -> Data Lake + Metadata (authoritative)
   -> Qlib Research Core
   -> Strategy API Layer (RiceQuant style)
   -> Backtrader Verification Engine
   -> Gateway Layer (Paper + miniQMT + future brokers)
```

### 3.1 域划分

1. Data：采集、校验、PIT、发布快照。
2. Research：Qlib 因子/模型/workflow/experiment。
3. Strategy：统一策略 API、调度、信号。
4. Backtest：Backtrader 订单生命周期与成本验证。
5. Trading：模拟盘与实盘下单、成交、对账。
6. Risk：事前/事中/事后三层风控。
7. Ops：任务编排、状态机、审计、恢复。

## 四、可替换设计（关键）

这一节的核心不是“为了抽象而抽象”，而是解决一个非常现实的问题：

如果今天系统只围绕 Tushare + A 股 + 本地单机去写，短期看实现最快；
但一旦后面要接美股、分钟数据、第二数据源、第二存储后端，整个系统就会被迫重写。

所以这里先把两个最容易耦合死的点拆开：

1. 数据从哪里来：`DataProvider`
2. 数据最终存在哪里：`StorageBackend`

### 4.1 数据源可替换

统一 Provider 接口，不把上层逻辑绑定到 Tushare：

```python
class DataProvider:
    def fetch_instruments(self, market: str): ...
    def fetch_calendar(self, market: str, start: str, end: str): ...
    def fetch_bars(self, market: str, symbols: list[str], freq: str, start: str, end: str): ...
    def fetch_fundamental(self, market: str, symbols: list[str], fields: list[str], start: str, end: str): ...
    def fetch_events(self, market: str, symbols: list[str], start: str, end: str): ...
```

默认实现：TushareProvider（CN，全A）。
未来实现：USProvider、RQDataProvider、PolygonProvider 等。

### 4.2 存储可替换

统一 StorageBackend 接口，不把上层逻辑绑定到 Parquet+DuckDB：

```python
class StorageBackend:
    def upsert(self, dataset: str, df, partition: dict): ...
    def read(self, dataset: str, filters: dict): ...
    def list_partitions(self, dataset: str): ...
    def snapshot(self, date: str): ...
```

默认实现：ParquetDuckDBBackend。
未来实现：PostgresBackend、ClickHouseBackend、DolphinDBBackend。

### 4.3 第一个数据库到底用什么

这是当前最重要的讨论点。

先给结论：

1. **第一版权威底座不应该直接用 Qlib 格式**。
2. **第一版建议使用 Parquet 作为持久化格式，DuckDB 作为查询和编排引擎**。
3. **Qlib 格式应该是研究服务视图，而不是底层唯一真相源**。

### 4.4 为什么不直接把 Qlib 格式当数据库

Qlib 的二进制格式很适合“研究侧高性能读取”，但它不适合承担“系统底座数据库”的职责。

原因有 6 个：

1. Qlib 格式本质上是研究引擎优化的数据组织方式，不是通用数据底座。
2. 它对 OHLCV、表达式因子、时间序列研究很友好，但对公告、财务、事件、元数据、审计日志这类异构数据并不自然。
3. Qlib 上层更偏“取数和算因子”，不擅长做日常数据治理、表结构演进、回填修复、问题排查。
4. 如果把唯一原始数据只存成 Qlib 格式，后续接别的研究框架、BI 分析、SQL 检查都会变得很别扭。
5. Qlib 更适合作为“派生视图”，而不是“可审计原始底座”。
6. 未来如果要支持多市场、多数据源并存，原始层需要更中性的组织方式。

所以，Qlib 在 Vortex 里的定位应该是：

1. 研究引擎输入层
2. 因子和模型数据服务层
3. 从权威底座派生出来的研究视图层

而不是：

1. 全系统唯一数据库

### 4.5 为什么第一版建议 Parquet + DuckDB

这里也不是说 Parquet 神圣不可替代，而是它非常适合第一版。

原因有 8 个：

1. Parquet 是列式格式，天然适合行情、财务、因子这种宽表和时序分析。
2. 它对 Python 生态兼容极好，Pandas、Polars、DuckDB、PyArrow 都能直接处理。
3. 文件就是数据，天然适合版本化、快照化、分区化。
4. 本地开发极轻，不需要先部署数据库服务。
5. DuckDB 可以直接查询 Parquet，既保留文件式简单性，又拥有 SQL 能力。
6. 对研究型工作负载非常合适，特别是单机和中小规模团队阶段。
7. 后续如果要迁移到 ClickHouse、Postgres、对象存储湖仓，成本相对可控。
8. 它非常适合先把“数据流程正确性”做扎实，而不是过早做基础设施工程。

### 4.6 这里说的“数据库”到底是什么意思

这里要区分 3 个概念：

1. **持久化格式**：Parquet、Qlib bin、CSV
2. **查询引擎/数据库**：DuckDB、Postgres、ClickHouse
3. **系统权威底座**：哪个层被认定为“唯一真相源”

在当前设计里：

1. 持久化格式优先选 Parquet
2. 查询引擎优先选 DuckDB
3. 权威底座是“Parquet 数据湖 + 元数据清单”，不是 Qlib bin

### 4.7 第一版建议的落地形态

第一版不要把问题想成“只能二选一”。

建议直接采用双层：

1. **权威层**：Parquet + DuckDB
2. **研究层**：Qlib bin

也就是：

1. 原始与标准化数据，先落 Parquet
2. 研究需要的日频/分钟频视图，再构建成 Qlib bin
3. 任何研究结果都能追溯回 Parquet 权威层

这比“全上 Qlib”稳，也比“只上 SQL 数据库”更适合研究型场景。

## 五、端到端流程

### 5.1 研究流程

1. Provider 增量取数。
2. 质量校验与 PIT 对齐。
3. 写入权威存储并发布快照。
4. 构建 Qlib 视图并运行研究 workflow。
5. 输出 SignalSnapshot。

### 5.2 执行流程

1. SignalSnapshot -> PortfolioCompiler -> TargetPortfolio。
2. TargetPortfolio -> OrderPlan。
3. Backtrader 进行执行可行性验证（关键策略）。
4. 通过后进入模拟盘/miniQMT 执行。
5. 生成 ExecutionReport 并自动对账。

### 5.3 失败策略

1. 关键质量规则失败：直接阻断，不允许继续。
2. PIT 失败：阻断研究与交易。
3. 执行前风控失败：阻断下单。

## 八、从使用者角度，这个系统应该怎么用

前面几节主要是工程设计视角，但真正决定系统是否好用的，不是类分得多漂亮，而是使用者有没有一个稳定、统一、低心智负担的使用方式。

这里先给结论：

1. 使用者不应该直接理解 `Provider`、`StorageBackend`、`DatasetRepository` 这些内部类。
2. 使用者应该围绕少数几个稳定对象工作：`profile`、`snapshot`、`workflow`、`report`。
3. 日常动作应该是“声明我要做什么”，而不是“自己拼内部模块调用顺序”。

### 8.1 使用者真正关心的不是内部类，而是 4 个对象

从使用者视角，系统最好只暴露这 4 个核心对象：

1. `profile`：我在什么市场、什么数据源、什么环境下工作。
2. `snapshot`：我这次研究、回测、执行到底使用哪一版数据。
3. `workflow`：我要运行哪种流程，例如同步、研究、回测、调仓。
4. `report`：这次流程的结果是什么，哪里失败了，是否可复现。

这 4 个对象的意义是：

1. `profile` 解决“环境与口径”问题。
2. `snapshot` 解决“版本一致性”问题。
3. `workflow` 解决“入口统一”问题。
4. `report` 解决“结果可追踪”问题。

如果没有这 4 个对象，使用者很快就会退化成：

1. 手动改脚本参数
2. 手动记住某天的数据路径
3. 手动判断数据是否可用
4. 手动追踪一次运行到底用了什么版本

这会让系统越来越像一堆脚本，而不是一个平台。

### 8.2 角色视角下，系统应该怎么被使用

建议把使用者分成 4 类，每类人看到的系统都不一样。

#### 8.2.1 数据运营 / 平台维护者

这个角色不关心 alpha，不关心具体策略；他关心的是：

1. 今天的数据有没有同步成功。
2. 哪个数据集被阻断了。
3. 今晚 22:00 的 research snapshot 有没有发布成功。
4. 如果失败，应该重跑哪一段。

对他来说，系统的使用动作应该是：

1. 运行同步流程。
2. 查看健康报告和 PIT 报告。
3. 重新执行失败数据集。
4. 发布或回滚快照。

#### 8.2.2 研究员 / 因子开发者

这个角色不应该关心 Parquet 路径，也不应该自己猜数据是否可用。

他真正关心的是：

1. 我今天该用哪个 snapshot 做研究。
2. 这个 snapshot 包含哪些市场、哪些频率、哪些字段。
3. 我的 Qlib workflow 跑完以后，结果挂在哪个 experiment 下。
4. 结果能不能和上次实验公平对比。

对他来说，系统的使用动作应该是：

1. 选择一个 snapshot。
2. 运行 research workflow。
3. 查看实验结果和因子评测报告。
4. 导出 signal snapshot 给策略或组合层。

#### 8.2.3 策略开发者 / 回测用户

这个角色最容易犯的错误，是“拿研究时随手导出的数据就去回测”。

所以对他来说，系统应该强制建立一个习惯：

1. 先选 snapshot。
2. 再选 strategy config。
3. 再跑 backtest。
4. 最后比较报告，而不是比较零散的 notebook 输出。

他真正关心的是：

1. 这次回测用的是哪一版信号和哪一版数据。
2. 订单约束、手续费、滑点、调仓规则是什么。
3. 回测结果能不能稳定复现。

#### 8.2.4 交易执行 / 运营同学

这个角色不应直接接触研究中间产物，而是接收已经通过风控和版本固化的目标组合。

他真正关心的是：

1. 今天该执行哪个 target portfolio。
2. 这个组合来自哪个 snapshot 和哪个 strategy run。
3. 下单失败后如何重试。
4. 收盘后如何对账。

对他来说，系统的使用动作应该是：

1. 读取 target portfolio。
2. 执行 paper 或 live rebalance。
3. 查看 execution report 和 reconciliation report。

### 8.3 使用者看到的入口应该尽量少

从平台设计角度，最忌讳的是把入口做成一堆脚本名。

更合适的方式，是统一成一个根命令，例如：

```bash
vortex data sync
vortex data inspect
vortex data publish

vortex research run
vortex research inspect

vortex strategy backtest
vortex strategy compile

vortex trade paper rebalance
vortex trade live rebalance
vortex trade reconcile
```

这样做的原因很直接：

1. 使用者记住的是“领域动作”，不是文件名。
2. 后续 CLI、API、Web UI 可以共用同一套动作语义。
3. 文档会更稳定，因为入口名和领域动作一致。

### 8.4 从使用者角度，参数应该怎么设计

使用者日常不应该频繁显式指定 provider、storage、路径、DuckDB 文件位置。

这些东西应该尽量被 `profile` 吃掉。

也就是说，用户更常写的应当是：

```bash
vortex data sync --profile cn_stock_daily --as-of 2026-04-04
vortex research run --profile cn_alpha_daily --snapshot 2026-04-04
vortex strategy backtest --profile dividend_classic --snapshot 2026-04-04
vortex trade paper rebalance --profile dividend_classic --snapshot 2026-04-04
```

这里的设计意图是：

1. `profile` 负责声明市场、频率、数据源、存储、时区、交易日历等环境信息。
2. `snapshot` 负责声明数据版本。
3. 业务命令只表达动作本身。

这比让用户每次写十几个底层参数更接近真实可用的平台体验。

### 8.5 使用者不应该自己找路径，而应该找快照

这是一个非常关键的产品设计原则。

研究员、策略开发者、执行同学都不应该自己写：

1. 某个 Parquet 路径
2. 某个 DuckDB 文件名
3. 某个 Qlib 目录

他们应该只写：

1. 我用 `2026-04-04` 的 research snapshot
2. 我运行 `dividend_classic` 策略
3. 我生成今天的 rebalance 订单

也就是说，系统对外暴露的定位键应该是：

1. `profile`
2. `snapshot`
3. `strategy`
4. `run_id`

而不是内部物理路径。

### 8.6 一个真实的日常使用路径应该长什么样

下面给一个从晚上到第二天开盘前的完整用户旅程。

#### 8.6.1 晚间数据阶段

数据运营同学运行：

```bash
vortex data sync --profile cn_stock_daily --as-of 2026-04-04
vortex data publish --profile cn_stock_daily --as-of 2026-04-04
```

系统输出：

1. 是否同步成功。
2. 哪些数据集通过质量门禁。
3. 是否成功发布 `2026-04-04` 的 research snapshot。

#### 8.6.2 夜间研究阶段

研究员运行：

```bash
vortex research run --profile cn_alpha_daily --snapshot 2026-04-04
```

系统输出：

1. 本次实验对应的 `experiment_id`。
2. 使用的是哪一个 snapshot。
3. 产出的 signal snapshot 在哪里。

#### 8.6.3 回测验证阶段

策略开发者运行：

```bash
vortex strategy backtest --profile dividend_classic --snapshot 2026-04-04
```

系统输出：

1. 本次回测使用的 snapshot。
2. 本次回测使用的策略配置版本。
3. 绩效、换手、回撤、成交约束验证报告。

#### 8.6.4 模拟盘 / 实盘阶段

交易运营同学运行：

```bash
vortex trade paper rebalance --profile dividend_classic --snapshot 2026-04-04
```

或者：

```bash
vortex trade live rebalance --profile dividend_classic --snapshot 2026-04-04
```

系统输出：

1. 本次 target portfolio 来源于哪个 strategy run。
2. 生成了哪些订单。
3. 是否全部成交。
4. 是否完成对账。

### 8.7 对使用者来说，最重要的不是“跑命令成功”，而是“系统是否给出可判断结果”

所以每一个用户动作，都应该返回统一风格的结果对象或终端摘要，至少包含：

1. `status`
2. `run_id`
3. `profile`
4. `snapshot`
5. `artifact_manifest`
6. `blocking_reason`（如果失败）

这会让系统具备两个很重要的能力：

1. 人能看懂。
2. 机器也能接着编排。

### 8.8 从使用者视角反推，对工程设计的约束是什么

如果接受上面的使用方式，那么工程上就必须满足这些约束：

1. 所有领域都必须围绕 `profile` 和 `snapshot` 对齐。
2. 不能把“版本选择”下放给脚本作者自己处理。
3. 数据、研究、回测、交易都要产出统一 manifest。
4. 每个动作都要能被 CLI、API 和调度系统复用。
5. 失败必须结构化，而不是只打一段日志。

换句话说，使用者视角不是文档包装层，而是反过来决定内部接口设计的上游约束。

## 九、Profile 配置模型（摘要）

上面已经明确：`profile` 是使用者的第一配置对象。

但这里再强调一个关键结论：

1. `profile` 必须统一命名。
2. `profile` 不能做成一个超级大配置文件。
3. `profile` 应该按领域分成 `DataProfile`、`ResearchProfile`、`StrategyProfile`、`TradeProfile`。

它们的关系建议是：

1. `DataProfile` 负责定义数据口径、质量策略、PIT 策略与快照生成方式。
2. `ResearchProfile` 负责定义如何消费 snapshot 做 Qlib 研究。
3. `StrategyProfile` 负责定义如何把信号转成可回测、可验证的目标组合。
4. `TradeProfile` 负责定义如何把目标组合送入模拟盘或实盘执行。

这里最重要的工程约束是：

1. `profile` 负责长期配置。
2. `snapshot` 负责具体数据版本。
3. 用户可见字段和内部派生字段必须分开。
