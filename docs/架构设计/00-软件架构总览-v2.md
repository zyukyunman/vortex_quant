## 一、系统上下文

### 1.1 用户角色

| 角色 | 关注点 | 典型动作 |
|------|--------|---------|
| **数据运营** | 数据可用性、同步健康 | bootstrap / update / repair / publish / status |
| **研究员** | 因子评测、信号生成 | register / run (factor_eval) / publish-signal |
| **策略开发者** | 回测验证、组合编译 | backtest / compile / compare |
| **交易执行** | 模拟/实盘下单、对账 | rebalance / reconcile / retry |
| **平台管理员** | 配置管理、全局运维 | profile explain / resolve、调度编排、日志审计 |

### 1.2 外部系统

```
                        ┌────────────────────────────────────────┐
                        │           Vortex 系统边界               │
                        │                                        │
                        │  Data → Research → Strategy → Trade    │
                        │         ↑ 共享契约层 (配置/时间/ID/审计) ↑  │
                        └──┬─────────────────────────────────┬───┘
                           │                                 │
                    外部数据源                            外部交易通道
                    ├─ Tushare API                       ├─ miniQMT (xtquant)
                    ├─ (未来) RQData                     ├─ PaperGateway (内置)
                    └─ (未来) Wind / Polygon             └─ (未来) CTP / IB
                                                         │
                                                    外部通知渠道
                                                    ├─ 飞书 Bot
                                                    └─ (未来) Webhook
```

### 1.3 系统边界约定

1. **Vortex 是本地优先的单机量化平台**。不设计分布式部署。
2. **Vortex 不提供 Web UI**。入口为本地常驻服务 + CLI + 配置文件 + HTML 报告。
3. **Vortex 不托管用户策略代码**。用户策略以 Python 文件形式存在于本地文件系统。
4. **Vortex 不提供实时行情推送**。数据以日频/分钟频批量更新为主。
5. **外部系统通过 Gateway/Provider 协议接入**，内部逻辑不依赖任何具体外部 API。

### 1.4 运行形态：本地服务 + CLI thin client

Vortex 的产品运行形态不是“每个 CLI 命令都直接在终端进程里执行业务逻辑”，而是：

```text
用户命令 → vortex CLI → 本地 Vortex 服务进程 → 领域编排（Data / Research / Strategy / Trade）
                                      ↓
                               工件落盘 + 状态记录 + HTML 报告
```

这一定义很重要，因为它决定了系统的稳定性边界：

1. **CLI 是薄客户端（thin client）**：负责参数校验、向本地服务发起命令、展示结果摘要；不直接承载长耗时研究/回测逻辑。
2. **服务进程是控制面入口**：负责持有工作空间（workspace）、调度长任务、串联 Profile 解析、领域模块调用、工件落盘与状态持久化。
3. **数据面仍然是文件工件**：服务执行完成后，把数据快照、factor panel、signal snapshot、回测报告写入工作空间；CLI 返回的是摘要 + 工件路径，而不是把大结果直接塞回终端。
4. **单机场景下优先本地 HTTP/JSON 协议**：不引入外部分布式组件；服务与 CLI 默认通过 localhost 通信，保持可观测和可脚本化。
5. **服务崩溃不应破坏已发布工件**：控制面失败只能中断当前 run，不允许回滚或覆写已发布的 snapshot / signal / report。

> **当前实现映射（Phase 1）**：`vortex server start` 已是后台服务语义；Data 域写命令（`bootstrap / update / backfill / repair / publish`）默认先写入控制面 `task_queue`，再由后台 worker 进程执行。对应的控制面观察入口也已经固定为 `data status --watch`、`data logs --follow`、`data cancel`。也就是说，当前代码虽然还没有把所有 worker 完全收敛到单一常驻 server 进程内，但**对用户暴露的产品语义已经统一为“提交任务，由控制面负责观察和取消”，而不是前台阻塞跑完整链路**。

> **演进路径**：上述边界是当前的范围限定，不是架构天花板。系统通过协议隔离（DataProvider、Gateway、NotificationChannel、CLI 入口层）保证未来扩展的可能性。但需要明确：**"无需重构"指的是无需推翻域边界**，不同演进目标的成本差异如下：
>
> | 演进目标 | 影响范围 | 需要新增的内容 |
> |---------|---------|--------------|
> | 新数据源（如 Wind） | Data 域 | 新 DataProvider 实现 |
> | Web UI | 输入/输出层 | API 适配层（REST/WebSocket），不改域内部逻辑 |
> | 秒频/tick 数据 | Data + Research + Trade | 实时/回放子协议、高频回测引擎适配、tick 级撮合模型 |
> | 多资产（期货/期权） | 全域 | 订单模型扩展、成本模型扩展、资产特定风控规则 |

## 二、架构原则与约束

### 2.1 核心原则

| # | 原则 | 含义 | 违反后果 |
|---|------|------|---------|
| P1 | **确定性优先** | 相同输入 + 相同环境 = 相同输出。任何引入不确定性的操作必须显式声明并提供确定性替代。 | 回测不可复现，因子评测无意义 |
| P2 | **Fail-Closed**（失败即阻断） | 质量、PIT、风控校验失败时阻断流程，不允许跳过或静默降级。 | 脏数据进入研究/交易链路 |
| P3 | **可替换** | 数据源（Provider）和存储后端（StorageBackend）必须通过协议接入，内部逻辑不绑定具体实现。 | 无法接入第二数据源或迁移存储 |
| P4 | **严格 PIT**（Point-In-Time，时点对齐） | 任何数据只有在公告日之后才对研究/回测可见。绝不允许未来信息泄露。 | 回测收益虚高，实盘无法复现 |
| P5 | **快照即版本** | 跨域数据传递以不可变快照为单位。禁止使用 latest 等可变引用。 | 上下游数据版本不一致 |
| P6 | **血统可追溯** | 任何产出物必须能追溯到：使用了哪个快照、哪个配置版本、哪次运行。 | 无法审计、无法复现 |
| P7 | **控制面/数据面分离** | 调度、状态、配置（控制面）与行情、因子、信号、订单（数据面）在存储和访问路径上分离。 | 控制逻辑与数据逻辑耦合 |

### 2.2 约束条件

1. **技术栈约束**：Python 3.10+；Parquet + DuckDB（权威层）；Qlib（研究引擎）；Backtrader（回测引擎）；miniQMT/xtquant（实盘执行）。
2. **单机约束**：所有组件运行在同一台机器上，不考虑网络分区、分布式事务。
3. **时区约束**：默认 Asia/Shanghai；交易日历与时区必须配置化，为多市场预留。
4. **安全约束**：API Key / 账户凭证不进入代码仓库，通过 .env 或环境变量注入。

## 三、域分解与职责边界

### 3.1 域划分

系统分为 **5 个业务域** + **1 个共享契约层**：

```
┌──────────────────────────────────────────────────────────┐
│              用户入口 (CLI thin client)                  │
├──────────────────────────────────────────────────────────┤
│   本地服务进程（命令路由 / 调度 / 工作空间 / 状态持久化）   │
├──────┬───────┬──────────┬────────┬──────────────────────┤
│ Data │Resrch │ Strategy │ Trade  │ Notification         │
│  域  │  域   │    域    │   域   │ (跨域通知)           │
├──────┴───────┴──────────┴────────┴──────────────────────┤
│      共享契约层（Profile / 时间 / ID / 审计 / 错误）       │
├──────────────────────────────────────────────────────────┤
│      基础设施层（存储 / Provider / Gateway / 本地 HTTP）   │
└──────────────────────────────────────────────────────────┘
```

### 3.2 各域职责

| 域 | 职责 | 拥有的核心数据 | 不做什么 |
|----|------|-------------|---------|
| **Data** | 采集、校验、PIT 对齐、快照发布、衍生指标 | 权威行情/财务/事件、sync_manifest、snapshot_descriptor | 不做因子计算、不做信号生成 |
| **Research** | 因子注册/评测/缓存、模型训练、信号发布 | factor_registry、factor_cache、signal_snapshot | 不做回测、不做组合构建 |
| **Strategy** | 信号→组合编译、多维度回测、评分归档 | strategy_runs、backtest_results、target_portfolio | 不做订单执行 |
| **Trade** | 模拟/实盘执行、三层风控、对账、重试 | executions、orders、fills、reconciliations | 不做策略决策 |
| **Notification** | 跨域通知投递、订阅管理、路由决策 | 通知模板、投递记录、订阅规则、Watch 数据源 | 不做业务逻辑判断 |
| **共享契约层** | 配置解析、时间/日历、ID 生成、审计、错误模型 | Profile YAML、preset/pack、交易日历 | 不做业务逻辑 |

### 3.3 域间数据所有权规则

**核心规则：每份数据有且仅有一个 owner。**

| 数据 | Owner | 消费者 | 传递方式 |
|------|-------|--------|---------|
| 权威行情/财务 | Data | Research, Strategy | snapshot_descriptor（不可变路径引用） |
| Qlib 研究视图 | Data（派生） | Research | 目录路径（Data publish 生成） |
| 因子值/评测结果 | Research | Strategy | factor_archive / signal_snapshot |
| 信号快照 | Research | Strategy | signal_snapshot_descriptor.json |
| 目标组合 | Strategy | Trade | target_portfolio.json |
| 订单/成交 | Trade | Notification, 审计 | execution_report.json |
| 对账结果 | Trade | Notification, 审计 | reconcile_report.json |
| 配置 | 共享契约层 | 所有域 | ResolvedProfile 对象 |

## 四、依赖规则与允许的调用方向

### 4.1 域间依赖矩阵

```
依赖方向: 行 → 列（行依赖列）

            Data  Research  Strategy  Trade  Notification  共享契约
Data          -     x         x        x      emit          dep
Research    dep     -         x        x      emit          dep
Strategy    dep    dep        -        x      emit          dep
Trade       ro      x       dep        -      emit          dep
Notification  x     x         x        x       -            dep
共享契约       x     x         x        x       x             -

dep = 允许依赖  ro = 只读已发布数据  x = 禁止依赖  emit = 仅发送事件
```

**关键规则**：
1. 依赖方向单向流动：Data → Research → Strategy → Trade。不允许逆向依赖。
2. Trade 不直接依赖 Research。Trade 可**只读**已发布的 Data 快照（交易日历、参考价格、涨跌停标志等风控所需的市场数据），但不依赖 Data 的可变状态（如正在同步的中间数据）。
3. 所有域可以依赖共享契约层，但共享契约层不依赖任何业务域。
4. Notification 是纯事件消费者，不被业务域依赖。
5. 域间传递**仅通过不可变工件**（JSON 文件/快照描述符），不通过内存引用或共享数据库连接。

### 4.2 信号发布是冻结边界

Research 域的 publish-signal 是整条链路中最关键的冻结点：

```
Data snapshot（不可变）→ Research 因子计算 → publish-signal → signal_snapshot（不可变）
                                                              |
                                              Strategy 只能引用已发布的 signal_snapshot
```

一旦 signal 发布，其内容不可修改。Strategy 和 Trade 只能引用已发布的快照 ID。

### 4.3 交易幂等性与订单安全

上述快照冻结语义约束的是数据流方向。但在交易域，还存在一组同样属于架构级的不变量：订单的幂等性与安全性。这些规则之所以在总览文档而非 05-交易域中定义，是因为它们约束的不仅是 Trade 域的内部实现，更是整条 Data→Research→Strategy→Trade 链路末端的安全底线——一旦违反，资金损失不可逆。

Trade 域必须满足以下架构级不变量（详细设计见 05-交易域）：

1. **确定性订单标识**：每笔订单生成唯一 `client_order_id`，格式为 `{exec_id}_{seq}`。**恢复重放**（crash replay）沿用同一 `client_order_id` 以实现网关去重；**业务重试**（`vortex trade retry`）生成新 `exec_id` 和新 `client_order_id`，通过 `parent_exec_id` / `root_exec_id` 关联历史链路。
2. **意图持久化**：订单意图（OrderIntent）在提交到 Gateway 之前必须先持久化到本地状态，并携带 canonical intent hash。如果进程崩溃，恢复后可从持久化的 intent 重建状态。重复提交时逐字段比对，完全一致视为幂等重放，否则 fatal conflict 阻断。
3. **状态机约束**：订单状态转换是单向的，终态（filled/rejected/cancelled）不可逆转。部分成交（partial_fill）可前进到 filled、timeout 或 cancelled（撤余场景：`cancelled + filled_qty > 0` 表示部分成交后主动/被动撤销剩余份额）。提交后网络超时进入 `ack_pending` 中间态，需通过 reconcile 确认实际状态。
4. **Fail-Closed 执行**：任何预交易风控失败 → 整批订单阻断，零部分执行。
5. **重连恢复**：Gateway 断连后重连，必须先执行 reconcile（对齐网关持仓与本地记录），再决定是否继续执行未完成订单。

## 五、数据血统链

端到端数据流的完整链路，用于审计和复现：

```
外部数据源 ──→ Data 域 ──→ Research 域 ──→ Strategy 域 ──→ Trade 域
(Tushare)    ① 采集      ④ 因子计算     ⑦ 组合编译      ⑨ 生成订单
             ② 质量校验   ⑤ 模型训练     ⑧ 回测验证      ⑩ 风控校验
             ③ PIT+发布   ⑥ 信号发布                     ⑪ 执行成交
                  |             |              |           ⑫ 对账
           snapshot_desc  signal_snapshot  target_portfolio    |
                                                        exec_report
```

### 5.1 血统记录要求

每个域运行后产出的结果文件统称为"产出工件"（artifact），如 Data 域的 snapshot_descriptor、Research 域的评测报告、Strategy 域的 target_portfolio 等。每个产出工件必须记录以下血统信息：

**基础字段**（所有工件必须包含）：
```json
{
  "lineage": {
    "run_id": "backtest_20260403_223118_c3d4",
    "profile": "dividend_classic",
    "snapshot_id": "snap_20260403_220000_b2c3",
    "parent_run_id": null,
    "created_at": "2026-04-03T22:31:18+08:00",
    "vortex_version": "0.2.0",
    "config_hash": "sha256:abc123..."
  }
}
```

> **字段可空性说明**：基础字段在该工件首次产出时全部必填。其中 `parent_run_id` 在非重试场景下为 null（仅当 repair/retry 产生的子 run 才填写父 run ID）；`snapshot_id` 在 Data 域自身产出的工件中即为自己的快照 ID。

上述基础字段对所有域通用。但每个域的工作流还有自己特有的可复现信息需要记录（如因子代码版本、回测参数、执行网关类型等），因此在基础字段之上追加"扩展字段"：

**工作流特定扩展字段**（按域追加）：

| 域 | 扩展字段 | 用途 |
|----|---------|------|
| Research (factor_eval) | factor_name, factor_code_hash, label_spec, preprocess_config | 因子复现 |
| Research (model_train) | feature_set, train_split, val_split, test_split, model_type, hyperparams | 模型复现+泄露审计 |
| Strategy (backtest) | signal_id, backtest_mode, seed, benchmark, cost_model, rebalance_freq | 回测复现 |
| Trade (execution) | target_portfolio_id, gateway_type, risk_pack, client_order_id_prefix | 执行审计 |

### 5.2 快照版本语义

**核心概念**：为了兼顾"数据可修正"与"历史可复现"，Data 域的快照采用**逻辑快照 + 物理修订**双层标识：

- **逻辑快照（as_of）**：代表某个交易日的数据切面，是业务层面的标识。
- **物理修订（revision_id）**：每次 publish 生成唯一的 `revision_id`（格式：`snap_{as_of}_{publish_timestamp}_{hash_4}`）。同一 `as_of` 可以有多个 revision，系统维护 `latest_revision_for(as_of)` 映射。

**Data 域 publish 规则**：同一 `as_of` 日期可以重新 publish，生成新的 `revision_id`。新 revision 成为该 `as_of` 的当前版本，但**旧 revision 的物理文件不会被删除**（仅标记为 `superseded`）。这确保历史 run 可严格复现。

**下游引用规则**：下游域（Research/Strategy/Trade）的每次 run 在血统中记录 `snapshot_revision_id`（而非仅 `snapshot_id`）。无论 Data 是否重新 publish 了同一 `as_of`，旧 run 的 lineage 指向的物理数据始终可访问。新 run 默认使用 `latest_revision_for(as_of)`。

**revision 生命周期管理**：旧 revision 默认保留。用户可通过 `vortex data gc --before <date>` 清理指定日期前无下游引用的 superseded revision（需扫描全域 lineage 确认无引用后才允许物理删除）。

**signal_snapshot 规则**：publish-signal 后不可修改、不可覆盖。如需修正，必须重新 publish 生成新的 signal_id。这是比 data snapshot 更严格的不可变性，因为 signal 是交易链路的直接输入。

### 5.3 不可变性约定

| 工件 | 不可变性级别 | 违反处理 |
|------|------------|---------|
| snapshot_descriptor | 可重发布（生成新 revision_id，旧 revision 保留为 superseded） | 下游 run 通过 lineage 中的 `snapshot_revision_id` 精确追溯 |
| signal_snapshot | **严格不可变** | 如需修正，生成新 signal_id |
| target_portfolio | 编译后不可修改 | 需重新编译生成新版本 |
| execution_report | 执行后不可修改 | 追加 reconcile_report 记录偏差 |
| factor_cache | code_hash 变更时自动失效 | 重新计算 |

## 六、控制面与数据面分离

### 6.1 概念定义

| 面 | 内容 | 存储介质 | 访问特征 |
|----|------|---------|---------|
| **控制面** | Profile、preset/pack、run 状态、调度元数据、审计日志 | YAML + 轻量 SQLite/JSON | 低频读写、事务性 |
| **数据面** | 行情、财务、因子值、信号、组合权重、订单成交 | Parquet + DuckDB | 高频读、批量写、列式查询 |

### 6.2 分离规则

1. 控制面操作（创建 run、更新状态、写审计日志）不经过 DuckDB/Parquet。
2. 数据面操作（读行情、算因子、跑回测）不修改控制面状态（由调用方在数据操作前后更新）。
3. 控制面的"指向"（如 snapshot_id）是不可变引用，不是数据副本。

### 6.3 存储布局（概念级）

```
vortex_home/
+-- config/                  # 控制面：配置
|   +-- profiles/            #   用户 Profile YAML
|   +-- presets/             #   平台预设
+-- data/                    # 数据面：权威数据
|   +-- authoritative/       #   Parquet 分区数据
|   +-- derived/             #   衍生指标
|   +-- qlib_view/           #   Qlib 研究视图
+-- research/                # 数据面：研究产出
|   +-- factor_cache/
|   +-- signals/
|   +-- experiments/         #   每个 run 内含 HTML 报告
+-- strategy/                # 数据面：策略产出
|   +-- backtests/           #   每个 run 内含 HTML 报告
|   +-- portfolios/
```

## 七、跨域交互模式

### 7.1 交互风格

域间交互仅使用两种模式：

1. **文件传递**：上游域写入不可变工件，下游域通过 ID 引用读取。这是主要交互方式。
2. **事件通知**：域完成关键动作后发出事件，Notification 服务消费并投递。不影响数据流。

**不使用的模式**：
- 域间直接函数调用
- 共享数据库连接
- 消息队列（不引入额外中间件）
- 域间回调

### 7.2 事件清单

| 事件 | 发布者 | 语义 | 通知级别 |
|------|--------|------|---------|
| data.sync.completed | Data | 日更完成 | info（默认静默） |
| data.sync.failed | Data | 日更失败 | error（强制通知） |
| data.quality.blocked | Data | 数据质量检查未通过，流程被拦截 | error |
| data.publish.completed | Data | 快照发布成功 | info |
| research.eval.completed | Research | 因子评测完成 | info |
| research.signal.published | Research | 信号发布 | info |
| strategy.backtest.completed | Strategy | 回测完成 | info |
| strategy.compile.completed | Strategy | 组合编译完成 | info |
| trade.execution.completed | Trade | 实盘执行完成 | warn（需关注） |
| trade.execution.paper_completed | Trade | 模拟盘执行完成 | info |
| trade.execution.failed | Trade | 执行失败 | error |
| trade.risk.blocked | Trade | 风控阻断 | error |
| trade.reconcile.abnormal | Trade | 对账异常 | error |

**通知级别语义**：
- info：默认静默，用户可配置接收。
- warn：默认推送，用户可关闭。
- error：强制推送，不可关闭。

### 7.3 通知架构（扩展）

Notification 不仅是简单的事件投递，还承担订阅管理和定期数据推送：

**核心组件**：
1. **路由引擎**：根据事件类型、通知级别、用户订阅规则决定投递目标和渠道。
2. **订阅存储**：用户配置的订阅规则（哪些事件、什么频率、推送到哪里）。
3. **投递历史**：记录每次投递的状态（成功/失败/静默跳过），用于排查和审计。
4. **Watch Provider**：定期数据拉取插件机制。用于"用户关心但不由业务域主动推送"的场景，如：
   - 估值百分位日报
   - 策略定期分析（仅分析+推送，不执行调仓）
   - 市场宏观指标监控

**Watch Provider 架构**：
- 每个 Watch Provider 实现统一协议：`pull(config) -> WatchResult`
- 调度器按用户配置的 schedule 定期调用
- WatchResult 通过路由引擎投递到指定渠道
- Watch Provider 是纯只读的——只能读取已发布的快照和已归档的结果，不能触发业务流程

> **数据来源与处理逻辑**：Watch Provider 不直接访问原始数据源（rawdata），而是通过**域查询服务（Domain Reader Protocol）**读取已发布的快照数据和已归档的运行结果。Watch Provider **不应直接构造存储路径或直接读取 Parquet/DuckDB 文件**——所有数据读取必须经过与 StorageBackend 对齐的 Reader 接口，以保证存储后端可替换。数据采集与清洗由 Data 域的 Provider + 质量检查流程完成——这发生在 Watch 之前。Watch Provider 的代码是一个实现了 `pull()` 协议的 Python 类，放在框架的 `notification/watch/` 目录下。例如"估值百分位日报"Provider 通过 DataReader 读取已发布快照中的 PE/PB 指标，计算历史百分位，输出一段文字摘要作为 WatchResult。框架开发者可以按同样的协议新增 Watch Provider，详见下方"§十九 框架扩展点"。

**渠道适配**：
- 飞书 Bot（文本消息）
- 未来：Webhook、Telegram、邮件
- 渠道通过 NotificationChannel 协议接入，与业务逻辑解耦

### 7.4 运维与控制面子系统

控制面需要一个显式的运维子系统来统一管理跨域的运行状态、调度和恢复：

**Run Registry（运行注册表）**：
- 所有域的 run 统一注册，提供 `status / inspect / list` 查询能力
- 状态枚举：pending → running → success / failed / blocked
- 每条 run 记录关联 lineage、artifacts、耗时、错误信息

**调度编排**：
- 支持 cron 式定时触发（如每晚 22:00 执行数据→研究→策略→模拟盘全链路）
- 步骤间依赖：上一步 success 才触发下一步；任何 failed/blocked 终止后续
- 调度配置存储在 Profile 的 schedule 字段中

**故障恢复**：
- 每个域的 run 失败后，控制面记录失败点（failed_units）
- repair/retry 从失败点恢复，不重跑已成功的部分
- 恢复后的 run 关联 parent_run_id

**审计日志**：
- audit.log 是不可删除的追加日志
- 所有实盘操作、配置变更、手动确认必须写入
- 日志格式：时间戳 + 域 + 动作 + 操作人 + run_id + 结果

**清理与归档**：
- 回测结果/因子缓存按策略可配置的保留期（如 30 天）自动标记为可清理
- 用户标记为 protected 的 run 不被清理
- 清理前检查是否有下游依赖

**版本兼容与 Schema 迁移**：
- 控制面 SQLite 内置 `schema_version` 表，记录当前库版本
- 每次 `vortex init` 或 `server start` 自动检测并执行增量迁移，用户无需手动删库
- 迁移仅允许 additive 变更（加表、加列、加索引），不允许删列或改类型
- 对没有 `schema_version` 表的旧库，通过已有表结构特征自动推断版本号
- 具体迁移注册表和执行流程见 06 代码细节 §1.7

```
研究员               Research域                Strategy域

  |                     |                     backtest()

```

### 8.3 风控阻断与故障恢复

Trade 域提交调仓计划后，风控引擎在三个阶段进行校验：盘前（pre_trade）、盘中（intra_trade）、盘后（post_trade）。任一阶段检查失败，整批订单立即阻断（不做部分执行），同时通知用户。用户修正后可通过 `retry` 重新提交，但风控引擎会基于最新市场状态重新校验——即使之前通过的规则也可能因市场变化而失败。

```
Trade域           风控引擎        Gateway       飞书
  |                 |              |             |
  +--submit plan--> |              |             |
  |              pre_trade_check() |             |
  |                 |              |             |
  |  ALL PASS ----> |              |             |
  |              send to gateway-->|             |
  |                 |           fill/reject      |
  |              intra_trade()     |             |
  |                 |              |             |
  |  (收盘后)        |              |             |
  |              post_trade()      |             |
  |                 |              |             |
  |  ANY FAIL ----> |              |             |
  |              BLOCK entire batch|             |
  |              emit: blocked ----+------------>通知
  |                 |              |             |
  | (用户修正后)      |              |             |
  +--retry--------> |              |             |
  |              re-check (市场已变) |             |
```

### 8.4 调度失败与自动恢复

数据日更（update）按数据集分批执行。如果某批次失败（如网络超时），系统只记录失败的数据集，不回滚已成功的部分。之后可以通过 `repair` 命令只重跑失败的数据集，而不需要重跑全量。全部数据集成功后，流程状态变为 success 并发出完成通知。

```
调度器           Data域           State存储
  |               |                |
  +--update-----> |                |
  |            run_state=running ->|
  |            chunk 1... OK       |
  |            chunk 2... FAIL     |
  |            failed_units -----> |
  |            run_state=failed -> |
  |               |                |
  | (自动/手动)    |                |
  +--repair-----> |                |
  |            read failed_units <-|
  |            retry failed only   |
  |            ALL OK              |
  |            run_state=success ->|
  |            emit: completed     |
```

## 九、技术选型与理由

### 9.1 核心技术栈

| 层 | 技术 | 选型理由 | 替换条件 |
|----|------|---------|---------|
| **持久化** | Parquet | 列式、生态兼容、文件即数据、天然快照化 | 需要实时更新或事务时 |
| **查询引擎** | DuckDB | 零部署、直查 Parquet、SQL 能力 | 需要并发写入或分布式查询时 |
| **研究引擎** | Qlib | 因子/模型/workflow 一体化 | Qlib 不再维护或需求超出 |
| **回测引擎** | Backtrader | 成熟、事件驱动、自定义 Analyzer | 需要高频 tick 级回测时 |
| **实盘执行** | miniQMT (xtquant) | 国内券商接入、Python SDK | 更换券商或需 CTP 时 |
| **通知** | 飞书 Bot | 团队已用、API 简单 | 更换 IM 时 |
| **AI 辅助** | LLM (openclaw / API) | 因子分类、定性辅助 | 不需要 AI 时 |
| **CLI** | Click / Typer | Python 原生、自动帮助 | — |

### 9.2 关键决策：Parquet + DuckDB 为权威底座

**Qlib bin 是研究派生视图，不是系统数据库。**

理由：
1. Qlib bin 适合研究读取，不适合数据治理（回填、修复、审计）。
2. Qlib 对异构数据（公告、事件、元数据）不自然。
3. 唯一存 Qlib 格式，接其他框架或 SQL 分析很别扭。
4. Parquet 是中性格式，任何工具都能读取。

## 十一、错误模型

### 11.1 错误层次

```
VortexError (基类)
+-- ConfigError          # 配置解析/验证失败
+-- DataError            # 数据域
|   +-- ProviderError    #   数据源获取失败
|   +-- QualityError     #   质量校验不通过
|   +-- PITError         #   PIT 对齐失败
+-- ResearchError        # 研究域
|   +-- FactorError      #   因子计算失败
|   +-- ModelError       #   模型训练失败
+-- StrategyError        # 策略域
|   +-- BacktestError    #   回测执行失败
|   +-- CompileError     #   组合编译失败
+-- TradeError           # 交易域
|   +-- RiskBlockError   #   风控阻断
|   +-- GatewayError     #   网关通信失败
|   +-- ReconcileError   #   对账异常
+-- NotificationError    # 通知投递失败（不阻断主流程）
```

### 11.2 错误处理策略

| 错误类型 | 处理方式 | 是否阻断 | 是否通知 |
|---------|---------|---------|---------|
| QualityError | 记录 + 阻断 + 写 health_report | 阻断 | error |
| PITError | 记录 + 阻断 + 写 pit_report | 阻断 | error |
| ProviderError (网络) | 重试 3 次 + 记录失败单元 | 部分阻断 | warn |
| RiskBlockError | 记录 + 阻断整批 + pre_trade_result | 阻断 | error |
| GatewayError | 重试 + 记录 + 阻断 | 阻断 | error |
| NotificationError | 记录，不阻断主流程 | 不阻断 | 日志 |

### 11.3 错误输出契约

```json

  "error_type": "QualityError",
  "domain": "data",
  "run_id": "data_20260404_220000_a1b2",
  "message": "critical rule 'not_empty' failed on dataset 'daily_bars'",
  "blocking": true,
  "details": {},
  "timestamp": "2026-04-04T22:01:30+08:00"
}
```

## 十二、确定性架构

### 12.1 威胁模型

| 威胁 | 来源 | 防御 |
|------|------|------|
| 浮点精度漂移 | 不同 CPU/库版本 | 排序用 rank（整数），关键指标定义精度容忍度 |
| 随机性 | 随机日期回测、模型训练 | 种子固定：seed = hash(profile + snapshot_id) |
| 排序不稳定 | 相同分值排序不确定 | 所有排序定义二级键（score desc, symbol asc） |
| 并发竞争 | 多 run 同时执行 | 每个 run 独立 workdir，缓存按 code_hash+snapshot 隔离 |
| 时间依赖 | datetime.now() | 所有时间显式传参，禁止隐式获取当前时间 |
| 依赖版本漂移 | pip 更新 | lockfile 固定；run 记录 vortex_version |
| 数据修订 | 数据源修正历史 | 快照不可变；修正生成新快照 |

### 12.2 环境契约

1. Python >= 3.10
2. 依赖版本与 lockfile 一致
3. 时区正确配置
4. 磁盘空间足够（bootstrap 前检查）

### 12.3 可复现性验证

- **golden master**：固定输入 + 固定种子，输出与基准逐字节比对。
- **差分测试**：同一输入运行两次，结果必须完全一致。
- 详见 07 测试方案。

## 十三、时间与日历语义

### 13.1 时间约定

| 概念 | 定义 | 示例 |

### 13.2 日历所有权

- 交易日历由 Data 域获取并缓存（来自 Provider）。

- 默认 A 股日历；多市场通过配置扩展。

### 13.3 迟到数据与修正

1. **迟到数据**：下次 update 自动补入；已发布快照不受影响。
2. **数据修正**：手动 backfill + 重新 publish 新快照。旧快照不变。

## 十四、研究与生产隔离

### 14.1 三种运行时

| 运行时 | 用途 | 隔离要求 |
|--------|------|---------|
| **探索研究** | 研究员日常试验 | run 级隔离（独立 workdir） |
| **回测验证** | 策略验证 | 确定性要求最高 |
| **实盘执行** | 真实下单 | 安全要求最高；审计日志 |

### 14.2 隔离规则

1. 探索研究的中间产物**不自动流入**回测或实盘。必须显式 publish-signal。
2. 回测用历史快照，实盘用最新快照——两者可能不同。
3. 实盘执行禁止读取未发布的研究中间产物。
4. 三种运行时共享 Profile 配置，但各自有独立的 run_id 和工件目录。

| 全样本回测（5 年日频） | < 10 分钟 | Backtrader 单线程 |
| 模拟盘调仓（30 标的） | < 1 分钟 | 含风控 |
| Qlib 视图构建（增量） | < 5 分钟 | 全量 < 30 分钟 |

### 16.2 设计原则

1. **批量优先**：避免逐条处理。
2. **增量优先**：日更只处理增量。
3. **可配置并发**：max_concurrent_tasks 默认 3。
4. **缓存利用**：因子缓存按 code_hash + snapshot 键。

## 十八、API 设计原则

### 18.1 CLI 约定

1. **统一根命令**：vortex {domain} {action} [--options]
2. **Profile 驱动**：通过 --profile 指定环境，不需要写路径。
3. **Snapshot 显式**：需要数据版本的命令必须指定 --snapshot。
4. **Dry-run 支持**：破坏性操作支持 --dry-run。
5. **统一输出**：默认人类可读；--format json 机器可解析。

### 18.2 输出契约

每个命令返回至少包含：

```
status:    success | failed | blocked
run_id:    {domain}_{YYYYMMDD}_{HHMMSS}_{random4}
profile:   {profile_name}
snapshot:  {snapshot_id}  (如适用)
artifacts: [文件路径列表]
blocking:  {原因}  (如 blocked)
elapsed:   {耗时}
```

### 18.3 退出码

| 退出码 | 含义 |
|--------|------|
| 0 | 成功 |
| 1 | 一般错误 |
| 2 | 配置错误 |
| 3 | 质量/PIT 阻断 |
| 4 | 风控阻断 |
| 5 | 网关/外部系统错误 |

## 十九、框架扩展点与开发指引

Vortex 是一个框架——它定义骨架（协议、流程、数据契约），但不包含所有具体业务实现。框架开发者需要在扩展点填入实际逻辑。本节列出所有可扩展位置和操作方式。

### 19.1 扩展点清单

| 扩展点 | 协议/基类 | 典型场景 | 需要做什么 |
|--------|----------|---------|-----------|
| **数据源** | DataProvider | Tushare 新增 API、接入 Wind/AKShare 等新数据源 | 实现 `fetch_*` 方法，注册到 Provider 工厂 |
| **数据集** | dataset 配置 | Tushare 有新接口但框架未定义对应数据集 | 在数据集配置中新增 dataset 条目（名称、字段映射、质量规则） |
| **存储后端** | StorageBackend | 从 Parquet 迁移到 ClickHouse | 实现 `upsert/read/list_partitions/snapshot` 方法 |
| **质量规则** | QualityRule | 对某个数据集增加特殊校验 | 编写规则函数，注册到数据集的 quality_rules 配置 |
| **因子** | 因子注册协议 | 开发新因子 | 代码因子：写 Python 函数 + `register`；表因子：写 SQL 定义 |
| **交易网关** | Gateway | 接入 CTP/富途/IB | 实现 `submit_order/cancel_order/query_*` 等方法 |
| **Watch Provider** | WatchProvider | 新增"北向资金日报"推送 | 实现 `pull(config) -> WatchResult` |
| **通知渠道** | NotificationChannel | 增加 Telegram/邮件 | 实现 `send(message) -> result` |
| **风控规则** | RiskRule | 增加"行业集中度"风控 | 编写规则函数，注册到 risk_pack 配置 |

### 19.2 扩展操作流程（以 Tushare 新 API 为例）

以"Tushare 新增了分红明细接口 `dividend_detail`，框架需要接入"为例：

1. **定义数据集**：在数据集配置中新增 `dividend_detail` 条目，声明字段映射（Tushare 返回字段 → Vortex 内部字段名）和质量规则（如 `not_empty`、`no_duplicate`）。
2. **实现 fetch 方法**：在 TushareProvider 中新增 `fetch_dividend_detail()` 方法，调用 Tushare API 获取数据，返回标准 DataFrame。
3. **注册到流程**：将新数据集加入 `update` 的数据集列表，使得日更自动采集该数据。
4. **验证**：运行 `vortex data update --dataset dividend_detail` 确认采集 + 质量检查通过。

其他扩展点的操作模式类似：定义配置 → 实现协议方法 → 注册到框架 → 验证。

### 19.3 扩展注意事项

1. **不改框架骨架**：新增扩展点实现时，不应修改框架核心流程代码（如调度器、质量检查引擎、风控引擎的控制逻辑）。如果发现需要修改骨架才能实现某个扩展，说明该扩展点的协议设计不够——应先讨论协议变更。
2. **遵循数据契约**：新数据集的输出格式必须符合 02-共享契约中定义的 schema 规范。
3. **写测试**：每个新扩展至少覆盖：正常路径、空数据、网络失败。详见 07-测试方案。
4. **样例工程**：06-代码细节设计说明书将提供一个端到端样例脚本，从 `bootstrap` 到 `rebalance(paper)` 走完全过程，帮助开发者和测试人员理解框架使用方式。
