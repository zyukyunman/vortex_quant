# 策略域设计说明书 v2

> **定位**：本文档是 Strategy 域的架构级设计说明，回答"信号如何变成可交易的目标组合、回测如何多维度验证"。
> 对应产品冻结 v0.2 中策略组合（§三）与回测生命周期（§四）。`signal` 命令为只读查询入口——基于用户"在策略层查看可用信号及其评测摘要"的 UX 需求（见 [[用户场景与体验增强设计|07 §三]]）设计。
> 域间边界与跨域序列图见 [[软件架构总览]]；代码级签名见 [[代码细节设计说明书|06]]。
>
> 核心设计原则：回测引擎可替换（BacktestEngine 协议）、compile 与 backtest 分离、信号标准化消费（只接受 published signal_snapshot）、Fail-Closed、文件级域间解耦、compile 幂等性。
>
> **优先级**：若本文档与产品原型文档（07）在实现细节上有差异（如路径命名、枚举值），以本文档和 [[软件架构总览|00-总览]] 为准。

## 一、设计目标与产品映射

### 1.1 CLI 动作清单

本文档覆盖 9 个 CLI 动作，分为三组：

| 组 | 动作 | 一句话 | 关键输入 | 关键输出 |
|----|------|--------|----------|----------|
| **回测** | `backtest` | 多维度运行回测 | `--profile` `--snapshot` `--signal` | 多维度报告 + HTML + 评分 |
| | `inspect` | 查看指定回测记录 | `--run-id` | 终端格式化展示 |
| | `compare` | 并排比较两次回测 | `--left` `--right` | 差异表 + 关键指标对比 |
| **生命周期** | `tag` | 为回测打标签（受保护） | `--run-id` `--tag` | 标签写入 manifest |
| | `restore` | 恢复已归档的回测 | `--run-id` | 工件恢复至 active |
| | `cleanup` | 智能清理建议 | `--profile` | 分级清理建议 + 用户确认 |
| | `status` | 策略线概览 | `--profile` | 最近 run、当前 portfolio、异常汇总 |
| **编译** | `compile` | 信号 → 目标持仓 | `--profile` `--snapshot` `--signal` | `target_portfolio.json` + diff |
| **信号** | `signal` | 查看策略可用信号列表 | `--profile` | 已发布 signal 清单 + 评测摘要 |

> **`signal` 与 Research `publish-signal` 的边界**：`signal` 是策略域的只读查询动作——列出当前 profile 可消费的已发布 signal snapshot 及其关联的因子评测摘要。它不创建、不修改、不发布任何信号。信号的生成（因子计算 / 模型预测）和发布（`vortex research publish-signal`）完全属于 Research 域。只有 Research 域显式 publish 并将 status 设为 `published` 的 signal snapshot，才会出现在 `signal` 的查询结果中。

### 1.2 核心设计目标

| # | 目标 | 含义 |
|---|------|------|
| G1 | **信号标准化消费** | 策略域只接受已 publish 的 signal snapshot，不接受裸文件或临时 DataFrame |
| G2 | **回测引擎可替换** | 第一版用 Backtrader，上层通过 BacktestEngine 协议隔离 |
| G3 | **compile 与 backtest 分离** | backtest 是验证工具，compile 才是产出 target portfolio 的动作 |
| G4 | **全链路可追溯** | 每次执行产出 run_id + manifest + lineage，与数据域、研究域追溯体系一致 |
| G5 | **多维度验证** | 默认四种回测维度同时运行，避免单次全样本回测掩盖过拟合 |
| G6 | **回测即资产** | 每次回测自动保存、自动评分、可标记保护、支持智能清理 |

### 1.3 与其他域的集成概要

策略域处于 Research → **Strategy** → Trade 的中间位置：

- **上游 Research**：消费已发布的 signal_snapshot（symbol / score / rank），不依赖研究域内部因子中间表。
- **上游 Data**：通过 snapshot_id 获取行情数据和基准数据，只读已发布快照。
- **下游 Trade**：通过 `target_portfolio.json` 文件交付目标持仓，文件级解耦。
- **共享契约层**：委托 ProfileService（定义见 02-共享契约与配置模型）解析配置，消费 CalendarService 获取交易日历。StrategyProfile 的 schema 定义、必填字段校验、preset 继承规则由 02 文档统一管理，策略域只做 resolve 后的二次校验（如 alpha.signals 引用的 signal 是否存在、权重之和是否为 1）。

详细集成契约见 00-总览 §四。

## 二、域边界、核心对象与不变量

### 2.1 策略域拥有的核心对象

| 对象 | 可变性 | 说明 |
|------|--------|------|
| **BacktestRun** | 不可变工件 + 可变元数据 | 一次回测的完整产出。工件（report / equity_curve / trade_log / HTML）创建后不可修改；元数据（tag / archive_state / score）可变 |
| **DimensionResult** | 不可变 | 多维度回测中每个维度的独立结果（全样本 / 随机日期 / Walk-Forward / Regime） |
| **QualityScore** | 不可变 | 基于 BacktestRun 的 5 维评分快照。如评分算法升级，生成新版 score 而非覆盖旧版 |
| **TargetPortfolio** | 不可变 | compile 产出的目标持仓，冻结后交付给 Trade 域 |
| **RebalanceDiff** | 不可变 | compile 时计算的调仓差异 |
| **StrategyManifest** | 不可变 | run/compile 级执行元数据，包含完整 lineage |

### 2.2 状态模型

BacktestRun 有两组正交状态：**执行状态**和**保留状态**。

**执行状态**（不可逆）：

```
running → success | success_with_warnings | failed
```

**保留状态**（可回退）：

```
┌──────────┐  tag   ┌───────────┐
│  active  │ ─────→ │ protected │     protected 的 run 不参与 cleanup 建议
│          │ ←───── │           │     移除所有 tag 后自动回退
└──────────┘ untag  └───────────┘
     │
     │ cleanup + 用户确认
     ▼
┌──────────┐  30天后自动  ┌───────────┐
│ archived │ ──────────→ │  purged   │   工件物理删除，仅保留 manifest
└──────────┘             └───────────┘
     ↑   │
     └───┘ restore（30天内）
```

- `active`：默认保留状态，参与清理评估。
- `protected`：打过 tag 的 run，不参与自动清理建议。
- `archived`：经 cleanup 用户确认后标记，30 天内可 restore。注意：系统不自动 archive，必须经用户确认。
- `purged`：工件已物理删除，仅保留 manifest 元数据用于审计。

**TargetPortfolio 状态机**：

```
┌─────────┐  Pipeline 成功   ┌──────────┐  同 profile 新 compile  ┌─────────────┐
│  draft  │ ──────────────→ │  frozen  │ ────────────────────→  │ superseded  │
└─────────┘                 └──────────┘                        └─────────────┘
     │                           │
     │ Pipeline 失败              │ Trade 消费约束：只有 frozen 可被 rebalance 引用
     ▼                           │ archive/cleanup 不适用于 portfolio（永久保留供审计）
 （丢弃，不持久化）
```

- `draft`：Pipeline 执行中的中间态，仅存在于内存。Pipeline 失败时 draft 不持久化。
- `frozen`：Pipeline 成功后写入磁盘，此后内容不可变。Trade 域唯一可消费的状态。
- `superseded`：同一 profile 使用新 signal / snapshot 重新 compile 成功后，旧 frozen portfolio 自动标记 superseded。不可被新 rebalance 引用，但永久保留供审计追溯。

### 2.3 不变量

1. **发布边界**：Strategy 域不生成信号，不训练模型。Alpha 阶段仅对已发布的 signal snapshot 做融合 / 路由 / 筛选。
2. **快照不可变**：任何 BacktestRun 的工件（JSON / CSV / HTML）创建后不可修改。需要修正时重新执行，生成新 run_id。
3. **compile 幂等**：相同 profile + signal + snapshot 输入产出完全相同的 target_portfolio。重复执行返回已有结果。
4. **cleanup 不丢失引用**：已被 compile 或 Trade 引用的 BacktestRun 不可 purge（仅可 archive，保留 manifest）。
5. **评分不阻断**：QualityScore 计算失败不阻断回测完成，run 标记 `score_status: failed`，后续可重算。

### 2.4 血统（Lineage）字段

血统记录回测和编译的完整依赖链，支持从最终交易决策反向追溯至原始数据和因子版本。

**BacktestRun manifest lineage 字段**：

```
run_id / profile / snapshot_id
signal_ids[]                        # 有序数组，支持多 signal 输入
signal_snapshot_versions[]          # 每个 signal 对应的 snapshot 版本号
signal_published_at[]               # 每个 signal 的 publish 时间戳
research_run_ids[]                  # 每个 signal 来源的 Research 域 run_id，支持反向追溯
backtest_mode / dimensions_enabled / seed
benchmark_config / cost_model / slippage_model
config_hash / vortex_version / created_at
```

**TargetPortfolio manifest lineage 字段**：

```
portfolio_id / profile / snapshot_id
signal_ids[] / signal_snapshot_versions[]
compile_run_id                      # 产出本 portfolio 的 compile 运行 ID
source_backtest_run_ids[]           # 验证本 portfolio 的回测 run_id 列表
config_hash / vortex_version / created_at
```

策略输入的唯一身份由 **ordered signal_ids + snapshot_id + config_hash** 三元组确定。compile 幂等和 run 血统追溯均基于此三元组。

**完整溯源链**：Trade execution → `source_portfolio_id` → TargetPortfolio → `compile_run_id` / `source_backtest_run_ids` → BacktestRun → `signal_ids` + `research_run_ids` → Research run → Data snapshot。Walk-Forward 和随机日期回测的每个子 run 额外记录 `window_start / window_end / trial_index`。

**signal_id 命名约定**：`{factor_family}_{variant}_{version}`，例如 `momentum_12m_v2`、`dividend_yield_ttm_v1`。版本号随 Research 域因子迭代递增，publish 后不可变。文内 profile 示例中的简写名（如 `sig_momentum_20d`）为 profile 内部 alias，由 ProfileService resolve 为完整 signal_id。compile 的 manifest 复用上述 BacktestRun lineage 基础字段，额外记录 `compile_run_id` 自身标识。

## 三、Pipeline 四阶段架构

### 3.1 Pipeline 作为核心执行抽象

策略域的核心计算模型是一条**四阶段固定顺序 Pipeline**。compile 和 backtest 共用同一 Pipeline，区别仅在于执行维度（单截面 vs 多期回放）。Pipeline 具有以下架构特征：

- **纯函数优先**：每个 Stage 的输出仅由其输入决定，不访问外部可变状态（磁盘持仓、网络等）。
- **显式输入/输出契约**：Stage 之间通过 typed result 对象传递，不共享全局变量。
- **Fail-Closed**：任一 Stage 产出无效结果（空集、NaN、约束冲突），Pipeline 立即阻断并记录 `blocking_reason`。
- **可截断执行**：`--stage --dry-run` 可在任意阶段截断，输出该阶段的标准中间结果。

### 3.2 Stage Protocol（域级协议）

四个阶段的统一协议定义如下（代码级签名见 06 §5.2）：

```
PipelineStage Protocol
  ├── name: str                          # 阶段标识
  ├── execute(input: StageInput, ctx: PipelineContext) -> StageResult
  └── validate_output(result: StageResult) -> ValidationResult
```

每个 Stage 接收上一阶段的 StageResult（首阶段接收 PipelineContext 中的初始输入），返回自己的 StageResult。validate_output 在 execute 之后由 Orchestrator 自动调用，不通过则触发 Fail-Closed。

**四阶段协议与数据流**：

```
┌──────────────┐         ┌──────────────┐         ┌───────────────────┐         ┌──────────────┐
│ UniverseStage│    →    │  AlphaStage  │    →    │ PortfolioStage    │    →    │  RiskStage   │
│ 股池选择      │         │ 信号融合      │         │ 组合构建           │         │ 静态约束      │
└──────────────┘         └──────────────┘         └───────────────────┘         └──────────────┘
  输入: MarketSnapshot     输入: UniverseResult     输入: AlphaResult            输入: PortfolioDraft
        + Profile                + SignalSet               + Constraints              + ConstraintSet
  输出: UniverseResult     输出: AlphaResult        输出: PortfolioDraft          输出: TargetHoldings
```

**Stage 输出的公共字段**（每个 StageResult 必须携带）：

| 字段 | 说明 |
|------|------|
| `stage_name` | 阶段标识 |
| `as_of` | 数据截面日期 |
| `snapshot_id` | 依赖的数据快照 |
| `signal_ids` | 消费的信号列表（Alpha 阶段填入） |
| `warnings` | 非阻断性告警列表 |
| `blocking_reason` | 若有值则 Pipeline 阻断（互斥于正常输出） |
| `elapsed_ms` | 阶段耗时 |

**各阶段的专属输出**：

| 阶段 | 输出对象 | 关键字段 |
|------|---------|---------|
| Universe | `UniverseResult` | `symbols: list[str]`（候选标的）, `filter_log: list[FilterRecord]`（过滤明细） |
| Alpha | `AlphaResult` | `scores: dict[str, float]`（标的→分数）, `ranks: dict[str, int]`（标的→排名）, `mode: str`（融合模式标记） |
| Portfolio | `PortfolioDraft` | `holdings: list[Holding]`（标的+权重）, `scheme: str`（赋权方案）, `total_weight: float`（归一化前后） |
| Risk | `TargetHoldings` | `holdings: list[Holding]`（约束后持仓）, `adjustments: list[Adjustment]`（约束调整记录）, `violations: list[Violation]`（约束冲突记录） |

### 3.3 PipelineOrchestrator

PipelineOrchestrator 负责从 profile 组装 Pipeline 实例，按序执行四阶段，管理阶段间的上下文传递。

**PipelineContext 字段**：

| 字段 | 说明 | 来源 |
|------|------|------|
| `snapshot_id` | 数据快照 ID | CLI `--snapshot` |
| `signal_ids` | 消费的信号列表 | CLI `--signal` |
| `as_of` | 数据截面日期 | 从 snapshot 推断或 CLI 指定 |
| `profile_config` | 已解析的 ResolvedProfile | ProfileService |
| `seed` | 确定性随机种子 | hash(profile + snapshot_id) |
| `current_holdings` | 当前持仓快照（可选） | CLI `--current-holdings`；未提供时为空列表 |

> `current_holdings` 是可选输入。compile 模式下通过 `--current-holdings` 注入；backtest 模式下由 BacktestEngine 在每个 rebalance_date 自动填入上一期的实际持仓。此字段对 Universe 和 Alpha 阶段均可见——Universe 用于合并候选池，Alpha 用于保留式筛选（见 §3.7 `preserve_current_holdings`）。

**职责**：
1. **resolve stages**：根据 profile 配置，从 registry 中查找各阶段的具体实现。
2. **build context**：构建 PipelineContext（含上述全部字段）。
3. **sequential execute**：按 Universe → Alpha → Portfolio → Risk 顺序依次执行。
4. **validate between stages**：每阶段执行后调用 validate_output，不通过则立即 Fail-Closed。
5. **truncate for dry-run**：`--stage` 参数指定截断点，Orchestrator 执行到该阶段后返回标准中间结果。

**compile vs backtest 的执行差异**：

| | compile | backtest |
|---|---------|----------|
| 调用方 | CompileRunner | BacktestRunner（委托 BacktestEngine） |
| 时间维度 | 单截面：对当前 as_of 执行一次 Pipeline | 多期回放：按 rebalance_dates 重复执行 Pipeline |
| Pipeline 实例 | 一次性创建并执行 | 每个 rebalance_date 复用同一 Pipeline 实例 |
| 输出 | 最终 TargetHoldings → target_portfolio.json | 每期 TargetHoldings → BacktestEngine 撮合 → 净值曲线 |
| 幂等 | 严格幂等（相同输入 → 相同 portfolio_id） | 每次执行产生新 run_id |

### 3.4 错误模型

Pipeline 的错误分为两类：

**阻断型（Hard Fail → Fail-Closed）**：

| 阶段 | 故障场景 | blocking_reason |
|------|---------|----------------|
| Universe | 过滤后候选池为空 | `"universe is empty after filters"` |
| Alpha | 输出含 NaN 或 score 全相同 | `"alpha output invalid: {detail}"` |
| Alpha | weighted_fusion 权重之和 ≠ 1.0 | `"alpha weights do not sum to 1.0"` |
| Portfolio | 归一化后权重异常 | `"portfolio weights invalid"` |
| Risk | 约束冲突导致组合清空 | `"all positions eliminated by constraints"` |
| Risk | 约束间不可调和（如 max_weight 裁剪后违反 min_positions） | `"constraint conflict: {rule_a} vs {rule_b}"` |

阻断型错误通过 StageResult.blocking_reason 返回，Orchestrator 立即终止 Pipeline，不执行后续阶段。compile 返回失败；backtest 在该 rebalance_date 标记 `period_failed`，后续 date 继续执行。

**告警型（Warning → 记录但不阻断）**：

- Universe 过滤比例超过 80%（过度筛选提醒）
- Alpha 多信号融合中某信号覆盖率低于 50%（信号稀疏提醒）
- Risk 约束触发大量权重调整（约束过紧提醒）

告警通过 StageResult.warnings 累积，最终写入 run manifest。

### 3.5 风险分层：静态约束 vs 运行时策略

Risk 阶段的约束分为两类，分别适用于不同场景：

**一、静态组合约束（StaticConstraint）——compile 和 backtest 共用**

这类约束作用于单个截面的目标持仓，不依赖历史路径。compile 和 backtest 使用完全相同的约束逻辑，保证回测结论在实盘可复现。

| 约束 | 参数 | 行为 |
|------|------|------|
| `max_weight` | 如 5% | 超限标的裁剪至上限，剩余重新归一化 |
| `min_positions` | 如 15 | 不足时按 score 补入候选池 |
| `max_positions` | 如 30 | 超出时按 score 淘汰尾部 |
| `sector_limit` | 如 20% | 行业超限时按 score 裁剪该行业持仓 |

约束按声明顺序逐条应用。每条约束修改权重后，后续约束基于修改后的权重继续检查。所有约束应用完毕后做最终归一化。

**二、运行时风险策略（RuntimeRiskPolicy）——仅 backtest 可用，compile 不执行**

这类策略依赖时间序列或持仓演化路径，无法在单截面 compile 中执行：

| 策略 | 参数 | 行为 |
|------|------|------|
| `drawdown_stop` | 如 -15% | 回测中累计回撤触及阈值则全部清仓，后续 rebalance 跳过 |
| `volatility_scale` | 如 target_vol=15% | 按近期波动率动态调整仓位规模 |

配置中二者明确分离：

```yaml
risk:
  constraints:                # 静态约束 → compile + backtest 共用
    - max_weight: 0.05
    - max_positions: 30
    - sector_limit: 0.20
  runtime_policies:           # 运行时策略 → 仅 backtest
    - drawdown_stop: -0.15
```

compile 仅执行 `constraints`，忽略 `runtime_policies`。backtest 两者都执行。

### 3.6 扩展与注册机制

Pipeline 的每个阶段支持通过 Registry 注册新的实现，框架开发者和用户均可扩展：

| Registry | 内置实现 | 扩展方式 |
|----------|---------|---------|
| `UniverseFilterRegistry` | `index_members`（指数成分）、`liquidity_min`、`listing_days_min`、`exclude_st` | 实现 `UniverseFilter Protocol`，放入 `strategy/universe/` 目录，自动发现 |
| `AlphaModeRegistry` | `weighted_fusion`（加权融合）、`cascading`（级联筛选） | 实现 `AlphaMode Protocol`，放入 `strategy/alpha/` 目录 |
| `PortfolioSchemeRegistry` | `equal_weight`、`signal_weighted`、`risk_parity` | 实现 `PortfolioScheme Protocol`，放入 `strategy/portfolio/` 目录 |
| `ConstraintRegistry` | `max_weight`、`min_positions`、`max_positions`、`sector_limit` | 实现 `Constraint Protocol`，放入 `strategy/risk/` 目录 |

注册机制与 01 的 `DataProviderRegistry`、03 的 `WorkflowTemplateRegistry` 设计一致——基于目录扫描自动发现、Protocol 约束接口、profile 中按名称引用。

### 3.7 内置实现说明

以下是 Pipeline 各阶段的内置实现及其 profile 配置。这些是框架预置的默认能力，用户可通过 §3.6 的扩展机制新增。

#### Universe 阶段

配置示例：

```yaml
universe:
  base: "000300.SH"           # 基础池：沪深300成分股
  filters:
    - liquidity_min: 5000000  # 日均成交额 ≥ 500万
    - listing_days_min: 120   # 上市满 120 个交易日
    - exclude_st: true        # 排除 ST 股票
```

Universe 输出 `UniverseResult.symbols`（候选标的列表），同时记录 `filter_log`（每条 filter 淘汰了哪些标的、淘汰数量），供审计和 dry-run 时展示。

#### Alpha 阶段：两种内置信号组合模式

通过 profile 的 `alpha.mode` 字段选择模式：

**模式一：加权融合（weighted_fusion）**——多个信号按权重合并为综合分数：

```yaml
alpha:
  mode: weighted_fusion
  signals:
    - signal_id: sig_momentum_20d
      weight: 0.3
    - signal_id: sig_value_bp
      weight: 0.3
    - signal_id: sig_quality_roe
      weight: 0.2
    - signal_id: sig_volatility
      weight: 0.2
```

融合规则：对每只标的，`weight × normalize(score)` 加权求和。权重之和必须为 1.0，否则 Fail-Closed 阻断。

**模式二：级联筛选（cascading）**——多个步骤按顺序执行，上一步输出为下一步输入：

```yaml
alpha:
  mode: cascading
  stages:
    - name: style_rotation
      signal_id: sig_regime_detector
      action: route
    - name: stock_selection
      signal_id: sig_small_growth
      action: filter_top_n
      params: { top_n: 50 }
    - name: timing_overlay
      signal_id: sig_momentum_timing
      action: score_filter
      params: { threshold: 0.6 }
```

两种模式可嵌套——级联的某一步可使用加权融合。

**`filter_top_n` 的 `preserve_current_holdings` 参数**：

当 `preserve_current_holdings: true` 时，`filter_top_n` 的输出 = top-N 标的 ∪ PipelineContext.current_holdings 中仍在候选池的标的。这使得当前持仓不会因为跌出价值排名而被立即剔除，而是留到后续阶段（如动量打分）决定去留。

```yaml
alpha:
  mode: cascading
  stages:
    - name: value_selection
      signal_id: sig_value_composite
      action: filter_top_n
      params: { top_n: 50, preserve_current_holdings: true }
    - name: momentum_timing
      signal_id: sig_momentum_20d
      action: score_filter
      params: { threshold: 0.0 }
```

上例中，value_selection 阶段输出 = 价值排名前 50 只 + 当前持仓中仍在 Universe 范围内的标的（即使它们不在前 50）。momentum_timing 再对这个扩展集合打分，动量 ≤ 0 的标的被移除——包括跌出价值前 50 且动量转负的当前持仓。

> **关键边界**：Alpha 不做因子计算和模型训练——这些属于 Research 域。Alpha 只对已发布的 signal_snapshot 做融合/路由/筛选。多 signal 输入时，每个 signal 必须是独立已发布的 signal_snapshot。

#### Portfolio Construction 阶段

| 方案 | 逻辑 | 适用场景 |
|------|------|---------|
| `equal_weight` | 入选标的等权 | 因子投资、分散化 |
| `signal_weighted` | 按 score 排名加权 | 信号置信度分化大 |
| `risk_parity` | 按波动率倒数加权 | 风险预算均衡 |

赋权后执行归一化，确保 `total_weight = 1.0`。

### 3.8 阶段独立测试

得益于 Stage Protocol 的显式输入/输出契约，每个阶段可独立验证：

```bash
# 只执行到 Alpha 阶段并输出中间结果（不涉及组合构建和风控）
$ vortex strategy backtest --profile my_strategy --stage alpha --dry-run

# 只测 Risk 约束效果（给定一个目标持仓，看约束调整了什么）
$ vortex strategy backtest --profile my_strategy --stage risk --dry-run
```

`--stage` + `--dry-run` 由 PipelineOrchestrator 执行到指定阶段后截断，输出该阶段的标准 StageResult（JSON 格式），不实际运行回测引擎。

## 四、组合编译（Compile）

### 4.1 编译流程

compile 是策略域对 Trade 域的核心交付动作，将信号转化为可执行的目标持仓：

```
signal snapshot(s) → Pipeline 四阶段 → target_portfolio.json
                                        ├── rebalance_diff.csv
                                        └── order_plan_preview.csv
```

编译步骤：

1. 解析 profile（委托 ProfileService）
2. 校验所有引用的 signal snapshot 已 publish
3. 构建 PipelineContext（含 `current_holdings`，来自 `--current-holdings` 参数；未提供时为空）
4. 执行 Pipeline 四阶段（Universe → Alpha → Portfolio Construction → Risk）
5. 如 `current_holdings` 非空，基于 Pipeline 输出的 TargetHoldings 与 current_holdings 计算调仓差异
6. 写入工件到 `strategy/portfolios/{portfolio_id}/`
7. 返回 manifest

### 4.2 幂等性保证

compile 的幂等性通过三层机制保证：

1. **portfolio_id 确定性生成**：`tp_{profile}_{snapshot_id_hash_6}_{signals_hash_6}`，基于 ordered signal_ids + snapshot_id + config_hash 三元组，相同输入 → 相同 ID。
2. **写入前检查**：如 `strategy/portfolios/{portfolio_id}/` 已存在且 config_hash 一致，直接返回已有结果。
3. **不依赖外部可变状态**：compile 只读取 signal snapshot（不可变）和 profile（版本化），不读取交易系统实时状态。

### 4.3 工件 Schema

`target_portfolio.json`：

```json
{
  "portfolio_id": "tp_dividend_classic_b2c3a1_f7e8d9",
  "signal_ids": ["momentum_12m_v2", "dividend_yield_ttm_v1"],
  "snapshot_id": "snap_20260403_220000_b2c3",
  "profile": "dividend_classic",
  "pipeline_summary": {
    "universe": "000300.SH, 筛后 248 只",
    "alpha_mode": "weighted_fusion",
    "portfolio_scheme": "signal_weighted",
    "constraints_applied": ["max_weight_5pct", "sector_limit_20pct"]
  },
  "holdings": [
    { "symbol": "600036.SH", "weight": 0.048, "target_shares": 1200, "reason": "score=0.82, rank=3" }
  ],
  "total_positions": 20,
  "benchmark": "000300.SH",
  "compile_time": "2026-04-03T22:31:18+08:00",
  "status": "frozen",
  "lineage": {
    "compile_run_id": "compile_20260403_223118_a1b2",
    "signal_snapshot_versions": ["v3", "v1"],
    "research_run_ids": ["research_20260402_m1", "research_20260401_d2"],
    "source_backtest_run_ids": ["backtest_20260403_210000_c3d4"],
    "config_hash": "e7f8a9b0"
  }
}
```

### 4.4 compile 冲突场景

| 场景 | 行为 |
|------|------|
| **相同输入重复 compile** | 幂等机制：检测到 portfolio_id 已存在且 config_hash 一致，直接返回已有结果 |
| **并发 compile 相同输入** | 先完成的写入成功；后到的检测到已存在，直接返回（无锁竞争，利用文件系统原子性） |
| **已有 frozen portfolio 被新 compile 取代** | 旧 portfolio 标记 `superseded`，新 portfolio 获得新 portfolio_id（因 `signal_ids` / `snapshot_id` / `config_hash` 任一变化） |
| **Pipeline 约束冲突** | Fail-Closed 阻断（见 §3.5 约束冲突处理） |

### 4.5 compile 与 backtest 的关系

二者共用同一套 Pipeline 四阶段逻辑和 profile 解析，区别在于：

| | compile | backtest |
|---|---------|---------|
| 目的 | 生产级目标持仓交付 | 历史验证 |
| 时间维度 | 单次、当前截面 | 多期、历史区间 |
| 输出 | target_portfolio + diff | report + equity_curve + trade_log + HTML |
| 幂等 | 严格幂等 | 每次执行产生新 run_id |
| 静态约束（§3.5） | 执行 `constraints` | 执行 `constraints`（与 compile 一致） |
| 运行时策略（§3.5） | **不执行** `runtime_policies` | 执行 `runtime_policies`（如 drawdown_stop） |

静态约束的一致性保证回测结论在实盘可复现。运行时策略仅在回测中提供额外的路径依赖验证。

## 五、多维度回测系统

### 5.1 设计动机

单次全样本回测容易掩盖过拟合——策略可能刚好在样本期表现好，换个起始日期就失效。多维度回测是**默认行为**，每次回测自动执行四种模式，从不同角度验证策略鲁棒性。

### 5.2 四种回测维度

#### 5.2.1 全样本回测（Full-Sample）

使用 snapshot 中的完整历史区间。**始终开启，不可关闭**，是基础参照。

- 输入：完整历史行情 + signal snapshot
- 输出：BacktestReport（标准指标集 + 资金曲线 + 交易记录）

#### 5.2.2 随机日期回测（Random-Dates）

从可用日期范围中随机采样 N 组起止日期，各自运行独立回测：

- 默认采样次数：20 次
- 最小窗口：252 个交易日（约 1 年）
- **种子固定**：`seed = hash(profile_name + snapshot_id)`，保证相同输入下采样结果完全一致
- 输出：N 次 Sharpe 的均值 / 标准差 / 中位数 / 胜率（Sharpe > 0.5 的比例）

随机日期回测的核心价值是检验策略对起止日期选择的敏感度。如果 20 次里只有 5 次 Sharpe > 0.5，说明全样本的好看数据大概率是运气。

#### 5.2.3 Walk-Forward 验证

滚动窗口方式，模拟"用过去训练、用未来检验"的场景：

- 训练窗口（train_window）：默认 252 交易日
- 测试窗口（test_window）：默认 63 交易日
- 步进间隔（step）：默认 63 交易日（每次向前滚动一个测试窗口）
- 输出：每个窗口的样本外 Sharpe、整体衰减率（样本外 / 样本内 Sharpe 比值均值）

衰减率 < 0.5 强烈暗示过拟合。衰减率 > 0.8 说明策略对训练期的依赖度低。

#### 5.2.4 牛熊市分段回测（Regime）

将历史区间按市场状态（Regime）分段，分别评估策略在不同市况下的表现：

- 分类方法：默认 MA120 + 斜率
  - 价格 > MA120 且 MA120 斜率 > 0 → 牛市
  - 价格 < MA120 且 MA120 斜率 < 0 → 熊市
  - 其他 → 震荡
- 分类依据的行情数据必须来自 **Data snapshot 内的 PIT（Point-In-Time，时点对齐）可追溯数据**，不使用未来信息
- 输出：各 regime 下的 Sharpe / 年化收益 / 最大回撤

### 5.3 回测配置

通过 StrategyProfile 的 `backtest.dimensions` 配置各维度参数：

```yaml
backtest:
  dimensions:
    full_sample: true                # 始终开启
    random_dates:
      enabled: true
      n_trials: 20
      min_window: 252
    walk_forward:
      enabled: true
      train_window: 252
      test_window: 63
      step: 63
    regime:
      enabled: true
      method: ma120                  # ma120 | ai | custom
  rebalance_freq: monthly            # daily | weekly | monthly
```

用户可通过 `--skip random_dates,walk_forward` 临时跳过耗时维度。

### 5.4 部分成功处理

当某个维度无法执行时（如历史不足导致 Walk-Forward 窗口数为 0），该维度标记为 `skipped`，其余维度正常完成。整体 run 状态为 `success_with_warnings`，manifest 记录各维度执行状态：

```json
{
  "dimensions": {
    "full_sample": { "status": "success" },
    "random_dates": { "status": "success", "trials": 20 },
    "walk_forward": { "status": "skipped", "reason": "历史不足: 需要 315 交易日, 实际 200" },
    "regime": { "status": "success", "segments": 8 }
  }
}
```

## 六、双基准系统

### 6.1 设计动机

只看绝对收益无法判断策略的真实 Alpha。指数点位基准简单但忽略了涨跌停 / 停牌等实际约束，导致超额收益偏高。Alpha 基准通过成分股仿真排除这些干扰。

### 6.2 两种基准

| 基准 | 计算方式 | 用途 |
|------|---------|------|
| **指数价格基准** | 直接使用基准指数收盘价序列计算净值 | 快速判断跑赢大盘幅度 |
| **Alpha 基准** | 以指数成分股和权重构建虚拟组合，与策略共用成交规则（涨跌停、停牌、手续费） | 衡量真实 Alpha |

Alpha 基准有两种权重模式：

| 模式 | 说明 | 性能开销 |
|------|------|---------|
| `quarterly` | 季报公布的成分股和权重，季报间保持不变 | 低 |
| `dynamic` | 每个调仓日重新获取成分股和权重 | 约增加 30-50% 回测时间 |

### 6.3 基准数据来源

Alpha 基准的成分股 / 权重数据必须来自 Data snapshot 内的 PIT 可追溯来源：
- 成分股列表取自指数成分数据（Data 域提供）
- 回测中每个时点使用的成分股是**当时已公布**的版本，不使用未来信息
- 如成分股数据缺失，Alpha 基准维度标记 `skipped`，不阻断回测

### 6.4 配置

```yaml
benchmark:
  index: "000300.SH"
  show_price_benchmark: true
  show_alpha_benchmark: true
  alpha_weight_mode: quarterly     # quarterly | dynamic
```

## 七、质量评分系统

### 7.1 评分模型

每次回测完成后自动从 5 个维度计算质量分数，综合评估策略的收益能力、一致性、韧性、实用性和泛化能力：

| 维度 | 计算方式 | 权重 |
|------|---------|------|
| **收益风险比** | Sharpe ratio 的统计显著性（bootstrap p-value） | 25% |
| **稳定性** | 滚动 6 个月 Sharpe 的标准差，越小越好 | 25% |
| **回撤控制** | 最大回撤 / 恢复时间的综合评分 | 20% |
| **成本效率** | (收益 - 预估成本) / 换手率 | 15% |
| **过拟合风险** | 全样本 vs Walk-Forward 样本外 Sharpe 衰减率 + 随机日期稳定性 | 15% |

每个维度独立评分为 0-100 分，加权求和得到综合分。

### 7.2 字母评级

| 评级 | 综合分 | 含义 |
|------|--------|------|
| A | ≥ 85 | 优秀——建议保留并进一步验证 |
| B+ | ≥ 75 | 良好偏优——值得重点关注 |
| B | ≥ 70 | 良好——值得保留观察 |
| B- | ≥ 60 | 尚可——需谨慎 |
| C | ≥ 50 | 一般——不建议直接使用 |
| D | ≥ 35 | 较差——建议分析原因 |
| F | < 35 | 无效——建议清理 |

> 回测评级与因子评级共用 A / B+ / B / B- / C / D 体系，回测额外增加 F 级。

### 7.3 评分容错

- 如果 Walk-Forward 被 skip（历史不足），过拟合风险维度退化为仅看随机日期稳定性，权重不变。
- 如果随机日期也被 skip，过拟合风险维度标记 N/A，其余 4 维按原比例重新归一化。
- 评分计算失败不阻断回测完成（不变量 2.3 第 5 条），run 记录 `score_status: failed`，用户可通过 `vortex strategy inspect --run-id X --rescore` 手动重算。

## 八、回测生命周期

### 8.1 生命周期全景

```
backtest 完成 → 自动保存 → 自动评分 → [用户 tag → protected]
                                     ↓
                              active（默认保留）
                                     ↓ cleanup 建议 + 用户确认
                              archived（30天可 restore）
                                     ↓ 30天后
                              purged（仅保留 manifest）
```

### 8.2 自动保存

每次回测结果自动持久化，包含：

| 保存内容 | 说明 |
|---------|------|
| 配置快照 | 完整的 ResolvedStrategyProfile 展开结果 |
| 绩效指标 | 全部标准指标（含各维度） |
| 资金曲线 | 每日净值时序（equity_curve.csv） |
| 交易记录 | 逐笔交易（trade_log.csv） |
| 流水线状态 | 各阶段输入输出摘要 |
| 质量评分 | 自动计算的 5 维分数 + 字母评级 |
| HTML 报告 | 6 页交互仪表盘（见 §九） |

### 8.3 标记保护（tag）

```bash
$ vortex strategy tag --run-id backtest_20260403_d8e3 --tag "baseline_v2"
```

- 一个 run 可以打多个 tag。
- 有 tag 的 run 状态变为 `protected`，不出现在 cleanup 建议中。
- 移除所有 tag 后自动回退为 `active`。

### 8.4 智能清理（cleanup）

```bash
$ vortex strategy cleanup --profile style_rotation_momentum
```

清理规则：
1. **系统只建议，用户做决定**——不自动删除任何回测。
2. **有 tag 的 run 受保护**——不出现在清理建议中。
3. **有下游依赖的 run 不可 purge**——如该 run 已被 compile 或 Trade 引用，仅可 archive 不可 purge，manifest 永久保留。依赖检测机制：扫描本域 `portfolios/` 目录下各 TargetPortfolio manifest 的 `source_backtest_run_ids[]` 字段，并通过 Run Registry 的 `inspect` / `list` 接口（见 00-总览 §7.4）检查 lineage 中是否存在对该 run_id 的引用。Trade 域通过在其 execution manifest 中记录 `source_portfolio_id` 建立跨域引用链。任一引用存在即阻断 purge。
4. **分级清理**——先建议清 F 级，再建议清 D 级，C 及以上默认保留。
5. **可恢复**——cleanup 只是标记为 `archived`，30 天内可 restore。

当某个 profile 下回测数量超过阈值（默认 50 条），回测完成后自动提醒用户考虑 cleanup。

### 8.5 恢复（restore）

```bash
$ vortex strategy restore --run-id backtest_20260401_e4f5
```

- 仅 `archived` 状态的 run 可恢复。
- 恢复后状态回到 `active`。
- 超过 30 天的 `archived` run 自动转为 `purged`（工件物理删除，manifest 保留）。

### 8.6 因子归档整合

策略可通过 `signal` 命令查看因子评测历史摘要。StrategyProfile 的 `alpha.signals` 引用的每个 signal_id 关联到 Research 域的 factor_archive，支持在策略层追溯"这个信号是由哪些因子、什么评测成绩产出的"。此关联为只读引用，不跨域修改数据。

## 九、HTML 回测报告架构

### 9.1 报告定位

每次回测自动生成自包含（self-contained）HTML 报告，内嵌数据和 JS，可离线查看。与因子评测 HTML 报告共享同一套设计系统（配色、布局、字体、组件）。

### 9.2 六页结构

| 页面 | 内容 | 核心数据依赖 |
|------|------|------------|
| **综合仪表盘** | 核心指标卡片 + 评级徽章 + 资金曲线（双基准）+ 质量雷达图 | nav / summary / quality_scores |
| **月度收益热力图** | 按月/年展示收益，识别季节性和异常月份 | monthly |
| **滚动指标** | 滚动 Sharpe / Beta / 波动率 / 回撤的时序图 | rolling |
| **风险归因** | 行业暴露热力图、风格暴露、个股集中度时序分解 | risk_attribution |
| **交易分析** | 持仓周期分布、盈亏比、最大连续亏损、换手率时序 | trades |
| **流水线拆解** | 各阶段（Universe / Alpha / Portfolio / Risk）对最终表现的贡献拆解 | pipeline |

### 9.3 数据嵌入

报告数据通过 `<script id="vortex-data" type="application/json">` 内嵌在 HTML 中，前端 JS 解析后渲染图表。内嵌 JSON == `backtest_report.json` 的 canonical schema（二者内容相同，不允许 HTML 端额外计算）。数据结构按页面分区（meta / nav / summary / quality_scores / monthly / rolling / risk_attribution / trades / pipeline），完整字段定义见 06 §5.7。共享渲染契约见 06 §4.9。

### 9.4 交互规范

- 页面切换、深浅色模式、图表缩放、悬浮提示遵循全平台共享设计系统（见 07-用户场景与体验增强设计 §2.7 设计体系）。
- 回测特有交互：基准切换（指数 / Alpha / 同时显示）、日期区间 range slider、流水线阶段复选开关、热力图年份筛选、行业暴露排序切换。
- 基准切换不仅切换资金曲线，也必须同步切换 `summary.relative_metrics` 与 `monthly.yearly_summary` 的展示口径；年度汇总由后端预计算，不允许 HTML 端从月收益热力图临时补算。
- 前端实现细节见产品原型文档 §4.6。

## 十一、域故障模式

策略域遵循全系统 Fail-Closed 原则：宁可阻断不执行，不可带着问题执行。

### 11.1 前置条件故障

| 故障 | 检查点 | 行为 | 严重度 |
|------|--------|------|--------|
| snapshot 不存在或未 publish | BacktestRunner / CompileRunner 启动前 | 阻断，`blocking_reason: "snapshot not published"` | error |
| signal snapshot 不存在或未 publish | BacktestRunner / CompileRunner 启动前 | 阻断，`blocking_reason: "signal not published"` | error |
| profile 不合法 | StrategyProfileResolver | 抛出 `ProfileValidationError` | error |
| Alpha 引用的多个 signal 中有任一未 publish | Pipeline Alpha 阶段 | 阻断整个 Pipeline | error |

### 11.2 回测引擎故障

| 故障 | 场景 | 行为 |
|------|------|------|
| Backtrader 内部异常 | Feed 数据格式错误、内存不足、Analyzer 异常 | 捕获异常，写 manifest `status: failed`，保留已产出的部分工件供排查 |
| Feed 数据不足 | snapshot 中标的数量 < min_positions | 阻断，`blocking_reason: "insufficient symbols in snapshot"` |
| 回测超时 | 回测运行超出配置的 max_duration（默认 30 分钟） | 强制终止，`status: failed`，记录已完成维度 |

### 11.3 Pipeline 阶段契约故障

> 完整的 Pipeline 错误模型（阻断型 vs 告警型）已在 §3.4 定义。本节列出典型故障场景速查表。

| 故障 | 阶段 | 行为 |
|------|------|------|
| Universe 输出空集 | Universe | 阻断，`blocking_reason: "universe is empty after filters"` |
| Alpha 输出含 NaN 或 score 全相同 | Alpha | 阻断，`blocking_reason: "alpha output invalid"` |
| Alpha weighted_fusion 权重之和 ≠ 1.0 | Alpha | 阻断，`blocking_reason: "alpha weights do not sum to 1.0"` |
| Portfolio 归一化后权重异常 | Portfolio | 阻断，`blocking_reason: "portfolio weights invalid"` |
| Risk 约束后组合清空 | Risk | 阻断，`blocking_reason: "all positions eliminated by constraints"` |
| Risk 约束间不可调和 | Risk | 阻断，`blocking_reason: "constraint conflict: {rule_a} vs {rule_b}"` |

### 11.4 多维度回测降级

| 故障 | 维度 | 行为 |
|------|------|------|
| 历史不足以切分窗口 | Walk-Forward | 该维度 `skipped`，其余维度正常完成 |
| 随机采样无法满足 min_window | Random-Dates | 该维度 `skipped` |
| MA120 数据不足以计算分类 | Regime | 该维度 `skipped` |
| 基准成分股数据缺失 | Alpha 基准 | Alpha 基准 `skipped`，指数价格基准正常 |

维度 skip 不阻断 run，整体状态为 `success_with_warnings`。

### 11.5 评分系统故障

| 故障 | 行为 |
|------|------|
| 某一维度分数无法计算（如 Walk-Forward 被 skip 导致过拟合维度缺失） | 该维度标 N/A，剩余维度重新归一化 |
| 评分算法整体失败 | run 标记 `score_status: failed`，不阻断回测 |

### 11.6 生命周期故障

| 故障 | 行为 |
|------|------|
| cleanup 尝试 purge 被 Trade 引用的 run | 阻断 purge，保留 manifest，提示用户"该 run 已被 portfolio X 引用" |
| restore 时工件已物理删除（超过 30 天） | 阻断，提示"已超过恢复窗口，仅 manifest 可查" |
| tag 不存在的 run | 返回 `run not found` |
| 并发 compile 相同输入 | 幂等机制保证：先完成的写入成功，后到的检测到已存在直接返回 |

### 11.7 CLI 退出码

| 退出码 | 含义 |
|--------|------|
| 0 | 成功 |
| 1 | 一般错误 |
| 2 | 配置错误（ProfileValidationError） |
| 3 | 前置条件阻断（snapshot / signal 未发布） |
| 4 | 回测引擎故障 |

## 十二、确定性保障

多维度回测引入了随机性和环境依赖，本节明确架构级确定性规则。

### 12.1 种子管理

| 使用随机性的场景 | 种子来源 | 保证 |
|----------------|---------|------|
| 随机日期采样 | `seed = hash(profile_name + snapshot_id)` | 相同 profile + snapshot → 相同采样 |
| 走不到的场景 | 不允许 `random()` | 编码规范禁止无种子随机调用 |

种子写入 manifest 的 lineage 字段，可追溯可复现。

### 12.2 时间与数据可复现

| 威胁 | 防御 |
|------|------|
| Regime 分类使用未来数据 | Regime 分类仅使用 snapshot 内的 PIT 数据，MA120 计算截止到每个回测时点 |
| Alpha 基准成分股使用未来数据 | 成分股 / 权重来自 Data snapshot 的 PIT 对齐版本 |
| Alpha 基准 dynamic 模式不可复现 | dynamic 模式每个调仓日使用的成分股 / 权重，由 snapshot 内 PIT 数据 + 交易日历确定性推导。不依赖外部实时查询，同一 snapshot 重跑结果一致 |
| Walk-Forward 窗口边界不确定 | 窗口参数 + 交易日历确定性推导，结果写入 lineage |
| 回测结果浮点漂移 | 排序使用 rank（整数）+ 二级键（score desc, symbol asc） |
| `datetime.now()` 泄露 | 所有时间显式传参，run_id 中的时间戳在入口处固定 |

### 12.3 可复现性验证

- **golden master**：固定 profile + snapshot + signal，输出与基准逐字节比对。
- **差分测试**：同一输入连续运行两次，结果必须完全一致。
- 详见 07-测试方案说明书。

## 十三、Backtrader 适配层

### 13.1 适配层定位

Backtrader 是选定的回测引擎，但其 API（Cerebro / Strategy 子类 / Analyzer）不应泄漏到策略域上层。适配层的职责：**Vortex 标准对象 ↔ Backtrader API 双向翻译。**

```
BacktestEngine (Protocol)
    ↑
BacktraderAdapter (实现)
    ├── FeedBuilder:       snapshot → bt.feeds.PandasData
    ├── BrokerConfigurator: CostModel + SlippageModel → Broker 设置
    ├── StrategyBridge:     Pipeline → bt.Strategy.next() 逻辑
    └── AnalyzerBridge:     bt.Analyzers → BacktestReport 标准指标
```

### 13.2 引擎可替换

上层（BacktestRunner / CompileRunner）通过 `BacktestEngine` 协议调用回测引擎。协议定义输入为 profile / feed / signal / cost_model / slippage_model，输出为标准化 BacktestReport。具体签名见 06-代码细节设计。第一版只实现 BacktraderAdapter；未来替换为 vnpy / zipline / 自研引擎，只需新增 adapter 不动 Runner 和 CLI。

### 13.3 成本模型

| 模型 | 参数 | 说明 |
|------|------|------|
| `CnStockDefaultCost` | 佣金万三 + 卖出印花税千一 | A 股默认 |
| `FixedBpsSlippage` | 默认 5bps | 固定基点滑点 |

成本模型通过 profile 的 `backtest.cost` 字段选择，由 preset 提供默认值。
