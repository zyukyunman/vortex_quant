# 交易域设计说明书 v2

> 把 target portfolio（目标组合）安全、幂等、可追溯地送进模拟盘或实盘，并在日终完成对账闭环。
>
> 本文档是交易域 API 契约、域内序列图、域故障模式的 **owner**（见 [[软件架构总览|00 §十九]] 文档归属矩阵）。
> 跨域不变量引用 [[软件架构总览]]；代码级签名与 DDL 见 [[代码细节设计说明书]]。

---

## 一、设计目标与产品映射

本文档对应产品冻结 §六「交易执行与风控」，落地以下 6 个 CLI 动作：

| 动作 | 一句话 |
|------|--------|
| `paper rebalance` | 模拟盘调仓：生成 order plan → 事前风控 → 提交 → 监听回报 |
| `live rebalance` | 实盘调仓：同 paper 流程，但网关为 miniQMT、风控更严、强制审计与确认 |
| `inspect` | 查看某次执行的全部工件（plan / risk result / report / audit） |
| `retry` | 对失败订单发起新执行（重新过事前风控，生成新 exec_id） |
| `reconcile` | 日终对账：网关持仓 vs 目标组合 vs 成交记录 → 三组偏差报告 |
| `status` | 查看当前交易线状态（网关连接、未完成订单、最近执行摘要） |

核心目标：

1. **安全**：三层风控 + Fail-Closed（失败即阻断），任何异常都不放行。
2. **幂等**：确定性订单标识 + 意图持久化 + 重连对齐，崩溃后可安全恢复。
3. **可追溯**：每次执行产出结构化工件，可回溯、可审计。
4. **可替换**：Gateway 抽象层让 PaperGateway 和 MiniQMTGateway 共享同一套上层逻辑。

## 二、域边界、输入输出工件与血统

### 2.1 Trade 域的输入

| 输入 | 来源 | 传递方式 | 说明 |
|------|------|---------|------|
| target portfolio | Strategy 域 | `target_portfolio.json`（不可变工件引用） | `{symbol: target_weight}` |
| ResolvedTradeProfile | 共享契约层 | ProfileService.prepare() → 内存对象（包含 gateway 类型、risk_pack 风控规则包、trading_window 交易时间窗等完整配置） | gateway / risk_pack / trading_window 等 |
| 已发布 Data 快照 | Data 域 | 只读引用 `snapshot_id` | 见 §2.1.1 |

#### 2.1.1 Trade 对 Data 快照的只读依赖

**关键约束**（引用 00 §4.1）：Trade **只读已发布的 Data 快照**，不依赖 Data 的可变中间状态（如正在同步的增量数据）。

Trade 消费的具体 Data 字段：

| 数据项 | 用途 | 消费方 | 新鲜度要求 |
|--------|------|--------|-----------|
| **交易日历** | 判断当天是否交易日、是否在 trading_window（当日允许交易的时段，如 09:30-15:00）内 | OrderPlanGenerator, CLI | 当日有效即可 |
| **参考价格**（最新收盘价/最新价） | PaperGateway 撮合基准、限价单参考 | PaperGateway, OrderPlanGenerator | 执行日的最近已发布快照 |
| **涨跌停标志** | 事前风控检查、PaperGateway 撮合拒绝 | RiskEngine, PaperGateway | 执行日当日（T 日盘前发布） |
| **停牌状态** | 事前风控检查、PaperGateway 撮合拒绝 | RiskEngine, PaperGateway | 执行日当日 |
| **ST / *ST 标记** | 事前风控检查 | RiskEngine | 执行日当日 |

**快照引用规则**：

- 每次执行在启动时锁定一个 `snapshot_id`，全流程（plan → risk → submit → report）使用同一快照。
- `snapshot_id` 记录在 `execution_report.lineage` 中，保证审计可复现。
- **live 模式**：快照必须存在且为执行日的最新发布版本，否则 Fail-Closed 阻断。
- **paper 模式**：快照缺失时可配置降级（使用上一交易日快照），但 lineage 中如实记录。

Trade **不直接依赖** Research 域。Strategy 域是唯一的信号消费者。

### 2.2 Trade 域的输出工件

Trade 的输出分属 **控制面** 和 **数据面**（引用 00 §6.2：控制面操作不经过 DuckDB/Parquet）：

| 工件 | 面 | 存储位置 | 写入时机 |
|------|---|---------|---------|
| `order_intent.json` | 控制面 | `state/trade/{exec_id}/` | Gateway 提交**前**（先写 intent 再提交）。Intent（订单意图）记录了每笔计划订单的参数，用于崩溃恢复时判定订单是否已提交 |
| `audit.log` | 控制面 | `state/trade/{exec_id}/` | 全程追加（live 强制），append-only 不可删除 |
| `order_plan.json` | 数据面 | `trade/executions/{exec_id}/` | plan 生成后 |
| `pre_trade_result.json` | 数据面 | `trade/executions/{exec_id}/` | 事前风控完成后（无论通过与否） |
| `execution_report.json` | 数据面 | `trade/executions/{exec_id}/` | 执行完成后 |
| `reconcile_report.json` | 数据面 | `trade/reconciliations/{date}/` | 对账完成后 |

### 2.3 Trade 域发出的事件

Trade 域通过 **fire-and-forget**（发后即忘）方式 emit 事件，由 Notification 域异步消费（见 00 §7.2-7.3）。Trade 不关心投递结果，不等待通知完成。

| 事件 | 语义 | 通知级别 |
|------|------|---------|
| `trade.execution.completed` | 执行完成 | live=warn, paper=info |
| `trade.execution.failed` | 执行失败 | error |
| `trade.risk.blocked` | 风控阻断 | error |
| `trade.reconcile.abnormal` | 对账异常 | error |

**payload 共性字段**：所有事件必含 `event`（事件名）、`exec_id` 或 `reconcile_id`（追溯入口）、`timestamp`。各事件完整 payload 示例见 §十 10.3。

### 2.4 血统字段

每个执行工件的 `lineage` 块至少包含：

```json
{
  "lineage": {
    "exec_id": "exec_20260404_093100_d4e8",
    "target_portfolio_id": "tp_dividend_classic_20260404_a",
    "signal_snapshot_id": "signal_20260403_b2c3",
    "snapshot_id": "snap_20260403_220000_b2c3",
    "gateway_type": "paper",
    "risk_pack": "cn_execution_v1",
    "parent_exec_id": null,
    "root_exec_id": "exec_20260404_093100_d4e8",
    "profile": "dividend_classic_paper",
    "vortex_version": "0.2.0"
  }
}
```

**signal 血统锁**：`signal_snapshot_id` 继承自 `target_portfolio.lineage.signal_id`。执行开始后此值不可变——即使 Research 域在执行期间重新发布了同一信号，当前执行仍引用启动时锁定的版本。新执行必须引用新 signal_snapshot_id。

## 三、Gateway 协议与执行语义

### 3.1 Gateway 接口

```python
class Gateway(ABC):
    """统一交易通道协议。灵感来自 vn.py BaseGateway。"""
    name: str   # 网关标识："paper" / "miniqmt"

    def connect(self, config: dict) -> None:
        """建立连接。失败抛 GatewayConnectionError。"""
    def submit_order(self, order: Order) -> str:
        """提交订单（必须携带 client_order_id），返回网关侧 order_id。"""
    def cancel_order(self, order_id: str) -> bool:
        """撤单，返回是否成功。"""
    def query_positions(self) -> list[PositionSnapshot]:
        """查询当前持仓快照。"""
    def query_orders(self, filter: OrderFilter | None = None) -> list[OrderStatus]:
        """查询订单状态（含成交回报）。"""
    def on_fill(self, callback: Callable[[FillRecord], None]) -> None:
        """注册成交回调。当前实现退化为 query_orders 轮询。"""
    def disconnect(self) -> None: ...
```

```python
class RiskRule(ABC):
    """可插拔风控规则基类。"""
    name: str
    level: RiskLevel  # PRE_TRADE / INTRA_TRADE / POST_TRADE

    def evaluate(self, context: RiskContext) -> RiskResult:
        """评估规则。context 按 level 分型。"""
```

设计约束：

- 所有 Gateway 方法同步返回，成交回报通过 `query_orders()` 轮询获取。
- 连接失败必须抛异常，不允许静默降级。
- submit_order 必须携带 `client_order_id`（见 §五幂等性）。

### 3.2 PaperGateway 撮合规则

PaperGateway 在内存中模拟撮合，行情来源为 **Data 域已发布快照** 中的参考价格。

**撮合逻辑**：

| 订单类型 | 撮合条件 | 结果 |
|---------|---------|------|
| market 买单 | 标的未涨停、未停牌 | 以参考价即时全额成交 |
| market 卖单 | 标的未跌停、未停牌 | 以参考价即时全额成交 |
| limit 买单 | 参考价 ≤ 限价 且 未涨停、未停牌 | 以参考价全额成交 |
| limit 卖单 | 参考价 ≥ 限价 且 未跌停、未停牌 | 以参考价全额成交 |
| limit 不满足价格条件 | — | 挂起，trading_window 结束标记 timeout |

**涨跌停与停牌检查**：涨停禁买（market/limit 均拒绝），跌停禁卖，停牌禁止一切交易。状态读自 Data 已发布快照。

**部分成交与多订单语义**：

- **PaperGateway 不模拟部分成交**。所有成交为 full fill 或 rejected/timeout。`partial_fill` 仅由实盘路径（MiniQMTGateway）产生。
- **多笔订单按 FIFO 顺序逐笔处理**，不做原子批量撮合。每笔独立判定。
- **与 Strategy 域回测引擎的关系**：PaperGateway 的撮合规则（全量成交、涨跌停/停牌检查）应与 Strategy 域 Backtrader 的成交模拟器对齐，避免模拟盘与回测结果出现系统性偏差。两者共享同一份 Data 快照中的涨跌停/停牌数据。

> 为什么不做部分成交模拟：模拟盘核心价值是验证全链路流程，而非精确模拟市场微观结构。

### 3.3 MiniQMTGateway 接入

MiniQMT 是国内券商提供的量化交易客户端，系统通过其 Python SDK（xtquant）与之通信：

- `connect()` → `xtquant.connect(endpoint)`，客户端必须已启动且登录。
- `submit_order()` → 转为 xtquant 委托参数提交，携带 `client_order_id`。
- `query_positions()` / `query_orders()` → 查询并转换为内部类型。
- `cancel_order()` → 通过 xtquant 撤单接口。

边界条件：不做自动重连，断开直接抛 GatewayConnectionError。

### 3.4 Gateway 可替换

新增网关只需：实现 Gateway 接口全部方法 → 注册网关名到类的映射 → 在 TradeProfile preset 中增加配置项。

## 四、订单生命周期与状态机

### 4.1 OrderPlan 生成

```
target portfolio + current positions → 计算持仓差异 → 过滤零差异 → 应用 order_policy → OrderPlan
```

- 持仓差异：`target_weight - current_weight` → 需买入/卖出的标的与数量。
- order_policy：决定市价/限价、是否拆单（默认不拆单）。
- OrderPlan 包含 orders 列表，每笔 order 含 symbol、side、order_type、price、quantity、reason。
- 生成时锁定 `snapshot_id`，全执行流程使用同一快照。

### 4.2 订单状态机

```
                          submit_order()
    ┌──────────┐      ─────────────────>     ┌───────────┐
    │ pending  │                             │ submitted │
    └──────────┘                             └─────┬─────┘
                                                   │
                           ┌───────────────────────┼────────────────────┐
                           │                       │                    │
                           ▼                       ▼                    ▼
                    ┌──────────────┐      ┌──────────────┐    ┌────────────┐
                    │ partial_fill │      │   rejected   │    │  timeout   │
                    └──────┬───┬──┘      └──────────────┘    └────────────┘
                           │   │                 ▲
                           │   └─────────────────┤
                           ▼                     │
                    ┌──────────────┐             │
                    │    filled    │     cancelled（用户主动 / 事中风控触发）
                    └──────────────┘
```

### 4.3 状态定义与转换规则

| 状态 | 含义 | 允许的后续状态 |
|------|------|--------------|
| `pending` | 已生成，等待提交 | submitted, cancelled |
| `submitted` | 已提交至网关，等待回报 | partial_fill, filled, rejected, timeout, cancelled |
| `partial_fill` | 部分成交 | filled, cancelled, timeout |
| `filled` | 全部成交（**终态**） | — |
| `rejected` | 被网关或交易所拒绝（**终态**） | — |
| `timeout` | trading_window 内未全部成交（**终态**） | — |
| `cancelled` | 被撤单（**终态**） | — |

**关键规则**：

1. **终态不可逆**：filled / rejected / timeout / cancelled 一旦设定，不允许回退。
2. **partial_fill → cancelled**：事中风控触发撤单时，已成交部分保留在 fills 记录中，剩余部分标记 cancelled。`cancelled` 状态的订单若 `filled_qty > 0`，表示"部分成交后撤余"。
3. **submit 超时 ≠ 订单 timeout**：网络超时意味着"不确定券商是否收到"。此时订单标记 `ack_pending = true`，恢复时必须先通过 reconcile 对齐（见 §五）。
4. **intent 先于状态机**：每次状态转换**前**必须先持久化 OrderIntent（见 §5.2），保证崩溃后可重建状态。

> **与 00 §4.3 的关系**：00 中 partial_fill "只能前进到 filled 或 timeout"是简化表述。实盘场景中事中风控撤单需要 partial_fill → cancelled 路径，本节是对 00 的细化补充。

### 4.4 ID 生成规则

| ID | 格式 | 说明 |
|----|------|------|
| `plan_id` | `plan_{YYYYMMDD}_{HHMMSS}_{short_uuid}` | 唯一标识一次 plan 生成 |
| `exec_id` | `exec_{YYYYMMDD}_{HHMMSS}_{short_uuid}` | 唯一标识一次执行 |
| `client_order_id` | `{exec_id}_{seq}` | 确定性生成，用于幂等去重 |
| `reconcile_id` | `recon_{YYYYMMDD}_{short_uuid}` | 唯一标识一次对账 |

`short_uuid` 取 UUID4 前 6 位十六进制，在日级别粒度下碰撞概率可忽略。

## 五、幂等性、恢复与订单安全

本节落地 00 §4.3 的架构级不变量。

### 5.1 确定性订单标识与去重协议

每笔订单的 `client_order_id = {exec_id}_{seq}`，其中 `seq` 是 OrderPlan 中的订单序号（从 0 开始，按 symbol 字典序确定）。

**去重范围**：`client_order_id` 的去重是**跨重连持久化**的，不是 per-session 内存级。PaperGateway 将 `client_order_id → order` 映射持久化到 `order_intent.json`；MiniQMTGateway 依赖 xtquant 报单引用字段 + 本地 intent 文件双重保障。

**Gateway 去重行为**：Gateway 收到已存在的 `client_order_id` 时，**幂等接受**——返回已有订单的当前状态，不创建新订单、不拒绝请求。这是恢复重放的基础。

**冲突检测**：去重不是"按 ID 盲合并"。每个 `client_order_id` 在 intent 持久化时记录 canonical hash（规范哈希值，即 `sha256(symbol + side + qty + price)`），作为订单内容的指纹。重复提交时逐字段比对：

- **完全一致** → 幂等重放，返回已有订单状态。
- **字段不一致** → **fatal conflict**，直接阻断并报错，绝不静默合并。这防止恢复逻辑 bug 或调用方错误把不同订单绑到同一 ID。

### 5.2 OrderIntent 持久化

**核心原则：先写 intent，再提交 gateway。Intent 持久化发生在状态机转换之前。**

```
1. 生成 OrderPlan → 计算每笔 client_order_id + canonical_hash
2. 写入 order_intent.json（每笔：client_order_id、参数、canonical_hash、状态="intent"）
3. 逐笔提交 Gateway.submit_order()
4. 每笔提交成功后，先更新 intent 状态为 "submitted"，再推进状态机
5. 收到成交回报后，先更新 intent 为终态，再推进状态机
```

如果进程在步骤 2 之后、步骤 3 之前崩溃 → intent 已持久化，订单未提交 → 恢复时可安全重提交。
如果在步骤 3 过程中崩溃 → 部分订单可能已提交 → 恢复时需先 reconcile。

### 5.3 恢复与重连

当进程崩溃或 Gateway 断连后恢复时，执行以下流程：

```
1. 读取最近的 order_intent.json
2. 连接 Gateway
3. 执行 reconcile：对齐网关持仓与 intent 记录
   - intent="submitted" 但网关无对应 client_order_id → 标记为"提交失败"，可安全重提交
   - intent="submitted" 且网关有对应订单 → 同步真实状态（含冲突检测）
   - intent="intent"（未提交） → 可安全重提交（沿用 client_order_id，Gateway 幂等接受）
4. 根据对齐结果决定：
   - 所有订单已完成 → 生成 execution_report，结束
   - 存在未完成订单 → 提示用户：继续执行 / 撤单放弃 / 人工介入
```

**非交互环境**：恢复时如果无法获得用户确认，默认 Fail-Closed——不继续提交，生成部分执行报告，触发通知。

### 5.4 恢复重放 vs 业务重试

两种场景使用不同的 ID 策略：

| 场景 | client_order_id | exec_id | 语义 |
|------|----------------|---------|------|
| **恢复重放**（crash replay） | 沿用原 client_order_id | 沿用原 exec_id | Gateway 幂等接受，不重复下单 |
| **业务重试**（`vortex trade retry`） | 新 exec_id + 新 seq | 新 exec_id | 全新的执行，关联 parent_exec_id |

> **与 00 §4.3 的关系**：00 中"重试时使用相同 client_order_id 防止重复提交"指的是**恢复重放**场景（crash replay）。用户主动发起的 `vortex trade retry` 是业务重试，生成新 ID。本节是对 00 的细化。

## 六、三层风控

### 6.1 总览

| 层级 | 触发时机 | 输入 | 阻断行为 |
|------|---------|------|---------|
| 事前（Pre-Trade） | order plan 生成后、提交前 | OrderPlan + Data 快照 | 不通过 → 不提交任何订单，整批阻断 |
| 事中（Intra-Trade） | 每笔成交回报到达时 | FillRecord + Tracker 状态 | 触发 → 撤回未成交订单 / 暂停执行 |
| 事后（Post-Trade） | 日终对账完成后 | ReconcileReport | 偏差超限 → 标记异常、通知、审计 |

### 6.2 事前规则

| 规则 | 检查逻辑 | 阻断条件 |
|------|---------|---------|
| `single_stock_concentration` | 单票目标权重 / 总组合 | 超过 `max_weight`（默认 10%） |
| `limit_up_down_check` | 查标的当日涨跌停标志（Data 快照） | 涨停禁买，跌停禁卖 |
| `st_check` | 查标的 ST / *ST 标记 | ST 标的禁止买入 |
| `suspension_check` | 查标的停牌状态 | 停牌标的不可交易 |

### 6.3 事中规则

| 规则 | 检查逻辑 | 阻断条件 |
|------|---------|---------|
| `slippage_limit` | 成交价与委托价偏差（bps） | 超过 `max_slippage_bps` → 撤回剩余 |
| `abnormal_fill_check` | 成交量 / 委托量异常 | 异常 → 暂停后续提交 |

### 6.4 事后规则

| 规则 | 检查逻辑 | 阻断条件 |
|------|---------|---------|
| `position_deviation` | 实际持仓 vs 目标偏差 | 超过 `max_deviation_pct` |
| `fill_price_deviation` | 成交均价 vs 委托价偏差 | 超过 `max_price_deviation_bps` |

### 6.5 risk_pack 配置

规则按 risk_pack 命名集合加载，定义在 YAML 中：

```yaml
# risk/packs/cn_execution_v1.yaml
name: cn_execution_v1
pre_trade:
  - rule: single_stock_concentration
    params: { max_weight: 0.10 }
  - rule: limit_up_down_check
  - rule: st_check
  - rule: suspension_check
intra_trade:
  - rule: slippage_limit
    params: { max_slippage_bps: 50 }
post_trade:
  - rule: position_deviation
    params: { max_deviation_pct: 2.0 }
```

### 6.6 Fail-Closed 原则

1. **规则执行异常视为不通过**：规则抛异常或超时 → 等同于阻断。
2. **规则缺失视为不通过**：risk_pack 引用未注册的规则名 → 初始化阶段报错。
3. **事前风控是原子判定**：任意一条规则不通过 → 整个 OrderPlan 不提交。
4. **风控结果必须持久化**：`pre_trade_result.json` 无论通过与否都写入，供审计。

## 七、对账引擎

### 7.1 三组偏差比较

对账需要三个数据源交叉验证：

| 比较组 | 含义 | 可能原因 |
|--------|------|---------|
| **actual（网关持仓）vs target（目标组合）** | 组合偏离程度 | 部分订单未成交、市场价格变动 |
| **target vs fills（成交记录）** | 执行达成度 | 部分订单被拒、超时未成交 |
| **fills vs actual（网关持仓）** | 网关一致性 | 网关同步延迟、外部干预、场外交易 |

每组逐标的计算偏差：`deviation = abs(source_weight - compare_weight)`。

### 7.2 偏差阈值与行动映射

对账偏差不只是"报告"，还驱动后续行动：

| 偏差幅度（单标的） | 状态 | 行动 |
|-------------------|------|------|
| < 1% | `within_threshold` | 仅写日志记录 |
| 1% – 3% | `minor_deviation` | 通知（warn 级别），记录但不强制干预 |
| > 3% | `abnormal` | 强制通知（error 级别）+ 审计 + 建议人工介入 |

```yaml
reconcile_policy:
  thresholds:
    log_threshold_pct: 1.0           # 低于此值仅日志
    notify_threshold_pct: 3.0        # 低于此值通知，高于此值 escalate
    max_total_deviation_pct: 5.0     # 组合总偏差上限
    max_fill_price_deviation_bps: 30 # 成交价格偏差上限
```

**对账与重试的关系**：reconcile_report 本身**不自动触发** retry。它是诊断工具——如果偏差由订单失败导致（rejected / timeout），用户可手动执行 `retry` 命令。不做对账→自动补单的链路，原因见 §12.3。

### 7.3 对账报告结构

```json
{
  "reconcile_id": "recon_20260404_g5h1j6",
  "exec_id": "exec_20260404_093100_d4e8",
  "date": "2026-04-04",
  "snapshot_id": "snap_20260403_220000_b2c3",
  "deviations": {
    "actual_vs_target": { "600519.SH": { "actual": 0.098, "target": 0.10, "deviation": 0.002 } },
    "target_vs_fills": { "600519.SH": { "target": 0.10, "filled": 0.098, "deviation": 0.002 } },
    "fills_vs_actual": { "600519.SH": { "filled": 0.098, "actual": 0.098, "deviation": 0.0 } }
  },
  "status": "within_threshold",
  "thresholds": { "log_threshold_pct": 1.0, "notify_threshold_pct": 3.0 }
}
```

## 八、重试机制

### 8.1 重试触发

`vortex trade retry --run-id <exec_id>` 用于对失败订单发起新执行。

### 8.2 重试流程

```
1. 加载 execution_report.json（指定 exec_id）
2. 筛选可重试订单：状态为 rejected / timeout 的订单
3. 用这些订单构建新 OrderPlan
4. 重新执行事前风控（市场状态可能已变化，如标的已涨停）
5. 通过后提交，生成新 exec_id
6. 新执行 lineage 记录 parent_exec_id 和 root_exec_id
```

### 8.3 追溯链

`parent_exec_id` 和 `root_exec_id` 是 **lineage 内部字段**（不是顶层独立字段），与其他血统信息统一管理：

```json
{
  "lineage": {
    "exec_id": "exec_20260404_100500_f2g3",
    "parent_exec_id": "exec_20260404_093100_d4e8",
    "root_exec_id": "exec_20260404_093100_d4e8",
    "snapshot_id": "snap_20260403_220000_b2c3"
  }
}
```

- **parent_exec_id**：直接父执行。`inspect` 时展示完整 retry 链。
- **root_exec_id**：追溯链根节点。多次 retry 后通过 root_exec_id 查询完整链。
- **reconcile 关联**：reconcile_report 通过 `exec_id` 字段关联到对应的执行。retry 后的新执行产生的 reconcile_report 可通过 parent chain 追溯到首次执行的 reconcile。

### 8.4 重试约束

- 每次重试必须重新过事前风控，不可跳过。
- retry 是**业务重试**（新 exec_id、新 client_order_id），不是恢复重放。
- 不做自动 retry。自动化场景可通过调度系统触发，但仍重新过风控。

## 九、实盘 vs 模拟盘差异

### 9.1 差异对照表

| 维度 | paper | live |
|------|-------|------|
| Gateway | PaperGateway（内存撮合） | MiniQMTGateway（xtquant） |
| 事前风控 | risk_pack 配置决定 | **必须配置** risk_pack，不可为空 |
| 执行确认 | 不需要 | 默认 Y/n 交互确认（见 §9.2） |
| 审计日志 | 可选（`--audit` 启用） | **强制写入** audit.log |
| 通知 | 可选 | **强制发送** |
| Data 快照 | 快照缺失可配置降级 | **必须存在**，否则 Fail-Closed |

### 9.2 实盘确认机制与审计链

`live rebalance` 在提交前展示执行摘要并要求确认：

```
┌─────────────────────────────────────────────────────┐
│  实盘调仓确认                                         │
│  Profile: dividend_classic_live                      │
│  Account: XXXXXXXX (miniQMT)                         │
│  买入: 5 笔, 约 ¥120,000                             │
│  卖出: 3 笔, 约 ¥85,000                              │
│  Risk Pack: cn_execution_v1 (4 规则全部通过)           │
│                                                      │
│  确认执行？ [Y/n]                                     │
└─────────────────────────────────────────────────────┘
```

**确认规则**：

- 交互终端（TTY）：默认展示摘要，等待用户输入 Y/n。
- `--confirm` 标志：跳过交互确认，适用于自动化调度。
- **非 TTY 环境且未指定 --confirm**：Fail-Closed，拒绝执行并报错。

**确认审计链**：确认动作本身必须审计，确保事后可追溯"谁在什么时候确认了什么"。

`execution_report.json` 包含 `confirmation` 字段：

```json
{
  "confirmation": {
    "method": "interactive",
    "confirmed_by": "user@tty",
    "confirmed_at": "2026-04-04T09:30:15+08:00",
    "confirm_flag": false,
    "plan_hash": "sha256:abc123..."
  }
}
```

`audit.log`（控制面，append-only）同步记录：

```
2026-04-04T09:30:15+08:00 CONFIRM exec_id=exec_20260404_093100_d4e8 method=interactive user=user@tty plan_hash=sha256:abc123...
```

两者通过 `exec_id` + `plan_hash` 交叉关联，可验证一致性。

## 十、事件与通知集成

### 10.1 Trade 域的通知职责

Trade 域只负责 **fire-and-forget emit 事件**。不负责订阅管理、路由策略、渠道适配（那是 Notification 域的职责，见 00 §7.3）。

Trade emit 事件后不等待投递结果、不关心投递是否成功。事件通过进程内事件总线发出，Notification 域异步消费。

### 10.2 事件触发规则

| 场景 | emit 事件 | 通知级别 | 触发条件 |
|------|----------|---------|---------|
| 实盘调仓完成 | `trade.execution.completed` | warn | live execution 完成（payload `mode: live`） |
| 模拟盘调仓完成 | `trade.execution.completed` | info | paper execution 完成（payload `mode: paper`） |
| 执行失败 | `trade.execution.failed` | error | 不可恢复的执行错误 |
| 事前风控阻断 | `trade.risk.blocked` | error | 事前风控不通过 |
| 事中风控撤单 | `trade.risk.blocked` | error | 事中风控触发撤单 |
| 对账偏差超限 | `trade.reconcile.abnormal` | error | 任一组偏差超 notify 阈值 |

同一事件名 `trade.execution.completed` 通过 payload `mode` 字段区分 paper/live。Notification 路由引擎根据 `mode` 决定通知级别（live=warn, paper=info）。

### 10.3 各事件 payload 示例

**trade.execution.completed**：

```json
{ "event": "trade.execution.completed", "exec_id": "exec_20260404_093100_d4e8",
  "mode": "live", "profile": "dividend_classic_live",
  "summary": { "total": 12, "filled": 10, "rejected": 1, "timeout": 1 },
  "needs_retry": true, "timestamp": "2026-04-04T09:35:00+08:00" }
```

**trade.execution.failed**：

```json
{ "event": "trade.execution.failed", "exec_id": "exec_20260404_093100_d4e8",
  "error_type": "GatewayConnectionError", "message": "miniQMT 客户端连接超时",
  "timestamp": "2026-04-04T09:30:05+08:00" }
```

**trade.risk.blocked**：

```json
{ "event": "trade.risk.blocked", "exec_id": "exec_20260404_091501_a1b2",
  "blocked_rules": ["single_stock_concentration"],
  "affected_orders": [{"symbol": "000858.SZ", "target_weight": 0.072, "limit": 0.05}],
  "timestamp": "2026-04-04T09:15:02+08:00" }
```

**trade.reconcile.abnormal**：

```json
{ "event": "trade.reconcile.abnormal", "reconcile_id": "recon_20260404_g5h1j6",
  "max_deviation_pct": 3.5, "abnormal_symbols": ["000858.SZ", "601318.SH"],
  "timestamp": "2026-04-04T15:10:00+08:00" }
```

### 10.4 通知级别语义（引用 00 §7.2）

- **info**：默认静默，用户可配置接收。
- **warn**：默认推送，用户可关闭。
- **error**：强制推送，不可关闭。

## 十一、域内序列图

以下序列图标注了同步/异步边界。Trade 内部 plan→risk→submit 为**同步链路**；成交回报归集为**同步轮询**；跨域通知为 **fire-and-forget 异步**。

### 11.1 Paper 调仓完整流程

```
用户: vortex trade paper rebalance --profile dividend_paper --target tp_latest

  CLI                                                              [同步链路]
   │
   ├──→ TradeProfileResolver.resolve()          → ResolvedTradeProfile
   │
   ├──→ Gateway.connect()                        → 成功 / 失败阻断
   │
   ├──→ 锁定 Data 快照引用                         → snapshot_id（全流程不变）
   │
   ├──→ OrderPlanGenerator.generate()
   │      ├── 读 target_portfolio
   │      ├── Gateway.query_positions()           → 当前持仓
   │      └── 计算差异 → OrderPlan
   │
   ├──→ RiskEngine.check_pre_trade(plan)
   │      ├── 逐条规则检查（读 Data 快照中的涨跌停/停牌/ST）
   │      ├── 全部通过 → 继续
   │      └── 任一不通过 → 阻断，写 pre_trade_result
   │                        emit risk.blocked ─────────→ [异步] Notification
   │
   ├──→ 写入 order_intent.json（控制面）            → 意图持久化（先于提交）
   │
   ├──→ 逐笔 Gateway.submit_order()                                [同步]
   │      ├── PaperGateway: 内存撮合 FIFO（涨跌停/停牌→成交/拒绝）
   │      └── 每笔完成后先更新 intent 再推进状态机
   │
   ├──→ ExecutionTracker.collect_fills()                            [同步轮询]
   │      ├── 归集成交回报
   │      └── RiskEngine.check_intra_trade() 每笔
   │
   ├──→ 生成 execution_report.json（数据面）
   │
   └──→ emit trade.execution.completed ────────→ [异步] Notification
```

### 11.2 Live 调仓含确认流程

```
用户: vortex trade live rebalance --profile dividend_live --target tp_latest

  CLI                                                              [同步链路]
   │
   ├──→ TradeProfileResolver.resolve()
   │      └── 校验 risk_pack 非空、gateway = miniqmt
   │
   ├──→ Gateway.connect()                        → miniQMT 连接
   │
   ├──→ 锁定 Data 快照引用
   │      └── 校验快照存在（Fail-Closed：快照缺失 → 阻断）
   │
   ├──→ OrderPlanGenerator.generate()             → OrderPlan
   │
   ├──→ RiskEngine.check_pre_trade(plan)          → 通过 / 阻断
   │
   ├──→ ★ 实盘确认 ★                                               [同步阻塞]
   │      ├── TTY: 展示摘要，等待 Y/n
   │      ├── --confirm: 自动通过
   │      └── 非 TTY 且无 --confirm: Fail-Closed 拒绝
   │      └── 审计：写 audit.log CONFIRM 记录 + execution_report.confirmation
   │
   ├──→ 写入 order_intent.json（控制面）
   │
   ├──→ 逐笔 Gateway.submit_order()                                [同步]
   │      └── 每笔携带 client_order_id
   │
   ├──→ 轮询 Gateway.query_orders()                                [同步轮询]
   │      └── RiskEngine.check_intra_trade() 每笔
   │
   ├──→ 生成 execution_report.json + audit.log（强制）
   │
   └──→ emit trade.execution.completed ────────→ [异步] Notification
```

### 11.3 重试流程

```
用户: vortex trade retry --run-id exec_20260404_093100_d4e8

  CLI                                                              [同步链路]
   │
   ├──→ 加载 execution_report.json (exec_id=指定值)
   │
   ├──→ 筛选 needs_retry 订单（rejected / timeout）
   │      └── 无可重试订单 → 提示并退出

   ├──→ 构建新 OrderPlan（仅含失败订单）
   │
   ├──→ 重新执行事前风控（市场状态可能已变）

   └──→ emit trade.execution.completed ────────→ [异步] Notification
```

## 十二、域故障模式

### 12.1 故障目录

| # | 故障场景 | 处理方式 | 风险 |
|---|---------|---------|------|
| F1 | Gateway 连接失败 | 抛异常，阻断执行 | 无 |
| F2 | submit 网络超时 | 标记 ack_pending；恢复时 reconcile 对齐 | 中 |
| F3 | 部分成交后进程崩溃 | 读 order_intent + reconcile 对齐 | 高 |
| F4 | Gateway 执行中断连 | 停止后续提交；恢复后 reconcile | 高 |
| F5 | 券商拒单 | 标记 rejected + needs_retry | 低 |
| F6 | 事前风控不通过 | 整批阻断，零提交 | 无 |
| F7 | 事中风控触发 | 撤回未成交，暂停执行 | 中 |
| F8 | 对账偏差超限 | abnormal + 通知 + 审计 | 低 |
| F9 | Data 快照不可用 | live: Fail-Closed；paper: 可降级 | 无 |
| F10 | target portfolio 不存在 | 启动报错，不进入执行 | 无 |

### 12.2 F2/F3 详细恢复流程

**F2 submit 网络超时**：标记 `ack_pending = true`（先写 intent） → 默认暂停后续提交 → 恢复时 reconcile 查 gateway：有对应 `client_order_id` 则同步状态（幂等去重保证），无则标记 rejected。

**F3 部分成交后崩溃**：读 `order_intent.json` → 连接 Gateway → reconcile 逐笔比对 intent 与实际状态 → 已成交同步为 filled/partial_fill → 未提交（intent="intent"）可安全重提交（沿用 client_order_id，幂等接受） → 已提交但状态不明按 F2 处理 → 提示用户决策：继续 / 撤单 / 人工介入。

### 12.3 不做自动补单

不做自动补偿。自动补偿在异常市场条件下可能放大风险。失败后由人工判断是否 retry 更安全。自动化场景可通过调度触发 retry，但每次 retry 都重新过风控。

## 十三、工件目录结构

控制面与数据面分开存放（引用 00 §6.2）：

```
{vortex_home}/
  state/                              # 控制面
    trade/
      {exec_id}/
        order_intent.json             # 幂等恢复用，先于 Gateway 提交写入
        audit.log                     # append-only，live 强制，不可删除

  trade/                              # 数据面
    executions/
      {exec_id}/
        order_plan.json               # 调仓计划
        pre_trade_result.json         # 事前风控结果
        execution_report.json         # 执行报告（含 confirmation、lineage）
    reconciliations/
      {date}/
        reconcile_report.json         # 对账报告（含三组偏差）
```
