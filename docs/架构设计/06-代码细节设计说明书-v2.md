## 1. 跨域统一约束

### 1.1 确定性契约（Determinism Contract）

**目标**：相同输入 + 相同配置 + 相同代码版本 → bit-identical 输出。

**排序规则**（全系统统一）：

| 场景 | 主排序 | 次排序 | 说明 |
|------|--------|--------|------|
| 行情数据 | date ASC | symbol ASC | 所有域读数据的默认排序 |
| 因子排名 | score DESC | symbol ASC | 分组时 score 相同有确定性顺序 |
| 组合权重 | weight DESC | symbol ASC | Top-N 截断确定性 |
| IC 排名 | ic DESC | factor_name ASC | 因子比较确定性排序 |

**reproducibility_key 计算**：

```python
def compute_reproducibility_key(resolved_profile: ResolvedProfile) -> str:
    """基于冻结配置 + 代码版本生成可复现键。"""
    payload = {
        "profile_hash": hash_dict(resolved_profile.to_public_dict()),
        "vortex_version": vortex.__version__,
        "python_version": sys.version,
    }
    return hashlib.sha256(json.dumps(payload, sort_keys=True).encode()).hexdigest()[:16]
```

### 1.2 舍入规则（Rounding Convention）

全系统统一精度标准：

| 数据类型 | 精度 | 舍入方式 | 依据 |
|---------|------|---------|------|
| 股价 | 2 位小数 | 四舍五入 | A 股最小变动单位 0.01 元 |
| 权重 | 4 位小数 | 四舍五入 | 1.0000 = 100% |
| IC / RankIC | 6 位小数 | 四舍五入 | 统计量需高精度 |
| Sharpe | 4 位小数 | 四舍五入 | 行业惯例 |
| 收益率 | 6 位小数 | 四舍五入 | 累计时精度要求高 |
| 手数 | 整数 | **向下取整** | A 股 100 股/手，`floor(shares/100)*100` |
| 佣金 | 2 位小数 | **向上取整** | ceiling 到分，保护模拟精度 |
| 滑点 (bps) | 1 位小数 | 四舍五入 | `(fill-limit)/limit × 10000` |
| 百分位排名 | 2 位小数 | 四舍五入 | 0.00–100.00 |

**关键不变量**：
- 组合权重和：`|sum(weights) - 1.0| < 1e-4`
- 所有权重非负：`all(weights >= 0)`

### 1.3 时间与时区

#### 1.3.1 系统唯一时区

```
系统时区 = Asia/Shanghai (UTC+8)
所有 datetime 字段存储带时区 ISO 8601 格式（如 2026-04-04T22:30:00+08:00）
禁止依赖数据库 CURRENT_TIMESTAMP（可能不带时区）
写入时由应用层统一生成 UTC+8 带 offset 的时间字符串

多市场兼容：
  时区和交易时段下沉到 market/calendar 配置
  核心时间逻辑（as_of 判定、PIT 对齐）接受 timezone 参数，不硬编码 UTC+8
  timezone 参数默认值为 Asia/Shanghai
```

#### 1.3.2 Snapshot as_of 判定

```
规则：当日 22:00 后生成的 snapshot 归属当日 as_of
      22:00 前生成的归属前一个交易日

示例：
  2026-04-04 22:30 生成 → as_of = 2026-04-04
  2026-04-04 21:50 生成 → as_of = 2026-04-03（前一个交易日）
```

#### 1.3.3 交易日历

- 由 `DataProvider.fetch_calendar()` 提供，缓存在 `trading_calendar` 表
- 非交易日：跳过，不填充
- 半日市（如春节前一天）：正常处理，仅数据量少
- 分钟频数据边界：09:30–11:30, 13:00–15:00，非交易时段数据丢弃

#### 1.3.4 PIT（Point-In-Time）判定规则

```
规则 1：财报数据以 ann_date（公告日期）为准，NOT report_date
规则 2：effective_from = ann_date 的下一个交易日 09:30
规则 3：若 ann_date 在交易时段内（09:30-15:00），当日 15:00 后生效
规则 4：快报、预告、正式报各自独立 PIT，互不覆盖
规则 5：ann_date 为非交易日 → effective_from = 下一个交易日 09:30
```

### 1.4 并发与隔离

```
域间隔离：各域独立运行、独立 run_id、独立目录
域内并发：
  - Data：按 dataset 分组并行拉取，单 dataset 内串行
  - Research：因子评测可按 factor_name 并行，队列管理
  - Strategy：回测任务可并行（不同 backtest_run_id）
  - Trade：同一 exec_id 串行，不同 profile 可并行

文件锁：控制面 SQLite（state/）使用写锁，数据面 Parquet 不加锁（只追加/覆盖）
```

### 1.5 错误码枚举

全系统使用结构化错误码，格式 `{DOMAIN}_{CATEGORY}_{DETAIL}`：

| 错误码 | 域 | 含义 | 默认动作 |
|--------|------|------|---------|
| `DATA_PROVIDER_FETCH_FAILED` | Data | Provider 请求失败 | 重试 3 次后 abort |
| `DATA_PROVIDER_AUTH_FAILED` | Data | Provider 认证失败 | 立即 abort |
| `DATA_QUALITY_CHECK_FAILED` | Data | 质量门禁未通过 | 阻断发布 |
| `DATA_SNAPSHOT_CONFLICT` | Data | 快照 ID 冲突 | 覆盖（data 允许） |
| `DATA_CALENDAR_MISSING` | Data | 交易日历缺失 | abort |
| `RESEARCH_SNAPSHOT_NOT_FOUND` | Research | 依赖的数据快照不存在 | abort |
| `RESEARCH_FACTOR_NOT_REGISTERED` | Research | 因子未注册 | abort |
| `RESEARCH_EVAL_DIVERGED` | Research | 评测结果不可复现 | warning |
| `RESEARCH_SIGNAL_IMMUTABLE` | Research | 尝试覆盖已发布信号 | abort |
| `STRATEGY_SIGNAL_NOT_FOUND` | Strategy | 依赖的信号快照不存在 | abort |
| `STRATEGY_BACKTEST_ENGINE_ERROR` | Strategy | 回测引擎内部错误 | abort |
| `STRATEGY_QUALITY_SCORE_LOW` | Strategy | 评分低于阈值 | warning |
| `TRADE_RISK_BLOCKED` | Trade | 风控规则阻断 | abort（fail-closed） |
| `TRADE_GATEWAY_TIMEOUT` | Trade | Gateway 超时 | 标记 ack_pending |
| `TRADE_GATEWAY_CONNECTION_FAILED` | Trade | Gateway 连接失败 | abort |
| `TRADE_ORDER_REJECTED` | Trade | 券商拒单 | 记录 + 通知 |
| `TRADE_IDEMPOTENCY_CONFLICT` | Trade | client_order_id 冲突 | 查询已有订单 |
| `CONFIG_VALIDATION_ERROR` | Config | Profile 校验失败 | abort |
| `CONFIG_RESOLUTION_ERROR` | Config | Profile 解析失败 | abort |
| `CONFIG_SECRET_MISSING` | Config | 密钥缺失 | abort |

### 1.6 审计字段

所有 run manifest 和控制面记录必须包含以下审计字段：

```python
@dataclass
class AuditFields:
    """所有运行记录的审计元数据。"""
    run_id: str               # 格式: {domain}_{date}_{time}_{hash6}
    profile: str              # profile 名称
    vortex_version: str       # 如 "0.2.0"
    started_at: str           # ISO 8601 带时区
    finished_at: str | None   # ISO 8601 带时区，运行中为 None
    status: str               # pending / running / success / failed / aborted
    reproducibility_key: str  # §1.1 定义
    trigger: str              # "manual" | "scheduled" | "retry"
    parent_run_id: str | None # 重试链追溯
```

**audit.log 格式**（Trade 域强制、其他域可选）：

```
每行一条 JSON，append-only，不可删除。
{"ts": "2026-04-04T09:31:00+08:00", "event": "order_submitted", "detail": {...}}
```

### 1.7 Schema 版本管理

```
每类工件（manifest、snapshot descriptor、report）都包含 schema_version 字段。
当前统一版本：schema_version = "1"
未来升级时增加迁移脚本，启动时自动检测并迁移。
```

### 1.8 确定性验收双层标准

| 层 | 说明 | 验收标准 |
|----|------|---------|
| **严格层** | 因子值、IC、组合权重 | bit-identical（同输入同输出） |
| **宽松层** | Sharpe、最大回撤、总收益 | 误差 < 1e-6（浮点累计误差容忍） |

验收方法：Golden Master 对比——首次运行产出基准文件，后续运行与之 diff。

### 2.3 合并逻辑

```python
def resolve_profile(user_yaml_path: str, cli_overrides: dict | None = None) -> ResolvedProfile:
    """Profile 解析主流程。"""
    # 1. 加载框架默认值
    defaults = load_defaults(profile_type)       # 从 defaults/{type}.yaml

    # 2. 加载用户 Profile（只含差异字段）
    user_config = load_yaml(user_yaml_path)

    # 3. 若 extends 存在，递归加载父 Profile
    if user_config.get("extends"):
        parent = resolve_parent(user_config["extends"])
        defaults = deep_merge(defaults, parent)

    # 4. 深度合并：defaults + user_config
    merged = deep_merge(defaults, user_config)

    # 5. 应用 CLI 覆盖（最高优先级）
    if cli_overrides:
        merged = deep_merge(merged, cli_overrides)

    # 6. 校验
    validate(merged)

    # 7. 运行时解析（路径、密钥、快照别名）
    resolved = resolve_runtime(merged)

    # 8. 输出 .resolved.yaml
    dump_resolved(resolved, path=f".vortex/{profile_name}.resolved.yaml")

    return resolved
```

**合并规则**：
- **嵌套 dict**：递归深度合并
- **list**：整体替换（不做元素级合并）
- **null / ~**：显式清除（从合并结果中移除该键）
- **标量**：用户值覆盖默认值

### 4.3 评测引擎 10 维指标计算口径

| 维度 | 指标 | 计算方法 | 精度 |
|------|------|---------|------|
| **IC** | Information Coefficient | `pearsonr(factor_rank, forward_return)` 每期截面 | 6 位 |
| **RankIC** | Rank IC | `spearmanr(factor_rank, forward_return_rank)` 每期截面 | 6 位 |
| **ICIR** | IC Information Ratio | `mean(IC_series) / std(IC_series)` | 4 位 |
| **IR** | Information Ratio | `mean(excess_return) / std(excess_return)` 年化 | 4 位 |
| **多空收益** | Long-Short Return | `top_group_return - bottom_group_return` 年化 | 6 位 |
| **换手率** | Turnover | `mean(|position_change|)` 每期 | 4 位 |
| **IC 衰减** | IC Decay | IC_1d, IC_5d, IC_20d 序列，衰减速率 | 6 位 |
| **分组收益** | Group Return | 按因子值等分 N 组，各组平均收益，检查单调性 | 6 位 |
| **最大回撤** | Max Drawdown | 多头组合的历史最大回撤 | 4 位 |
| **稳定性** | Stability | `R² of IC_cumsum regression on time` | 4 位 |

### 5.4 多维度回测参数

| 维度 | 目的 | 关键参数 | 结果 |
|------|------|---------|------|
| **全样本** | 基础绝对表现 | 完整区间 | 必须运行 |
| **随机日期** | 起始日依赖检验 | `n_trials=50`, `min_window=252d` | 分布统计 |
| **Walk-Forward** | 过拟合检验 | `train=756d`, `test=252d`, `step=63d` | 多段拼接 |
| **Regime** | 不同市况下表现 | 市场状态指标 + 分段 | 各市况独立评测 |

### 5.5 质量评分公式

**五维评分**：

| 维度 | 权重 | 指标来源 | 归一化方法 |
|------|------|---------|-----------|
| 收益风险 | 25% | 全样本 Sharpe, 年化收益, 年化超额 | 百分位映射 |
| 稳定性 | 25% | 随机日期 Sharpe 分布的 std / mean | 反向百分位 |
| 稳健性 | 20% | Walk-Forward OOS Sharpe / IS Sharpe | 比值越接近 1 越好 |
| 成本 | 15% | 换手率、滑点影响 | 反向百分位 |
| 风险 | 15% | 最大回撤、尾部风险（CVaR） | 反向百分位 |

```python
def compute_quality_score(metrics: dict) -> tuple[float, str]:
    """计算综合评分与字母评级。"""
    score = (
        0.25 * normalize(metrics["return_risk"]) +
        0.25 * normalize(metrics["stability"]) +
        0.20 * normalize(metrics["robustness"]) +
        0.15 * normalize(metrics["cost"]) +
        0.15 * normalize(metrics["risk"])
    )
    grade = score_to_grade(score)  # A/B+/B/B-/C/D/F
    return score, grade
```

### 6.5 三层风控

```python
class RiskRule(ABC):
    """可插拔风控规则。"""
    name: str
    level: RiskLevel  # PRE_TRADE | INTRA_TRADE | POST_TRADE

    @abstractmethod
    def evaluate(self, context: RiskContext) -> RiskResult: ...


class RiskEngine:
    """三层风控编排。"""

    def pre_trade_check(self, plan: OrderPlan, positions: list[PositionSnapshot]) -> RiskReport:
        """事前风控。任一 REJECT → 整体 abort（fail-closed）。"""
        ...

    def intra_trade_check(self, filled_orders: list[OrderStatus]) -> RiskReport:
        """事中风控。触发 → 撤回未成交 + 暂停执行。"""
        ...

    def post_trade_check(self, execution_report: ExecutionReport) -> RiskReport:
        """事后风控。仅记录 + 通知，不回滚。"""
        ...
```

**内置风控规则**：

| 规则 | 层 | 说明 |
|------|-----|------|
| `single_stock_concentration` | PRE | 单只持仓不超过总资产的 N% |
| `total_order_value` | PRE | 单次调仓总金额不超过 M 元 |
| `price_limit_filter` | PRE | 涨跌停标的不下单 |
| `slippage_limit` | INTRA | 成交价偏离限价超过 N bps → 撤回剩余未成交订单 |
| `position_deviation` | POST | 实际持仓偏离目标超过 N% → 通知 |

### 6.6 幂等性实现

本章展示一个完整的 Vortex 使用流程，从数据初始化到模拟交易。
目的是为测试同学演示框架的使用过程和输出格式。

### 8.1 环境准备

```bash
# 安装 Vortex
pip install -e .

# 配置 Tushare Token
export TUSHARE_TOKEN=your_token_here

# 查看 Profile 解释（产品冻结命令：vortex profile explain）
vortex profile explain --profile cn_stock_daily
```

### 8.2 数据初始化（bootstrap）

```bash
# 用户 Profile（只写差异）
cat > profiles/cn_stock_daily.yaml << EOF
name: cn_stock_daily
type: data
provider: tushare
history_start: "20200101"

# 执行 bootstrap
vortex data bootstrap --profile cn_stock_daily
```

## 附录 A：存储路径与文件格式约定

### A.1 Parquet 分区与路径约定

数据面（data plane）使用 Parquet + DuckDB，控制面（control plane）使用 SQLite/YAML/JSON。两者物理分离，不混存。

**数据面路径结构**：

```
data/
├── authoritative/{profile}/              # 已发布的权威数据（只读视图）
│   ├── bars/date={YYYYMMDD}/data.parquet
│   ├── fundamental/{table}/end_date={YYYYMMDD}/data.parquet
│   ├── calendar/data.parquet
│   └── instruments/data.parquet
├── raw/{profile}/{provider}/             # 原始拉取数据（可重跑）
│   └── {dataset}/{date_range}/raw.parquet
├── signals/{signal_id}/                  # 信号快照（不可变）
│   └── signal.parquet
└── catalog.duckdb                        # DuckDB catalog（只读视图，可重建）
```

**控制面路径结构**：

```
state/
├── manifests/{profile}/                  # 运行清单
│   └── sync_manifest.db                  # SQLite
├── snapshots/{profile}/                  # 快照描述符
│   └── snapshot_descriptors.db
├── research/{profile}/                   # 研究域控制面
│   ├── factor_registry.db
│   ├── experiment_record.db
│   └── signal_snapshot_descriptor.db
├── strategy/{profile}/
│   ├── backtest_runs.db
│   └── target_portfolios.db
├── trade/{profile}/
│   ├── executions.db
│   └── orders.db
└── notification/
    ├── notification_log.db
    └── watch_results.db
```

**Parquet 写入规则**：
- 压缩格式：Snappy（默认）
- 行组大小：128MB 或 1M 行（先到为准）
- 分区键：日线数据按 `date` 分区；财务数据按 `end_date` 分区
- 文件名：统一为 `data.parquet`，不使用随机后缀
- 覆盖策略：数据快照允许 overwrite 同 `as_of`；信号快照严格不可变

### A.2 DuckDB 读取模式

```python
# 标准读取模式：通过 catalog.duckdb 创建视图
import duckdb

conn = duckdb.connect("data/catalog.duckdb", read_only=True)
df = conn.execute("""
    SELECT * FROM bars
    WHERE date BETWEEN '20260101' AND '20260404'
    AND symbol = '600519.SH'
""").fetchdf()
```

catalog.duckdb 是可重建的只读视图层——基于 Parquet 文件自动注册外部表。丢失后可从 Parquet 重建。
