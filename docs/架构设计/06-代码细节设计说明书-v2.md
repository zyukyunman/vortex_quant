## 1. 跨域统一约束

### 1.1 确定性契约（Determinism Contract）

**目标**：相同输入 + 相同配置 + 相同代码版本 → bit-identical 输出。

**排序规则**（全系统统一）：

| 场景 | 主排序 | 次排序 | 说明 |
|------|--------|--------|------|
| 行情数据 | date ASC | symbol ASC | 所有域读数据的默认排序 |
| 因子排名 | score DESC | symbol ASC | 分组时 score 相同有确定性顺序 |
| 组合权重 | weight DESC | symbol ASC | Top-N 截断确定性 |
| IC 排名 | ic DESC | factor_name ASC | 因子比较确定性排序 |

**reproducibility_key 计算**：

```python
def compute_reproducibility_key(resolved_profile: ResolvedProfile) -> str:
    """基于冻结配置 + 代码版本生成可复现键。"""
    payload = {
        "profile_hash": hash_dict(resolved_profile.to_public_dict()),
        "vortex_version": vortex.__version__,
        "python_version": sys.version,
    }
    return hashlib.sha256(json.dumps(payload, sort_keys=True).encode()).hexdigest()[:16]
```

### 1.2 舍入规则（Rounding Convention）

全系统统一精度标准：

| 数据类型 | 精度 | 舍入方式 | 依据 |
|---------|------|---------|------|
| 股价 | 2 位小数 | 四舍五入 | A 股最小变动单位 0.01 元 |
| 权重 | 4 位小数 | 四舍五入 | 1.0000 = 100% |
| IC / RankIC | 6 位小数 | 四舍五入 | 统计量需高精度 |
| Sharpe | 4 位小数 | 四舍五入 | 行业惯例 |
| 收益率 | 6 位小数 | 四舍五入 | 累计时精度要求高 |
| 手数 | 整数 | **向下取整** | A 股 100 股/手，`floor(shares/100)*100` |
| 佣金 | 2 位小数 | **向上取整** | ceiling 到分，保护模拟精度 |
| 滑点 (bps) | 1 位小数 | 四舍五入 | `(fill-limit)/limit × 10000` |
| 百分位排名 | 2 位小数 | 四舍五入 | 0.00–100.00 |

**关键不变量**：
- 组合权重和：`|sum(weights) - 1.0| < 1e-4`
- 所有权重非负：`all(weights >= 0)`

### 1.3 时间与时区

#### 1.3.1 系统唯一时区

```
系统时区 = Asia/Shanghai (UTC+8)
所有 datetime 字段存储带时区 ISO 8601 格式（如 2026-04-04T22:30:00+08:00）
禁止依赖数据库 CURRENT_TIMESTAMP（可能不带时区）
写入时由应用层统一生成 UTC+8 带 offset 的时间字符串

多市场兼容：
  时区和交易时段下沉到 market/calendar 配置
  核心时间逻辑（as_of 判定、PIT 对齐）接受 timezone 参数，不硬编码 UTC+8
  timezone 参数默认值为 Asia/Shanghai
```

#### 1.3.2 Snapshot as_of 判定

```
规则：当日 22:00 后生成的 snapshot 归属当日 as_of
      22:00 前生成的归属前一个交易日

示例：
  2026-04-04 22:30 生成 → as_of = 2026-04-04
  2026-04-04 21:50 生成 → as_of = 2026-04-03（前一个交易日）
```

#### 1.3.3 交易日历

- 由 `DataProvider.fetch_calendar()` 提供，缓存在 `trading_calendar` 表
- 非交易日：跳过，不填充
- 半日市（如春节前一天）：正常处理，仅数据量少
- 分钟频数据边界：09:30–11:30, 13:00–15:00，非交易时段数据丢弃

#### 1.3.4 PIT（Point-In-Time）判定规则

```
规则 1：财报数据以 ann_date（公告日期）为准，NOT report_date
规则 2：effective_from = ann_date 的下一个交易日 09:30
规则 3：若 ann_date 在交易时段内（09:30-15:00），当日 15:00 后生效
规则 4：快报、预告、正式报各自独立 PIT，互不覆盖
规则 5：ann_date 为非交易日 → effective_from = 下一个交易日 09:30
```

### 1.4 并发与隔离

```
域间隔离：各域独立运行、独立 run_id、独立目录
域内并发：
  - Data：按 dataset 分组并行拉取，单 dataset 内串行
  - Research：因子评测可按 factor_name 并行，队列管理
  - Strategy：回测任务可并行（不同 backtest_run_id）
  - Trade：同一 exec_id 串行，不同 profile 可并行

文件锁：控制面 SQLite（state/）使用写锁，数据面 Parquet 不加锁（只追加/覆盖）
```

### 1.4.1 本地服务与 CLI 协议

前文定义了域内并发和文件隔离，但还缺一层“**谁来驱动这些域模块真正运行**”。Vortex 在代码实现上必须补齐这一层：

```text
vortex CLI（参数解析 / 结果展示）
        ↓ localhost HTTP/JSON
VortexServer（命令路由 / 工作空间绑定 / 长任务编排）
        ↓
Data / Research / Strategy / Trade 领域模块
```

**职责边界**：

1. **CLI**：
   - 解析命令行参数
   - 连接本地服务
   - 输出 run_id、摘要、工件路径
   - 不直接执行长耗时数据同步 / 因子评测 / 回测逻辑
2. **VortexServer**：
   - 绑定单个 workspace
   - 校验命令请求
   - 调用 ProfileService、Provider、BacktestEngine 等领域对象
   - 写入状态文件、report、manifest、signal/snapshot 工件
3. **领域模块**：
   - 保持“纯业务能力”属性
   - 不关心 HTTP/CLI 协议细节

**工作空间约定**：

服务进程启动时必须绑定 `VORTEX_WORKSPACE`，所有 profile、数据、研究与策略产物都相对该目录解析：

```text
{workspace}/
├── profiles/
│   ├── data/
│   ├── research/
│   └── strategy/
├── data/
│   ├── authoritative/
│   └── manifests/
├── research/
│   ├── panels/
│   ├── signals/
│   └── reports/
├── strategy/
│   └── reports/
└── state/
```

**命令协议约定**（MVP）：

| CLI 命令 | 服务动作 | 同步返回 |
|---------|---------|---------|
| `vortex init` | 初始化工作空间与 profile 模板 | 创建结果摘要 |
| `vortex server start` | 后台启动本地服务 | pid / log_path / workspace |
| `vortex data bootstrap` | 提交数据抓取后台任务 | task_id + run_id + log_path |
| `vortex data status` | 读取控制面任务状态与最近一次 run 摘要 | 活跃任务进度 + latest_run |
| `vortex data logs` | 解析任务日志路径并输出 tail / follow | log_path + tail |
| `vortex data cancel` | 标记任务 cancelled 并向 worker 发信号 | task_id + run_id + cancel_ack |
| `vortex research register` | 注册/发现因子实现 | factor_ids + registry_path |
| `vortex research eval` | 生成 factor panel、评测报告、最新信号 | run_id + report_path + panel_paths |
| `vortex strategy backtest` | 消费研究工件执行动态回测 | run_id + report_path + equity_curve_path |

### 1.7 Schema 版本管理

Vortex 控制面 SQLite 采用内置版本化迁移机制，保证前后版本兼容，用户无需手动删库。

**核心组件**：

| 组件 | 说明 |
|------|------|
| `schema_version` 表 | 每个 SQLite 库内置，记录当前版本号和应用时间 |
| `SCHEMA_VERSION` 常量 | `database.py` 顶层，代表代码要求的最新版本 |
| `_MIGRATIONS` 注册表 | `dict[int, list[str]]`，key = 目标版本号，value = SQL 列表 |
| `_detect_legacy_version()` | 对没有 `schema_version` 表的旧库，通过表结构特征推断当前版本 |

**迁移执行流程**（`initialize_tables()` 内部）：

```
启动 / init
  ├─ 读 schema_version 表 → 得到 current_version
  ├─ 若表不存在 → _detect_legacy_version() 通过结构推断
  ├─ 若 current >= SCHEMA_VERSION → 跳过（已是最新）
  └─ 否则：
       for target in range(current+1, SCHEMA_VERSION+1):
           执行 _MIGRATIONS[target] 中全部 SQL
       写入 schema_version = SCHEMA_VERSION
```

**迁移指令支持两种格式**：

1. 标准 SQL：`CREATE TABLE ...`、`CREATE INDEX ...`、`ALTER TABLE ...`
2. 安全列添加：`_migrate_add_column:表名:列名:类型` — 先检测列是否存在，不存在才 ALTER，避免重复执行报错

**版本历史**：

| 版本 | 变更内容 |
|------|---------|
| v1 | 初始建表：audit_log、task_queue + 基础索引 |
| v2 | task_queue 增加 resource_key 列 + 复合索引（任务幂等） |

**新增迁移规范**（后续域开发时遵守）：

1. 递增 `SCHEMA_VERSION`
2. 在 `_MIGRATIONS` 注册表中添加对应版本的 SQL 列表
3. 新增列使用 `_migrate_add_column:` 前缀确保幂等
4. 不允许删列、改列类型、删表——仅允许 additive 变更
5. 补充对应的迁移测试（模拟旧库 → 调用 `initialize_tables()` → 验证新列/数据保留）

**数据面工件的 schema_version**：

每类工件（sync_manifest、snapshot_descriptor、report）的行内 `schema_version` 字段仍然保留，用于标识该条记录写入时的结构版本。与库级 `schema_version` 表配合使用：库级版本驱动迁移，行级版本驱动向后兼容读取。

### 1.8 确定性验收双层标准

| 层 | 说明 | 验收标准 |
|----|------|---------|
| **严格层** | 因子值、IC、组合权重 | bit-identical（同输入同输出） |
| **宽松层** | Sharpe、最大回撤、总收益 | 误差 < 1e-6（浮点累计误差容忍） |

验收方法：Golden Master 对比——首次运行产出基准文件，后续运行与之 diff。

### 2.3 合并逻辑

```python
def resolve_profile(user_yaml_path: str, cli_overrides: dict | None = None) -> ResolvedProfile:
    """Profile 解析主流程。"""
    # 1. 加载框架默认值
    defaults = load_defaults(profile_type)       # 从 defaults/{type}.yaml

    # 2. 加载用户 Profile（只含差异字段）
    user_config = load_yaml(user_yaml_path)

    # 3. 若 extends 存在，递归加载父 Profile
    if user_config.get("extends"):
        parent = resolve_parent(user_config["extends"])
        defaults = deep_merge(defaults, parent)

    # 4. 深度合并：defaults + user_config
    merged = deep_merge(defaults, user_config)

    # 5. 应用 CLI 覆盖（最高优先级）
    if cli_overrides:
        merged = deep_merge(merged, cli_overrides)

    # 6. 校验
    validate(merged)

    # 7. 运行时解析（路径、密钥、快照别名）
    resolved = resolve_runtime(merged)

    # 8. 输出 .resolved.yaml
    dump_resolved(resolved, path=f".vortex/{profile_name}.resolved.yaml")

    return resolved
```

**合并规则**：
- **嵌套 dict**：递归深度合并
- **list**：整体替换（不做元素级合并）
- **null / ~**：显式清除（从合并结果中移除该键）
- **标量**：用户值覆盖默认值

针对 Tushare 这类按接口限频的数据源，`DataPipeline` / `SyncPlanBuilder` 的实现约束再补充如下：

1. **先做缺口规划，再做抓取调用**  
   `bootstrap / update / repair` 不得把"写入幂等"当成"抓取幂等"。在调用 Provider 前，必须先基于 `StorageBackend.list_partitions()` / `read()` 和控制面的 `SyncManifest.dataset_partition_coverage` 计算：
   - `target_chunks`
   - `existing_chunks`
   - `covered_chunks`
   - `missing_chunks`
   - 本轮是否 `skip / partial_fetch / full_fetch`

2. **request budget 是全局共享的，不按自然天数硬编码**  
   调度器不应假设"每个 API 都有独立 500/min 窗口"。对于默认 5000 积分档位，普通积分接口的安全阈值可收敛为 `effective_global_rpm ≈ 400`；其他接口若文档还有更低的单接口上限，则再取两者较小值。  
   也就是说，预算单位始终是"请求次数"，并且首先受**全局预算**约束。

3. **全市场时间序列优先按 `trade_date` 遍历**  
   遵循 Tushare 官方 `doc_id=230` 的建议：像 `daily` / `daily_basic` 这种全市场历史时间序列，不应默认按 `ts_code` 全量循环，而应优先按 `trade_date` 获取单日全市场数据，再写入 `date=...` 分区。

4. **季频财务接口按 `quarter × symbol_batch` 调度**  
   对 `income` / `balancesheet` / `cashflow` / `fina_indicator` / `forecast` / `express`，先判断缺失季度；5000 分档位优先走 `*_vip(period=季度末)` 做季度全市场批量抓取，积分不足时再回退普通单股票接口。  
   此外，Provider 应接受显式的 `partition_values` 精确缺口列表，避免把中间已覆盖季度因为 `start~end` 宽区间又捎带重抓。

5. **日志必须暴露调度决策**  
   worker 日志最少要打印：

    ```text
    dataset=valuation dedup: partition_key=date target=545 existing=398 missing=147
    rate_limit effective_global_rpm=400 api=daily_basic
    dataset=fundamental dedup: partition_key=end_date target=84 existing=60 missing=23
    dataset=fundamental dedup_coverage: partition_key=end_date covered=1
    ```

   这样用户才能确认系统是否真的执行了"跳过已覆盖分区"和"全局限流"两层逻辑。

### 4.3 评测引擎 10 维指标计算口径

| 维度 | 指标 | 计算方法 | 精度 |
|------|------|---------|------|
| **IC** | Information Coefficient | `pearsonr(factor_rank, forward_return)` 每期截面 | 6 位 |
| **RankIC** | Rank IC | `spearmanr(factor_rank, forward_return_rank)` 每期截面 | 6 位 |
| **ICIR** | IC Information Ratio | `mean(IC_series) / std(IC_series)` | 4 位 |
| **IR** | Information Ratio | `mean(excess_return) / std(excess_return)` 年化 | 4 位 |
| **多空收益** | Long-Short Return | `top_group_return - bottom_group_return` 年化 | 6 位 |
| **换手率** | Turnover | `mean(|position_change|)` 每期 | 4 位 |
| **IC 衰减** | IC Decay | IC_1d, IC_5d, IC_20d 序列，衰减速率 | 6 位 |
| **分组收益** | Group Return | 按因子值等分 N 组，各组平均收益，检查单调性 | 6 位 |
| **最大回撤** | Max Drawdown | 多头组合的历史最大回撤 | 4 位 |
| **稳定性** | Stability | `R² of IC_cumsum regression on time` | 4 位 |

### 4.8 因子评测 HTML Tear Sheet Payload Schema

> **正则关系**：`<script id="vortex-data">` 内嵌 JSON == `research_report.json` 的 canonical schema。
> 二者内容相同，不允许 HTML 端额外计算或使用未定义字段。
> 若某个展示块无法由下列字段驱动，说明 schema 需要扩展，而不是由前端临时补算。

### 4.9 HTML 报告共享渲染契约

因子评测报告与回测报告共享一套渲染基础设施。本节定义**稳定契约**——实现必须遵守，但不规定具体 CSS 类名或 DOM 结构。

#### 数据嵌入点

所有报告数据通过 `<script id="vortex-data" type="application/json">` 内嵌。渲染 JS 解析此节点后驱动图表。不允许 XHR / fetch 外部数据。

#### 页面路由

顶部导航区提供页面切换。每个页面对应一个稳定 `page_id`（因子报告 7 个、回测报告 6 个），page_id 与 payload key 一一对应。页面切换时触发图表 resize（解决 Plotly 隐藏容器尺寸为 0 的问题）。

#### 响应式要求

| 约束 | 阈值 | 行为 |
|------|------|------|
| 报告最大宽度 | ≤ 1400px | 居中 + 两侧留白 |
| 小屏断点 | ≤ 800px | 侧栏指标折叠到主区下方，表格水平滚动 |
| 表格溢出 | 内容超出视口 | 水平滚动包装，不破坏外层布局 |
| 图表容器 | 隐藏→显示切换 | 必须触发 resize / relayout |

#### 主题支持

支持浅色 / 深色两种主题。切换时 Plotly 图表配色同步刷新。主题偏好可持久化（localStorage）。

#### 指标 Tooltip 契约

每个侧栏指标卡片和核心指标卡片支持 hover/focus 触发的 tooltip。Tooltip 文本由 EvalEmphasis 注册表（因子报告）或回测 QualityScorer 提供，前端通过 `metric_id` 查找，**不从 payload 动态读取**。

#### 图表交互基线

| 能力 | 所有报告统一 |
|------|------------|
| 框选缩放 / 双击还原 | ✅ |
| 悬浮提示（数值 + 日期） | ✅ |
| 全屏 / 导出 PNG | ✅（每个图表卡片右上角） |

#### 评级徽章

综合评级以徽章形式展示在报告头部，使用统一的颜色映射：A 绿 / B+ 浅绿 / B 蓝 / B- 黄 / C 橙 / D 红 / F 灰。徽章同时展示字母和数字分数。

### 5.4 多维度回测参数

| 维度 | 目的 | 关键参数 | 结果 |
|------|------|---------|------|
| **全样本** | 基础绝对表现 | 完整区间 | 必须运行 |
| **随机日期** | 起始日依赖检验 | `n_trials=50`, `min_window=252d` | 分布统计 |
| **Walk-Forward** | 过拟合检验 | `train=756d`, `test=252d`, `step=63d` | 多段拼接 |
| **Regime** | 不同市况下表现 | 市场状态指标 + 分段 | 各市况独立评测 |

### 5.5 质量评分公式

**五维评分**：

| 维度 | 权重 | 指标来源 | 归一化方法 |
|------|------|---------|-----------|
| 收益风险 | 25% | 全样本 Sharpe, 年化收益, 年化超额 | 百分位映射 |
| 稳定性 | 25% | 随机日期 Sharpe 分布的 std / mean | 反向百分位 |
| 稳健性 | 20% | Walk-Forward OOS Sharpe / IS Sharpe | 比值越接近 1 越好 |
| 成本 | 15% | 换手率、滑点影响 | 反向百分位 |
| 风险 | 15% | 最大回撤、尾部风险（CVaR） | 反向百分位 |

```python
def compute_quality_score(metrics: dict) -> tuple[float, str]:
    """计算综合评分与字母评级。"""
    score = (
        0.25 * normalize(metrics["return_risk"]) +
        0.25 * normalize(metrics["stability"]) +
        0.20 * normalize(metrics["robustness"]) +
        0.15 * normalize(metrics["cost"]) +
        0.15 * normalize(metrics["risk"])
    )
    grade = score_to_grade(score)  # A/B+/B/B-/C/D/F
    return score, grade
```

### 6.5 三层风控

```python
class RiskRule(ABC):
    """可插拔风控规则。"""
    name: str
    level: RiskLevel  # PRE_TRADE | INTRA_TRADE | POST_TRADE

    @abstractmethod
    def evaluate(self, context: RiskContext) -> RiskResult: ...


class RiskEngine:
    """三层风控编排。"""

    def pre_trade_check(self, plan: OrderPlan, positions: list[PositionSnapshot]) -> RiskReport:
        """事前风控。任一 REJECT → 整体 abort（fail-closed）。"""
        ...

    def intra_trade_check(self, filled_orders: list[OrderStatus]) -> RiskReport:
        """事中风控。触发 → 撤回未成交 + 暂停执行。"""
        ...

    def post_trade_check(self, execution_report: ExecutionReport) -> RiskReport:
        """事后风控。仅记录 + 通知，不回滚。"""
        ...
```

**内置风控规则**：

| 规则 | 层 | 说明 |
|------|-----|------|
| `single_stock_concentration` | PRE | 单只持仓不超过总资产的 N% |
| `total_order_value` | PRE | 单次调仓总金额不超过 M 元 |
| `price_limit_filter` | PRE | 涨跌停标的不下单 |
| `slippage_limit` | INTRA | 成交价偏离限价超过 N bps → 撤回剩余未成交订单 |
| `position_deviation` | POST | 实际持仓偏离目标超过 N% → 通知 |

## 附录 A：存储路径与文件格式约定

### A.1 Parquet 分区与路径约定

数据面（data plane）使用 Parquet + DuckDB，控制面（control plane）使用 SQLite/YAML/JSON。两者物理分离，不混存。

**数据面路径结构**：

```
data/
├── authoritative/{profile}/              # 已发布的权威数据（只读视图）
│   ├── bars/date={YYYYMMDD}/data.parquet
│   ├── fundamental/{table}/end_date={YYYYMMDD}/data.parquet
│   ├── calendar/data.parquet
│   └── instruments/data.parquet
├── raw/{profile}/{provider}/             # 原始拉取数据（可重跑）
│   └── {dataset}/{date_range}/raw.parquet
├── signals/{signal_id}/                  # 信号快照（不可变）
│   └── signal.parquet
└── catalog.duckdb                        # DuckDB catalog（只读视图，可重建）
```

**控制面路径结构**：

```
state/
├── manifests/{profile}/                  # 运行清单
│   └── sync_manifest.db                  # SQLite
├── snapshots/{profile}/                  # 快照描述符
│   └── snapshot_descriptors.db
├── research/{profile}/                   # 研究域控制面
│   ├── factor_registry.db
│   ├── experiment_record.db
│   └── signal_snapshot_descriptor.db
├── strategy/{profile}/
│   ├── backtest_runs.db
│   └── target_portfolios.db
├── trade/{profile}/
│   ├── executions.db
│   └── orders.db
└── control.db                  # task_queue / audit_log / notification_log
```

**Parquet 写入规则**：
- 压缩格式：Snappy（默认）
- 行组大小：128MB 或 1M 行（先到为准）
- 分区键：日线数据按 `date` 分区；财务数据按 `end_date` 分区
- 文件名：统一为 `data.parquet`，不使用随机后缀
- 覆盖策略：数据快照允许 overwrite 同 `as_of`；信号快照严格不可变
