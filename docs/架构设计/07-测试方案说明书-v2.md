## §1 测试策略与方法论

### 1.1 测试金字塔

Vortex 采用经典的三层测试金字塔（Test Pyramid），层级由下至上为：

| 层级 | 占比目标 | 执行速度 | 职责边界 |
|------|---------|---------|---------|
| **单元测试（Unit）** | ≥ 70% | < 50ms / case | 验证单个函数、类方法的输入输出正确性，包括边界值、异常路径和确定性合约 |
| **集成测试（Integration）** | ≈ 20% | < 2s / case | 验证跨模块协作流程，如 Data Pipeline 从 fetch 到 snapshot、Strategy Pipeline 从 signal 到 target_portfolio |
| **端到端测试（End-to-End）** | ≈ 10% | < 30s / case | 验证完整业务链路，如 `vortex data bootstrap` → `vortex research eval` → `vortex strategy compile` → `vortex trade reconcile` |
| **研究回归（Research Regression）** | 不纳入默认占比 | 可能较慢 | 使用本地真实 A 股数据复核因子、策略、持仓和报告，默认不在普通单测中执行 |

**设计原则**：

- **确定性优先（Determinism First）**：同一 `run_id` + 相同输入 + 相同代码 → 比特级一致输出（bit-identical），这是 P1 原则在测试中的体现
- **失败即关闭（Fail-Closed）**：所有涉及质量门禁（QualityGate）、风控拦截（Risk Interception）、约束冲突（Constraint Conflict）的路径必须有对应的失败断言
- **可追溯性（Lineage）**：测试用例应能通过 `run_id → compile_run_id → signal_ids → snapshot_id` 链路追溯数据血缘

### 1.2 确定性验收标准

根据 06 §1.8 双层确定性验收标准：

| 层级 | 验证对象 | 精度要求 | 验证方法 |
|------|---------|---------|---------|
| **严格层（Strict Layer）** | 因子值、IC、组合权重（portfolio weights） | 比特级一致 | `assert actual == expected`（Decimal 精确比较） |
| **宽松层（Loose Layer）** | 夏普比率（Sharpe）、最大回撤（max drawdown）、累计收益（total return） | 误差 < 1e-6 | `assert abs(actual - expected) < Decimal("1e-6")` |

**Golden Master 方法**：首次运行生成基准结果文件（golden master），后续运行与基准逐字段比对。基准文件纳入版本控制。

### 1.3 测试命名规范

所有测试函数遵循以下命名模式：

```
test_<模块>_<场景>_<预期结果>
```

示例：
- `test_round_price_half_up_boundary` — 验证价格在 0.5 边界的四舍五入行为
- `test_universe_merge_current_holdings_retained` — 验证 Universe 阶段合并持仓后持仓保留
- `test_paper_gateway_limit_buy_price_above_ref_filled` — 验证 PaperGateway 限价买入价高于参考价时成交

### 1.4 pytest 约定

```python
# 当前仓库采用“扁平文件 + 按职责拆模块”的 pytest 布局
# tests/conftest.py                 — 全局 fixture
# tests/test_config.py              — Config 域
# tests/test_data_components.py     — ProviderRegistry / dataset registry
# tests/test_tushare_provider.py    — Tushare provider 运行时行为
# tests/test_manifest_calendar.py   — SyncManifest / DataCalendar
# tests/test_data_pipeline.py       — DataPipeline 排序与入口编排
# tests/test_data_pipeline_dedup.py — DataPipeline 去重 / 续跑 / 恢复
# tests/test_cli_interactive.py     — CLI 交互、多选、init 流程
# tests/test_cli.py                 — CLI server / data 后台任务
# tests/test_storage.py             — StorageBackend
# tests/test_quality.py             — QualityEngine
# tests/test_pit.py                 — PIT 对齐
# tests/test_runtime.py             — Runtime / task queue / workspace
# tests/test_shared.py              — shared helpers / errors / 公共契约
```

**关键约定**：

- 新增单元测试默认使用 `tests/test_<domain>_<topic>.py` 命名；只有该域当前规模很小时，才允许继续落在 `tests/test_<domain>.py`
- 文件职责优先按“行为主题”拆分，而不是继续把同域所有测试堆进一个文件；例如 Data 域至少区分 `components / tushare_provider / manifest_calendar / pipeline / pipeline_dedup`
- 使用 `@pytest.fixture` 提供可复用的测试数据和 Mock 对象
- 使用 `@pytest.mark.parametrize` 实现同一逻辑的多场景覆盖
- 使用 `pytest.raises(ExceptionType, match="regex")` 断言异常
- 使用 `pytest.approx` 仅用于宽松层验证，严格层一律使用 `Decimal` 精确比较
- 标记体系：`@pytest.mark.unit`、`@pytest.mark.integration`、`@pytest.mark.e2e`、`@pytest.mark.determinism`、`@pytest.mark.research`、`@pytest.mark.local_data`

### 1.5 研究回归测试边界

研究代码分两类测试：

| 类型 | 例子 | 是否默认执行 |
|---|---|---|
| 单元 / 轻集成 | 合成数据验证 IC、事件日期映射、仓位上限、报告 writer | 是 |
| 研究回归 / 真实数据验收 | 用本地 A 股 parquet 复核业绩预告 v3 指标、因子 IC、持仓安全性 | 否 |

默认测试只要求快速、确定、无本地数据依赖。真实数据研究回归应放在：

```text
tests/research_regression/
```

并显式标记：

```python
@pytest.mark.research
@pytest.mark.local_data
```

推荐执行方式：

```bash
uv run python -m pytest tests/research_regression -q -m "research and local_data"
```

若本地数据、Tushare token 或权限不存在，研究回归必须跳过或给出清晰失败原因，不能影响默认 `uv run python -m pytest tests/ -q`。

### 1.6 Data 域闭环优先级

Data 域的测试不再只围绕“某个函数有没有返回 DataFrame”展开，而必须同时覆盖三条主线：

| 维度 | 关注点 | 典型问题 | 必须回答的问题 |
|------|--------|----------|----------------|
| **流程正确性** | `bootstrap / cancel / rerun / status / server restart` 是否形成闭环 | 任务已取消但 manifest 仍显示 running；server restart 把活跃 worker 误判成 stale | 控制面状态是否一致、用户操作是否可恢复 |
| **抓取效率** | 已安全落盘或已登记 coverage 的数据是否被跳过 | `bars` / `fundamental` / `events` 在 rerun 时重复抓取 | rerun 是不是只重抓真正不完整的那部分 |
| **数据安全** | cancel / retry / crash recovery 后是否会留下半成品或误判完成 | 稀疏事件表第一次没跑完却被当成 complete；季度表覆盖账本和物化分区不一致 | 是不是宁可保守重抓，也不会误跳过不完整数据 |

因此，Data 域测试必须分成两层：

1. **模块回归层**：provider / manifest / pipeline / cli / runtime 各自用快速单测守住局部不变量。  
2. **生命周期闭环层**：使用 `tests/test_data_lifecycle.py` 把 `bootstrap → cancel → rerun dedup → server/status/cancel` 串成完整流程，验证控制面与数据面是否一致收尾。

### 2.2 MockDataProvider

`MockDataProvider` 实现 `DataProvider` 协议，返回确定性的合成数据，不依赖外部数据源（Tushare）。
所有价格、成交量均基于固定种子（seed=42）生成，保证每次调用返回相同结果。

```python
# tests/conftest.py

import pandas as pd
import numpy as np
from dataclasses import dataclass


@dataclass
class MockDataProvider:
    """确定性数据提供器，实现 DataProvider 协议。

    所有数据基于 seed=42 生成；支持 bars、calendar、fundamentals。
    """

    seed: int = 42

    @property
    def name(self) -> str:
        return "mock"

    @property
    def supported_markets(self) -> list[str]:
        return ["cn_stock"]

    def fetch_bars(
        self,
        symbols: list[str],
        freq: str,
        start: date,
        end: date,
    ) -> pd.DataFrame:
        """生成确定性日线数据。

        价格围绕基准价 ± 10% 随机波动（固定种子），
        排序规则：date ASC, symbol ASC（符合 06 §1.1 排序契约）。
        """
        rng = np.random.default_rng(self.seed)
        trading_days = self.fetch_calendar("SSE")
        trading_days = [d for d in trading_days if start <= d <= end]
        rows = []
        base_prices = {s: 50.0 + i * 10 for i, s in enumerate(symbols)}
        for d in trading_days:
            for sym in sorted(symbols):
                bp = base_prices[sym]
                close = round(bp * (1 + rng.uniform(-0.1, 0.1)), 2)
                rows.append({
                    "ts_code": sym,
                    "trade_date": d,
                    "open": round(close * (1 + rng.uniform(-0.02, 0.02)), 2),
                    "high": round(close * 1.03, 2),
                    "low": round(close * 0.97, 2),
                    "close": close,
                    "volume": int(rng.integers(1_000_000, 50_000_000)),
                })
        df = pd.DataFrame(rows)
        return df.sort_values(["trade_date", "ts_code"]).reset_index(drop=True)

    def fetch_calendar(self, exchange: str) -> list[date]:
        """返回 2023 年全年交易日历（剔除周末和主要节假日）。"""
        all_days = pd.bdate_range(
            STANDARD_DATE_RANGE[0], STANDARD_DATE_RANGE[1], freq="B",
        )
        return [d.date() for d in all_days]

    def fetch_fundamentals(
        self,
        symbols: list[str],
        metrics: list[str],
    ) -> pd.DataFrame:
        """生成确定性基本面数据，含 ROE、FCF、operating_cf 等字段。"""
        rng = np.random.default_rng(self.seed + 1)
        rows = []
        for sym in sorted(symbols):
            rows.append({
                "ts_code": sym,
                "ann_date": date(2023, 4, 28),
                "roe": round(float(rng.uniform(0.05, 0.35)), 4),
                "fcf": round(float(rng.uniform(-1e8, 5e8)), 2),
                "operating_cf": round(float(rng.uniform(-5e7, 8e8)), 2),
                "enterprise_value": round(float(rng.uniform(1e9, 5e11)), 2),
            })
        return pd.DataFrame(rows)

    def fetch_instruments(self) -> pd.DataFrame:
        """返回标准股票池的证券信息。"""
        return pd.DataFrame(STANDARD_STOCK_POOL)

    def fetch_events(
        self,
        symbols: list[str],
        event_type: str,
        start: date,
        end: date,
    ) -> pd.DataFrame:
        """返回空事件表（默认无特殊事件）。"""
        return pd.DataFrame(columns=["ts_code", "event_date", "event_type", "detail"])
```

### 2.3 MockStorageBackend

```python
# tests/data/conftest.py

from typing import Any


class MockStorageBackend:
    """内存字典后端，实现 StorageBackend 协议。

    所有读写操作基于 dict 存储，不涉及文件 I/O，
    便于单元测试验证 upsert/read/list/snapshot 逻辑。
    """

    def __init__(self) -> None:
        self._store: dict[str, pd.DataFrame] = {}
        self._snapshots: dict[str, dict[str, Any]] = {}

    def upsert(self, dataset: str, df: pd.DataFrame) -> int:
        """写入或追加数据，返回受影响行数。"""
        if dataset in self._store:
            existing = self._store[dataset]
            self._store[dataset] = pd.concat(
                [existing, df],
            ).drop_duplicates().reset_index(drop=True)
        else:
            self._store[dataset] = df.copy()
        return len(df)

    def read(
        self,
        dataset: str,
        columns: list[str] | None = None,
        filters: dict | None = None,
    ) -> pd.DataFrame:
        """读取指定数据集，支持列筛选与条件过滤。"""
        if dataset not in self._store:
            return pd.DataFrame()
        df = self._store[dataset]
        if columns:
            df = df[columns]
        if filters:
            for col, val in filters.items():
                df = df[df[col] == val]
        return df.copy()

    def list_partitions(self, dataset: str) -> list[str]:
        """列出已有数据集名称。"""
        return list(self._store.keys())

    def snapshot(self, snapshot_id: str, datasets: list[str]) -> dict:
        """创建快照：冻结指定数据集的当前版本。"""
        snap = {}
        for ds in datasets:
            if ds in self._store:
                snap[ds] = self._store[ds].copy()
        self._snapshots[snapshot_id] = snap
        return {"snapshot_id": snapshot_id, "datasets": list(snap.keys())}


@pytest.fixture
def mock_storage() -> MockStorageBackend:
    return MockStorageBackend()
```

### 3.0 pytest 文件落点矩阵（当前仓库真实结构）

后续新增测试时，先按下面的文件职责落点；除非模块复杂度再次上升，否则不要把不同主题重新合并回大文件。

| 模块主题 | pytest 文件路径 | 当前职责 | 命名示例 |
|----------|----------------|----------|----------|

## §4 集成测试场景

集成测试验证跨模块协作流程的正确性。每个场景覆盖一条完整的业务链路，
但使用 Mock 组件替代外部依赖（如 Tushare、磁盘 I/O），确保测试速度和确定性。

### 4.1 Data 生命周期闭环：bootstrap → cancel → rerun dedup → control-plane recovery

Data 集成测试的核心不再是单一 happy path，而是**完整生命周期闭环**。  
代码层使用代表性 dataset 组合（`bars + fundamental + events`）来模拟真实“全量数据”中的三类关键行为；运维层再补真实 Tushare smoke，不把 52 张表全部塞进 pytest。

#### 4.1.1 代码侧闭环场景矩阵

| 场景编号 | 固定输入 | 目标 | 关键断言 | 对应测试 |
|----------|----------|------|----------|----------|
| **DL-1 首次成功后同范围 rerun** | 显式时间窗 `20260101 ~ 20260408`，代表 `trade_day_all + quarter + sparse events` 三类表 | 验证 rerun 真正做到“已完成不重抓” | `bars` 命中已有 `date` 分区；`fundamental` 命中已有季度分区；`events` 命中 exact-range coverage；provider 调用计数不增长 | `test_bootstrap_rerun_after_complete_range_skips_events_and_materialized_datasets` |
| **DL-2 cancel 后的安全续跑** | 同上，但第一次在 `events` 阶段主动取消 | 验证“已安全落盘部分跳过，未完成部分必须重抓” | 第一次 run = cancelled；rerun 时 `bars/fundamental` 不重抓；`events` 因上次未完成而重新抓取；最终只在成功 rerun 后登记 exact-range coverage | `test_bootstrap_cancel_then_rerun_skips_completed_datasets_but_refetches_incomplete_events` |
| **DL-3 server/status/cancel 控制面闭环** | 后台 bootstrap 任务，worker PID 仍存活，再执行 `server start` / `data status` / `data cancel` | 验证控制面状态不再打架 | server start 不误判 stale task；`status` 能看到活跃任务；`cancel` 后 `task_queue` 与 `sync_manifest` 同时变 `cancelled` | `test_server_status_cancel_form_a_closed_loop_for_running_bootstrap` |

#### 4.1.2 运维 smoke 样例（真实 CLI）

下面这条 smoke 不是单元测试，而是**用户体验验收脚本**。  
它直接对应“重新安装 → init/server → 选择全部数据 → `history_start=20260401` → bootstrap → cancel → rerun”的真实路径。

1. `uv tool install --reinstall .`
2. `vortex init`（provider=`tushare`，datasets=all，`history_start=20260401`）
3. `vortex server start`
4. `vortex data bootstrap`
5. 在 `events` 或任一长耗时 dataset 期间执行 `vortex data cancel`
6. 再次执行 `vortex data bootstrap`

此 smoke 至少要回答 3 个问题：

- **流程正确性**：`status / logs / cancel` 输出是否一致，取消后 run 是否以 `cancelled` 收尾；
- **抓取效率**：上一次已安全落盘或已登记 coverage 的 dataset 是否跳过，是否只重抓不完整部分；
- **数据安全**：cancel / rerun 后是否留下半成品分区、是否把不完整事件范围误判为 complete。

#### 4.1.3 为什么代码侧不用“52 张真实表全部跑完”

因为 pytest 的职责是验证**架构语义**，不是把真实外部 API 当回归环境。  
Data 闭环里真正需要守住的是：

- `trade_day_all` 型分区去重是否成立；
- `symbol_quarter_range` 型季度去重 / coverage 是否成立；
- `symbol_once + date` 稀疏事件表在 cancel / rerun 后是保守重抓还是正确跳过；
- `task_queue + sync_manifest + worker pid` 三方状态是否一致。

只要这四条语义在代表场景里闭环，pytest 就能高信噪比地发现结构问题；  
真实 52 张表的全量跑批，应作为 smoke / 验收层补充，而不是塞进单测里制造噪声。

```python
class TestAcceptanceHoldRetention:

    ) -> None:
```

### 场景 5：约束削减（Constraint Trim）

> **条件**：经过 Alpha 后有 35 只标的，超过 max_positions=30
> **预期**：Risk 阶段按最低得分削减至 30 只

```python
class TestAcceptanceConstraintTrim:
    """验收场景 5：风控约束削减超限持仓。

    背景：
        Alpha 阶段输出 35 只标的（均通过 value top-50 和 momentum > 0 筛选），
        但 Risk 阶段 max_positions=30，需要按得分从低到高移除 5 只标的。
        剩余 30 只重新归一化权重。
    """

    def test_constraint_trim_exceeds_max_positions_trims_lowest(
        self,
    ) -> None:
        # ── 构造输入 ──────────────────────────────────────────────
        # 35 只候选标的，均已通过 Alpha 阶段
        candidates = [f"TRIM_{i:04d}.SZ" for i in range(1, 36)]
        scores = {
            sym: Decimal(str(round(1.0 - i * 0.025, 4)))
            for i, sym in enumerate(candidates)
        }

        # 得分最低的 5 只
        sorted_by_score = sorted(scores.items(), key=lambda x: x[1])
        bottom_5 = {sym for sym, _ in sorted_by_score[:5]}
        top_30 = {sym for sym, _ in sorted_by_score[5:]}

        # ── 执行 Risk 阶段 ─────────────────────────────────────────
        risk_config = {
            "max_weight": 0.05,
            "min_positions": 20,
            "max_positions": 30,
            "sector_limit": 0.20,
        }
        risk_result = risk_stage_execute(
            candidates=candidates,
            scores=scores,
            weights={sym: Decimal("0.0286") for sym in candidates},  # 1/35 ≈ 0.0286
            config=risk_config,
        )

        # ── 断言 ────────────────────────────────────────────────
        final_symbols = set(risk_result["weights"].keys())
        assert len(final_symbols) == 30, (
            f"验收场景 5 失败：最终持仓数应为 30，实际为 {len(final_symbols)}"
        )
        # 被削减的应是得分最低的 5 只
        assert final_symbols & bottom_5 == set(), (
            "验收场景 5 失败：得分最低的 5 只标的应被移除"
        )
        assert top_30.issubset(final_symbols), (
            "验收场景 5 失败：得分较高的 30 只标的应保留"
        )
        # 权重归一化
        total_weight = sum(risk_result["weights"].values())
        assert abs(total_weight - Decimal("1.0")) < Decimal("1e-4"), (
            f"验收场景 5 失败：权重之和应 ≈ 1.0，实际为 {total_weight}"
        )
```

```python
class TestAcceptanceEmptyHoldingsFirstCompile:

        3. Alpha 阶段 2：score_filter threshold=0
        4. Portfolio：signal_weighted
        5. Risk：约束检查

    def test_empty_holdings_first_compile_pure_filter_no_preservation(
        self,
    ) -> None:

        # 空持仓
        current_holdings: dict[str, Decimal] = {}

        # 60 只候选标的
```

### 6.1 同 run_id 双跑比对

### 7.3 conftest.py 层级与加载顺序

pytest 的 fixture 按 conftest.py 的目录层级自下而上加载，
内层 conftest 可引用外层 fixture，外层不能引用内层。

```
加载顺序：
tests/conftest.py                  → 全局可用
  ├── tests/core/conftest.py       → core 测试可用
  ├── tests/data/conftest.py       → data 测试可用
  ├── tests/research/conftest.py   → research 测试可用
  ├── tests/strategy/conftest.py   → strategy 测试可用
  ├── tests/trade/conftest.py      → trade 测试可用
  └── tests/e2e/conftest.py        → e2e 测试可用
```

**分层原则**：

- **全局 conftest**：放置所有域都可能用到的 fixture，如 `MockDataProvider`、`stock_pool`、`date_range`、Profile 工厂
- **域 conftest**：放置仅该域使用的 fixture，如 `MockStorageBackend`（Data 域）、`PipelineContext 工厂`（Strategy 域）、`PaperGateway 实例`（Trade 域）
- **fixture 命名**：以功能而非实现命名，如 `ctx_with_holdings` 而非 `make_dict_with_three_stocks`

### 7.4 Golden Master 管理

Golden Master 文件用于确定性回归验证。管理规则：

1. **首次生成**：运行 `pytest --update-golden` 标志生成基准文件
2. **版本控制**：所有 Golden Master 文件纳入 Git
3. **更新条件**：仅当业务逻辑变更（如舍入规则调整、因子公式修改）时才更新
4. **比对精度**：严格层逐字段 Decimal 比对，宽松层允许 1e-6 容差
5. **文件格式**：JSON，所有 Decimal 序列化为字符串（避免 float 精度丢失）

```python
# 示例：Golden Master 比对工具
def assert_golden_master(actual: dict, golden_path: str, strict_keys: list[str]) -> None:
    """与 Golden Master 基准文件比对。

    Args:
        actual: 实际运行结果
        golden_path: 基准文件路径
        strict_keys: 需要比特级一致的键列表
    """
    with open(golden_path) as f:
        golden = json.load(f)
    for key in strict_keys:
        assert actual[key] == golden[key], (
            f"Golden Master 比对失败 [{key}]: "
            f"actual={actual[key]}, golden={golden[key]}"
        )
```
