# 工作台产品规格

## 一句话定义

工作台是你查看和调度 AI 量化公司的驾驶舱。它不是聊天产品，也不是报表集合；它把实盘、研究、数据、智能体、审批、通知和版本放在同一个运营视角下。

## 用户画像

主要用户是你：

1. 懂代码，但不希望每天靠翻代码和日志管理系统。
2. 关注收益和风险，也关注研究是否持续推进。
3. 希望 AI 智能体像团队一样工作，但不希望黑盒改实盘。
4. 需要手机或远程界面快速看状态、批动作、追问原因。

## 设计原则

1. **第一屏回答“系统现在是否正常”**。
2. **每个状态都能点进证据**：运行、产物、日志、智能体会议、决策。
3. **高密度但不混乱**：SaaS 运营工作台风格，避免营销页、巨型 hero 和装饰性卡片。
4. **聊天在右侧或抽屉中**：聊天用于解释当前对象，不取代结构化状态。
5. **所有图表消费标准产物**：前端不临时补算关键指标。
6. **审批动作少而明确**：批准、驳回、追问、暂停、重试。
7. **先服务因子研究流程**：研究实验室页面必须先匹配 [[因子研究操作流程]]，再讨论视觉布局。

## 页面信息架构

| 页面 | 核心问题 | 关键模块 | 主要动作 |
|---|---|---|---|
| 运营驾驶舱 | 现在是否正常，哪些事需要我处理？ | 实盘状态、数据状态、研究状态、智能体阻塞、待审批、飞书通知 | 批准、驳回、追问、进入运行 |
| 运行中心 | 每个运行到哪一步了，卡在哪里？ | 运行表、阶段进度、日志、产物、失败点 | 查看、重试、取消、归档 |
| 智能体办公室 | 哪些智能体在工作、讨论什么、结论是什么？ | 组织图、会议列表、分歧、任务、结论 | 打开会议、追问、批准动作 |
| 因子研究实验室 | 研究议题和因子候选走到哪一步了？ | 议题队列、候选、质量门禁、评测、版本血统 | 开会、晋升、拒绝、变异 |
| 策略工坊 | 哪些信号/策略可以进入下一阶段？ | 信号快照、回测、晋升流水线 | 对比、请求审查、晋升 |
| 交易台 | 今天该不该交易，交易是否完成？ | 目标组合、订单、成交、QMT 健康、对账 | 试运行、阻断、确认、对账 |
| 归档中心 | 历史结论和版本链在哪里？ | 产物搜索、决策日志、版本血统图 | 标记、打标签、对比、导出 |

## 运营驾驶舱第一屏

运营驾驶舱必须在一个桌面首屏内展示：

1. **状态条**：数据、研究、策略、交易、智能体五条状态。
2. **交易状态面板**：当前策略、账户模式、下一执行窗口、QMT 健康、待处理风险。
3. **研究推进面板**：正在研究的方向、最新候选、质量门禁/评测摘要。
4. **待处理箱**：所有等待你处理的动作，按风险排序。
5. **阻塞运行**：失败、阻塞、超时、等待人工输入的运行。
6. **智能体活动**：正在开的智能体会议、负责人、预计产出。
7. **通知记录**：最近推送到飞书的关键事件和确认状态。

状态条颜色语义：

| 状态 | 颜色语义 | 含义 |
|---|---|---|
| 正常 | 绿色 | 已完成或运行正常 |
| 运行中 | 蓝色 | 正在执行，未超时 |
| 等待中 | 灰色 | 等待交易日、数据、审批或调度窗口 |
| 警告 | 橙色 | 有风险但未阻断 |
| 已阻断 | 红色 | 已阻断，不能继续下游 |
| 已过期 | 紫色 | 状态过期或心跳缺失 |

## 运行中心

运行中心是事实源的主要读入口。

每个 run 行必须显示：

1. 运行编号、领域、类型、状态、负责岗位。
2. 当前阶段、总进度、开始时间、最近心跳。
3. 关键产物数量和最新产物。
4. failure summary 或 waiting reason。
5. 下游依赖和是否可重试。

运行详情页包含：

```text
页头：运行元数据 / 状态 / 风险等级 / 负责岗位
时间线：任务阶段和状态转换
产物：标准 JSON、HTML、快照、报告链接
日志：结构化日志和文本日志尾部
智能体会议：相关诊断或审查会议
决策：批准、驳回、晋升、事故记录
版本血统：上游输入和下游消费者
```

## 智能体办公室

智能体办公室不能做成“群聊列表”。它应该展示组织和会议：

1. 组织图：岗位、职责、当前负载、最近产出。
2. 会议列表：每次会议的主题、状态、主持人、参会角色。
3. 分歧面板：关键分歧、证据、待用户决策点。
4. 后续动作：会议后生成的任务。
5. 会议记录抽屉：需要时查看完整对话或推理摘要。

## 因子研究实验室

因子研究实验室的主线必须来自 [[因子研究操作流程]]：

```text
研究议题
  -> 研究入口
  -> 假设卡
  -> 智能体研究委员会
  -> 因子候选
  -> 质量门禁
  -> 计算与多周期评测
  -> 正交性和策略贡献
  -> 策略角色判定
  -> 归档 / 变异 / 晋升 / 降级
```

实验室页面要把研究流程产品化，而不是只展示报告。推荐按阶段拆成进度轨：

| 阶段 | 用户要看见什么 | 可操作动作 |
|---|---|---|
| 研究入口 | 失败、现象、数据、组合缺口四类来源及证据 | 新建假设卡、合并重复入口、驳回 |
| 假设卡 | 市场行为、目标策略、预期角色、字段、失败条件、候选预算 | 批准生成候选、要求补证据 |
| 候选生成 | 候选表达式、父代、生成方式、智能体分歧 | 送门禁、合并候选、驳回 |
| 质量门禁 | PIT、字段、覆盖率、重复度、表达式安全 | 通过、补证据、阻断归档 |
| 评测 | 多周期 IC、分层、多空、稳定性、换手 | 进入增量评估、降级观察、拒绝 |
| 策略贡献 | 与目标策略接入后的收益、回撤、换手、容量变化 | 判定角色、进入策略工坊、归档 |
| 归档演化 | 成功/失败原因、下一轮变异、禁止重复路径 | 建立父代、进入下一轮、保护产物 |

每个候选因子必须显示：

1. 研究假设、智能体来源、父代血统。
2. 来源入口、目标策略、预期角色和失败条件。
3. 所需字段、预测周期、预期方向。
4. 质量门禁结果。
5. 多周期评测摘要。
6. 与已有因子的相关性和重复风险。
7. 策略贡献和角色判定。
8. 下一步动作：拒绝、变异、降级、评测、提交信号审查。

研究实验室要给 UI 团队明确的状态分组：

1. `待定义`：只有研究入口，还没有假设卡。
2. `待审假设`：假设卡已生成，等待批准候选预算。
3. `候选生成中`：Agent 正在生成候选或变异。
4. `门禁中`：候选等待质量审查。
5. `评测中`：已通过门禁，正在计算和多周期评测。
6. `待定角色`：有评测结果，但还没判定排序/过滤/门控/风险/执行/解释。
7. `待决策`：需要用户批准晋升、变异、拒绝或归档。
8. `已归档`：结论和失败原因进入档案。

## 策略工坊

策略工坊只接收通过研究门禁的信号或策略候选。核心展示：

1. 策略版本流水线：草案 -> 候选 -> 影子跟踪 -> 模拟盘 -> 实盘。
2. 回测摘要、walk-forward、容量、交易约束、风险门禁。
3. 与当前实盘/模拟盘版本的对照。
4. 晋升会议记录和用户审批。

## 交易台

交易台默认失败即阻断。任何实盘相关动作都必须显示：

1. 当前账户模式：研究、影子跟踪、模拟盘、实盘。
2. 今日目标组合和来源策略运行。
3. 风控检查结果。
4. 订单、成交、撤单、废单和对账。
5. QMT bridge 健康、认证状态、最近心跳。
6. 手动确认记录。

## 归档中心

归档中心的目标是防止经验丢失和重复试错：

1. 按产物类型、策略、因子、运行、日期搜索。
2. 展示完整版本血统图。
3. 标记受保护、基线、已拒绝、已晋升。
4. 查看历史决策和事故复盘。
5. 将优秀结论反哺 skills 或研究档案。

## 技术边界

第一版 UI 推荐：

1. React + TypeScript + Vite。
2. TanStack Query 读取本地接口状态。
3. TanStack Table 展示运行、产物、待审批事项。
4. React Flow 展示版本血统和智能体组织图。
5. Plotly 渲染已有研究/回测图表。
6. FastAPI 提供本地 HTTP + SSE/WebSocket 适配层。

前端不直接读取数据面 Parquet；它只读控制面接口和标准产物。
