# 因子研究运行契约

## 一句话定义

因子研究运行契约把“研究想法”压成可执行、可审查、可归档的结构化对象。它服务三类消费者：脚本、智能体、UI。

第二版还服务第四类消费者：Codex Automations。Automation 不直接做研究判断，只读取状态卡、唤醒 Codex、推进一个明确阶段，并在需要人工判断时停止。

## 契约分层

```text
研究议题
  -> 研究假设卡
  -> 候选因子
  -> 质量门禁报告
  -> 因子值快照
  -> 多周期评测报告
  -> 证据评审和接力包
  -> 正交性和策略贡献报告
  -> 策略角色判定
  -> 归档记录
```

每一层都必须有产物编号、运行编号、输入来源和状态。

## Codex 自动化状态卡

Codex 客户端无 API key 版本使用状态卡承载“自动研究现在走到哪里”。状态卡可以先写成 Markdown 或 JSON 摘要；只有当 UI 或脚本需要稳定读取时，才把它固化成代码 schema。

```json

  "research_id": "research_20260516_001",
  "execution_mode": "codex_client_no_api_key",
  "automation_kind": "thread_heartbeat",
  "topic_id": "topic_20260516_001",
  "hypothesis_id": "hyp_20260516_001",
  "current_stage": "evidence_review",
  "last_completed_artifact": "eval_alpha_20260516_001",
  "last_handoff": "review_alpha_20260516_001",
  "next_agent": "CogAlpha 负责人",
  "next_wakeup_goal": "围绕同一假设生成 3 个低相关变形候选，并先做质量门禁",
  "allowed_actions": [
    "读取文档和研究 artifact",
    "更新研究 Markdown",
    "运行离线评测脚本",
    "生成接力包和归档摘要"
  ],
  "blocked_actions": [
    "修改交易配置",
    "下单或撤单",
    "删除受保护产物",
    "把候选推进实盘"
  ],
  "stop_conditions": [
    "没有明确下一步证据",
    "需要用户选择研究方向",
    "需要交易或配置审批",
    "连续多轮没有新增信息"
  ],
  "status": "active"

```

Codex 每次被唤醒时，只做一轮可审查推进：

```text
读取状态卡
  -> 读取相关 skill
  -> 读取最近产物和接力包
  -> 推进 current_stage 对应的一个产物
  -> 写回结论、下一位 Agent、停止条件
  -> 如果需要人工判断，停止并汇报
```

具体执行步骤和 thread automation prompt 放在 `$factor-research-operating-cycle` skill。本文只定义状态卡、契约字段、状态机和 UI 消费边界，避免和 skill 维护两套流程。

### 21-agent 自动化样例

样例目标不是继续做 shadow，也不是建设 API 服务，而是证明 Codex 客户端可以按固定节奏唤醒当前线程，让 21 个 CogAlpha 视角围绕同一张假设卡完成：

1. 第一轮 `AgentOpinionCard`。
2. 第二阶段证据推进、关闭或 handoff。
3. 风险官和执行运营的阻断/条件通过。
4. 归档审计和下一轮状态卡。

## 研究假设卡契约

研究假设卡连接“研究入口”和“候选生成”。它必须先于候选因子存在，防止 Agent 直接从公式开始随机搜索。

```json
{

  "hypothesis_id": "hyp_20260516_001",
  "topic_id": "topic_20260516_001",
  "entry_type": "failure_reuse",
  "source_evidence_refs": [
    {
      "type": "artifact",
      "id": "artifact_overlay_review_vtx_alpha_101"
    }
  ],

  "target_strategy": "earnings_forecast_drift_v3",
  "expected_role": "risk_budget",
  "allowed_candidate_forms": ["formula", "gate", "overlay_rule"],
  "required_fields": ["open", "high", "low", "close", "amount", "forecast_surprise"],
  "horizons": [5, 20],
  "candidate_budget": 6,
  "evaluation_focus": [
    "坏持仓命中率",
    "左尾改善",
    "收益牺牲",

  ],
  "failure_conditions": [
    "降低收益但不能显著改善左尾",

  ],
  "status": "ready_for_candidate_generation"
}
```

研究入口枚举：

| `entry_type` | 含义 |
|---|---|
| `failure_reuse` | 从失败、overlay 失败、拒绝因子中复用经验 |
| `market_phenomenon` | 从市场行为或交易制度现象出发 |
| `data_affordance` | 从新字段或已有字段能观察的行为出发 |
| `portfolio_gap` | 从当前策略缺口、回撤、容量、换手或可成交性出发 |

## 质量门禁报告

质量门禁不是收益评测。它只判断候选能不能进入评测。

```json
{
  "schema": "vortex.research.quality_gate.v1",
  "gate_id": "gate_alpha_20260516_001",
  "candidate_id": "alpha_20260516_001",
  "run_id": "run_20260516_quality_gate_001",
  "decision": "pass_with_warnings",
  "blocking_issues": [],
  "warnings": [
    {
      "code": "capacity_risk",
      "message": "候选可能偏向小成交额股票，需要容量评估"
    }
  ],
  "checks": {
    "field_visibility": "pass",
    "future_leakage": "pass",
    "coverage": "pass",
    "nan_inf": "pass",
    "distinct_ratio": "pass",
    "duplicate_factor": "warning",
    "economic_logic": "pass",
    "sandbox_required": "pass"
  },
  "required_follow_up": ["容量评估", "小市值暴露评估"],
  "reviewers": ["因子质量审查员", "风险官"]
}
```

门禁状态：

| 状态 | 含义 | 下游动作 |
|---|---|---|
| `pass` | 可以评测 | 进入因子值计算 |
| `pass_with_warnings` | 可以评测，但必须带着风险标签 | 进入评测，并生成补充检查 |
| `needs_evidence` | 缺证据，暂不评测 | 补证据后重跑门禁 |
| `reject` | 硬阻断 | 归档失败原因，不进入评测 |

## 证据评审和接力包

证据评审不是代码调度结果，而是研究审稿结果。它可以先以 Markdown 或 JSON 摘要存在，用于告诉下一位 Agent 应该读什么、做什么、何时停止。

```json
{
  "schema": "vortex.research.evidence_review_handoff.v1",
  "review_id": "review_alpha_20260516_001",
  "candidate_id": "alpha_20260516_001",
  "hypothesis_id": "hyp_20260516_001",
  "reviewer": "因子研究证据评审员",
  "evidence_refs": [
    "quality_gate_alpha_20260516_001",
    "eval_alpha_20260516_001"
  ],
  "conclusion": "informative_failure",
  "credible_evidence": [
    "20d RankIC 有正向线索",
    "硬过滤接入策略后收益牺牲过大"
  ],
  "weak_evidence": [
    "尚未证明样本外稳定",
    "可能与低波/小市值风格重复"
  ],
  "overfit_risks": [
    "窗口选择可能来自事后观察",
    "只在目标策略局部样本有效"
  ],
  "next_agent": "CogAlpha 负责人",
  "requested_work": "围绕同一假设做轻量 rerank、状态门控和风险预算三类变形",
  "stop_conditions": [
    "变形只提高 IC 但不能改善目标策略回撤",
    "与父代相关性过高且没有新增解释"
  ],
  "archive_tags": ["informative_failure", "mutation_parent"]
}
```

状态枚举：

| `conclusion` | 含义 | 下一步 |
|---|---|---|
| `invalid_experiment` | 实验不可采信 | 回到质量门禁或关闭 |
| `no_signal` | 没有有效信息 | 归档关闭 |
| `informative_failure` | 失败有复用价值 | 交给 CogAlpha 负责人变形 |
| `research_lead` | 有研究线索 | 补评测或继续同一假设 |
| `strategy_role_review` | 预测证据可用但角色不明 | 交给策略晋升专员 |
| `risk_review` | 预测证据可用但风险不清 | 交给风险官 |
| `archive_only` | 只需沉淀经验 | 交给归档审计员 |

## 状态机

```text
topic_draft
  -> hypothesis_draft
  -> ready_for_candidate_generation
  -> candidate_draft
  -> pending_quality_gate
  -> rejected
  -> needs_evidence
  -> ready_for_compute
  -> computing
  -> ready_for_evaluation
  -> evaluating
  -> evidence_reviewing
  -> handoff_ready
  -> needs_incremental_review
  -> role_decided
  -> waiting_human_decision
  -> archived
```

硬规则：

1. 没有 `hypothesis_card`，不能批量生成候选。
2. `rejected` 不能直接变成 `ready_for_compute`，必须新建候选或补证据重开门禁。
3. 没有 `quality_gate` 产物，不能进入计算。
4. 没有 `factor_snapshot`，不能进入评测。
5. 没有 `evidence_review_handoff`，不能进入策略角色判定。
6. 没有 `incremental_value`，不能进入策略候选。
7. 没有用户或策略晋升会议决策，不能进入模拟盘或实盘。

## UI 必备字段

候选因子详情页必须展示：

1. 候选编号、家族、来源议题、父代血统。
2. 研究假设卡：入口类型、来源证据、市场行为、目标策略、失败条件。
3. 研究假设、表达式、字段、预测周期、预期方向。
4. 当前状态、下一步动作、阻塞原因。
5. 质量门禁结果：硬阻断、软警告、补证据任务。
6. 多周期评测摘要：IC、RankIC、多空、换手、单调性。
7. 风险暴露：行业、市值、流动性、容量。
8. 正交性和策略贡献：与已有因子相关性、加入策略后的变化。
9. 智能体意见：赞成、反对、分歧、证据。
10. 决策记录：批准、驳回、追问、变异、归档。

质量门禁页必须展示：

1. 每个检查项的通过、警告、阻断状态。
2. 证据来源，不只展示结论。
3. 阻断项和可补证据项分开。
4. 哪个智能体有阻断权。
5. 下游动作是否允许点击。
