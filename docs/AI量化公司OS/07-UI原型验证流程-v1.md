# UI 原型验证流程

## 一句话定义

产品和 UI 不能只停留在文字描述里。每个关键页面都要用原型确认：用户第一眼能否看懂状态，能否找到阻塞，能否追溯证据，能否完成审批。

## 原型顺序

```text
因子研究操作流程
  -> 页面故事板
  -> 低保真信息架构
  -> 可点击 HTML / React 原型
  -> 截图走查
  -> 用户反馈
  -> 修订规格
```

先验证流程，再验证页面；先验证信息密度，再验证视觉细节。

## 第一批原型

| 原型 | 目标 | 必须回答的问题 |
|---|---|---|
| 运营驾驶舱 | 看公司是否正常 | 哪些任务在跑，哪些需要我管？ |
| 因子研究实验室 | 看研究流程 | 一个候选从想法到评测到归档走到哪了？ |
| 智能体办公室 | 看协作过程 | 哪些智能体在讨论，分歧是什么，结论是什么？ |
| 运行中心 | 看事实源 | 某个运行为什么失败，证据在哪里？ |
| 飞书通知详情 | 看移动端触达 | 我在飞书上能否快速知道要不要处理？ |

## 下一批原型优先级

当前已有的 `factor_research_workbench.html` 只能验证总览方向。下一批原型应按“最能暴露产品细节”的顺序补：

| 优先级 | 原型 | 验证重点 |
|---:|---|---|
| 1 | 候选因子详情页 | 假设、表达式、字段、父代、风险、当前状态是否一眼可读 |
| 2 | 质量门禁面板 | 硬阻断、软警告、证据、补证据任务是否清晰 |
| 3 | 因子评测结果页 | 多周期、分层、暴露、成本、稳定性和结论是否可比较 |
| 4 | 飞书消息详情页 | 通知内容、按钮、二次确认、审计写回是否可控 |
| 5 | 智能体会议详情页 | 分歧、证据、结论、后续任务是否能替代“聊天记录考古” |

## UI 专家技能要求

UI 智能体生成原型时必须：

1. 使用中文界面文案。
2. 按真实运行对象设计，不用虚假的营销数据。
3. 保持运营工具密度，不做宣传页。
4. 展示正常、运行中、等待、警告、阻断、过期状态。
5. 至少包含桌面和手机两种视口。
6. 每个模块标明数据来源字段。

## 原型验收清单

1. 首屏 10 秒内能判断系统是否正常。
2. 阻断任务能在 2 次点击内打开证据。
3. 候选因子能看出当前阶段、质量门禁和下一步。
4. 智能体会议能看出主持人、参会角色、分歧和结论。
5. 飞书通知能看出影响、建议动作和是否需要立即处理。
6. 页面不依赖用户阅读长篇说明。
7. 没有关键英文占位词、无意义卡片或装饰性图形。

## 推荐实现方式

第一轮原型可以是静态 HTML 或轻量 React，不需要后端。原型数据用手写 JSON，但字段必须对应 [[运行与产物契约]] 和 [[因子研究操作流程]]。

验证通过后，再进入真实 API 和工作台实现。
