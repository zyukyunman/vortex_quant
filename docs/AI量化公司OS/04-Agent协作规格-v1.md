# 智能体协作规格

## 一句话定义

智能体协作不是让多个模型自由聊天，而是让岗位化智能体在明确会议流程中产出可审计结论。每次协作都必须有输入、主持人、参会角色、分歧、证据、结论、下一步和产物。

## 第二版执行边界

第二版默认不建设独立的多 Agent API 服务，也不假设有 OpenAI API key。Codex 客户端是研究执行器；skills、Markdown 状态卡和研究产物是协作协议。

Codex Automations 只负责把当前线程按固定节奏唤醒。被唤醒后，Codex 必须先读状态卡和最后一条接力包，再决定本轮推进哪个阶段。Automation 不负责判断因子好坏，也不替代研究总监、质量审查员或证据评审员。

```text
Codex heartbeat
  -> 读取研究状态卡
  -> 读取最后接力包
  -> 按当前阶段调用对应 skill
  -> 推进一个明确产物
  -> 写回状态和下一步
  -> 如需人工判断则停止
```

## 什么时候开始写代码

当前阶段优先写文档、skill、状态卡和自动化 prompt。只有出现以下情况，才进入代码实现：

1. 同一个研究步骤反复人工执行，且需要确定性结果，例如读取评测 artifact、汇总状态、生成固定格式报告。
2. UI 或控制面需要稳定消费某个对象，例如研究状态卡、接力包、证据评审结论需要变成 schema。
3. 本地评测脚本缺少必要入口，导致 Codex 无法从候选因子推进到可验证结果。
4. Automation 需要安全边界，例如只允许写研究档案、不允许改交易配置，需要用脚本或配置限制动作。
5. 进入 shadow、模拟盘或实盘前，需要交易前检查、订单约束、审计记录等可复现机制。

不写代码的情况：定义角色、讨论研究流程、生成候选假设、写审稿意见、整理接力包、判断下一位 Agent。

## 智能体会议生命周期

```text
接收问题
  -> 制定议程
  -> 岗位初始意见
  -> 讨论和审查
  -> 证据核对
  -> 形成结论
  -> 必要时请求决策
  -> 归档
```

| 阶段 | 目标 | 输出 |
|---|---|---|
| 接收问题 | 明确问题、范围、风险等级 | 会议简报 |
| 制定议程 | 选择参会角色和流程 | 议程 |
| 岗位初始意见 | 各岗位给初始意见 | 岗位意见 |
| 讨论和审查 | 暴露分歧和风险 | 分歧清单 |
| 证据核对 | 绑定运行、产物、日志证据 | 证据引用 |
| 形成结论 | 形成建议或拒绝继续 | 结构化结论 |
| 请求决策 | 需要你处理时生成审批项 | 决策草案 |
| 归档 | 写入摘要和版本血统 | 会议产物 |

## 标准输出块

每个智能体会议的最终摘要必须包含：

1. `问题`：本次会议回答什么问题。
2. `上下文`：关联运行、产物、数据版本、策略版本。
3. `participants`：岗位，不是模型名。
4. `major_findings`：事实结论。
5. `关键分歧`：关键分歧和证据。
6. `风险`：未来函数、过拟合、容量、执行、系统风险。
7. `建议`：继续、拒绝、追问用户、阻断。
8. `行动项`：落成任务。
9. `决策请求`：需要你批准时才出现。

## 角色协作规则

1. 运营总协调负责拉会，不替代专业角色判断。
2. 产品和 UI 智能体不能决定研究或交易结论。
3. 研究和 CogAlpha 智能体不能跳过质量门禁。
4. 策略晋升专员不能绕过风险官把候选推进实盘。
5. 执行运营对交易链路有阻断权。
6. 归档审计员可以驳回“无法追溯”的产物。

## Agent 接力原则

Agent 接力不是在代码里写死一个调度链，也不是让多个模型自由聊天。它是一种研究工作协议：上一个岗位把当前证据、未解决问题、下一步建议写成接力包；下一个岗位按自己的 skill 接收、审查、补证据或拒绝继续。

每次接力只回答三个问题：

1. 上一位 Agent 已经确定了什么。
2. 下一位 Agent 为什么是它，而不是其他岗位。
3. 下一位 Agent 接手后必须产出什么，什么情况下必须停止。

### 接力包格式

```json
{
  "schema": "vortex.company.agent_handoff_note.v1",
  "handoff_id": "handoff_20260516_001",
  "source_agent": "因子研究证据评审员",
  "target_agent": "CogAlpha 负责人",
  "research_object": "hyp_20260516_001 / alpha_tail_risk_overlay_v1",
  "current_state": "rejected_informative",
  "evidence_refs": [
    "docs/因子研究/因子实验总表.md",
    "workspace/research/.../generation_summary.json"
  ],
  "what_we_know": [
    "硬过滤会牺牲收益",
    "轻量 rerank 可能改善回撤"
  ],
  "open_questions": [
    "是否应把 tail risk 从过滤变成 10% rerank 权重？",
    "是否只在高波动 regime 激活？"
  ],
  "requested_work": "围绕同一假设做 3-5 个变形，不要换题。",
  "stop_conditions": [
    "只复现父代相关性，没有新增解释",
    "变形不能改善目标策略回撤或坏持仓命中率"
  ]
}
```

这个接力包可以先存在 Markdown、会议摘要或研究档案里，不要求立刻进入代码 schema。产品 UI 后续可以把它渲染成“下一位 Agent / 需要读的证据 / 当前阻断项 / 下一步任务”。

## 时间维度

时间长短不是三套流程，而是同一流程的研究深度。更长的会话只是允许更多轮接力；是否继续取决于证据、队列和停止条件，不允许跳过证据评审。

| 会话深度 | 适合目标 | 必须留下的产物 |
|---|---|---|
| 30 分钟以内 | 复核一个候选、解释一次失败、整理下一步 | 一条接力包和一条归档结论 |
| 1 小时左右 | 完成一次候选生成、门禁、初步评审 | 假设卡、候选清单、门禁摘要、证据评审 |
| 2 小时以上 | 围绕同一假设做多轮变形和审稿 | 父代/子代链路、失败路径、下一轮队列、关闭路径 |

长时间研究的关键不是“跑得久”，而是每 30-60 分钟都能交付一个可读的中间结论。任何阶段都可以中断；下一次继续时先读最后一条接力包。

## CogAlpha Council 规则

CogAlpha 的 21 个智能体是研究视角，不是公司顶层岗位。它们归 CogAlpha 负责人管理。

标准流程：

```text
研究方向
  -> 选择 CogAlpha 视角
  -> 生成候选 / 变异 / 交叉
  -> 质量门禁
  -> 多周期评测
  -> 委员会审查
  -> 因子档案或下一轮队列
```

21 个视角输出必须被压缩为结构化候选，不允许把整段长对话当结果。

## 审批规则

智能体可以直接结束的情况：

1. 只读分析。
2. 研究候选进入下一轮离线评测。
3. 失败 run 的初步诊断。
4. 报告摘要和归档建议。

必须请求你确认的情况：

1. 改实盘配置。
2. 下单、撤单或改变交易通道。
3. 将策略候选推进影子跟踪、模拟盘或实盘。
4. 删除或覆盖受保护产物。
5. 修改 API key、账户、QMT bridge 或凭证。

## 失败处理

如果 session 无法形成结论，必须标记原因：

| 原因 | 处理 |
|---|---|
| 缺少证据 | 生成补证据任务 |
| 岗位分歧 | 升级给你追问 |
| 质量阻断 | 阻断后续，并写拒绝原因 |
| 执行风险 | 交给风险官和执行运营 |
| 上下文过期 | 重新读取运行和产物状态 |

失败 session 也要归档，避免重复召开同一类无效会议。

## 对模型后端的要求

1. 第二版默认后端是 Codex 客户端，不依赖 API key。
2. 后端以后可以扩展为 DeepSeek、OpenAI-compatible、本地模型或 Agents SDK，但那是后续实现选择。
3. 模型名不是产品对象；产品对象是岗位、智能体会议、接力包和决策。
4. 所有模型输出都必须进入结构化产物或人工审查。
5. 高风险任务需要更严格 review；交易和配置变更必须人工确认。
