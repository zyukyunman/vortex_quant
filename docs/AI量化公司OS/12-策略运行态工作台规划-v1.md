# 策略运行态工作台规划 v1

## 目标

策略运行态工作台要回答一个问题：现在公司到底在跑什么策略、跑到哪一步、为什么选这些股票、今天有没有异常、收益和风险是否正常。

第一版不追求支持很多策略。当前只有 `业绩预告漂移策略` 已经进入自动编排、QMT 交接和净值子账本，所以策略目录只展示这一条。其他研究方向只有通过策略晋升后，才进入运行目录。

## 产品原则

1. 当前运行策略和查看对象必须分开。点击某个区域不能改变“公司当前正在跑的策略”。
2. UI 展示结构化状态，不展示一堆 JSON 路径。
3. 所有状态都必须能追溯到本地产物：任务、目标组合、执行报告、净值台账、数据服务健康文件。
4. 实盘动作继续人工门禁；控制台可以检查、解释、生成任务，但不能悄悄下单。

## 核心对象

### 策略实例

代表一个可以持续运行的策略。

字段：

| 字段 | 说明 |
|---|---|
| `strategy_id` | 稳定 ID，例如 `earnings_forecast_drift` |
| `name` | 中文名称 |
| `strategy_version` | 当前运行版本，例如 `baseline_top110_large` |
| `mode` | shadow / paper / gated_live |
| `entrypoint` | 启动命令 |
| `review_skill` | 审查流程 |
| `status` | running / blocked / waiting / stopped |

### 每日运行

代表某个交易日的一次策略生命周期。

阶段：

1. 数据新鲜度门禁
2. 信号生成
3. 筛选漏斗
4. 持仓质量审查
5. 目标组合冻结
6. QMT 只读检查
7. 订单计划 / 执行
8. 对账
9. 净值快照
10. 盘后复盘

### 筛选漏斗

用于解释“为什么最后是这些股票”。

当前业绩预告漂移策略已有字段：

| 阶段 | 当前来源 |
|---|---|
| 原始信号 | `target_diagnostics.selection_funnel.raw_signal_count` |
| 正向信号 | `positive_signal_count` |
| 流动性过滤后 | `after_liquidity_count` |
| ST/风险过滤后 | `after_st_filter_count` |
| 市值门控后 | `after_market_cap_top50_count` |
| 开盘可交易过滤后 | `after_open_block_count` |
| 持仓质量审查后 | `after_quality_block_count` |
| 市场权限过滤后 | `after_permission_count` |
| 可执行候选 | `executable_candidate_count` |
| 最终目标持仓 | `selected_position_count` |

### 数据服务

数据服务不是策略内部细节，而是所有策略共享的基础设施。

当前可读事实源：

| 产物 | 说明 |
|---|---|
| `state/live-service-health-latest.json` | 服务健康、调度、最新更新、最新快照、当前任务 |
| `state/logs/data-update-*.log` | 每日数据更新日志 |
| `state/manifests/*` | 数据同步 manifest |
| `data/authoritative/default` | 已发布数据快照 |

第一版 UI 展示：

1. 调度时间。
2. 最新运行状态。
3. 最新成功快照。
4. 当前正在拉取的数据集。
5. 告警列表。

### 净值与风险

当前已有账户级策略子账本：

| 产物 | 说明 |
|---|---|
| `state/nav/*.json` | 策略账户绑定 |
| `trade/nav/*.csv` | 每日净值快照 |

第一版 UI 展示：

1. 最新净值。
2. 当日收益。
3. 基准收益。
4. 超额收益。
5. 最大回撤。
6. 总资产、可用现金、市值。

后续要补：

1. 收益分解：open-to-close、隔夜、成本、滑点、未成交、现金拖累。
2. 回撤区间、最长回撤天数、恢复天数。
3. 暴露：行业、规模、市值、持仓集中度。

## 页面规划

### 交易台 v1

保留一个页面，但结构改为运行驾驶舱：

1. 今日交易审查：总状态、阻断原因、QMT 检查、待处理任务。
2. 当前运行策略：只展示已接入自动编排的策略。
3. 策略运行流程：每个阶段的状态和说明。
4. 筛选漏斗：从信号到最终持仓的数量变化。
5. 目标持仓：股票、权重、股数、目标市值、原因。
6. 自动服务与数据：策略服务 tick、数据调度和当前拉取任务。
7. 净值与风险：最新净值、收益、超额、回撤。
8. 换仓 / 执行：计划买卖、最近执行、风险门禁。
9. QMT Bridge：只读检查、资产/持仓/委托/成交摘要。

### 后续页面拆分

当信息密度继续变高时拆成：

1. `策略运行台`：当前策略、流程、目标组合、执行。
2. `数据服务台`：数据更新、快照、分区、质量。
3. `绩效与归因`：净值、回撤、收益拆解。
4. `策略配置`：preset、资金、门禁、QMT、调度。

## 技术计划

### 阶段一：消费现有产物

已经可以做：

1. 从 pending task 读取策略版本、交易日、目标组合路径、筛选漏斗、质量审查。
2. 从 target portfolio 读取持仓。
3. 从 execution report 读取订单、成交、风控。
4. 从 nav ledger 读取净值。
5. 从 live-service health 读取数据服务。

### 阶段二：补策略运行 manifest

现在的问题是事实分散在多个文件。下一步应该新增：

```text
strategy_run_manifest.json
```

建议路径：

```text
workspace/strategy_runs/{strategy_id}/{trade_date}/strategy_run_manifest.json
```

它聚合：

1. 输入数据版本。
2. 筛选漏斗。
3. 目标组合。
4. 质量审查。
5. 执行计划。
6. 成交回报。
7. 对账结果。
8. 净值快照。
9. 异常和人工决策。

### 阶段三：统一策略接口

每个策略必须实现同一组接口：

```text
prepare()
build_signal()
build_target()
review()
handoff()
execute()
reconcile()
record_nav()
postmortem()
```

UI 不直接理解某个策略内部实现，只读统一 manifest。

## 当前缺口

1. 最新 UI 仍然直接读多个旧产物，短期可用，长期要收敛到 manifest。
2. 换仓摘要目前可以从目标组合差分估算，但真实执行差异要从 order/fill/reconcile 归因。
3. 净值目前只有一条快照，指标暂时不丰富。
4. 数据服务可以展示“正在拉什么”，但还没有 per-dataset SLA 和预计完成时间。
5. 当前 QMT Bridge 缺 Token/API Key，导致只读资产接口 401，交易台会正确显示阻断。

## 验收标准

1. 打开交易台，能一眼看到当前运行策略，而不是可变的选中项。
2. 能看到策略每天走到哪个阶段。
3. 能看到从多少信号过滤成多少目标持仓。
4. 能看到目标持仓、换仓计划、最近执行、净值、数据服务状态。
5. 任一关键链路失败时，交易状态明确变红并给出原因。
