# Vortex Quant 重写蓝图（Qlib + RiceQuant范式 + miniQMT实盘）

> 目标不是“跑一个策略”，而是构建“类似米筐、可承载多策略与持续迭代”的量化平台。
> 结论先行：Qlib 必须是一等公民，负责因子研究、模型训练、实验管理与在线信号更新。

---

## 一、系统定位与设计原则

### 1.1 系统定位

Vortex 2.0 定位为：

1. 研究-交易一体化平台（Research to Production）
2. 多策略并行平台（非单策略脚本）
3. 可复现实验平台（Experiment-first）
4. Mac 研究 + Windows miniQMT 执行的跨机架构平台

### 1.2 设计原则（对齐 RiceQuant + Qlib）

1. 约定优于配置：策略生命周期固定化（盘前、盘中、盘后）
2. 模块松耦合：Data / Research / Portfolio / Execution / Risk / Ops 分层
3. 实验可追溯：每次训练与回测必须有 Experiment/Recorder 记录
4. 双引擎回测：Qlib 组合回测 + 事件驱动精确回测
5. 在线可更新：模型、预测、信号按日增量更新，不重跑全量
6. 实盘可对账：目标仓位、订单计划、成交回报三账一致

## 三、目标架构（Qlib 为研究中枢）

### 3.1 分层架构

```
Interface Layer
  - CLI / API / Report UI

Workflow Layer
  - Pipeline Orchestrator
  - Qlib qrun / workflow_by_code
  - Experiment Manager (Qlib Recorder + MLflow backend)

Research Layer (Qlib Core)
  - DataHandlerLP / DatasetH
  - Alpha Formula Engine / Feature Engineering
  - Model Zoo / Custom Models
  - SignalRecord / SigAnaRecord / PortAnaRecord

Portfolio Layer
  - Signal Ensemble
  - Optimizer (risk constraints)
  - Rebalance Compiler

Execution Layer
  - Simulator (backtest execution)
  - MiniQMT Gateway (xqshare remote)

Infrastructure Layer
  - Data Lake (Parquet + DuckDB)
  - Qlib Bin Store
  - Scheduler / Cache / Logging / Artifact Store
```

### 3.2 关键定位

1. Qlib 负责：因子研究、模型训练、实验管理、在线预测更新、研究回测分析
2. 自研 Portfolio 负责：跨策略信号融合、交易约束落地、目标仓位编译
3. xqshare + miniQMT 负责：真实下单与账户状态获取

## 四、数据架构（Lakehouse + Qlib 双存储）

### 4.1 双存储策略

1. 权威底座：Parquet + DuckDB（全量原始与衍生数据）
2. 研究视图：Qlib Bin（供 Qlib 高性能检索与表达式计算）

### 4.2 日常数据流水线

1. Tushare 增量同步到 Data Lake
2. 数据质量校验（缺失、跳变、交易日对齐、复权一致性）
3. 将必要字段增量转换为 Qlib 格式（支持 csv/parquet -> dump_bin）
4. 更新 Qlib 缓存（ExpressionCache / DatasetCache）

### 4.3 PIT 与防未来函数

1. 财务字段使用公告日期对齐
2. 标签统一按可交易时点定义（T+1 买入、T+2 收益）
3. 所有 DataHandler 处理器分离 infer 与 learn 流程

## 五、研究架构（Qlib 全量启用）

### 5.1 因子研究

1. Formulaic Alpha：使用表达式引擎快速迭代
2. Python 因子：复杂逻辑走自定义 DataHandler/Processor
3. 因子评估：IC、RankIC、ICIR、分组收益、多空收益、稳定性

### 5.2 模型训练

1. 基线模型：LightGBM、XGBoost、MLP、LSTM
2. 动态训练：滚动重训、窗口更新
3. 模型版本化：按实验记录 + 参数哈希 + 数据切片版本

### 5.3 实验管理（必须）

1. 每次实验强制启动 Recorder
2. 记录 params / metrics / artifacts
3. 输出统一产物：
   - pred.pkl（预测）
   - ic_metrics.csv
   - backtest_report.pkl
   - config_snapshot.yaml

### 5.4 在线更新（Qlib Online Serving）

1. OnlineManager 管理在线模型集
2. Updater 增量更新预测与标签
3. 信号准备与策略解耦，支持多模型集成

## 六、组合与执行架构（RiceQuant范式落地）

### 6.1 核心思想

借鉴 RiceQuant 的 order_target_portfolio 思路，平台内部统一为：

1. Signal -> TargetWeight
2. TargetWeight + CurrentPosition -> OrderPlan
3. OrderPlan -> Gateway -> ExecutionReport

### 6.2 协议对象（域间强契约）

```python
from dataclasses import dataclass
from typing import Dict, List, Optional
import pandas as pd

@dataclass
class SignalSnapshot:
    date: str
    signal_id: str
    scores: pd.Series                 # ts_code -> score
    metadata: Dict

@dataclass
class TargetPortfolio:
    date: str
    strategy_id: str
    weights: Dict[str, float]         # ts_code -> target_weight
    cash_ratio: float
    constraints: Dict

@dataclass
class OrderInstruction:
    ts_code: str
    side: str                         # BUY / SELL
    qty: int
    order_type: str                   # MARKET / LIMIT
    limit_price: Optional[float]
    reason: str

@dataclass
class OrderPlan:
    date: str
    account_id: str
    orders: List[OrderInstruction]

@dataclass
class ExecutionFill:
    ts_code: str
    side: str
    qty: int
    price: float
    fee: float
    ts: str

@dataclass
class ExecutionReport:
    date: str
    account_id: str
    fills: List[ExecutionFill]
    rejected: List[OrderInstruction]
    cash_after: float
```

### 6.3 组合编译器（Portfolio Compiler）

输入：

1. 当日目标权重
2. 当前持仓
3. 可用资金
4. 交易约束（整手、最小成交额、单票上限、行业上限）

输出：

1. 可执行订单计划
2. 预计换手、交易成本、风险暴露变化

## 十三、模块级实现细节（第一版）

### 13.1 Data 模块

1. 输入接口：Tushare 拉取器 + 本地补丁数据
2. 输出接口：
   - Lake API（DuckDB SQL + Parquet）
   - Qlib Provider URI
3. 质量门槛：每日健康分 > 95 才允许进入研究流程

### 13.2 Research 模块

1. Workflow 模板：
   - 因子验证模板
   - 模型训练模板
   - 回测分析模板
2. 每个模板必须输出统一 artifacts
3. 支持 qrun 与 workflow_by_code 双模式

### 13.3 Portfolio 模块

1. 支持单策略与多策略融合
2. 融合方式：等权、ICIR加权、波动率逆权
3. 约束求解：线性约束优先，必要时二次规划

### 13.4 Execution 模块

1. 模拟与实盘共用同一 OrderPlan 协议
2. 执行回报标准化，便于统一对账
3. 实盘执行带风控闸门与人工确认开关

### 13.5 Risk 模块

1. 风控规则配置化（yaml）
2. 所有触发事件持久化，支持审计追溯
3. 关键规则支持紧急热更新

### 13.6 Ops 模块

1. 任务状态：PENDING/RUNNING/SUCCESS/FAILED/RETRYING
2. 支持补跑与断点恢复
3. 所有任务输出统一日志字段（run_id, module, cost_ms, status）

## 十四、与 RiceQuant / Qlib 的对应关系

| 目标能力 | RiceQuant范式 | Qlib范式 | Vortex落地 |
|---|---|---|---|
| 策略生命周期 | init/before/handle/after | workflow + strategy | Orchestrator + Strategy hooks |
| 目标组合调仓 | order_target_portfolio | WeightStrategyBase | PortfolioCompiler |
| 因子研究 | RQFactor | DataHandler + Alpha + SigAna | Research/alpha + evaluate |
| 回测分析 | RQAlpha | PortAnaRecord | backtest + report pipeline |
| 实验管理 | 平台记录 | Recorder/MLflow | experiments 服务 |
| 在线更新 | 平台级在线 | OnlineManager/Updater | ops + online signal update |

## 十五、本轮结论

1. Qlib 不是可选，而是研究中枢与实验底座
2. 架构按“大型私募平台”标准设计，不再按“单策略脚本”设计
3. RiceQuant 给出交易生命周期与目标组合范式，Qlib给出研究与实验范式
4. Mac -> Windows miniQMT 的执行链路以 xqshare 为标准方案
